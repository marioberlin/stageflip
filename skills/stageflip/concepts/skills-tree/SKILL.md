# The Skills Tree

`skills/stageflip/` is simultaneously **three things**:

1. Documentation for humans reading the codebase
2. Context for AI coding agents (Implementer, Reviewer, Validator)
3. The content of the published `@stageflip/plugin` (Claude plugin)

If those three would diverge, invariant I-8 (CI gate `check-skill-drift`)
fails the build. The skills tree is **the source of truth**. When code and a
skill disagree, fix the truth; don't "just update the skill to match."

## The four non-negotiables (T-221)

Every SKILL.md must satisfy:

1. **One-screen.** Body fits in one terminal screen at normal font (~60
   lines). If it doesn't, split into sibling skills.
2. **Examples over prose.** One concrete example is worth three paragraphs.
3. **Cross-linked.** Every concept this skill depends on is linked. No
   re-explaining what a sibling skill covers.
4. **Single source of truth.** Every fact lives in exactly one skill. Others
   link to it.

## Frontmatter contract

```yaml
---
title: <Human title>
id: skills/stageflip/<tier>/<name>
tier: concept | runtime | mode | profile | tools | workflow | reference | clip
status: substantive | placeholder | auto-generated
last_updated: YYYY-MM-DD
owner_task: T-NNN
related:
  - skills/stageflip/.../SKILL.md
---
```

`check-skill-drift` (T-014) validates:

- Every `.md` under `skills/stageflip/` has valid frontmatter
- `id` matches the file's actual path
- `related:` targets exist
- Auto-generated skills match their generator's current output

## Tiers

| Tier | Owns |
|---|---|
| concept | Invariants and vocabulary |
| runtime | One runtime kind |
| mode | A product (Slide / Video / Display) |
| profile | The schema/tools/validation bundle a mode uses |
| tools | One tool bundle (from I-9) |
| workflow | A multi-tool end-to-end flow |
| reference | Auto-generated catalogues |
| clip | Clip authoring + per-clip cards |

## Auto-generated skills

`skills/stageflip/tools/*/SKILL.md`, `reference/schema/SKILL.md`,
`reference/validation-rules/SKILL.md`, `reference/cli/SKILL.md`, and
`clips/catalog/SKILL.md` are generated by `@stageflip/skills-sync`. Editing
them by hand is pointless — next drift-check re-writes them.

## Package-name convention

Where skill files cross-reference packages, the naming convention is:

| Path | Package name |
|---|---|
| `packages/<name>` | `@stageflip/<name>` |
| `packages/runtimes/<kind>` | `@stageflip/runtimes-<kind>` (dash, not slash) |
| `packages/profiles/<kind>` | `@stageflip/profiles-<kind>` (dash, not slash) |

Occasionally docs refer to the nested kind as `@stageflip/runtimes/contract`
(slash form) — that is conceptual shorthand. The actual npm/workspace name
uses a dash: `@stageflip/runtimes-contract`.

## Implementation

- **`@stageflip/skills-core`** (T-013): `parseSkillFile`, `loadSkillTree`,
  `validateSkill`, `validateTree`. Zod frontmatter schema with a preprocessor
  for the YAML-date coercion gotcha. 14 unit + integration tests.
- **`@stageflip/skills-sync`** (T-034): generators that emit auto-generated
  skill files. First generator is `generateSchemaSkill` for
  `reference/schema/SKILL.md`. Pattern extends to tools (T-169), CLI (T-220),
  clips catalog (T-227), validation rules (T-107).
- **`scripts/check-skill-drift.ts`** (T-014): link-integrity + tier-coverage;
  parse errors reported as drift, not crashes.
- **`scripts/sync-skills.ts`** (T-034): orchestrator with a `--check` mode
  that fails CI on any drift between generator output and committed files.
- **CI**: both `pnpm check-skill-drift` and `pnpm skills-sync:check` run on
  every PR.
