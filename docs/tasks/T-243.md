# T-243 ‚Äî PPTX asset extraction (images) ‚Üí Firebase Storage

**Branch**: `task/T-243-pptx-asset-extraction`

## Goal

Resolve every `ParsedAssetRef.unresolved` in the parser-side tree T-240 emits. Read the asset bytes from the PPTX ZIP, upload via an abstract `AssetStorage` interface (real wiring through `@stageflip/storage-firebase`; mock for tests), and rewrite the reference to schema-typed `AssetRef` (`asset:<id>`). Clears every `LF-PPTX-UNRESOLVED-ASSET` flag.

**Image assets only.** Videos and fonts are deferred to follow-up tasks ‚Äî see Out of scope.

- T-240 merged (provides `ParsedAssetRef.unresolved` and `LF-PPTX-UNRESOLVED-ASSET`).

- T-230 merged (Firebase Storage rules).

## Architecture

`parsePptx` stays pure. T-243 adds a separate, async asset-resolution pass.

**New public surface in `@stageflip/import-pptx`**:

- `interface AssetStorage` ‚Äî abstract upload primitive:
  ```ts
  interface AssetStorage {
    put(content: Uint8Array, opts: { contentType: string }): Promise<{ id: string }>;
  }
  ```
- `resolveAssets(tree, entries, storage): Promise<CanonicalSlideTree>` ‚Äî walks the tree, uploads, rewrites refs, drops `LF-PPTX-UNRESOLVED-ASSET` flags. Returns a new tree with `assetsResolved: true`.
- `unpackPptx(buffer): ZipEntries` ‚Äî already exists internally; promote to public so callers can pass the same entries to both the parser and the asset resolver without double-unpacking.
- `inferContentType(path: string): string` ‚Äî small util mapping `.png` / `.jpg` / etc. to MIME types from the OOXML rel target.

**New public surface in `@stageflip/storage-firebase`** (or wherever the Firebase Storage adapter lives):

- An `AssetStorage` implementation that wraps the existing Firebase upload, using a stable storage path (`pptx-imports/{importId}/{assetId}`) and idempotent writes.

**Composition pattern callers will use**:

```ts
const entries = unpackPptx(buffer);
const tree = await parsePptx(buffer);
const resolved = await resolveAssets(tree, entries, firebaseStorage);
```

Caller can later invoke `documentSchema.parse(...)` on the post-resolution doc once T-242 + T-245 close out the geometry path.

## Type-layer additions

- `CanonicalSlideTree.assetsResolved?: boolean` ‚Äî idempotence marker, mirrors `transformsAccumulated` from T-241a. Re-running `resolveAssets` on already-resolved input returns a structural clone (no second upload).
- New `LossFlagCode` variant: `LF-PPTX-MISSING-ASSET-BYTES` ‚Äî emitted when a `ParsedAssetRef.unresolved` points at a path that's not in the ZIP entries (broken rel). Severity `error`, category `media`.
- New typed error `AssetResolutionError` with codes:
  - `'STORAGE_UPLOAD_FAILED'` ‚Äî wraps the storage adapter's error.
  - `'BROKEN_REL'` ‚Äî fall-through case when the ZIP entry can't be read at all (vs `LF-PPTX-MISSING-ASSET-BYTES` for the well-handled case where we emit a flag instead).

## Acceptance criteria

Each gets a Vitest test, written first and failing.

1. `resolveAssets(tree, entries, storage): Promise<CanonicalSlideTree>` returns a new tree with every `ParsedAssetRef.unresolved` replaced by `{ kind: 'resolved', ref: 'asset:<id>' }`. The id matches the storage's `put()` return.
2. Asset bytes uploaded to storage match the bytes read from `entries` at the rel-resolved path. Test pins this with a recording mock storage.
3. Each upload is keyed by content-hash (sha256) so re-running on a tree with duplicate references produces a single upload (dedup). The storage interface's `put()` is the source-of-truth for ids; the resolver hashes the bytes and passes the hash as a hint, but trusts the storage's returned id.
4. After resolution, **zero** `LF-PPTX-UNRESOLVED-ASSET` flags remain in `tree.lossFlags`.
5. When a `ParsedAssetRef.unresolved.oocxmlPath` is absent from `entries` (broken rel), the function emits a single `LF-PPTX-MISSING-ASSET-BYTES` flag for that ref and leaves the ref `unresolved`. Does NOT throw; the rest of the tree resolves normally.
6. When `storage.put()` rejects, `resolveAssets` throws `AssetResolutionError({ code: 'STORAGE_UPLOAD_FAILED', cause })`. No partial state is committed to the returned tree.
7. `inferContentType('ppt/media/image1.png')` ‚Üí `'image/png'`. Covers `.png`, `.jpg/.jpeg`, `.gif`, `.webp`, `.svg`. Unknown extension ‚Üí `'application/octet-stream'`.
8. `assetsResolved` marker is set; second call short-circuits and returns a structural clone (no `storage.put()` invocations).
9. Coverage ‚â•85% on the new `assets/` module.
10. Determinism: the resolver is deterministic given a deterministic storage adapter. (`Math.random` and `Date.now` not used; tested via the recording mock.)

## Notes for the Orchestrator

1. **Scope narrowing**: the plan's T-243 includes videos + fonts; this spec narrows to images only. Videos and fonts become explicit follow-ups (T-243b, T-243c). The plan should be amended in a separate doc PR after T-243 lands.
2. **Storage-adapter PR**: the concrete `@stageflip/storage-firebase` adapter is intentionally not in T-243's scope. It's a small follow-up (`T-243-storage-adapter`) that wraps the abstract interface around the existing T-230 Firebase Storage primitives. Keeps T-243 focused on the importer side.
3. **Idempotence with mutating storage**: `assetsResolved` marker prevents re-uploads on the same tree, but if the storage backend mutates (e.g., a bucket gets cleared), re-running on a stale tree is silently fine ‚Äî the asset ids stay valid. If they need to be re-uploaded, the caller must drop the marker; document as caller responsibility.
