# T-244 — `@stageflip/import-google-slides` — OAuth + Slides API v1 + native grouping + render-PNG pixel matching

**Branch**: `task/T-244-import-google-slides`

## Goal

Convert the empty `@stageflip/import-google-slides` package stub into the foundational Google Slides importer. The importer takes a presentation id + an auth-token getter, fetches `presentations.get` plus a per-slide rendered PNG via `presentations.pages.getThumbnail` (Slides API v1's documented thumbnail endpoint), and produces a `CanonicalSlideTree` (the same parser-side tree shape `@stageflip/import-pptx` emits). The output preserves native grouping, tables with merged cells, and placeholder inheritance via the `inheritsFrom` field landed in T-251.

The critical contribution beyond a naive Slides API client is the **pixel-candidate pipeline**: for every slide, a deterministic computer-vision pass on the rendered PNG produces text-line / shape / region candidates. A matching pass pairs each Slides API element with one or more candidates to recover information the API alone does not expose — exact text-line bounds, padding, autofit-scaled font sizes, and stroke / fill information that the API exposes only via theme references. Candidates that fail to match deterministically are flagged for the T-246 Gemini multimodal fallback loop.

**The rendered PNG is the source of truth.** API geometry is reference; pixels are ground truth.

## Architectural decisions

### 1. Output shape: same `CanonicalSlideTree` as `@stageflip/import-pptx`

`packages/import-google-slides/src/types.ts` re-exports `CanonicalSlideTree`, `ParsedAssetRef`, `ParsedGroupElement`, etc. from `@stageflip/import-pptx`. Adding a sibling parser-side type definition would split the canonical tree shape and force every consumer (editor canvas, RIR compiler, parity harness) to handle two intermediate shapes. **Decision**: re-export and reuse.

The exception: the `LossFlagCode` union is per-importer-local. T-244 defines its own `GSlidesLossFlagCode` enum, and the canonical `LossFlag.code` field stays `string` (per `@stageflip/loss-flags` types).

### 2. Public surface

```ts
// packages/import-google-slides/src/index.ts
export interface GoogleAuthProvider {
  /** Returns a fresh OAuth access token with `presentations.readonly` scope. */
  getAccessToken(): Promise<string>;
}

export interface CvCandidateProvider {
  /**
   * Run the deterministic CV pipeline on a rendered slide PNG.
   * Returns ranked candidates per detector (text-lines, contours, masks).
   * The production impl is an HTTP client; the test impl reads canned JSON.
   */
  detect(pageImage: Uint8Array, opts: CvDetectOptions): Promise<CvCandidates>;
}

export interface ParseGoogleSlidesOptions {
  presentationId: string;
  auth: GoogleAuthProvider;
  cv: CvCandidateProvider;
  /**
   * Thumbnail size for the per-slide PNG. Slides API constrains this to a
   * fixed enum; the actual returned PNG dimensions depend on the slide
   * aspect ratio. Default LARGE (≥1600 px on the longest side; height
   * scales to preserve aspect — for the standard 16:9 page that is
   * 1600×900).
   */
  thumbnailSize?: 'SMALL' | 'MEDIUM' | 'LARGE';
  /** Confidence threshold for deterministic matches. Below: emits residual for T-246. Default 0.78. */
  matchConfidenceThreshold?: number;
```

`parseGoogleSlides` returns a `CanonicalSlideTree` with `assetsResolved: false` (image bytes not yet uploaded — same staged-narrowing pattern as T-240/T-243a). The caller is expected to subsequently invoke `resolveAssets` (re-exported from `@stageflip/import-pptx`) to upload image bytes and rewrite the references.

### 3. Geometry: API affine → render-pixel boxes

The Slides API exposes per-element `transform`: `{ scaleX, scaleY, translateX, translateY, shearX, shearY, unit }`. Slide page size is `presentation.pageSize` in EMU. The thumbnail comes from `presentations.pages.getThumbnail` at the requested `thumbnailSize` enum; the returned `Thumbnail.{width,height}` give the actual rendered dimensions (height scales with aspect). Conversion:

```
renderSize = { width: thumbnail.width, height: thumbnail.height }
emu_to_px = renderSize / { pageSize.width, pageSize.height }   // per-axis
elementBboxPx = applyAffineToUnitSquare(elementTransform * emu_to_px) ∩ slideBoundsPx
```

The parser reads the actual rendered dimensions off the API response — it does not assume 1600×900 even for the default `LARGE` size, because non-16:9 page sizes produce different heights.

For nested elements (group children), the parser composes parent-group transforms into the child transform before applying `emu_to_px`. **The math is NOT identical to T-241a's PPTX accumulator** — `packages/import-pptx/src/transforms/accumulate.ts` operates on a domain-specific `GroupFrame` (chOff/chExt + rotation-around-center) and explicitly notes (lines 150-157) that proper 2×3 affine matrix composition is out of scope for T-241a. The Slides API exposes a true 2×3 affine (`{scaleX, scaleY, translateX, translateY, shearX, shearY}`); composition is the standard matrix product. **Decision**: T-244 ships its own `composeAffines(parent, child)` private to `@stageflip/import-google-slides/src/geometry/affine.ts`. **No cross-package extraction; `@stageflip/import-pptx` is untouched in this PR.** A future T-241a refactor (or a successor task) may unify the two on a 2×3 representation, but that is out of scope here.

### 4. Pixel-candidate pipeline behind `CvCandidateProvider`

The interface keeps the heavy CV stack out of `@stageflip/import-google-slides`'s direct dependencies. Two implementations ship:

- **`StubCvProvider`** (test-only, in `packages/import-google-slides/src/cv/stub.ts`): reads candidate JSON from a fixture path and returns it verbatim. Used by every Vitest test in this PR; no Python, no network, deterministic.
- **`HttpCvProvider`** (production, in `packages/import-google-slides/src/cv/http.ts`): POSTs the page image to a `CV_WORKER_URL` env var, expects the same JSON shape as the stub. Implementation has only fetch + retry + timeout. The Python sidecar (PaddleOCR + OpenCV + SAM 2) is owned by `T-244-cv-worker` (separate task), not this PR.

`CvCandidates` shape:

```ts
interface CvCandidates {
  textLines: Array<{
    polygonPx: number[][];   // [[x,y], ...] in render-pixel coords
    text: string;             // OCR-recognized text
    confidence: number;       // 0..1
  }>;
  contours: Array<{
    bboxPx: { x: number; y: number; width: number; height: number };
    /** Whether the contour is rectangular, rounded, or other (per OpenCV approxPolyDP). */
    shapeKind: 'rect' | 'rounded-rect' | 'ellipse' | 'polygon';
    /** Sampled fill color from the contour interior. RGBA 0..255. */
    fillSample: [number, number, number, number];
    confidence: number;
  }>;
  masks?: Array<{
    bboxPx: { x: number; y: number; width: number; height: number };
    /** Run-length-encoded mask (COCO RLE). Optional; SAM 2 is the only producer. */
    rle?: string;
    confidence: number;
  }>;
}
```

### 5. Matching: API element → CV candidate

For every API `pageElement` walked (recursively into groups), the matcher attempts in priority order:

1. **Text-content equality** (for `shape.text` / `table.tableRows[].tableCells[].text`): build the API element's text string by concatenating its `textElements[].textRun.content`; find the CV `textLine` whose normalized text matches (Unicode NFC + collapsed whitespace + case-sensitive). One match → `confidence = 1.0` for the text-content axis.

2. **Center-inside containment**: an API element bbox center must fall inside a CV candidate's bbox (or polygon). For text matches, the surviving candidate must contain the API element's center; for shapes (no text), at least one contour must satisfy containment. `confidence_position = 1 - distance(elementCenter, candidateCenter) / max(slideDim)`.

3. **Z-order plausibility**: the slide-level pageElement order maps to CV candidate paint order via the rendering algorithm Google uses (latest in the page-elements array paints on top). Each candidate gets a z-rank from its overlap pattern; mismatched z-rank lowers confidence by 0.15 per delta step.

`overallConfidence = min(content, position) * (1 - zPenalty)`. Above `matchConfidenceThreshold`: emit the canonical element with values from the candidate (precise bbox, sampled fill, OCR-derived text bounds, autofit font size). Below threshold: emit the canonical element with API-only values, push a residual into `tree.pendingResolution[slideId][elementId]`, and emit `LF-GSLIDES-LOW-MATCH-CONFIDENCE`.

### 6. Native grouping, tables, placeholders

- **Groups**: `pageElement.elementGroup` recurses with `children: ParsedElement[]`; the parser emits `ParsedGroupElement` (already in `@stageflip/import-pptx`'s `types.ts`; reused). No flattening — confirmed by T-251's pre-flight audit #1. Group transforms compose into children via the package-local `composeAffines` helper from §3.
- **Tables**: `pageElement.table` emits `TableElement` (confirmed by T-251's pre-flight audit #2 — `TableCell.colspan` / `rowspan` exist with `default(1)`). For every cell with `rowSpan > 1` or `colSpan > 1`, the parser materializes one `TableCell` with the spans set; the "covered" sibling cells are absent from the cells array. If the API ever returns inconsistent span data (e.g., overlapping spans), the parser emits `LF-GSLIDES-TABLE-MERGE-LOST` and falls back to one independent cell per slot.
- **Placeholders**: `pageElement.shape.placeholder.{type, index, parentObjectId}` populates `ElementBase.inheritsFrom: { templateId: parentObjectId, placeholderIdx: index }`. The parent template is either the slide's layout (`page.slideProperties.layoutObjectId`) or the master (transitively). Layouts and masters parse into `Document.layouts` / `Document.masters` ahead of slides; their placeholder shapes likewise emit canonical elements. If `parentObjectId` doesn't match any parsed layout / master, the parser emits `LF-GSLIDES-PLACEHOLDER-INLINED` and inlines the API's flat geometry.

### 7. Image bytes: emit `ParsedAssetRef.unresolved` with the API's `contentUrl`

Image elements (`pageElement.image`) carry a `contentUrl` from the API — a short-lived public URL pointing at the image bytes. T-244 emits `ParsedAssetRef.unresolved` with `oocxmlPath` repurposed as the URL string (the field name is OOXML-flavored but `resolveAssets` in `@stageflip/import-pptx` already treats it as an opaque locator). A T-243-style follow-on can optionally narrow the field to a discriminated union; not in this PR.

`resolveAssets` already handles the protocol-agnostic upload path: bytes-fetch → content-hash → `AssetStorage.put`. To reuse it, T-244 ships a small `gslidesUrlFetcher(url): Promise<Uint8Array>` shim — image `contentUrl`s, like thumbnail `contentUrl`s, are short-lived public links and do NOT take the OAuth token. The shim is tested with a mock fetch (no live HTTP).

### 8. Loss flags taxonomy

The plan v1.24 row enumerates six `LF-GSLIDES-*` codes. T-244 lands all six:

| Code | Severity | Category | Emitted when |
|---|---|---|---|
| `LF-GSLIDES-PADDING-INFERRED` | `info` | `shape` | A text element's API geometry doesn't match its CV text-line bbox; the parser infers `padding` from the delta. Pin the inferred value in the message. |
| `LF-GSLIDES-FONT-SUBSTITUTED` | `warn` | `font` | An API `textRun.style.fontFamily` is not present in the local font cache; the parser substitutes the system stack. |
| `LF-GSLIDES-IMAGE-FALLBACK` | `warn` | `media` | An element can't be modeled cleanly (gradient fills, theme effects, unknown shape kind); the parser emits a placeholder `ParsedAssetRef.unresolved` pointing at the bbox slice for T-245's `rasterizeFromThumbnail` to materialize. |
| `LF-GSLIDES-LOW-MATCH-CONFIDENCE` | `warn` | `other` | An element's deterministic match confidence falls below `matchConfidenceThreshold`. The element ships with API-only values; T-246 may upgrade. |
| `LF-GSLIDES-PLACEHOLDER-INLINED` | `warn` | `shape` | An element's `placeholder.parentObjectId` doesn't resolve to a parsed layout / master; the parser inlines the API's flat geometry instead of emitting `inheritsFrom`. |
| `LF-GSLIDES-TABLE-MERGE-LOST` | `error` | `shape` | A table's API spans are internally inconsistent (overlapping or zero); the parser falls back to per-slot independent cells. |

The wrapper `emitLossFlag` mirrors `@stageflip/import-pptx/src/loss-flags.ts`: a `CODE_DEFAULTS` table holds the severity / category, and the wrapper auto-fills `source: 'gslides'`.

### 11. Determinism

This package's TS code is **not** in the determinism-restricted scope (see CLAUDE.md §3 — only `frame-runtime`, `runtimes`, `renderer-core/clips` are restricted). However, importer determinism still matters for parity / round-trip tests. Two non-determinism sources to control:

- **CV stub provider**: reads candidates from fixture JSON; bit-deterministic.
- **Network fetch**: tests use a mock `fetch` that returns recorded API responses byte-identically.

Real (non-stubbed) calls to Google's API + the CV worker are non-deterministic by nature (Google may rebrand thumbnails); production use accepts this and the loss flags absorb the divergence into well-typed signals.

### 12. RIR / renderer impact

**None.** T-244 produces a `CanonicalSlideTree` matching the existing `@stageflip/import-pptx` shape; downstream of `documentSchema.parse`, the document flows through the same RIR + renderer paths. T-251 already added the `applyInheritance` pass that handles `inheritsFrom`. No new RIR pass, no renderer dispatcher change.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### API client layer

1. `apiClient.fetchPresentation(presentationId, auth)` issues a single GET to `https://slides.googleapis.com/v1/presentations/{id}` with `Authorization: Bearer <token>`. Pin via mock-fetch test.
2. `apiClient.fetchSlideThumbnail(presentationId, slideObjectId, thumbnailSize, auth)` issues a GET to `https://slides.googleapis.com/v1/presentations/{presentationId}/pages/{slideObjectId}/thumbnail?thumbnailProperties.thumbnailSize={size}&thumbnailProperties.mimeType=PNG` (the documented `presentations.pages.getThumbnail` REST endpoint), parses the `Thumbnail` resource, then issues a second GET to `Thumbnail.contentUrl` (without the auth header — the URL is a short-lived public link) for the PNG bytes. Returns `{ bytes: Uint8Array; width: number; height: number }`. Pin the URL format and the two-step fetch.
3. On HTTP 401, the client retries once after re-acquiring the token via `auth.getAccessToken()` (covers expired-token case). On second 401, throws `GoogleApiError({ code: 'AUTH_FAILED' })`. Pin via mock fetch with sequenced 401 / 200.
4. On HTTP 5xx, the client retries with exponential backoff (3 attempts, 250ms / 500ms / 1000ms) before throwing `GoogleApiError({ code: 'API_UNAVAILABLE' })`. Pin via mock fetch.
5. Timeouts: per-request timeout 30s; configurable via options. Pin via mock that delays past the timeout.

### Geometry layer

6. `composeAffines(parent, child)` returns the matrix product such that applying `compose` to the child's local unit square yields the world-space coordinates. Pin via fixture: a child with `translateX = 100` inside a group with `translateX = 200, scaleX = 2` produces `worldTranslateX = 200 + 2 * 100 = 400`.
7. `emuToPx({ pageSizeEmu, renderSize })` returns per-axis scale factors. Pin via 9144000 EMU × 5143500 EMU (default 16:9 page) at 1600×900 → `{ x: 0.0001749…, y: 0.0001749… }` (within 1e-9).
8. `walkPageElements(page, parentTransform)` is recursive over `elementGroup`; emits a flat list of `(element, worldTransform)` tuples. Pin via a 3-deep group fixture.

### CV provider contracts

9. `StubCvProvider` reads JSON from a fixture path and returns it verbatim. Used by every other test in this package.
10. `HttpCvProvider` POSTs the page image (multipart/form-data with `image: <bytes>` + JSON `options`) to `CV_WORKER_URL`. Pin headers + body shape via mock fetch.
11. `HttpCvProvider` retries idempotently on 5xx (3 attempts) and times out at 60s. Pin via mock fetch.
12. The `CvCandidates` shape parses with a Zod validator at the provider boundary. Malformed responses throw `CvProviderError`. Pin: missing `confidence` field → throws.

### Matching layer

13. **Text-content equality**: an API element with `shape.text` = "Quarterly Revenue" matches a CV `textLine` with `text` = "Quarterly Revenue" with `confidence_content = 1.0`. Pin via fixture.
14. **NFC normalization**: API `"café"` (composed) and CV `"café"` (decomposed) match. Pin via fixture.
15. **Whitespace collapse**: API `"  Hello   World  "` and CV `"Hello World"` match. Pin via fixture.
16. **Case sensitivity**: API `"Hello"` and CV `"hello"` do NOT match. Pin via fixture (regression: do not silently lowercase).
17. **Center-inside containment**: a candidate whose bbox does not contain the API element's center is rejected even if text matches. Pin: a `textLine` adjacent to but not containing the element bbox center → `confidence_position = 0`.
18. **Z-order plausibility**: matching to a candidate one z-step out costs 0.15; two steps costs 0.30. Pin via fixture.
19. **`overallConfidence` aggregation**: `min(content, position) * (1 - zPenalty)`. Pin: `(content=1.0, position=0.9, zPenalty=0.15)` → `0.9 * 0.85 = 0.765`.
20. **Threshold semantics**: an element below `matchConfidenceThreshold` (default 0.78) emits `LF-GSLIDES-LOW-MATCH-CONFIDENCE` and goes into `tree.pendingResolution`; the canonical element ships with API-only values.

### Loss flags

21. `emitLossFlag` wrapper auto-fills `source: 'gslides'`. Pin: a flag emitted with `code: 'LF-GSLIDES-PADDING-INFERRED'` has `source === 'gslides'` and `severity === 'info'`.
22. All six `GSlidesLossFlagCode` variants resolve to a `CODE_DEFAULTS` entry. Pin via test that iterates the union.
23. `LF-GSLIDES-PLACEHOLDER-INLINED`: emitted exactly once for an element whose `placeholder.parentObjectId` doesn't resolve. The element is still emitted with API geometry. Pin via fixture.
24. `LF-GSLIDES-TABLE-MERGE-LOST`: emitted once per table whose spans are inconsistent; the parser falls back to per-slot cells. Pin via fixture with `rowSpan: 2` on a 1-row table.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- The Slides API's documented `pageElement` types diverge from the in-repo schema in a way that needs a new `Element` variant. T-244 expects every API element to map cleanly to an existing canonical element type (shape / image / table / group / line). If the API surfaces a kind we don't model (e.g., a video embed that doesn't fit `VideoElement`), escalate; do NOT silently drop or coerce.
- The `presentations.pages.getThumbnail` `contentUrl` returns a redirect chain that doesn't terminate, or returns content with the wrong content-type. Investigate; if the API contract has shifted, escalate before working around it.
- The API's affine matrix produces a child world-transform whose `scale` or `translation` differs by more than 0.5 px from the rendered PNG bbox (at the actual `Thumbnail.{width,height}` returned). The math should be exact; a delta > 0.5 px means either the API row order is wrong (column-major vs row-major confusion) or the page-size unit is being misread (EMU vs PT). Escalate before fudging.
- A real-world deck surfaces a placeholder that targets a chain of three or more templates (e.g., master → layout → another layout → slide). Slides' data model doesn't allow this in theory; if it appears, escalate before extending `inheritsFrom` to a list.
- The CV worker's HTTP response shape diverges from the Zod validator at the provider boundary. The validator is the contract; ratify any extension via the spec, not by relaxing the validator.
- The matching threshold of 0.78 turns out to be wrong on the canned fixtures (e.g., causes false positives on the `low-confidence-residual` fixture). Profile the candidate confidence distribution; propose a new threshold + a fixture justifying it. Do NOT lower the threshold to make tests pass.
- An OAuth scope check rejects the token even though the user granted both required scopes. Investigate at the auth-provider seam first; if it's a Slides API quirk, escalate.
- Coverage on `matching/match.ts` falls below 90% because some branches are only triggered by malformed CV responses. Add fixtures that synthesize the malformed case rather than lowering the bar.
