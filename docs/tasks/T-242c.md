# T-242c — PPTX preset geometries, second wave

**Branch root**: `task/T-242c-batch1-<short-slug>`, `task/T-242c-batch2-<short-slug>`

## Goal

Extend `PRESET_GENERATORS` from the 16 presets shipped via T-242a (#176) and T-242b first-wave (#177) toward the parent T-242 commitment of 36 custom-path presets. T-242c covers **17 of the remaining 20**; the trailing **3 arc-bearing presets (`chord`, `pie`, `donut`)** are deferred to T-242d, which lands `<a:arcTo>` support and the SVG `A` segment helper they require.

After T-242c + T-242d merge, every preset in the parent T-242 spec's 36-preset commitment ships, and `LF-PPTX-PRESET-GEOMETRY` clears for the full 50-preset committed set.

## Dependencies

- T-242a merged (#176): geometry library infra + 6 presets.

## Architectural decisions

### Preset coverage shipped in T-242c

**Batch 1 — arrows + callouts (9 presets)**

| Family | Presets |
|---|---|
| Arrows | `leftRightArrow`, `upDownArrow`, `bentArrow`, `curvedRightArrow` |
| Callouts | `wedgeRoundRectCallout`, `wedgeEllipseCallout`, `cloudCallout`, `borderCallout1`, `borderCallout2` |

Rationale: arrows and callouts use the most aggressive `<a:gd>` formula reads (arrow head/shaft ratios, callout pointer offsets). Grouping them keeps the deeper-math work in one diff.

**Batch 2 — banners + misc (8 presets)**

| Family | Presets |
|---|---|
| Banners | `ribbon2`, `verticalScroll`, `horizontalScroll`, `star10`, `star12` |
| Misc | `moon`, `lightningBolt`, `noSmoking` |

Rationale: simpler polygon math + the trailing star variants (extending the T-242a stars pattern) + three single-shape misc presets.

**Deferred to T-242d (3 presets)**

`chord`, `pie`, `donut` — all are circle-with-cuts shapes where the OOXML definition uses arc segments. Cubic-Bézier approximation would intentionally repeat the `cloud`-preset shape that handover §4 carry-forward #2 already calls out for re-derivation. T-242d ships the `arcTo` helper they need.

### Adjustment defaults

Per parent T-242 spec, T-242c uses constants for `<a:gd>` formula values (no formula evaluator). Each generator picks **defensible defaults derived from ECMA-376 §20.1.9** — these are simplifications of the spec's `<a:gd>` formula defaults expressed as round constants for Implementer ergonomics, not verbatim quotes from the spec. The Implementer should consult the spec's per-preset `<a:gd>` formula defaults (typically `val NNNNN` in EMU thousandths) and may pick numerically equivalent values:

- Arrows: head : shaft = 1 : 2 (50% box length for the head). Approximates `adj1 = 50000` / `adj2 = 50000`.
- `bentArrow`: bend point at 50% of box width and 50% of remaining height. Approximates `adj1` / `adj2` ≈ `50000`.
- Callouts: pointer length defaults to 25% of the shorter box dimension. Approximates spec defaults that vary slightly per callout family.
- Banners: ribbon notch depth defaults to 10% of the band width. Implementer-friendly approximation.
- `noSmoking`: prohibition bar at 30° from the horizontal. Approximates the spec default ratio.

If the Implementer finds a closer-to-spec constant during reference-PR work, that's preferred over the round numbers above; this list is a starting point, not a contract.

If a fixture sets `<a:avLst>` adjustments T-242c does not honor, the existing `LF-PPTX-PRESET-ADJUSTMENT-IGNORED` flag emits as it does for unhonored adjustments today.

### Test ergonomics

Each preset gets a Vitest test pinning the exact SVG `d` string for one canonical box (typically `{ x: 0, y: 0, width: 200, height: 100 }` or square `100×100` where shape is naturally square). Snapshot-style comparisons keep diffs reviewable.

## Acceptance criteria

Each preset gets a Vitest test, written first and failing before implementation.

### Batch 1 (9 presets, arrows + callouts)

1. `geometryFor(prst, box)` returns a non-empty SVG path string for each of the 9 batch-1 preset names.
2. Each preset's generator is pure (no `Date`, no `Math.random`); deterministic across runs.
3. `PRESET_GENERATORS` keys grow from 16 → 25; `COVERED_PRESETS` reflects the additions.
4. Existing 16-preset test snapshots remain unchanged (no regression on T-242a / T-242b output).
5. `LF-PPTX-PRESET-GEOMETRY` no longer emits for the 9 newly covered presets in `parsePptx` round-trip tests.
6. Coverage on the geometries module remains ≥85%.

### Batch 2 (8 presets, banners + misc)

1. `geometryFor(prst, box)` returns a non-empty SVG path string for each of the 8 batch-2 preset names.
2. Same purity / determinism / regression guarantees as batch 1.
3. `PRESET_GENERATORS` keys grow from 25 → 33.
4. `LF-PPTX-PRESET-GEOMETRY` no longer emits for the 8 newly covered presets.
5. Coverage remains ≥85%.

After both batches: 33 of the 36 custom-path presets covered. The trailing 3 (`chord`, `pie`, `donut`) clear when T-242d merges.
