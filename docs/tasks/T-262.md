# T-262 — Auth + org tenancy

**Branch**: `task/T-262-auth-org-tenancy`

## Goal

Stand up the auth + tenancy layer that converts every request into one of four resolved principals (`user` / `org-membership` / `api-key` / `mcp-session`) with a role attached. Architecture is **already ratified** in `skills/stageflip/concepts/auth/SKILL.md` — T-262 implements it. No ADR needed.

T-262 ships:

1. **`@stageflip/auth-schema`** — Zod types for `User`, `Org`, `Membership`, `ApiKey`, `Role`.
2. **Cloud Functions** — invite, join, leave, role-change, active-org-switch (issues fresh JWT with custom claims `{ orgId, role }`), api-key issuance + revoke.
3. **`@stageflip/auth-middleware`** — Connect-style HTTP middleware: `requireAuth`, `requireRole(role)`, `resolvePrincipal` (parses Authorization header → JWT or api-key → returns resolved principal).
4. **`@stageflip/auth-client`** — Frontend hooks: `useCurrentUser`, `useCurrentOrg`, `useRole`, `switchOrg(orgId)`. React-only; consumed by `apps/stageflip-*`.
5. **Firestore data**: `orgs/{orgId}/members/{userId}`, `orgs/{orgId}/apiKeys/{keyId}`, `users/{userId}`.

T-262 does **not** wire individual API routes or Cloud Functions to use the middleware — that's a per-surface follow-up. T-262 stands up the engine + idiom.

## Architectural decisions

All deferred to `skills/stageflip/concepts/auth/SKILL.md`. This spec restates only the load-bearing summary:

- **Auth provider**: Firebase Auth (email/password, Google SSO, GitHub SSO).
- **Principals**: `user`, `org-membership` (user × active-org × role), `api-key` (org-scoped), `mcp-session` (T-223; already shipped).
- **Roles**: `viewer` < `editor` < `admin` < `owner` (strict hierarchy; `requireRole('editor')` accepts `admin` and `owner`).
- **Multi-org per user**: yes. One "active org" tracked in JWT custom claims.
- **JWT custom claims**: `{ orgId: string, role: 'viewer'|'editor'|'admin'|'owner' }`. Set by `setActiveOrg` Cloud Function. Refreshed on next ID-token refresh (Firebase SDK handles).
- **API key**: org-scoped, hashed at rest with `bcrypt` (or workspace-standard hash; check), prefix `sf_<env>_<random>` displayed once at issuance, role attached.
- **Firestore schema** (locked at `docs/architecture.md:307-322`):
  ```
  users/{userId}                   # { email, displayName, photoURL, createdAt }
  orgs/{orgId}                     # { name, slug, createdAt, ownerId, plan, region? }
    members/{userId}               # { role, joinedAt, invitedBy, lastModifiedBy }
    apiKeys/{keyId}                # { name, hashedKey, prefix, role, createdAt, createdBy, lastUsedAt?, revokedAt? }
  ```

### D-T262-1. Active-org switch via JWT custom claims (Firebase Auth idiom)

The `setActiveOrg(orgId)` Cloud Function:

1. Verifies the caller is a member of `orgId` (read `orgs/{orgId}/members/{callerId}`).
2. Calls `admin.auth().setCustomUserClaims(callerId, { orgId, role })`.
3. Returns success; the client calls `await user.getIdToken(true)` to force-refresh the token.

Subsequent requests carry the refreshed JWT with `orgId` + `role` in claims. Middleware reads them directly without a Firestore round-trip per request.

### D-T262-2. API key issuance + verification

- Issuance: `createApiKey({ name, role })` Cloud Function generates a 32-byte random key, formats as `sf_<env>_<base64url>`, returns the **plaintext** key ONCE in the response (caller must capture). Server stores `bcrypt(key)` + the prefix `sf_<env>_<first-6-chars>` for identification.
- Verification: middleware sees `Authorization: Bearer sf_<env>_xxx`, queries `orgs/*/apiKeys` indexed by prefix (Firestore composite index required), bcrypt-compares the suffix.
- **Implementer note**: prefix-only Firestore queries against an unbounded `orgs/*` collection-group are potentially slow. Acceptable to scope: api keys are per-org; the request must carry an `X-Org-Id` header that bounds the lookup. Document this requirement.

### D-T262-3. Determinism posture

`packages/auth-*/**` is NOT clip / runtime code. `pnpm check-determinism` rules don't apply. Auth uses `Date.now()`, `crypto.randomBytes()`, network I/O — all expected.

## Acceptance criteria

Each AC has a Vitest test, written first and failing.

### `auth-schema`

1. `userSchema.parse({ id, email, displayName, photoURL?, createdAt })` accepts the user record per architecture §Firestore.
2. `orgSchema.parse({ id, name, slug, createdAt, ownerId, plan, region? })` accepts the org record.
3. `membershipSchema.parse({ role, joinedAt, invitedBy, lastModifiedBy })` accepts.
4. `apiKeySchema.parse({ id, name, hashedKey, prefix, role, createdAt, createdBy, lastUsedAt?, revokedAt? })` accepts.
5. `roleSchema` is a Zod enum `['viewer', 'editor', 'admin', 'owner']` (in hierarchy order; `viewer` index 0 = lowest).
6. `checkRoleAtLeast(have, need)` returns `true` iff `roleSchema.options.indexOf(have) >= roleSchema.options.indexOf(need)`. Pin every pair (4×4 = 16 fixture rows).

### `auth-middleware`

7. `resolvePrincipal(authHeader)` parses `Bearer <jwt>` → `{ kind: 'user', userId, orgId, role }` via Firebase Admin `verifyIdToken`. Pin via test using a stub admin client.
8. `resolvePrincipal(authHeader)` parses `Bearer sf_<env>_<key>` → `{ kind: 'apiKey', orgId, keyId, role }` via bcrypt-compare against `orgs/<x-org-id>/apiKeys/*`. The `X-Org-Id` header is REQUIRED for api-key auth (D-T262-2); missing → 400.
9. `resolvePrincipal` throws `AuthError({ code: 'missing-token' })` on no header; `AuthError({ code: 'invalid-token' })` on parse failure; `AuthError({ code: 'expired-token' })` on Firebase verifyIdToken expired.
10. `requireAuth` middleware calls `resolvePrincipal`, attaches `req.principal`, and either `next()` or 401 with structured body.
11. `requireRole(need)` middleware calls `requireAuth` then checks `checkRoleAtLeast(req.principal.role, need)`. On rejection: 403 with `{ code: 'insufficient-role', have, need }`.
12. **Bcrypt-compare cache**: api-key verification result is cached in-process for 60 s keyed by the full key (NOT the hash) — bcrypt is slow (~100ms); without cache every request pays the latency. Pin via test that first call takes >50ms (with mock-slow bcrypt) and second call is sub-ms.
13. **Cache invalidation on revoke**: `revokeApiKey` Cloud Function, when called, invalidates the in-process cache by key. Tricky in distributed deployments (multi-instance); document the limitation and accept eventual consistency (cache TTL of 60s bounds the window).

### `auth-client`

14. `useCurrentUser()` returns the Firebase Auth user (or null) reactively. Pin via React Testing Library + Firebase Auth emulator.
15. `useCurrentOrg()` parses JWT custom claims into `{ orgId, role } | null`. Returns `null` if no claims set.
16. `useRole(need)` returns boolean `checkRoleAtLeast(currentRole, need)`.
17. `switchOrg(orgId)` calls `setActiveOrg` Cloud Function, then awaits `user.getIdToken(true)`, then returns. After returning, `useCurrentOrg()` reflects the new org.
18. **No-org-yet flow**: a user with zero memberships has `useCurrentOrg() === null`. Apps must redirect to an org-creation surface; T-262 doesn't ship that surface but the hook semantics support it.

### Cloud Functions

19. `setActiveOrg({ orgId })` callable: verifies caller is a member, sets custom claims `{ orgId, role }`, returns `{ success: true }`. Pin via firebase-functions-test.
20. `createApiKey({ name, role })` callable: requires `admin` role on caller's active org, generates key, hashes, stores, returns plaintext key in response (one-time). Pin includes assertion that the plaintext is NOT logged anywhere.
21. `revokeApiKey({ keyId })` callable: requires `admin` role; sets `revokedAt`; key cannot be used after this. Pin.
22. `inviteMember({ email, role })` callable: requires `admin` role; creates `orgs/{orgId}/invites/{token}` doc with TTL 7 days; returns the invite token. Pin (email delivery is operational; out of scope).
23. `acceptInvite({ token })` callable: any authenticated user; verifies token; creates `orgs/{orgId}/members/{userId}` with the invite's role.
24. `removeMember({ userId })` callable: requires `admin`. Cannot remove `owner` (only owner-transfer can; out of scope for T-262).
25. `changeMemberRole({ userId, newRole })` callable: requires `admin`. Cannot change own role; cannot grant `owner` (owner-transfer is a separate flow).

### Firestore rules

26. `users/{userId}` — read: self only; write: self only.
27. `orgs/{orgId}` — read: members only; write: `admin` or `owner` only.
28. `orgs/{orgId}/members/{userId}` — read: org members; write: `admin` or `owner` (with same self-protection rules from AC #24/#25).
29. `orgs/{orgId}/apiKeys/{keyId}` — read: `admin`+ only; write: never (Cloud Functions only via `service-account` bypass).
30. `orgs/{orgId}/documents/**` — read/write per role per existing T-230 rules; T-262 verifies the role check now reads from `members/{auth.uid}.role` not from a placeholder. Pin via firebase-rules-unit-testing.

## Escalation triggers (CLAUDE.md §6)

- The auth/SKILL.md role hierarchy doesn't compose with an existing consumer's expectations (e.g., `@stageflip/engine` or a CLI handler reads `req.user` in a shape T-262 doesn't match). Escalate; the principal type may need a discriminator we missed.
- Firestore rules tests fail because T-230's existing rules conflict with T-262's additions (e.g., T-230 has a `match /{document=**}` catch-all that T-262's role checks can't override). Escalate; rule-merge strategy is ADR-worthy.
- The `bcrypt` dep adds significant cold-start latency to Cloud Functions (Firebase Functions cold-start is sensitive). If profile shows >500ms cold-start hit from bcrypt, switch to `node:crypto.scrypt` and document. Escalate if both options are inadequate.
- The plan row says T-262 is L. If you find scope expanding past 8 commits, escalate; it may need to split into T-262-base (schema + middleware + functions) + T-262-rider (client hooks + rules tightening), like T-253 did.
- API-key prefix lookup against `orgs/*/apiKeys` collection-group requires a Firestore composite index that hasn't shipped. The spec says one is added in `firestore.indexes.json`; if it doesn't deploy or has unexpected restrictions, escalate.
- Auto-create personal org on signup: spec says no (out-of-scope), but if Reviewer pushes for it on UX grounds, escalate to Orchestrator before reversing — the call has billing implications (T-267).
