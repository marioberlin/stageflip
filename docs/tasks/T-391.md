# T-391 ‚Äî `LiveDataClip` `liveMount` (endpoint fetch)

**Branch**: `task/T-391-live-data-clip-live-mount`

## Goal

Ship the third Œ≥-live family: `LiveDataClip` `liveMount`. A schema-validated `InteractiveClip` with `family: 'live-data'`, registered in `interactiveClipRegistry`, mountable via the T-306 `ClipFactory` contract. Fetches a single response from a configured endpoint at mount time, surfaces the parsed body (and parse errors) through a typed `MountHandle`, emits typed lifecycle telemetry. The clip itself does NOT render the data ‚Äî the host supplies a `render(data, root)` callback OR consumes the data via `MountHandle.onData(handler)`. T-406 (chart family) lands separately and can be wired in a future task once on `main`.

T-391 is the **fourth application of the second Œ≥ pattern** (event-driven, no ¬ß3 reuse). After T-389 + T-390 + T-391, the pattern has run on three different clip families (voice, ai-chat, live-data); this spec re-evaluates the `ProviderSeam<T>` extraction eligibility flagged in T-389 D-T389-10 and rules on it (D-T391-10 below).

T-391 ships `liveMount` only. **T-392 ships `staticFallback`** (cached snapshot rendered as text ‚Äî chart support deferred to T-406's landing) as a separate S-sized task.

## Architectural decisions

### D-T391-1. `LiveDataClip` lives entirely in `runtimes-interactive/src/clips/live-data/`

No new package; no ¬ß3 runtime to reuse. Same posture as T-387 (D-T387-1) and T-389 (D-T389-1). The clip is event-driven (one-shot fetch resolution + manual refresh), not frame-driven. Confirms the second Œ≥ pattern across four families.

### D-T391-2. Schema narrowing ‚Äî `liveDataClipPropsSchema`

```ts
// packages/schema/src/clips/interactive/live-data-props.ts (NEW)
export const liveDataClipPropsSchema = z.object({
  /** Endpoint URL. Absolute; the clip does not resolve relative paths. */
  endpoint: z.string().url(),
  /** HTTP method. v1 supports GET / POST. */
  method: z.enum(['GET', 'POST']).default('GET'),
  /**
   * Allowlisted request headers. Keys are case-insensitive and restricted
   * to a fixed set (no Authorization in the schema ‚Äî host injects auth via
   * the fetcher; ADR-005 ¬ßD7 keeps credential scoping out of clip props).
   */
  headers: z.record(z.string(), z.string()).optional(),
  /** Optional JSON request body for POST. Stringified by the provider. */
  body: z.unknown().optional(),
  /**
   * Response parse mode. `'json'` (default) parses the response body as
   * JSON; `'text'` returns the raw string. Other parsers are future tasks.
   */
  parseMode: z.enum(['json', 'text']).default('json'),
  /**
   * Refresh policy. v1: `'mount-only'` (one-shot at mount; default) or
   * `'manual'` (host calls `MountHandle.refresh()` to re-fetch). Polling
   * is future.
   */
  refreshTrigger: z.enum(['mount-only', 'manual']).default('mount-only'),
  /** Frame at which `staticFallback` poster is sampled (T-392 consumes). */
  posterFrame: z.number().int().nonnegative().default(0),
}).strict();
```

Browser-safe (pure Zod). Mirrors the discriminator pattern from T-383/T-384/T-387/T-389. **No `Authorization` header in the schema** ‚Äî host injects auth via the fetcher; documented per D-T391-3.

### D-T391-3. Permissions ‚Äî `['network']`; pre-prompt off

The clip declares `permissions: ['network']`. Per the existing shim (T-385), `network` is a no-op grant in v1; tenant allowlists land later (T-403). Pre-prompt is OFF by default for `network`-only clips per the precedent established by T-389 D-T389-3 (the no-op grant short-circuits without any user-visible browser dialog). Documented in the skill.

### D-T391-4. `liveMount` API surface

The factory mounts a minimal React tree by default; the host can override the rendered surface entirely via the `render` factory option (D-T391-5). When omitted, the clip renders a single `<output data-role="live-data" />` element whose textContent is the JSON-pretty-printed body (v1 default; chart rendering is T-406-gated and out of scope here).

The factory exposes lifecycle callbacks via `MountHandle`:

```ts
interface LiveDataClipMountHandle extends MountHandle {
  /** Re-fetch using the current endpoint/method/body/headers config. */
  refresh(): Promise<void>;
  /** Latest resolved data (parsed per `parseMode`). `undefined` until first fetch resolves. */
  getData(): unknown | undefined;
  /** Latest resolved status code. `undefined` until first fetch resolves. */
  getStatus(): number | undefined;
  /** Subscribe to data resolution events. Returns unsubscribe. */
  onData(handler: (event: DataEvent) => void): () => void;
  /** Subscribe to error events. Returns unsubscribe. */
  onError(handler: (event: ErrorEvent) => void): () => void;
}

type DataEvent = {
  kind: 'resolved';
  status: number;
  data: unknown;
  durationMs: number;
};

type ErrorEvent = {
  kind: 'fetch-error' | 'parse-error' | 'aborted';
  message: string;
};
```

`refresh()` is rejected with a typed `RefreshTriggerError` when `refreshTrigger: 'mount-only'`. Pin via test (AC #14).

### D-T391-5. `LiveDataProvider` interface ‚Äî host-injected fetcher

```ts
// packages/runtimes/interactive/src/clips/live-data/live-data-provider.ts (NEW)
export interface LiveDataProvider {
  /** Resolve a single endpoint fetch. */
  fetchOnce(args: {
    url: string;
    method: 'GET' | 'POST';
    headers: Record<string, string>;
    body: string | undefined;
    signal: AbortSignal;
  }): Promise<{ status: number; bodyText: string; contentType: string | undefined }>;
}

export interface Fetcher {
  (url: string, init: { method: string; headers: Record<string, string>; body: string | undefined; signal: AbortSignal }): Promise<{
    status: number;
    text(): Promise<string>;
    headers: { get(key: string): string | null };
  }>;
}
```

Two implementations ship with T-391:

1. `HostFetcherProvider({ fetcher })` ‚Äî wraps a host-supplied `Fetcher` (typically `globalThis.fetch.bind(globalThis)`). The clip itself never touches the global. The host vetted the egress at the network-permission gate.
2. `InMemoryLiveDataProvider({ scripted })` ‚Äî for tests; resolves a scripted response by URL.

A `@stageflip/http-abstraction` package mirroring `@stageflip/llm-abstraction` is a future asset-gen task ‚Äî Phase 14 ADR-006 covers the pattern. T-391 ships only the seam.

### D-T391-6. No convergence test, no `frameSource` dependency

Confirms second Œ≥ pattern. The factory does NOT throw `MissingFrameSourceError`; it ignores `MountContext.frameSource`. Pin via test (mounting with `frameSource: undefined` succeeds).

### D-T391-7. Resource cleanup on dispose

`MountHandle.dispose()` MUST:

1. `signal.abort()` on any in-flight `fetchOnce` call.
2. Drop the resolved data reference (no leaked references for GC).
3. Unsubscribe all `onData` / `onError` handlers.
4. Be idempotent.

Less heavy than T-389's LLM-stream graph (no per-token state, no accumulated history), but the abort discipline still matters ‚Äî a leaked fetch costs the host's network budget.

### D-T391-8. Telemetry ‚Äî lifecycle + fetch events

- `live-data-clip.mount.start` (attributes: `family: 'live-data'`, `endpoint`, `method`, `parseMode`, `refreshTrigger`)
- `live-data-clip.mount.success`
- `live-data-clip.mount.failure` (reasons: `'invalid-props'`, `'fetcher-unavailable'`, `'permission-denied'`)
- `live-data-clip.fetch.started` (attributes: `requestId`, `endpoint`, `method`)
- `live-data-clip.fetch.resolved` (attributes: `requestId`, `status`, `durationMs`, `bodyByteLength`, `parseMode`)
- `live-data-clip.fetch.error` (attributes: `requestId`, `errorKind` from `'network' | 'parse' | 'aborted'`)
- `live-data-clip.dispose`

**Privacy posture**: response bodies NEVER appear in telemetry attributes. `bodyByteLength` is an integer length only. Same posture as T-389 D-T389-8 / T-390 D-T390-4. Pin via test (AC #18).

- Subpath export `./clips/live-data` added to `runtimes-interactive/package.json`.

### D-T391-10. Pattern-evaluation ‚Äî fourth application; `ProviderSeam<T>` ruling

T-389 D-T389-10 deferred the `ProviderSeam<T>` extraction question to T-391 (the third provider seam). Compare the three:

- `TranscriptionProvider.start({ stream, onPartial, onFinal, signal })` ‚Äî streaming with two callbacks.
- `LLMChatProvider.streamTurn({ systemPrompt, history, userMessage, model, ..., onToken, signal })` ‚Äî streaming with one callback + many input fields.
- `LiveDataProvider.fetchOnce({ url, method, headers, body, signal })` ‚Äî request/response, no callbacks, returns `Promise<Response>`.

The three seams share:
- A `signal: AbortSignal` for cancellation.
- A test-double pattern (`InMemory*Provider`).
- A "host supplies the client" production path.

The three seams diverge:
- Streaming vs one-shot ‚Äî different control flow.
- Callback shape ‚Äî different.
- Domain payload ‚Äî entirely different.

**Decision**: NO `ProviderSeam<T>` extraction. The shared surface (`signal`, in-memory test double, host-supplied client) is convention, not duplication. A generic seam would either be too narrow (cancellation only) or fight all three concrete shapes.

Skill update: `skills/stageflip/concepts/runtimes/SKILL.md` records the ruling ‚Äî "The three frontier-clip provider seams (transcription, LLM chat, live data) share a host-injection convention but no extractable abstraction. Future seams should follow the convention without inheriting a generic shape."

## Acceptance criteria

### Schema

1. `liveDataClipPropsSchema` parses a valid payload (endpoint, method, headers, body, parseMode, refreshTrigger, posterFrame).
2. Non-URL `endpoint` ‚Üí schema error.
3. `method: 'PUT'` ‚Üí schema error (v1 enum is GET / POST).
4. `parseMode: 'xml'` ‚Üí schema error.
5. `check-preset-integrity` rejects `family: 'live-data'` with bad props; accepts valid fixture.

### Factory + registry

6. `interactiveClipRegistry.resolve('live-data')` returns the factory after side-effect import.
7. Re-importing throws `InteractiveClipFamilyAlreadyRegisteredError`.
8. Mount with `MountContext.frameSource: undefined` succeeds ‚Äî no `MissingFrameSourceError`.

### Fetch lifecycle

9. Mount with `refreshTrigger: 'mount-only'` calls the provider's `fetchOnce` exactly once and resolves `getData()` to the parsed body.
10. `parseMode: 'json'` and a non-JSON body ‚Üí `onError` fires with `{ kind: 'parse-error' }`; `getData()` stays `undefined`.
11. Provider rejection ‚Üí `onError` fires with `{ kind: 'fetch-error' }`; `getData()` stays `undefined`.
12. `parseMode: 'text'` returns the raw body string from `getData()`.
13. POST + JSON body ‚Üí the provider receives the stringified body and `Content-Type: application/json` header.
14. `refresh()` with `refreshTrigger: 'mount-only'` rejects with `RefreshTriggerError`; pinned via test.
15. `refresh()` with `refreshTrigger: 'manual'` invokes a fresh `fetchOnce` and updates `getData()`.

### Resource cleanup

16. `MountHandle.dispose()` calls `signal.abort()` on the in-flight call (verified via spy).
17. `MountHandle.dispose()` is idempotent.

### Telemetry

18. Each documented event (D-T391-8) is emitted with the documented attribute shape, observed via spy. **Response body NEVER appears in any attribute**; `bodyByteLength` is an integer length. Pin via grep on captured event payloads.

### Permission posture

## Escalation triggers (CLAUDE.md ¬ß6)

- The host-injected `Fetcher` shape doesn't fit a real-world `globalThis.fetch` adapter (e.g., headers iteration, body shape, AbortSignal propagation). Escalate; we may need a `@stageflip/http-abstraction` package up front rather than as a Phase-14 follow-up.
- A reviewer pushes back on the "no default real fetcher" stance and wants T-391 to ship a `RealLiveDataProvider` that calls `globalThis.fetch` with a `// determinism-safe:` escape comment. Escalate; the ¬ß3 rule's intent is that the determinism floor stays clean even when the architectural floor permits.
- The `parseMode: 'json'` parse step needs to be deterministic across hosts (some JS engines preserve key order differently). Escalate; we may need to canonicalise.
- The `RefreshTriggerError` semantics conflict with how a host wants to provide a manual-refresh button (e.g., they want a no-op silent rejection rather than a typed throw). Escalate.
