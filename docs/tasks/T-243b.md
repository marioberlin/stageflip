# T-243b — PPTX video asset extraction → AssetStorage

**Branch**: `task/T-243b-pptx-video-extraction`

## Goal

Extend `@stageflip/import-pptx` to (1) parse `<p:videoFile>` elements during the structural pass and (2) resolve their bytes through the existing `AssetStorage` interface. After this PR, a PPTX with embedded video round-trips through `parsePptx → resolveAssets` into a `CanonicalSlideTree` whose `ParsedVideoElement` instances carry resolved `AssetRef`s — no `LF-PPTX-UNRESOLVED-VIDEO` flags emit on success.

Mirrors the architecture T-243a (#172) shipped for images: structural-pass parser emits a `ParsedVideoElement` with `ParsedAssetRef.unresolved`; the post-walk `resolveAssets` pass uploads the bytes and rewrites the ref. Zero schema changes (`@stageflip/schema`'s `VideoElement` already exists at `packages/schema/src/elements/video.ts`).

## Architecture

### Parser-side (extends `parts/sp-tree.ts` + new `elements/video.ts`)

`<p:videoFile>` appears inside `<p:nvSpPr><p:nvPr><p:videoFile r:link="rId.."/></p:nvPr></p:nvSpPr>` (per ECMA-376 §19.5.41) — i.e., it's a **shape extension**, not a top-level element kind. The current `walkSpTree` dispatches on `<p:sp>` / `<p:pic>` / `<p:grpSp>` / etc.; videos surface via `<p:sp>` whose `<p:nvSpPr>` carries a `<p:videoFile>` child.

T-243b's walker change:

1. In `walkSpTree`, after dispatching `<p:sp>` to `parseShape`, **inspect the same `<p:sp>` node** for a `<p:nvSpPr><p:nvPr><p:videoFile r:link="…">` child.
2. If present, emit a `ParsedVideoElement` instead of (or alongside) the shape result. The `<p:sp>`'s geometry becomes the video element's frame; the shape body is dropped.
3. Resolve the `r:link` (or `r:embed`) relId against `ctx.rels`, build a `ParsedAssetRef.unresolved` pointing at the resolved part path.
4. Emit `LF-PPTX-UNRESOLVED-VIDEO` (new `LossFlagCode`).

**Where the parser lives**: a new file `packages/import-pptx/src/elements/video.ts`, mirroring `picture.ts` (the image precedent). Exports `parseVideo(sp: OrderedXmlNode, ctx: ElementContext): { element?: ParsedVideoElement; flags: LossFlag[] }`.

### Type-layer additions

In `packages/import-pptx/src/types.ts`:

```ts
export type ParsedVideoElement = Omit<VideoElement, 'src'> & {
  src: ParsedAssetRef;
};
```

Mirrors the existing `ParsedImageElement` shape (lines 32-36 of `types.ts`).

`LossFlagCode` (the PPTX-local union in `types.ts:117-125`) extends with one new member:

```ts
export type LossFlagCode =
  | 'LF-PPTX-CUSTOM-GEOMETRY'
  | …existing…
  | 'LF-PPTX-UNRESOLVED-VIDEO';  // NEW
```

`CODE_DEFAULTS` in `loss-flags.ts` adds the matching severity/category entry: `{ severity: 'info', category: 'media' }`.

### Asset-resolution-side (extends `assets/resolve.ts`)

The existing `resolveAssets` walker dispatches by `element.type`. Add a `'video'` branch that mirrors the `'image'` branch:

1. If `element.src.kind === 'unresolved'`, read bytes via `readBinaryEntry(entries, element.src.oocxmlPath)`.
2. If bytes are missing, drop a `LF-PPTX-MISSING-ASSET-BYTES` flag (existing flag; same semantics for video as image).
3. Otherwise `storage.put(bytes, { contentType: inferContentType(path) })` → `{ id }`.
4. Rewrite `element.src` to `{ kind: 'resolved', ref: 'asset:' + id }`.
5. Drop the `LF-PPTX-UNRESOLVED-VIDEO` flag from the tree's `lossFlags` array (the same way `resolveAssets` drops `LF-PPTX-UNRESOLVED-ASSET` for images).

`inferContentType(path)` already handles video extensions in spirit (it's a simple ext → MIME map). Verify it returns sensible values for `.mp4` / `.mov` / `.webm`; if not, extend the map in this PR.

### Content-type extension

`packages/import-pptx/src/assets/content-type.ts` currently covers image MIMEs. Extend with the standard video MIMEs:

```ts
'.mp4':  'video/mp4',
'.m4v':  'video/mp4',
'.mov':  'video/quicktime',
'.webm': 'video/webm',
'.avi':  'video/x-msvideo',
'.wmv':  'video/x-ms-wmv',
```

Source: IANA media-type registry (`https://www.iana.org/assignments/media-types/media-types.xhtml#video`).

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### Parser layer

1. `<p:sp>` with a `<p:nvSpPr><p:nvPr><p:videoFile r:link="rId7"/></p:nvPr></p:nvSpPr>` child parses to a `ParsedVideoElement` (not a `ParsedShapeElement`) with `src.kind === 'unresolved'` and `src.oocxmlPath` matching the relId's resolved target.
2. Same fixture but with `r:embed` instead of `r:link`: same result (both map to the same byte-extraction path).
3. `<p:sp>` with **no** `<p:videoFile>` child parses to a `ParsedShapeElement` as before (no regression on T-242a/b/c shape parsing).
4. The `<p:sp>`'s `<p:spPr><a:xfrm>` becomes the `ParsedVideoElement.transform` (frame coords). Pin coordinates in a fixture.
5. Element `id` derives from the same `<p:cNvPr id="…">` source as `parsePicture` uses (existing `makeElementId` helper).
6. Element `name` inherits from `<p:cNvPr name="…">`.
7. `LF-PPTX-UNRESOLVED-VIDEO` emits with `severity: 'info'`, `category: 'media'`, `location.elementId` matching the parsed video id, `originalSnippet` set to the relId.

### Asset-resolution layer

8. `resolveAssets` reads bytes for the video relationship target, calls `storage.put(bytes, { contentType: 'video/mp4' })` (or appropriate MIME), and rewrites `element.src` to `{ kind: 'resolved', ref: 'asset:<id>' }`.
9. `LF-PPTX-UNRESOLVED-VIDEO` drops from `tree.lossFlags` after `resolveAssets` succeeds.
10. If video bytes are missing from the ZIP, `LF-PPTX-MISSING-ASSET-BYTES` emits (existing flag) and the ref stays unresolved.
11. `inferContentType('.mp4')` → `'video/mp4'`; `.mov` → `'video/quicktime'`; `.webm` → `'video/webm'`. Pin via test.
12. Idempotence: running `resolveAssets` twice is a no-op (reuses the existing `assetsResolved: true` marker).

### End-to-end

13. The full pipeline (`parsePptx → resolveAssets`) on a synthetic fixture with one video and one image emits **zero** unresolved-asset flags after the resolve pass.
14. Existing 121+ tests in `@stageflip/import-pptx` continue to pass with no expected-output changes (no regression on image / shape parsing).

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- A real PPTX fixture exposes a `<p:videoFile>` whose `<p:nvPr>` slot also carries an `<p:audioFile>` or `<p:contentPart>` sibling. Spec assumes one media-asset extension per shape; if a fixture violates this, escalate before fudging the dispatcher.
- A fixture uses `r:link` to reference an external URL (not a part inside the ZIP). Spec defers `LF-PPTX-LINKED-VIDEO` handling; the parser should fall through to `LF-PPTX-UNSUPPORTED-ELEMENT`. Confirm with Orchestrator if this happens.
- A real fixture's video MIME isn't in the standard set (e.g., `.mkv`, `.flv`, `.3gp`). Add the entry, cite IANA, no escalation needed.
- The walker dispatcher change introduces a TypeScript inference regression in `walkSpTree` that requires more than 30 minutes to chase. The change is intentionally surgical — material drag means the design assumption is wrong.
- A gate fails after 2 rounds of debugging.

## Notes for the Orchestrator

1. **T-243c (font extraction) is the natural next sibling.** T-243c follows the same pattern (parser surface change + `resolveAssets` extension + new `LossFlagCode`). Authoring T-243c after T-243b lands lets the Implementer reuse the just-validated extension pattern.
2. **No schema changes.** `VideoElement` is a stable shape. T-243b is purely an importer extension.
3. **`<p:videoFile>` is a shape extension, not a top-level element kind.** The walker's existing `<p:sp>` dispatch finds it; the parser routes the shape to either `parseShape` or `parseVideo` based on `<p:nvPr>` content. This shape-vs-video disambiguation is the most error-prone part — pin it with AC #1 + #3 (both with-video and without-video on the same `<p:sp>` walker entry).
4. **`r:link` and `r:embed` are both valid** for embedded videos in OOXML. Spec requires both to map to the same `ParsedAssetRef.unresolved` path (AC #1 + #2). Linked external URLs (a separate concern) emit `LF-PPTX-UNSUPPORTED-ELEMENT` until a future task adds `LF-PPTX-LINKED-VIDEO`.
5. **Dispatch convention** (this session): foreground Implementer, no `isolation: worktree`. Reviewer dispatch likewise foreground. Per `feedback_subagent_shared_worktree.md`, when a sub-agent is running git ops in the shared tree, the main thread holds off on git work; sequence dispatches accordingly.
