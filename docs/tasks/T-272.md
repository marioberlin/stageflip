# T-272 — Backup + point-in-time recovery

**Branch**: `task/T-272-backup-pitr`

## Goal

Stand up backup + PITR for the canonical state stores (Firestore, Storage), document the restore procedure, and run a restore drill to validate the procedure works end-to-end. T-272 is operational + procedural — minimal code, heavy ops.

T-272 ships:

1. **Firestore PITR** enabled on both `(default)` and `eu-west` databases (T-271).
2. **Daily backup schedule** for Firestore — exports to `gs://stageflip-backups/firestore/<region>/<date>` retained 30 days.
3. **Daily backup schedule** for Firebase Storage — bucket-level snapshots retained 30 days.
4. **Restore runbook** — `docs/ops/restore-procedure.md` documenting full + partial restore.
5. **Restore drill** — execute the runbook against a staging tenant + record results.
6. **Backup-verification Cloud Function** — daily scheduled function that asserts the latest backup exists and is well-formed.

## Dependencies

- T-271 (EU Firestore region) — merged. Backups must cover both regions.
- T-264 (observability) — merged. Backup-verification function emits metrics + Sentry alerts on failure.
- **Operational**: GCS bucket `gs://stageflip-backups` provisioned with 30-day lifecycle policy. (Out of scope = automating bucket provisioning; in scope = documenting + verifying it exists.)

## Out of scope

| Item | Why deferred |
|---|---|
| RTDB (presence) backup | Presence is ephemeral by design (ADR-006 §D5; never persisted). No backup needed. |
| Upstash Redis backup | Queue is ephemeral; jobs replay on retry. No backup needed. |
| BullMQ job replay logic | Already exists in BullMQ; T-272 doesn't extend it. |
| Asset CDN cache restoration | CDN regenerates from Storage on cache miss; no separate backup. |
| Cross-region disaster recovery (whole-region failure) | Future task. T-272 covers data loss within a region (PITR for Firestore handles); whole-region failover is a separate plan. |
| Encrypted backups (customer-managed keys) | T-272 uses GCP-managed encryption; CMEK is a future enterprise feature. |
| Automated restore (one-click) | Restore is a high-stakes operation; manual procedure with checklist is safer for v1. |

## Architectural decisions

### D-T272-1. PITR for Firestore is configuration, not code

Firestore's PITR is a per-database setting in the Firebase console (or via `gcloud firestore databases update --enable-pitr`). T-272 documents the enablement + verifies via Admin SDK that it's on. No code-side enablement.

PITR retains 7 days of fine-grained history. For longer retention, daily exports (D-T272-2) handle the 7-day-to-30-day window.

### D-T272-2. Daily exports via Cloud Scheduler

A scheduled Cloud Function (`scripts/backup-firestore.ts` deployed as a Cloud Function) runs daily at 02:00 UTC, calls `firestore.exportDocuments()` for each database, writes to `gs://stageflip-backups/firestore/<region>/<YYYY-MM-DD>`.

GCS lifecycle policy on the backups bucket auto-deletes objects > 30 days old.

Same pattern for Firebase Storage: the backup function lists the assets bucket and copies to the backups bucket (or uses GCS bucket-level snapshots if the project has Object Versioning enabled — Implementer's choice, document).

### D-T272-3. Backup verification

A second scheduled Cloud Function runs at 03:00 UTC (1 hour after backup) and asserts:

1. The latest expected backup file exists.
2. File size is non-zero.
3. (Best-effort) The backup metadata file is well-formed (JSON parse succeeds).

On failure, calls `captureError` (T-264) which emits to Sentry. Ops responds.

### D-T272-4. Restore is manual

`docs/ops/restore-procedure.md` documents full restore + per-collection restore + per-document restore (PITR). Each path is explicit:

- **Full restore** (catastrophic loss): import from backup; expected RTO 1 hour; covered by drill.
- **Per-collection restore** (e.g., one collection corrupted): export-only-collection from backup, import.
- **Per-document PITR** (e.g., user deleted important data): use PITR within 7 days.

The drill (D-T272-5) verifies the full-restore path; per-collection and PITR paths are documented but not drilled (lower-stakes recovery).

### D-T272-5. Restore drill

Once during T-272 implementation:

1. Spin up a staging Firebase project (or use existing staging).
2. Seed it with realistic data (T-269 seed script may help).
3. Wait for a daily backup to land.
4. Execute the full-restore procedure.
5. Verify the restored data matches the seed.
6. Document timing + any procedural issues found in the runbook.

The drill runs against staging, NOT prod. Document the drill in `docs/ops/restore-drill-2026-04.md` (or current year-month).

### D-T272-6. Determinism posture

`firebase/functions/src/backup/**` is not in determinism scope (timestamps, network I/O are expected).

## Acceptance criteria

### Backup Cloud Functions

1. `backupFirestore` callable: triggered daily by Cloud Scheduler at 02:00 UTC. Calls `firestore.exportDocuments()` for both `(default)` and `eu-west` databases (per T-271). Writes to `gs://stageflip-backups/firestore/<region>/<YYYY-MM-DD>/`. Pin via firebase-functions-test with mock admin SDK.
2. `backupStorage` callable: lists the production assets buckets, copies to `gs://stageflip-backups/storage/<bucket-name>/<YYYY-MM-DD>/`. (Or uses GCS Object Versioning — document choice.) Pin via mock.
3. Both backup functions emit success/failure logs via `@stageflip/observability` logger.
4. Failures surface via `captureError` (Sentry).

### Verification function

5. `verifyBackup` runs daily at 03:00 UTC. Asserts: (a) latest expected backup file exists at the predicted path; (b) file size > 0; (c) (best-effort) JSON metadata parses.
6. On any assertion failure, emits a Sentry alert via `captureError` AND logs at `error` level (which auto-promotes to Sentry per T-264 logger semantics).
7. Pin via test using mock GCS that returns expected/missing/empty files.

### Runbook

8. `docs/ops/restore-procedure.md` documents the three restore paths (full / per-collection / PITR) with explicit `gcloud` / `firebase` commands.
9. Each path includes: prerequisites, step-by-step commands, verification steps, rollback plan.
10. The runbook is reviewed by a second pair of eyes (Reviewer agent counts) and pins assumptions about authn (which service account, which IAM roles).

### Drill

11. `docs/ops/restore-drill-2026-04.md` documents an actual drill: timing, results, issues found, runbook revisions.
12. Drill produces a successful restore (the staging tenant's data is recoverable). If the drill FAILS, escalate per CLAUDE.md §6 — that's a finding, not a doc gap.

### CLI helper

13. `scripts/backup-restore.ts --dry-run --target=<staging|prod> --backup-date=<date>` prints planned operations.
14. `scripts/backup-restore.ts --execute --target=staging` runs the staging restore (prod is gated behind a manual confirmation prompt).

### Backward compatibility

15. The full workspace test suite passes unchanged.
16. Existing Firestore + Storage operations are unaffected (backup runs in admin context, doesn't interact with app traffic).

### Coverage + gates

17. Coverage on `firebase/functions/src/backup/**` ≥85%.
18. Coverage on `scripts/backup-restore.ts` ≥80% (excluding interactive prompts).
19. `pnpm typecheck`, `pnpm lint`, `pnpm test`, `pnpm check-licenses`, `pnpm check-remotion-imports`, `pnpm check-determinism`, `pnpm check-skill-drift`, `pnpm size-limit` all green.

## Escalation triggers (CLAUDE.md §6)

- The restore drill FAILS — that's a real finding, escalate. Don't merge T-272 if the procedure doesn't actually work.
- Firestore export takes longer than the verification window (>1 hour) so the verifier reports a missing backup. Escalate; verification timing must adapt to actual export duration.
- The backup bucket permissions don't allow the Cloud Function service account to write. Escalate IAM setup before continuing.
- PITR is not supported on a database that hosts > N collections (Firestore limit). Escalate; we may need to split databases or accept reduced PITR scope.

## Notes for the Orchestrator

1. **M-sized; five commits.** Backup functions + verifier + CLI + runbook + drill.
2. **The drill is mandatory.** Don't merge T-272 with a paper runbook that hasn't been executed; the value of the runbook is precisely that it works.
3. **Reviewer focus**: AC #5 (verifier asserts the right things — file existence is necessary but not sufficient; size + JSON parse catch the "empty backup" failure mode); AC #11 (drill log is real, not synthesized); restoring against staging (not prod) — verify the runbook's `--target=staging` gate is real.
4. **No determinism issues.** Backup functions use `Date.now()` and network I/O — both fine for ops code.
5. **GCS bucket lifecycle policy** is operational; T-272 verifies it exists but doesn't provision it. If the policy is missing, the 30-day retention claim doesn't hold — flag immediately.
