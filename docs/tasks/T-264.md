# T-264 — OpenTelemetry + Sentry observability

**Branch**: `task/T-264-otel-sentry`

## Goal

Stand up the `@stageflip/observability` package — OpenTelemetry tracing + Sentry error reporting + structured-log helpers, wired through every long-lived service surface (collab/sync, storage adapters, render farm, cv-worker bridge) with W3C `traceparent` propagation across service hops. T-264 ships the package + initialization helpers; instrumenting individual handlers is a per-package follow-up coordinated with each owner.

T-264 is a foundation task. Before T-264, the workspace has no telemetry; after, every service can opt in with one import.

## Dependencies

- T-263 (rate limiting) — **strongly preferred to merge first** so that `rate_limit_exceeded` becomes a measurable event from day one. Not strictly blocking; T-264 ships the SDK, T-263's hooks plug in afterwards.
- T-262 (auth + tenancy) — preferred for `org_id` / `user_id` attributes on every span. If T-262 hasn't shipped, traces still emit; just without those tags.
- Sentry org + project provisioned (operational; coordinate with infra at PR time).
- Honeycomb / Tempo / Grafana Cloud OTel endpoint provisioned (operational).

## Architectural decisions

### D-T264-1. SDK choice: OpenTelemetry JS + Sentry JS

- **OpenTelemetry JS** for tracing + metrics. Vendor-neutral; we can switch backends without rewriting instrumentation.
- **Sentry JS** for error reporting + breadcrumbs + release tracking. OTel does errors via span events but Sentry's UX is better for triage; both coexist.
- **Both wired in one package** (`@stageflip/observability`) so consumers do `import { initObservability } from '@stageflip/observability'` once at process start.

### D-T264-2. Sampling

- **Production**: 1% trace sampling, 100% error sampling. Standard for cost control.
- **Staging**: 10% trace sampling, 100% error sampling.
- **Dev**: 100% trace sampling (helpful for debugging), 100% error sampling.
- Configurable per-service via env (`STAGEFLIP_OTEL_TRACE_SAMPLE_RATE`).
- **Critical paths always 100%**: T-264 exposes `withForcedTrace(name, fn)` to opt specific spans (auth flow, payment, data export) into 100% sampling regardless of base rate. Use sparingly.

### D-T264-3. Trace propagation: W3C traceparent

- Standard W3C `traceparent` header for HTTP propagation. OTel JS bundles `W3CTraceContextPropagator` — use it.
- For non-HTTP service hops (BullMQ jobs, Yjs storage delta calls, RTDB presence writes): T-264 exposes `getTraceContext()` / `withTraceContext(ctx, fn)` helpers so the caller can attach trace metadata to the job payload / event header, and the callee restores context on dequeue. Per-package follow-up wiring.

### D-T264-4. Standard span attributes

Every span SHOULD carry (when available — T-262 provides these):

- `org.id`
- `user.id`
- `service.name` (from package.json name)
- `service.version` (from package.json version)
- `deployment.environment` (`dev` | `staging` | `prod`)

Plus per-domain attributes: span owners decide. T-264 doesn't enumerate; each consumer's instrumentation rollout PR adds its specific attributes (e.g., `collab.docId`, `render.renderId`).

### D-T264-5. Sentry release + sourcemap upload

- `release` = git SHA at build time. Set via env `STAGEFLIP_RELEASE` from CI.
- Sourcemaps uploaded to Sentry from CI per-package after build. Implementer ships the upload script (`scripts/sentry-upload-sourcemaps.ts`); CI integration is operational.
- Sentry `environment` field matches `deployment.environment`.

### D-T264-6. Structured logs (bonus surface)

OTel covers tracing; Sentry covers errors. **Plain logs** still need a home. T-264 exports a thin `createLogger(name)` that:

- Wraps `pino` (workspace-standard? if not, the de-facto Node JSON logger).
- Auto-emits `traceparent` and `span_id` fields when called inside an active OTel span context.
- Auto-promotes `level: 'error'` logs to Sentry breadcrumbs.

This keeps three signals (logs / traces / errors) correlatable without manual stitching.

### D-T264-7. Determinism posture

`packages/observability/**` is NOT clip / runtime code. Standard pnpm-determinism rules don't apply. The SDK is wall-clock + network-driven by nature.

## Acceptance criteria

Each AC has a Vitest test, written first and failing.

### `init.ts`

1. `initObservability({ serviceName, environment, traceSampleRate? })` registers OTel SDK + Sentry SDK with the configured exporters and sampling.
2. **Idempotent**: calling `initObservability` twice is a no-op (idempotency tracked via a module-level singleton). Pin via test.
3. **`SENTRY_DSN` empty/undefined** → Sentry initialization is skipped silently; OTel still initializes. Pin via test (covers staging environments where Sentry isn't desired).
4. **`OTEL_EXPORTER_OTLP_ENDPOINT` empty/undefined** → OTel uses a console-exporter fallback (so dev still shows traces in stdout). Pin via test.

### `tracer.ts`

5. `withForcedTrace(name, fn)` runs `fn` inside a span that's always sampled regardless of base rate. Pin: when `traceSampleRate=0`, the forced span still exports.
6. `getTraceContext()` returns a serializable `{ traceparent: string, tracestate?: string }` object suitable for embedding in BullMQ payloads or event headers.
7. `withTraceContext(ctx, fn)` runs `fn` with the OTel active span set to the ctx-derived span. The created span IS a child of the upstream span (verifiable via `traceparent` parent-ID match).

### `errors.ts`

8. `captureError(err, context?)` forwards to Sentry. Pin: a thrown `Error` with `cause` chain is captured with the cause attached.
9. **Off-Sentry mode** (DSN unset): `captureError` no-ops without throwing. Pin via test.
10. `captureBreadcrumb(message, data?)` adds a Sentry breadcrumb to the current scope.

### `logger.ts`

11. `createLogger('my-service').info('hello', { foo: 'bar' })` emits a JSON log line with `name: 'my-service'`, `msg: 'hello'`, `foo: 'bar'`, `level: 'info'`.
12. Inside a `withForcedTrace`/`startSpan` context, the log line carries `traceparent` and `span_id` fields. Pin via test.
13. `logger.error(err)` automatically calls `captureError(err)` (via Sentry promotion). Pin via spy on Sentry.

### `config.ts`

14. Reads `OTEL_EXPORTER_OTLP_ENDPOINT`, `STAGEFLIP_OTEL_TRACE_SAMPLE_RATE`, `SENTRY_DSN`, `STAGEFLIP_RELEASE`, `NODE_ENV` (mapped to `deployment.environment`). Defaults documented.
15. Invalid `STAGEFLIP_OTEL_TRACE_SAMPLE_RATE` (non-numeric, < 0, > 1) throws at module load with clear message.

### Sentry-upload-sourcemaps script

16. `scripts/sentry-upload-sourcemaps.ts --release=<sha> --org=<org> --project=<proj> --path=<dir>` uploads sourcemaps to Sentry.
17. `--dry-run` mode prints what would be uploaded without calling Sentry.
18. Script exits non-zero on missing required args; usage message printed.

### Backward compatibility

20. No existing package gains observability instrumentation in this PR. (Per-package wiring is a follow-up.)

## Escalation triggers (CLAUDE.md §6)

- The workspace already has a logging convention (e.g., `winston` already in use somewhere). Escalate; D-T264-6 picks `pino` but consistency wins. Do NOT introduce a second logger.
- OTel SDK initialization conflicts with `@stageflip/cv-worker` (Python service) or any subprocess that has its own OTel setup. Escalate; cross-runtime trace propagation may need additional Python-side wiring.
- Sentry project + DSN aren't provisioned at PR time. Ship the package with the off-Sentry-mode default (AC #3); flag that ops needs to provision before consumers wire in.
- `@opentelemetry/sdk-node` introduces a peer-dependency conflict with another package (e.g., a runtime that already bundles a different OTel API version). Escalate; pinning OTel-API at the workspace root may be the fix but is ADR-worthy.

## Notes for the Orchestrator

1. **M-sized; six commits.** Init + tracer + errors + logger + config + skill.
2. **Reviewer focus**: AC #5 (forced trace ALWAYS samples — easy to ship a sampling bug where forced still gets dropped); AC #7 (parent-child relationship across `withTraceContext` — verifiable via `traceparent` parent-ID); AC #13 (Sentry promotion from logger — must not double-capture if user explicitly called `captureError`).
3. **No package gains instrumentation in this PR.** Per-package rollouts are follow-ups. The contract: each consumer adds `initObservability(...)` at its entry point + spans around its hot paths.
4. **Sentry DSN ops dependency**: the package ships off-Sentry-mode by default (AC #3); ops provisioning before prod-mode is a follow-up coordination.
5. **Sourcemap upload script** must run from CI after `pnpm build`. Implementer ships the script; the GitHub Actions wiring is operational. Don't block T-264 on that wiring.
6. **OTel + Sentry double-counting errors**: Sentry's instrumentation can also create OTel spans for HTTP requests (Sentry's auto-instrumentation). Verify the two don't double-emit; if they do, disable Sentry's auto-instrumentation in the init helper and let OTel handle HTTP. Document the choice.
