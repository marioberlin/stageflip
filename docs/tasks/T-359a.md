# T-359a — extend T-313 parity generator for multi-variant + bind productionRenderer

**Branch**: `task/T-359a-parity-multivariant-renderer`

## Goal

Close the two structural blockers documented in T-359's escalation that forced the first Cluster E preset (`f1-sector-purple-green`) to ship with `signOff.parityFixture: pending-user-review` instead of `signed:<date>`:

1. **`scripts/generate-preset-parity-fixture.ts` is single-frame, single-variant per preset id.** It writes one `golden-frame-<n>.png` and a one-element `referenceFrames` array. T-359's preset has three semantically distinct visual states (`sessionBest` / `personalBest` / `neutral`) that must live under one preset id (D-T359-5 explicitly rejected the three-preset-files alternative).
2. **`productionRenderer` (lines ~542–553 of the same script) is a stub** that always throws `RenderUnavailableError`. No preset has yet produced real golden PNGs end-to-end; the only existing fixture dir (`parity-fixtures/news/cnn-classic/`) is empty.

T-359a fixes both in one PR. After it lands, every existing preset stub across all clusters (44+ today) can flip from `pending-user-review` to `signed:<date>` over time as their owners run the generator. The unblock is **the whole preset corpus**, not just T-359.

## Architectural decisions

### D-T359a-1. Manifest format — object-keyed by variant name

Two viable shapes for multi-variant `referenceFrames`:

- **Array-of-arrays**: `referenceFrames: [[60], [60], [60]]` paired with a `variants: ['sessionBest', 'personalBest', 'neutral']` sibling.
- **Object-keyed**: `variants: { sessionBest: { frames: [60] }, personalBest: { frames: [60] }, neutral: { frames: [60] } }`.

T-359a picks **object-keyed**. Rationale:

- Each variant carries metadata beyond its frame list (the `state` prop value, optional per-variant description, future per-variant prop overrides). An object-keyed shape grows cleanly; an array-of-arrays does not.
- Diffs are stable across variant ordering (variant names are the keys; insertion order is irrelevant for downstream consumers).
- Single-variant manifests preserve the existing `referenceFrames: [60]` shape; multi-variant manifests opt in via a new top-level `variants` field. Backward compat is structural (D-T359a-5).

The new manifest TypeScript type extends `PresetFixtureManifest` with an optional `variants` field. When present, `referenceFrames` may be omitted (or kept for v1 single-variant clarity — Implementer's call).

### D-T359a-2. Sign-off — per-manifest atomic, not per-variant

`signOff.parityFixture` lives in the **preset** frontmatter, not the manifest. The signed shape (`signed:YYYY-MM-DD`) refers to the whole fixture bundle (manifest + all goldens + thresholds). T-359a does NOT introduce per-variant sign-off in frontmatter.

`--mark-signed` requires **all declared variants to render cleanly**. If any variant fails to render (or `--variant` was not invoked for every declared variant), `--mark-signed` refuses to mutate the frontmatter. Re-sign guard from T-313 D-T313-4 (`--force` to overwrite an existing `signed:...`) is preserved unchanged.

Rationale: the consumer of `signOff.parityFixture` (cluster-eligibility gate + integrity gate) treats the value as a single bit ("this preset's fixture is reviewed and signed"). Per-variant sign-off would require schema changes to frontmatter + integrity gate + cluster-eligibility — far outside this task's scope. Atomic per-manifest sign-off is sufficient and minimally invasive.

### D-T359a-3. `productionRenderer` binding — injection hook in `parity-cli`

Three options for replacing the `RenderUnavailableError` stub:

- **(a) Wire directly into `packages/parity-cli`** and require the script be invoked from there (loses standalone `pnpm tsx scripts/generate-preset-parity-fixture.ts ...` invocation).
- **(b) Build a self-contained renderer in the script** (too much for one task; would duplicate the puppeteer/CDP pipeline `packages/parity-cli/src/puppeteer-primer.ts` already runs).
- **(c) Export a `bindProductionRenderer(renderer: FixtureRenderer): void` hook** that `parity-cli` calls at process start. The script remains standalone-invokable for tests; production invocation goes through `parity-cli` which binds the real renderer.

T-359a picks **(c)**. Rationale:

- Preserves the existing `FixtureRenderer` DI interface; the test path keeps working unchanged.
- Localizes the renderer dependency in `parity-cli` (where the puppeteer + CDP machinery already lives — `packages/parity-cli/src/prime.ts`, `puppeteer-primer.ts`).
- Adds a new bin entry to `parity-cli` (e.g., `parity-cli generate-fixture --preset=<id> ...`) that internally calls `bindProductionRenderer(...)` then defers to `runGenerate(...)`. The standalone script invocation continues to throw `RenderUnavailableError` with the same helpful message — pointing users to the `parity-cli` bin.
- The hook lives in `scripts/generate-preset-parity-fixture.ts` itself (not a new module) — a setter that swaps the `productionRenderer` object's internal `_impl` reference. Mirror the pattern used by other late-bound singletons in the codebase if any precedent exists; otherwise the simplest setter is acceptable.

### D-T359a-4. clipKind → component resolution — injected map at the renderer boundary

The bound renderer needs to know **what React component** to render for a given `clipKind`. There is no semantic-clip → frame-clip dispatcher today (T-359 D-T359-4 documents this in prose). T-359a does NOT ship a dispatcher.

Instead, the production renderer accepts a `clipKindResolver: (clipKind: string) => ComponentType | undefined` map at construction time. The `parity-cli` bin populates this map with the known clipKinds (at minimum: `bigNumber → animated-value`, plus any others that already have prose-documented bindings in shipped preset frontmatter — `bar`, `line`, `area`, `pie`, `donut`, `scatter`, `combo` from T-406 chart family if they have parity fixtures).

The resolver returns `undefined` for unknown clipKinds; the renderer surfaces this as a clean `RenderUnavailableError` (`no component bound for clipKind '<x>'`) so the CLI's exit-1 message is helpful.

Implementer's note: the initial map can be small (just `bigNumber` + the chart family). Each cluster's first preset to actually request a render will add its clipKind to the map. The map is a one-line addition per clipKind; the dispatcher (which would auto-derive the map) is a future task.

### D-T359a-5. Backward compatibility — single-variant manifests unchanged

Existing single-variant invocation must continue to work bit-for-bit:

```
pnpm tsx scripts/generate-preset-parity-fixture.ts --preset=cnn-classic --frame=60
```

…produces the same `manifest.json` (with `referenceFrames: [60]`, no `variants` field) and the same `golden-frame-60.png` (no `-<variant>` suffix). The new `--variant` flag is opt-in. When absent, the script behaves exactly like T-313.

The manifest reader/consumer (parity harness, cluster-eligibility check) treats a missing `variants` field as "single-variant, legacy shape". This is the interop contract.

### D-T359a-6. T-359 re-promotion — IN SCOPE for this PR

After landing the multi-variant generator + production renderer, T-359a runs the generator on T-359's preset (`f1-sector-purple-green`) with all three variants and `--mark-signed`, producing the three goldens + manifest and flipping the frontmatter from `signed-pending-user-review` to `signed:<today>`. This is the proof-of-life for the multi-variant pipeline AND closes the T-359 carve-out in the same PR that unblocks it.

If the renderer-binding work runs over budget (D-T359a-3 + D-T359a-4 are non-trivial), the Implementer may **descope T-359 re-promotion to a follow-up PR** and ship T-359a as the unblock-only task. In that case the AC for T-359 re-promotion (AC #5 below) becomes optional. The Implementer flags this trade-off explicitly in the PR description.

### D-T359a-7. Other 44+ preset stubs — NOT touched in this PR

T-359a does not flip any preset's `signOff.parityFixture` other than (optionally, per D-T359a-6) T-359's. The 44+ stubs across all clusters remain `pending-user-review` after T-359a lands. Their owners run the generator individually as their cluster batches reach merge readiness.

Rationale: owner-controlled sign-off is the ADR-004 §D5 contract. T-359a is infrastructure; mass sign-off is operational + per-cluster.

### D-T359a-8. CLI shape — `--variant` repeatable, with declared-variants check

```
pnpm tsx scripts/generate-preset-parity-fixture.ts \
  --preset=f1-sector-purple-green \
  --variant=sessionBest \
  --variant=personalBest \
  --variant=neutral \
  --frame=60 \
  --mark-signed
```

Repeatable `--variant=<name>` accumulates into an array. Comma-separated form (`--variant=a,b,c`) is also accepted (Implementer picks one as canonical for the help text; both parse). When at least one `--variant` is supplied, the manifest is written in multi-variant shape per D-T359a-1.

When `--variant` is absent and the preset declares variants in its body (e.g., T-359's `## Rules` section names `sessionBest` / `personalBest` / `neutral`), the script does NOT auto-discover them — the CLI requires explicit `--variant` invocations. Auto-discovery from preset body is out-of-scope for v1 (would require parsing prose; brittle).

The renderer is invoked once per variant; each variant's golden is written as `golden-frame-<n>-<variant>.png`. Variant names must be `^[a-z][a-zA-Z0-9]*$` (camelCase, no hyphens) — enforced by `parseArgs`.

### D-T359a-9. Determinism + browser-bundle posture

Script remains Node-only (T-313 D-T313-5 unchanged). The production renderer (in `parity-cli`) uses puppeteer + CDP — already exempt from the determinism gate (puppeteer is the existing parity infrastructure, not new clip code). T-359a introduces zero new code paths inside `clips/**` or `frame-runtime/**`.

The `bindProductionRenderer` setter is module-scoped state. This is acceptable for a script's late binding (operator invokes once per process); it is NOT acceptable for runtime/clip code. The determinism gate's scope (CLAUDE.md §3) does not include `scripts/**` or `packages/parity-cli/**`.

## Acceptance criteria

### `--variant` flag + multi-variant manifest (AC #1–#6)

1. `--variant=<name>` accepted; repeated invocations accumulate. Comma-separated form (`--variant=a,b,c`) also accepted.
2. Variant names validated against `^[a-z][a-zA-Z0-9]*$`; invalid names exit non-zero with clear error.
3. With one or more `--variant` flags, the renderer is invoked once per variant; goldens written as `golden-frame-<n>-<variant>.png`.
4. The manifest's `variants` field is populated with the object-keyed shape (D-T359a-1); each entry has at least `frames: [<n>]`.
5. `--mark-signed` mutates frontmatter to `signed:<today UTC>` only when ALL declared variants rendered cleanly. Partial failure → no mutation, exit non-zero.
6. Re-running the generator on an existing multi-variant manifest is idempotent: same goldens overwrite (atomic write), same manifest produced.

### `productionRenderer` binding (AC #7–#10)

7. `productionRenderer` no longer throws `RenderUnavailableError` when invoked from `packages/parity-cli`'s `generate-fixture` bin.
8. `bindProductionRenderer(renderer)` swaps the module-scoped impl; `parity-cli`'s bin calls this at module load.
9. Calling the standalone script `pnpm tsx scripts/generate-preset-parity-fixture.ts ...` (without `parity-cli`) still produces the AC #4 error from T-313 (`rendering pipeline unavailable: ...`) — backward-compat for that error path.
10. The renderer's `clipKindResolver` map includes at minimum `bigNumber → animated-value`. Unknown clipKinds throw `RenderUnavailableError` with message `no component bound for clipKind '<x>'`.

### Backward compatibility (AC #11–#13)

11. Existing single-variant invocation `--preset=<id> --frame=<n>` (without `--variant`) produces the same `manifest.json` shape as T-313 (no `variants` field; `referenceFrames: [<n>]`) and `golden-frame-<n>.png` (no `-<variant>` suffix).

## Test plan

T-359a ships vitest tests covering:

1. **`parseArgs` — `--variant` parsing**: repeated invocations, comma-separated, validation of variant-name regex, dedup of duplicate variant names.
2. **`buildManifest` — multi-variant shape**: with `variants: ['a', 'b']`, manifest has `variants: { a: { frames: [n] }, b: { frames: [n] } }` and the expected metadata.
3. **`buildManifest` — single-variant backward compat**: with no variants, manifest matches T-313's shape exactly (no `variants` field; `referenceFrames: [n]`).
4. **`runGenerate` — multi-variant render loop**: stub renderer invoked once per variant; golden filenames include the `-<variant>` suffix; partial failure aborts before frontmatter mutation.
5. **`runGenerate` — single-variant backward compat**: existing T-313 tests pass unchanged (no `--variant` → same behavior).
6. **`rewriteSignOff` — atomic per-manifest sign-off**: all variants required for `--mark-signed` to mutate; one missing → no mutation.
7. **`bindProductionRenderer` — swap + invoke**: stub component injected, render produces a non-empty Uint8Array, manifest references the variant.
8. **`productionRenderer` — unknown clipKind**: throws `RenderUnavailableError` with the documented message.
9. **`parity-cli/generate-fixture` — round-trip**: bind a stub renderer, invoke the bin against a synthetic preset fixture, verify exit 0 + expected files written.

Test environment does NOT require real Chrome + ffmpeg. The `parity-cli/generate-fixture.test.ts` test uses a stub component; the puppeteer-backed renderer is exercised separately in the existing `prime.test.ts` infrastructure.

## Escalation triggers (CLAUDE.md §6)

- `packages/parity-cli` does not have a clean injection seam for `bindProductionRenderer` and the binding requires a refactor of more than ~50 LOC there. Escalate; the alternative (option (a) — wire directly, lose standalone invocation) is a Reviewer-visible architectural shift.
- The puppeteer/CDP renderer in `parity-cli/prime.ts` cannot accept an arbitrary React component for a single-frame render (it may be tightly coupled to the existing fixture-prime workflow). Escalate; this would require a renderer abstraction not in scope for T-359a.
- The `clipKindResolver` map approach (D-T359a-4) requires more than just `bigNumber` to be useful in v1 — Reviewer demands the chart family (T-406 — `bar`/`line`/`area`/`pie`/`donut`/`scatter`/`combo`) all be wired up. Escalate; this is a meaningful scope expansion (each clipKind needs a component binding + render contract verified).
- The `cnn-classic` empty fixture dir turns out to encode a contract (e.g., a manifest schema shipped but never populated) the new multi-variant shape conflicts with. Escalate; the spec assumes the dir is genuinely empty / pre-pipeline.
- `--mark-signed` partial-failure semantics (D-T359a-2 — all variants required) conflict with operator workflow (e.g., reviewer signs variant-by-variant over multiple sessions). Escalate; per-variant sign-off requires schema changes outside T-359a's scope.
- The standalone script invocation (without `parity-cli`) needs to keep working for `--mark-signed` use cases that don't render (e.g., sign-off after manual fixture inspection). Escalate; the spec assumes `--mark-signed` always implies a fresh render.
- The Implementer's render budget (D-T359a-3 + D-T359a-4) approaches L-task ceiling and T-359 re-promotion (D-T359a-6) becomes infeasible in the same PR. Descope T-359 re-promotion per the D-T359a-6 fallback; flag the trade-off in the PR description and open a follow-up ticket.
