# T-270 — `@stageflip/storage-postgres` (Supabase)

**Branch**: `task/T-270-storage-postgres-supabase`

## Goal

Implement the storage contract (T-025) on a SECOND, categorically-different backend to prove the abstraction holds. Firebase/Firestore is the primary (T-036, document store, eventual consistency, onSnapshot push). Postgres is the alternate: relational, ACID transactions, LISTEN/NOTIFY for real-time. Target deployment: Supabase (managed PG) for dev-grade; the same adapter works against self-hosted Cloud SQL or Neon for prod.

T-270 ships:

1. **`@stageflip/storage-postgres`** package implementing `StorageAdapter` (snapshot + delta + patch tiers).
2. **Schema migrations** (SQL DDL) for the four tables: `documents`, `snapshots`, `updates`, `changesets`.
3. **LISTEN/NOTIFY plumbing** for `subscribeUpdates` to deliver real-time deltas.
4. **Vitest fixtures** running against a Postgres test container OR pg-mem (in-memory PG) — Implementer's call.
5. **Supabase deployment doc** — `docs/ops/supabase-setup.md` with project provisioning, connection string format, RLS posture.

T-270 does NOT migrate any production data; it stands up the adapter and pins it via tests.

## Architectural decisions

### D-T270-1. Schema

Four tables (DDL in `migrations/0001_init.sql`):

```sql
-- documents: org_id is FK-conceptually; actual FK to orgs(id) only if we move orgs to PG (we don't yet).
CREATE TABLE documents (
  id           TEXT PRIMARY KEY,
  org_id       TEXT NOT NULL,
  current_version  BIGINT NOT NULL DEFAULT 0,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX documents_org_idx ON documents(org_id);

-- snapshots: append-only, never updated. The latest is fetched via documents.current_version.
CREATE TABLE snapshots (
  doc_id       TEXT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  version      BIGINT NOT NULL,
  content      JSONB NOT NULL,
  updated_at   TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (doc_id, version)
);

-- updates: binary CRDT deltas.
CREATE TABLE updates (
  id           BIGSERIAL PRIMARY KEY,
  doc_id       TEXT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  payload      BYTEA NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX updates_doc_idx ON updates(doc_id, id);

-- changesets: JSON-Patch ops with version control.
CREATE TABLE changesets (
  id              TEXT PRIMARY KEY,
  doc_id          TEXT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  parent_version  BIGINT NOT NULL,
  ops             JSONB NOT NULL,
  actor           TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL
);
CREATE INDEX changesets_doc_idx ON changesets(doc_id, created_at);
```

`StorageVersionMismatchError` semantics from the contract are enforced in a transaction: `applyPatch` reads `documents.current_version`, asserts equality with `parent_version`, INSERTs the changeset, increments `current_version`. A FOR UPDATE row lock prevents lost-update.

### D-T270-2. LISTEN/NOTIFY for `subscribeUpdates`

When `applyUpdate` writes to `updates`, it also fires `NOTIFY updates_<doc_id> '<update_id>'` (channel name namespaced per doc). `subscribeUpdates` opens a dedicated connection, runs `LISTEN updates_<doc_id>`, and on each notification reads the update by id.

**Why per-doc channel name**: simpler subscribe filter than parsing a payload discriminator on a global channel; pg's channel name limit (63 chars) accommodates `updates_<orgId>_<docId>` if needed.

**Why NOTIFY payload = update id, not the update itself**: pg's NOTIFY payload is limited to 8000 bytes; Yjs updates can exceed this. Pass the id, fetch the row.

**Connection topology**: `subscribeUpdates` checks out a dedicated long-lived connection from a separate pool (the LISTEN connection cannot service other queries). On `AbortSignal.abort`, the connection is returned to the pool.

### D-T270-3. Test infrastructure

Two paths, both supported:

- **`pg-mem`** (in-memory PG): unit tests run against this. Fast, no Docker, runs in CI without setup. Caveat: pg-mem's LISTEN/NOTIFY is mock-quality, not byte-exact pg semantics.
- **Postgres test container** (e.g., via `testcontainers`): integration tests + the LISTEN/NOTIFY-specific tests run against this. Slower (~2s startup); guarded by `STAGEFLIP_TEST_PG_INTEGRATION=1` env so default `pnpm test` is fast.

CI runs both paths; locally, default is unit-only.

### D-T270-4. Connection management

`StorageAdapter` constructor accepts a `Pool` (from `pg`). For LISTEN, a separate dedicated connection is obtained via `pool.connect()` and held until disposal.

The adapter does NOT own the pool — consumers create the pool, pass it in, and dispose. Mirrors the auth-middleware DI pattern (T-262 / T-263 precedent).

### D-T270-5. Determinism posture

`packages/storage-postgres/**` is NOT clip / runtime code. Standard `pnpm check-determinism` rules don't apply. PG adapter uses `Date.now()`, network I/O, transactions — all expected.

## Acceptance criteria

### Snapshot tier

1. `getSnapshot(docId)` returns the latest snapshot or `null` for unknown docId.
2. `putSnapshot(docId, snapshot)` inserts the snapshot, updates `documents.current_version` and `documents.updated_at`. Pin via test.
3. **Concurrent putSnapshot** with the same `version` for a docId fails the second call (PRIMARY KEY violation surfaces as a clear error).
4. The schema-typed `content` JSONB round-trips byte-identical (Postgres JSONB normalizes whitespace; verify the round-trip is acceptable or use TEXT).

### Update tier

5. `applyUpdate(docId, bytes)` inserts a row in `updates` and fires `NOTIFY` on the per-doc channel. Pin via test.
6. `subscribeUpdates(docId, { signal })` opens a LISTEN connection, yields incoming updates as they arrive. Pin via test (testcontainers integration only).
7. **AbortSignal cleanup**: aborting the signal closes the LISTEN connection and returns it to the pool. No connection leaks.
8. **Multi-subscriber**: two parallel `subscribeUpdates(docId)` calls both receive every update (each opens its own LISTEN connection).
9. **Per-doc isolation**: `applyUpdate('doc1', ...)` does NOT show up in `subscribeUpdates('doc2')`.

### Patch tier

10. `applyPatch(docId, changeset)` runs in a transaction: reads `documents.current_version` with FOR UPDATE, asserts `parentVersion === current_version`, INSERTs the changeset, increments `current_version`. Pin via test.
11. **Version mismatch**: applying a changeset with a stale `parentVersion` throws `StorageVersionMismatchError` from the contract. Pin via test.
12. `getHistory(docId, opts?)` returns changesets ordered by `created_at`. Respects `limit` and `after` options.

### Migration runner

13. `runMigrations(pool)` applies all `.sql` files in `migrations/` in lexicographic order. Pin via test that runs against a fresh pg-mem DB.
14. **Idempotent**: running migrations twice doesn't error (uses a `__migrations` table for tracking applied ones).
15. Each migration runs in a single transaction; on failure, rolls back cleanly.

### Error mapping

16. PG-specific errors (constraint violations, deadlocks, connection drops) are mapped to clear application errors. `unique_violation` on snapshots becomes `StorageVersionMismatchError`-shaped; `connection_failure` propagates as-is.

### Backward compatibility

17. The full workspace test suite passes unchanged. `@stageflip/storage` and `@stageflip/storage-firebase` are not touched.
18. **Contract conformance**: the new adapter passes the SAME contract test suite as `InMemoryStorageAdapter` and `FirebaseStorageAdapter`. If the contract test suite is shared across adapters, run it against PostgresStorageAdapter; if not, document why (it may need to be extracted as a shared fixture in this PR — that's acceptable scope).

### Coverage + gates

19. Coverage on `packages/storage-postgres/src/adapter.ts` ≥85% (LISTEN paths are integration-only; ≥85% acceptable).
20. Coverage on `packages/storage-postgres/src/migration-runner.ts` ≥90%.

- A real Yjs binary update exceeds Postgres BYTEA practical size limits (>1 GB pathological). Spec assumes < 1 MB updates; flag if a fixture surfaces a real >1 MB update.
