## Sub-PR 1 — `preserveOrder: true` workspace refactor

### Why

`fast-xml-parser` with `preserveOrder: false` (current default) collapses repeated child tags into arrays grouped by tag name and drops the interleaving order between different tag names. For an `<a:pathLst>` with `<a:moveTo>` → `<a:arcTo>` → `<a:lnTo>` → `<a:close>`, the parsed object today looks like:

```js
{ "a:moveTo": [{...}], "a:arcTo": [{...}], "a:lnTo": [{...}], "a:close": [{}] }
```

— losing the document-order needed to translate `<a:arcTo>` (which depends on the current pen position from prior commands).

With `preserveOrder: true`, the parser emits an **ordered array** of single-key objects:

```js
[
  { "a:moveTo": [...children], ":@": {...attrs} },
  { "a:arcTo":  [...children], ":@": {...attrs} },
  { "a:lnTo":   [...children], ":@": {...attrs} },
  { "a:close":  [],            ":@": {} },
]
```

— preserving order; attributes move to a synthetic `:@` key per node.

### What changes

The single line at `packages/import-pptx/src/opc.ts:19` flips from `false` to `true`. **Every consumer of `parseXml` must adapt to the new node shape.** Affected modules (per `grep -rn "@_\|tag\[" packages/import-pptx/src --include="*.ts"`, expanded by the Implementer before starting):

### Recommended helper (Implementer's call)

The new node shape is more verbose. A small internal helper in `opc.ts` keeps callsites readable:

```ts
/** Get the first child node matching `tagName`, or undefined. */
export function firstChild(node: OrderedXmlNode, tagName: string): OrderedXmlNode | undefined;
/** Get all children matching `tagName`, in document order. */
export function children(node: OrderedXmlNode, tagName: string): OrderedXmlNode[];
/** Get the attribute map (`:@`) for a node. */
export function attrs(node: OrderedXmlNode): Record<string, string>;
/** Get an attribute value by name, or undefined. */
export function attr(node: OrderedXmlNode, name: string): string | undefined;
```

Implementer is encouraged to introduce these (or equivalents) — verbosity creep is otherwise high.

## Sub-PR 2 — `<a:arcTo>` support + chord/pie/donut presets

### `<a:arcTo>` translation math (ECMA-376 §20.1.9.3)

OOXML `<a:arcTo wR="rx" hR="ry" stAng="α" swAng="β">`:

- `wR` / `hR` are the ellipse radii (in EMU; same coordinate system as `<a:pathLst w h>`).
- `stAng` is the start angle in **60000ths of a degree** (so divide by 60000 to get degrees, then convert to radians).
- `swAng` is the sweep angle, same units. Positive = clockwise; negative = counter-clockwise (per OOXML convention; SVG is the opposite — see flag mapping below).

The arc starts at the **current pen position** (the previous command's end point). To translate to SVG:

1. **Convert angles**: `stRad = stAng * π / (180 * 60000)`; `swRad = swAng * π / (180 * 60000)`.
2. **Compute the implicit ellipse center** from the current pen position `(px, py)`:
   - `cx = px - wR * cos(stRad)`
   - `cy = py - hR * sin(stRad)`
3. **Compute the arc end point**:
   - `endX = cx + wR * cos(stRad + swRad)`
   - `endY = cy + hR * sin(stRad + swRad)`
4. **Map to SVG `A rx ry 0 large-arc sweep endX endY`**:
   - `rx = wR`, `ry = hR`, x-axis-rotation = `0`.
   - `large-arc-flag = abs(swAng) > 180 * 60000 ? 1 : 0`.
   - `sweep-flag = swAng >= 0 ? 1 : 0` (SVG `1` = positive-angle direction = clockwise in SVG's y-down coordinate system, matching OOXML's positive `swAng`).
5. **Update pen position** to `(endX, endY)` for subsequent commands.

This requires the cust-geom walker to maintain pen-position state — which the existing walker already does for `<a:moveTo>` / `<a:lnTo>` / `<a:cubicBezTo>` / `<a:quadBezTo>` / `<a:close>` (verify in `cust-geom/parse.ts` before starting; if not, add it).

### chord / pie / donut presets

All three are circle-with-cuts shapes from ECMA-376 §20.1.9 reference geometry. Each ships in `packages/import-pptx/src/geometries/presets/basics.ts`:

| Preset | Construction |
|---|---|
| `pie` | `M cx cy L startX startY A rx ry 0 large sweep endX endY Z`. Default `<a:gd>` start angle = `0`, sweep = `270000` (270°). Use spec defaults; honor `<a:avLst>` `adj1` (start angle) / `adj2` (sweep angle) if present. |
| `chord` | `M startX startY A rx ry 0 large sweep endX endY Z` — closing line is the chord. Same default angles as `pie`. |
| `donut` | `M w 0 A rx ry 0 1 1 0 cy A rx ry 0 1 1 w cy Z` for the outer ring, then `M (w-t) cy A (rx-t) (ry-t) 0 1 0 t cy A (rx-t) (ry-t) 0 1 0 (w-t) cy Z` for the inner cutout (CCW for fill-rule). Default `<a:gd>` `adj1` (ring thickness) = `25000` (25% of width). |

The `noSmoking` preset shipped in T-242c batch 2 already uses the donut idiom (outer + inner ring with cubic-Bézier ellipse approximation); after T-242d the Implementer **may** opportunistically replace `noSmoking`'s cubic-Béziers with real arcs for consistency, but this is **non-blocking** (it's listed as a follow-up like cloud/cloudCallout).
