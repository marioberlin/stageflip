## Sub-PR 1 ‚Äî `preserveOrder: true` workspace refactor

### Why

`fast-xml-parser` with `preserveOrder: false` (current default) collapses repeated child tags into arrays grouped by tag name and drops the interleaving order between different tag names. For an `<a:pathLst>` with `<a:moveTo>` ‚Üí `<a:arcTo>` ‚Üí `<a:lnTo>` ‚Üí `<a:close>`, the parsed object today looks like:

```js
{ "a:moveTo": [{...}], "a:arcTo": [{...}], "a:lnTo": [{...}], "a:close": [{}] }
```

‚Äî losing the document-order needed to translate `<a:arcTo>` (which depends on the current pen position from prior commands).

With `preserveOrder: true`, the parser emits an **ordered array** of single-key objects:

```js
[
  { "a:moveTo": [...children], ":@": {...attrs} },
  { "a:arcTo":  [...children], ":@": {...attrs} },
  { "a:lnTo":   [...children], ":@": {...attrs} },
  { "a:close":  [],            ":@": {} },
]
```

‚Äî preserving order; attributes move to a synthetic `:@` key per node.

### Required helpers

The new node shape is more verbose. A small set of internal helpers in `opc.ts` keeps callsites readable. **These are required, not optional** ‚Äî adding them but leaving callsites with raw `node[':@']` / array indexing produces a half-finished diff. Add the helpers AND adopt them at every callsite in the same Sub-PR.

```ts
/** Get the first child node matching `tagName`, or undefined. */

/** Get all children matching `tagName`, in document order. */

/** Get an attribute value by name, or undefined. */
```

## Sub-PR 2 ‚Äî `<a:arcTo>` support + chord/pie/donut presets

### `<a:arcTo>` translation math (ECMA-376 ¬ß20.1.9.3)

OOXML `<a:arcTo wR="rx" hR="ry" stAng="Œ±" swAng="Œ≤">`:

- `wR` / `hR` are the ellipse radii (in EMU; same coordinate system as `<a:pathLst w h>`).
- `stAng` is the start angle in **60000ths of a degree** (so divide by 60000 to get degrees, then convert to radians).
- `swAng` is the sweep angle, same units. Positive = clockwise; negative = counter-clockwise (per OOXML convention; SVG is the opposite ‚Äî see flag mapping below).

The arc starts at the **current pen position** (the previous command's end point). To translate to SVG:

1. **Convert angles**: `stRad = stAng * œÄ / (180 * 60000)`; `swRad = swAng * œÄ / (180 * 60000)`.
2. **Compute the implicit ellipse center** from the current pen position `(px, py)`:
   - `cx = px - wR * cos(stRad)`
   - `cy = py - hR * sin(stRad)`
3. **Compute the arc end point**:
   - `endX = cx + wR * cos(stRad + swRad)`
   - `endY = cy + hR * sin(stRad + swRad)`
4. **Map to SVG `A rx ry 0 large-arc sweep endX endY`**:
   - `rx = wR`, `ry = hR`, x-axis-rotation = `0`.
   - `large-arc-flag = abs(swAng) > 180 * 60000 ? 1 : 0`.
   - `sweep-flag = swAng >= 0 ? 1 : 0` (SVG `1` = positive-angle direction = clockwise in SVG's y-down coordinate system, matching OOXML's positive `swAng`).
5. **Update pen position** to `(endX, endY)` for subsequent commands.

This requires the cust-geom walker to maintain pen-position state ‚Äî which the existing walker already does for `<a:moveTo>` / `<a:lnTo>` / `<a:cubicBezTo>` / `<a:quadBezTo>` / `<a:close>` (verify in `cust-geom/parse.ts` before starting; if not, add it).

### chord / pie / donut presets

All three are circle-with-cuts shapes from ECMA-376 ¬ß20.1.9 reference geometry. Each ships in `packages/import-pptx/src/geometries/presets/basics.ts`:

**Unit convention for adjustment values**: OOXML angle adjustments are in **60000ths of a degree** (so 90¬∞ = `5400000`, 180¬∞ = `10800000`, 270¬∞ = `16200000`, 360¬∞ = `21600000`). EMU-based percentage adjustments (e.g., `donut.adj1` for ring thickness) are in **thousandths of one** (so 25% = `25000`, 50% = `50000`). Spec values below are quoted in raw OOXML units with parenthetical degree/percent comments to keep the convention visible.

| Preset | Construction |
|---|---|
| `pie` | `M cx cy L startX startY A rx ry 0 large sweep endX endY Z`. Default `<a:gd>`: start angle `adj1 = 0` (0¬∞), sweep `adj2 = 16200000` (270¬∞). Honor `<a:avLst>` `adj1` (start angle) / `adj2` (sweep angle) if present. |
| `chord` | `M startX startY A rx ry 0 large sweep endX endY Z` ‚Äî closing line is the chord. Same default `<a:gd>` values as `pie`. |
| `donut` | **Outer ring (CW)**: `M w cy A rx ry 0 1 1 0 cy A rx ry 0 1 1 w cy Z`. **Inner cutout (CCW)**: `M t cy A (rx-t) (ry-t) 0 1 0 (w-t) cy A (rx-t) (ry-t) 0 1 0 t cy Z` where `t = w * adj1/100000`. Default `<a:gd>` `adj1 = 25000` (25% of width). |

**Donut fill-rule**: the path above is authored for SVG's default `fill-rule="nonzero"`. The opposite winding directions of the two subpaths (outer CW + inner CCW) cancel inside the donut hole, leaving it transparent. **No `fill-rule="evenodd"` attribute should be emitted** ‚Äî verify renderer-core's path dispatcher doesn't override the default.

The `noSmoking` preset shipped in T-242c batch 2 already uses the donut idiom (outer + inner ring with cubic-B√©zier ellipse approximation); after T-242d the Implementer **may** opportunistically replace `noSmoking`'s cubic-B√©ziers with real arcs for consistency, but this is **non-blocking** (it's listed as a follow-up like cloud/cloudCallout).
