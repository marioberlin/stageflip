# T-386 — Fast variant-generation mode (size × message × locale matrix over RIR)

**Branch**: `task/T-386-variant-generation`

## Goal

Ship the variant-generation primitive that turns one canonical Document into a matrix of variants — the differentiator for ad-tech / marketing / display-mode workflows where teams ship hundreds of permutations from a single source. Per the plan task description ("size × message × locale matrix over RIR") and ADR-005 §D6 (variant generation operates over RIR trees that may include interactive clips; export matrix routes per ADR-003 §D4).

**Scope reduction for v1**: T-386 ships **message + locale axes only**. Size axis is deferred — it requires constraint-based layout adaptation across aspect ratios that is a meaty design problem in its own right. Documented as out-of-scope with a follow-up task ID (T-386a).

T-386 unblocks the export-matrix routing layer (T-408) by giving it concrete variant outputs to route, and the agent's `arrange_variants` semantic tool (new tool category).

## Architectural decisions

### D-T386-1. New package `@stageflip/variant-gen`

Variant generation is a transformation pipeline that operates on `CanonicalDocument` and produces N transformed Documents per the matrix spec. It belongs as a separate package (not inside `@stageflip/engine`) because:

1. The engine's handler bundles do *semantic* edits (compose-this, validate-that). Variant-gen does *mechanical permutation*. Different concern.
2. Editor surfaces and headless agent surfaces both need variant-gen; living in engine couples it to the agent's tool-handler lifecycle unnecessarily.
3. Sizing precedent: `@stageflip/asset-cache`, `@stageflip/render-farm`, `@stageflip/observability` are all separate packages of similar scope.

### D-T386-2. Input contract

```ts
// @stageflip/variant-gen
export interface VariantMatrixSpec {
  /** Variants of message slots (text content). */
  messages?: ReadonlyArray<MessageVariantAxisEntry>;
  /** Locales to render the source + each message variant in. */
  locales?: ReadonlyArray<LocaleAxisEntry>;
  /** Strict cap on output count; prevents combinatorial blow-ups. Default 100. */
  maxVariants?: number;
}

export interface MessageVariantAxisEntry {
  /** Stable axis-coordinate id, used in output filenames + RIR cache keys. */
  id: string;
  /** Map of named-slot → text override. Slot names match the Document's variantSlots. */
  slots: Record<string, string>;
}

export interface LocaleAxisEntry {
  /** BCP-47 tag — e.g., 'en-US', 'de-DE', 'ja-JP'. */
  tag: string;
}

export interface VariantOutput {
  /** Resolved coordinate. */
  coordinate: { messageId?: string; locale?: string };
  /** Transformed Document — never references the source after construction. */
  document: CanonicalDocument;
  /** Stable cache key — sha256(sourceDocId + coordinate) — used by export pipeline. */
  cacheKey: string;
}

export function generateVariants(
  source: CanonicalDocument,
  spec: VariantMatrixSpec,
  options?: GenerateVariantsOptions,
): Iterable<VariantOutput>;
```

The function returns an iterable (synchronously) so callers can `for...of` and stop early. No streaming / async-iteration in v1; the matrix is bounded and the per-variant work is structural-sharing-cheap.

### D-T386-3. Schema additions — `variantSlots` on Document + `LocaleProvider` interface

Document gains an optional `variantSlots: Record<string, VariantSlotDef>` map at the root. Each `VariantSlotDef` references an Element ID and the text-property path (`'value' | 'placeholder' | ...`) where the slot's content lives. Authors mark text content as variant-slot-bound at authoring time; variant-gen substitutes at axis-resolution time.

```ts
// @stageflip/schema/variants
export const variantSlotDefSchema = z.object({
  elementId: z.string(),
  /** Property path inside the element. */
  path: z.string(),
}).strict();

export const variantSlotsSchema = z.record(variantSlotDefSchema);
```

Browser-safe (pure Zod). Per `feedback_t304_lessons.md`: any Node-only loader for variant-gen lives behind a subpath export.

`LocaleProvider` is a tiny interface in variant-gen:

```ts
export interface LocaleProvider {
  /** Translate a key+source-text into the requested tag. Returns the original on miss. */
  translate(args: { tag: string; key: string; source: string }): string;
}
```

Two implementations ship with T-386:

1. `InMemoryLocaleProvider({ catalogue })` — for tests; takes a pre-built catalogue.
2. `StaticBundleLocaleProvider({ bundle })` — reads a serialised JSON bundle (the same shape an app's `i18n/catalog.ts` produces). Browser-safe.

A network-fetching provider (Google Translate / DeepL) is a future task on the asset-gen / provider-seam pattern (T-415).

### D-T386-4. Determinism — subject to the broad §3 rule, not the shader sub-rule

Variant-gen operates outside `clips/{shader,three-scene}/**`. The broad §3 determinism rule applies: no `Date.now()`, no `Math.random()`, no rAF. Pure transformation over inputs. Run `pnpm check-determinism` to confirm.

The `cacheKey` is content-addressed (`sha256(sourceDocId + coordinate)`) so two runs with identical inputs produce identical keys. This is the foundation export-matrix routing (T-408) needs to dedupe across variant ↔ size combinations.

### D-T386-5. Performance — structural sharing across variants

L-sized scope warrants attention to performance. Most variants share most of the Document; only the varied text slots differ. T-386 implements **immutable-tree structural sharing**: a variant's Document tree shares Element nodes by reference with the source where unchanged, and constructs new objects only along the path from root to substituted slot.

This is straightforward with the existing immutable Document shape (no mutations are allowed; all transformations construct new objects). The implementation should NOT deep-clone the source per variant.

Pin via test: 100-element source with 1 substituted slot produces a variant whose `variant.elements[i] === source.elements[i]` for every unchanged index (reference equality).

### D-T386-6. `arrange_variants` agent tool — new bundle

Adds a new handler bundle at `packages/engine/src/handlers/arrange-variants/`. One tool:

```
arrange_variants({ matrixSpec, maxVariants? })
  → { variants: VariantOutput[] }
```

The tool calls `generateVariants` from `@stageflip/variant-gen`. Returns coordinates + Document IDs (the engine persists each variant Document to storage and returns IDs; full Documents inline would blow the agent context).

Tool-bundle registration follows the existing pattern from `qc-export-bulk` / `clip-animation` / etc.

Skill: new `skills/stageflip/tools/arrange-variants/SKILL.md` per CLAUDE.md §10.

### D-T386-7. `maxVariants` cap — prevents combinatorial blow-ups

3 messages × 5 locales = 15 variants. 10 messages × 20 locales = 200. Fine. But authors may casually add another axis later; a hard cap protects against accidentally renderingt thousands of variants and saturating storage / queue / billing.

Default cap: 100. Configurable via `VariantMatrixSpec.maxVariants`. Exceeded cap throws `VariantMatrixCapExceededError` synchronously (no partial output).

### D-T386-8. No size axis — explicit out-of-scope marker

T-386 ships a `size?: never` field on `VariantMatrixSpec`'s TypeScript type (with a comment explaining the deferral). Authors trying to add `size:` get a TypeScript error pointing them at T-386a's tracking row in the implementation plan. The Zod schema does NOT include the `size` field; passing it at runtime is silently ignored (Zod `strict()` mode would reject the spec — that's acceptable; rejection is more honest than silent ignore).

### D-T386-9. Export-matrix routing integration — opaque to T-386

T-408 (export-matrix routing) consumes `Iterable<VariantOutput>`. T-386 is its sole producer. The contract is the iterator + `cacheKey`; T-386 does not know how its output is rendered. This is correct decoupling — variant-gen produces transformed Documents; the export pipeline routes per ADR-003 §D4.

## Acceptance criteria

### Schema

1. `variantMatrixSpecSchema` parses a valid spec (`messages: [...]`, `locales: [...]`, optional `maxVariants`).
2. Empty spec (no axes, no maxVariants) is valid; produces zero variants on generate.
3. `maxVariants` of 0 → schema error.
4. `messages[].id` collisions → schema error (id uniqueness across the array).
5. `locales[].tag` not a valid BCP-47 → schema error (light regex check, not full ICU validation).
6. `variantSlots` schema validates `{ [name]: { elementId, path } }`; missing `elementId` → error.
7. Document with no `variantSlots` round-trips unchanged through schema parse (backward compat).

### `generateVariants`

8. Empty spec produces zero variants (an empty iterable).
9. 3 messages × 0 locales = 3 variants, one per message.
10. 0 messages × 5 locales = 5 variants, one per locale (source text translated).
11. 3 messages × 5 locales = 15 variants in row-major order: (m1,l1), (m1,l2), …, (m3,l5).
12. Each variant's `coordinate` matches its position in the matrix.
13. Each variant's `document` has the substituted text at the slot referenced by `variantSlots[name]`.
14. `maxVariants: 5` against a 4-output matrix returns all 4. Against an 8-output matrix throws `VariantMatrixCapExceededError`.

### Locale provider

15. `InMemoryLocaleProvider` returns the catalogue value when present.
16. `InMemoryLocaleProvider` returns the source text when the key is missing.
17. `StaticBundleLocaleProvider` reads a JSON bundle of the shape `{ tag: { key: translated } }`.

### Determinism + cache key

18. `generateVariants(source, spec)` called twice with identical args produces identical outputs (deep equality on each variant + cacheKey).
19. `cacheKey = sha256(sourceDocId + coordinate)`; variants from different coordinates have different keys; identical coordinates have identical keys.
20. `pnpm check-determinism` PASSES on `packages/variant-gen/**`.

### Structural sharing

21. A variant whose only difference from the source is one substituted text slot satisfies: every Element in the variant whose ID does not match the substituted slot's `elementId` is reference-equal to the source's corresponding Element. Pin via test on a 100-element source.

### Agent tool

22. `arrange_variants({ matrixSpec })` returns `{ variants: VariantOutput[] }` (Document is persisted to storage; the response carries Document IDs, not full payloads — verify via response-shape pin).
23. Empty matrixSpec via the tool returns an empty array.
24. Cap exceeded throws a typed engine-handler error mapped to the agent error contract (existing convention).

### Skill + drift

25. `skills/stageflip/tools/arrange-variants/SKILL.md` exists with the standard tool-skill structure (Purpose / Parameters / Output / Examples).
26. `skills/stageflip/concepts/variants/SKILL.md` exists with the variant-gen primitive concept overview.
27. `pnpm check-skill-drift` PASSES.

## Escalation triggers (CLAUDE.md §6)

- The `variantSlots` map's `path` field needs to address nested element properties (e.g., `liveMount.props.fragmentShader`); the simple dotted-path representation is insufficient. Escalate; we may need a JSON-pointer style reference.
- Structural-sharing tests reveal that the existing `CanonicalDocument` has hidden mutation paths (e.g., a clone-on-read pattern in the schema parse). Escalate; the immutability contract may need hardening before T-386 can guarantee ref-equality.
- The `arrange_variants` tool's return shape (Document IDs vs. full payloads) is wrong for the agent's downstream consumers. Escalate; the wire format may need iteration.
- BCP-47 validation regex over-matches or under-matches and rejects valid catalogue tags. Escalate; consider switching to `Intl.Locale` parsing.
