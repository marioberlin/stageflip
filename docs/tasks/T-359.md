# T-359 — Preset: `f1-sector-purple-green` (bigNumber, Cluster E)

**Branch**: `task/T-359-f1-sector-purple-green`

## Goal

Promote the existing `f1-sector-purple-green` preset stub from `status: stub` to `status: substantive` and ship it as the **first Cluster E preset to land**. The preset captures the universal Formula 1 broadcast convention for a single sector callout: a driver's most recent sector time, rendered as a large central number, where the **color of the number is the message** — purple if it is the session-best, green if it is the driver's personal-best, otherwise neutral / yellow.

T-359 is the bigNumber-only half of the Cluster E rollout. The data path is hard-coded (the preset takes a `state` prop ∈ `{ sessionBest, personalBest, neutral }`); the live-data binding to a `LiveDataClip` source is deferred to T-355 / T-356 / T-357. The `bigNumber` runtime primitive this preset binds to already exists in `frame-runtime-bridge` as `animated-value` (count-up text); T-359 introduces no new runtime clip.

T-359 establishes the **preset-PR workflow on real preset content** for the project: it is the first preset spec to pass the full integrity / parity / font-license / skill-drift gate stack end-to-end on a substantive (non-stub) document.

## Architectural decisions

### D-T359-1. On-disk location — `skills/stageflip/presets/data/f1-sector-purple-green.md`

The preset cluster directory is `data` (cluster E maps to `data` per `scripts/check-cluster-eligibility.ts` `LETTER_TO_NAME`). The file already exists at this path as a stub. T-359 **modifies** that file in place — it does NOT create a new file. The `id` slug stays `f1-sector-purple-green`; the cluster stays `data`.

This is intentional and matches the on-disk pattern set by T-304 (every cluster has a directory of preset `*.md` files alongside one `SKILL.md`).

### D-T359-2. Frontmatter shape — promote `status: stub` → `status: substantive`; rest unchanged

The existing stub frontmatter is already license-clean and schema-valid (it loads via `loadPreset` today). T-359's frontmatter delta is minimal:

```yaml
# CHANGES (red→green):
- status: stub
+ status: substantive
- signOff:
-   parityFixture: pending-user-review
-   typeDesign: na
+ signOff:
+   parityFixture: signed:YYYY-MM-DD   # auto-stamped by `generate-preset-parity-fixture --mark-signed`
+   typeDesign: na
```

All other fields (`id`, `cluster`, `clipKind: bigNumber`, `source`, `preferredFont`, `fallbackFont`, `permissions: []`) stay as the stub authored them. The `permissions: []` value is correct: the preset has no live-data network call, no mic, no camera. (The future T-355 LiveData wrap will declare `network`, but T-359 does not.)

### D-T359-3. Body content — substantive expansion of all six canonical H2 sections

The stub already populates every canonical H2 section (`Visual tokens`, `Typography`, `Animation`, `Rules`, `Acceptance (parity)`, `References`). T-359 expands each into substantive content:

- **`## Visual tokens`** — pin the three sector-time hex colors (`#6F2E9E` purple session-best, `#00B54A` green personal-best, `#F0C800` yellow neutral / slower-than-PB). Pin layout: number centered, driver code top-left, sector label (`S1` / `S2` / `S3`) top-right, all on a transparent or dark gradient background. Pin number size as a fraction of the canvas height (e.g., `min(canvas.height * 0.55, canvas.width * 0.40)`).
- **`## Typography`** — pin `preferredFont: Formula1 Display` weight 700 for the number, weight 600 for the driver code + sector label. Pin tabular numerals (mandatory per cluster convention). Pin the OFL fallback (`Barlow Condensed` weight 600) renders on every export medium where Formula1 Display is not BYO-cleared.
- **`## Animation`** — pin: count-up animation for the number using `AnimatedValue`'s default spring (damping 15, mass 0.8, stiffness 120) over the first ~24 frames @ 30fps; **no** state-transition animation in v1 (state is a prop, not a stream). Pin: fade-in for the driver code + sector label over the same ~24 frames.
- **`## Rules`** — pin: state palette (`sessionBest` → purple, `personalBest` → green, `neutral` → yellow) is universal F1 shorthand; do NOT re-theme. Pin: time always truncated, never rounded (`21.412` not `21.41`). Pin: driver code is uppercase 3-letter (`VER`, `HAM`, `LEC`). Pin: the canonical primitive binding is `animated-value` from `frame-runtime-bridge` (the `bigNumber` clipKind dispatcher does not exist yet; this prose binding is the contract until the dispatcher lands).
- **`## Acceptance (parity)`** — pin: three reference-frame fixtures, one per state, at `frame: 60` (mid-hold steady-state per ADR-004 §D5). PSNR ≥ 42 dB, SSIM ≥ 0.98 (stricter than the script default `35 / 0.95` because text-heavy big-number content tolerates tighter thresholds; mirrors the `f1-timing-tower` pin).
- **`## References`** — pin links to `docs/compass_artifact.md#formula-1`, `f1-timing-tower` sister preset, ADR-004, ADR-005 (LiveData posture for the future binding).

### D-T359-4. Bound primitive — `animated-value` (NOT `counter`, NOT a new clip)

The `bigNumber` clipKind enum entry is purely an integrity-gate sentinel today (in `VALID_CLIP_KINDS`). The preset documents — in the `## Rules` section — that the canonical render binding is `animated-value` from `frame-runtime-bridge` (`packages/runtimes/frame-runtime-bridge/src/clips/animated-value.tsx`). Reasoning:

- `AnimatedValue` accepts numeric `value`, optional `prefix` / `suffix`, `decimals` (for the `21.412` truncation), `color` (for the purple / green / yellow state palette), `fontSize`, `fontWeight`. Every visual-token requirement maps to a prop.
- `counter.tsx` is a sister primitive with broader scope (formatters, label slots) — it would also work, but `animated-value` has the tighter API surface and is already the building block kpi-grid composes from. The Reviewer may challenge.
- A bespoke primitive would violate "three similar lines beat a premature abstraction" — there are zero other consumers asking for an F1-specific primitive.

### D-T359-5. Three parity fixtures, one per state

ADR-004 §D5 + T-313 specify a single canonical mid-hold reference frame per preset. T-359 deviates **one step**: the preset has three semantically distinct visual outputs (purple / green / yellow) keyed by the `state` prop, and a single fixture would not exercise the state palette that is the entire point of the preset.

T-359 ships **three fixtures under one preset directory**:

```
parity-fixtures/data/f1-sector-purple-green/
  manifest.json                   # array of three composition variants, one per state
  golden-frame-60-sessionBest.png
  golden-frame-60-personalBest.png
  golden-frame-60-neutral.png
  thresholds.json
```

The `--mark-signed` flag stamps `signOff.parityFixture: signed:<UTC date>` after all three render clean. If the parity-fixture generator does not yet support multi-variant manifests for a single preset, the Implementer escalates per CLAUDE.md §6 — the alternative (three separate preset files) would balloon the cluster surface area.

### D-T359-6. Determinism — preset content is data; runtime determinism is enforced by the bound primitive

The preset `*.md` is markdown + YAML — no runtime code, no clip-scope determinism rule applies to the preset file itself. The bound primitive (`AnimatedValue`) is in `packages/runtimes/frame-runtime-bridge/**` and is already covered by `pnpm check-determinism`. T-359 introduces zero new code paths inside `clips/**` or `frame-runtime/**`; no new determinism surface.

The preset's parity contract — same `(state, frame, fixture inputs)` produces the same rendered PNG — is enforced by the existing parity harness, not by a new test in this PR.

### D-T359-7. Font-license posture — `Formula1 Display` (proprietary-byo) + `Barlow Condensed` (ofl) fallback

The font-license registry (T-307) already accepts the `proprietary-byo` + OFL fallback pair as license-clean. The fallback `Barlow Condensed` weight 600 is OFL — every render path that does not have `Formula1 Display` BYO-cleared falls back to it transparently. The `pnpm check-licenses` and font-license-registry whitelist gates pass without modification.

If a Reviewer challenges the BYO posture and demands a fully OFL preferred font, the Implementer escalates per CLAUDE.md §6. Promoting `Barlow Condensed` to `preferredFont` would meaningfully change the visual register and is an Orchestrator decision.

### D-T359-8. Cluster SKILL.md — no modification

The cluster `SKILL.md` at `skills/stageflip/presets/data/SKILL.md` is `status: substantive` already, owned by T-361. T-359 does NOT modify it. The link to `f1-sector-purple-green.md` already exists in the cluster SKILL's preset list. Skill-drift gate (`pnpm check-skill-drift`) passes without changes there.
