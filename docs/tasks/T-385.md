# T-385 — Permission envelope (mic / network / camera UX + enforcement)

**Branch**: `task/T-385-permission-envelope`

## Goal

Build the user-facing permission flow on top of the `PermissionShim` primitive shipped by T-306. The shim already handles tenant-policy gating, browser-API permission probes, grant caching, and denial telemetry — what's missing is the **state machine + React surface** that turns a denial into an actionable UX (denial banner, retry path, browser-settings hint) and an optional pre-prompt explanation modal that runs before the bare-bones browser permission dialog.

T-385 unblocks the five γ-live clip families that declare non-empty permissions: T-387 `VoiceClip` (`mic`), T-389 `AiChatClip` (`network`), T-391 `LiveDataClip` (`network`), T-393 `WebEmbedClip` (`network`), T-395 `AiGenerativeClip` (`network`). Without T-385, these clips would mount against a permissive default with no recourse for users who deny — silently falling back to static.

## Architectural decisions

### D-T385-1. Behaviour lives in `runtimes-interactive`; visual surface is text-prop-driven React components

The state machine, retry logic, telemetry hooks, and browser-API integration belong in `@stageflip/runtimes-interactive` so every consuming surface (editor, display, on-device player) shares the same enforcement primitives. Default visual components (`<PermissionDenialBanner>`, `<PermissionPrePromptModal>`) ship in the same package, but they accept all user-facing text as required props — no English-string defaults — so apps own localisation per CLAUDE.md §10.

### D-T385-2. `usePermissionFlow(clip, options)` hook + `PermissionFlowState` discriminator

A React hook encapsulates the state machine:

```ts
type PermissionFlowState =
  | { kind: 'idle' }
  | { kind: 'pre-prompt'; permission: Permission }
  | { kind: 'requesting'; permission: Permission }
  | { kind: 'granted'; permissions: ReadonlyArray<Permission> }
  | { kind: 'denied'; reason: 'tenant-denied' | 'permission-denied'; deniedPermission?: Permission };

interface UsePermissionFlowOptions {
  shim: PermissionShim;
  /** Show pre-prompt explanation BEFORE the browser dialog. Default false. */
  prePrompt?: boolean;
  /** Telemetry hook; no-op by default. */
  emitTelemetry?: EmitTelemetry;
}

interface UsePermissionFlowReturn {
  state: PermissionFlowState;
  start(): void;     // enters requesting (or pre-prompt if enabled)
  confirmPrePrompt(): void;
  cancelPrePrompt(): void;
  retry(): void;     // clears the single denied-permission cache entry; re-enters requesting
}
```

States are exhaustive; the discriminator drives UX rendering. `retry()` clears only the failed permission's cache entry (extends `PermissionShim` with a per-key `clearCacheEntry(family, permission)` method — see D-T385-3) so a successful re-prompt does not need to re-walk the full envelope.

### D-T385-3. `PermissionShim` extension — per-key cache invalidation

Add to the existing shim:

```ts
clearCacheEntry(family: InteractiveClipFamily, permission: Permission): void {
  this.grantCache.delete(`${family}:${permission}`);
}
```

The shim's broader `clearCache()` test-seam is preserved unchanged. The new method is the production-callable retry hook.

### D-T385-4. Pre-prompt is opt-in per mount

Per-permission research consistently shows that browser permission dialogs alone (the bare "site wants your mic — Allow / Block") yield higher denial rates than dialogs preceded by an in-app explanation. T-385 ships a pre-prompt option that's **off by default** and turned on per mount via `MountContext.permissionPrePrompt: boolean | undefined`. The flag is wired up the call chain:

- `MountContext` gains optional `permissionPrePrompt?: boolean`.
- The runtime mount-harness reads this and forwards to `usePermissionFlow`.
- A clip-family-specific factory (e.g., `VoiceClip`) may *recommend* `prePrompt: true` in its skill but does not unilaterally enable it — the host application owns the choice.

This change is backward-compatible (optional field; absent → off; T-306 contract preserved).

### D-T385-5. Telemetry events

Following T-306's emit-telemetry contract:

- `permission.pre-prompt.shown` — attribute: family, permission
- `permission.pre-prompt.confirmed` — attribute: family, permission
- `permission.pre-prompt.cancelled` — attribute: family, permission
- `permission.dialog.shown` — emitted just before `getUserMedia` (the browser dialog is observable only through the user's eyes; this event marks our best-effort proxy)
- `permission.retry.clicked` — attribute: family, permission, attemptNumber
- `permission.retry.granted` — attribute: family, permission, attemptNumber
- `permission.retry.denied` — attribute: family, permission, attemptNumber

Existing shim events (`tenant-denied`, `permission-denied`) are preserved unchanged.

### D-T385-6. `<PermissionDenialBanner>` + `<PermissionPrePromptModal>` components

Two React components ship as the default visual surface. Both:

- Accept all user-facing text as required props (no English defaults).
- Are unstyled / minimally-styled (CSS modules or inline styles using CSS variables) so apps can theme.
- Expose every callback the state machine emits (retry, confirm, cancel) as required props — no internal default behaviours that surprise.

```tsx
interface PermissionDenialBannerProps {
  state: Extract<PermissionFlowState, { kind: 'denied' }>;
  /** App-localised messages keyed by reason / permission. */
  messages: PermissionDenialMessages;
  onRetry(): void;
  onDismiss?(): void;
}

interface PermissionPrePromptModalProps {
  state: Extract<PermissionFlowState, { kind: 'pre-prompt' }>;
  messages: PermissionPrePromptMessages;
  onConfirm(): void;
  onCancel(): void;
}
```

Apps that don't want the default visual surface ignore these components and render against `usePermissionFlow`'s state directly.

### D-T385-7. Browser-settings hint — generic, not browser-detected

The denial banner exposes a `browserSettingsHint` prop the app supplies. T-385 ships a default-empty hint (the prop is required but may be empty string for "no hint"); apps that detect Chrome / Safari / Firefox ship their own copy. Browser-detection logic stays out of `runtimes-interactive` (would brittle, would require user-agent parsing).

### D-T385-8. No schema changes

Permissions are already typed in `@stageflip/schema/clips/interactive` (T-305 + T-383 + T-384). T-385 is runtime-only — no Zod additions, no `check-preset-integrity` extensions.

### D-T385-9. Mount-harness integration

The existing `mount-harness.ts` calls `permissionShim.mount(clip)` synchronously then mounts the factory or falls back. T-385 inserts the `usePermissionFlow` state machine between the call site and the actual `getUserMedia` invocation:

- If `prePrompt === false` (default), behaviour matches T-306 exactly: mount-harness calls `shim.mount()` and routes accordingly.
- If `prePrompt === true`, the harness yields control to a React render cycle that displays the pre-prompt modal; on confirm, the shim's `mount()` proceeds; on cancel, the harness routes to `staticFallback` with a new reason `'pre-prompt-cancelled'`.

This is a behavioural addition to `mount-harness`, not a contract change. Existing T-306 / T-383 / T-384 tests remain green.

## Acceptance criteria

### State machine

1. Initial state is `{ kind: 'idle' }`.
2. `start()` with no permissions declared transitions directly to `granted`.
3. `start()` with permissions and `prePrompt: false` transitions to `requesting`, then to `granted` or `denied` per shim outcome.
4. `start()` with `prePrompt: true` and at least one permission transitions to `pre-prompt`. `confirmPrePrompt()` advances to `requesting`. `cancelPrePrompt()` transitions to `denied` with reason `'pre-prompt-cancelled'` (a new value added to `PermissionFlowState['denied']`).
5. From `denied` with `permission-denied` reason, `retry()` clears the failed permission's cache entry, transitions to `requesting`, and re-runs the shim.
6. From `denied` with `tenant-denied` reason, `retry()` is a no-op (tenant policy is not user-overridable). Pin via test.

### Shim extension

7. `PermissionShim.clearCacheEntry(family, permission)` removes only the named entry; sibling entries are preserved. Pin via test.

### Mount-harness integration

8. With `MountContext.permissionPrePrompt === undefined`, behaviour matches T-306 baseline (existing tests green).
9. With `permissionPrePrompt: true` and a clip declaring `mic`, the harness yields a pre-prompt render cycle; confirming proceeds to `getUserMedia`; cancelling routes to `staticFallback` with reason `'pre-prompt-cancelled'`.

### Components

10. `<PermissionDenialBanner>` renders with required text props; missing required props is a TypeScript error (no runtime defaults).
11. `<PermissionDenialBanner>` calls `onRetry` exactly once per click; `onDismiss` is optional and absent click is a no-op.
12. `<PermissionDenialBanner>` does not call any browser API directly — it is a pure visual surface.
13. `<PermissionPrePromptModal>` renders confirm + cancel buttons; clicking each fires the respective callback exactly once.
14. Both components forward `data-testid` props for app-level test wiring.

### Telemetry

15. Each documented event (D-T385-5) is emitted with the documented attribute shape, observed via a spy on `MountContext.emitTelemetry`.
16. `permission.dialog.shown` is emitted exactly once per browser-prompt invocation, including retries (so attempt-1 + attempt-2 both emit).
17. Pre-prompt cancel does NOT emit `permission.dialog.shown` (the browser dialog never showed).

### i18n posture

18. No hardcoded English UI strings in `runtimes-interactive/src/permission-flow/**`. Verified by lint rule (a script-level grep is acceptable; no need to introduce a runtime ESLint rule).

### Backward compatibility

19. All pre-existing T-306 / T-383 / T-384 tests remain green at HEAD.
20. `MountContext.permissionPrePrompt` is optional; pre-existing consumers that construct `MountContext` literals continue to type-check.

## Escalation triggers (CLAUDE.md §6)

- The browser's `getUserMedia` rejection does not surface a distinguishable reason for "user denied" vs "no device available." Escalate; the denial banner copy may need to fork.
- Pre-prompt rendering inside the mount-harness's React tree creates a render-loop or layout-thrash issue that exceeds the M-sized budget. Escalate; consider routing pre-prompt rendering to the host application via a context callback.
- The `permissionPrePrompt` extension to `MountContext` is not backward-compatible (e.g., a `keyof` exhaustiveness check breaks). Escalate.
- A clip family's permission ordering matters semantically (e.g., asking for `mic` before `network` matters for VoiceClip's UX), and the linear iteration in the shim doesn't accommodate. Escalate; the shim may need a per-family ordering hook.
