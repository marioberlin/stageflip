# T-309a — Tighten `check-determinism` shader sub-rule

**Branch**: `task/T-309a-shader-rule-tighten`

## Goal

Resolve the structural concern raised by the T-383 PR review (Major M-1): the path-based shader sub-rule conflates "is a uniform updater" with "uses time correctly" and false-flags non-updater helper code in `clips/shader/**`. The Implementer of T-383 worked around this by wrapping the factory in a static-only class (methods aren't visited by the walker) — a workaround that, if propagated to T-384 / T-389+ / preset families, would erode the rule's value.

T-309a fixes the rule itself, two ways:

1. **Extend** `collectUniformUpdaters` to walk top-level `ClassDeclaration` members under path-matched files — a method on a class in `clips/shader/**` is inspected exactly like a top-level function.
2. **Drop** the "missing-frame-parameter" check (T-309 D-T309-2 / AC #5 / AC #6 first violation kind). The forbidden-API check (the actual hazard) is sufficient on its own. Functions / methods that don't take `frame` and don't call any forbidden API are by definition deterministic; flagging them as violations was overzealous defence and the source of the T-383 false-positive pressure.

After T-309a, T-383's static-class workaround can be unwound (`ShaderClipFactoryBuilder` can become top-level functions) without tripping the rule. T-384 starts on a clean rule.

**T-309a must land before T-384 starts** — explicit ordering constraint per the Reviewer recommendation in the T-383 PR thread.

## Architectural decisions

### D-T309a-1. Extend AST walk to method declarations on top-level classes (path-matched only)

In `collectUniformUpdaters`, when `pathMatched` is true and the visitor sees a top-level `ClassDeclaration`, walk its members and treat each `MethodDeclaration` (instance and static) as a uniform-updater for inspection.

```ts
// Pseudocode
if (ts.isClassDeclaration(node) && pathMatched) {
  for (const member of node.members) {
    if (ts.isMethodDeclaration(member)) {
      out.push({
        node: member,
        name: member.name?.getText(sourceFile) ?? '<anonymous>',
        pathMatched: true,
        decoratorTagged: hasUniformUpdaterTag(member),
      });
    }
  }
}
```

Decorator-tagged methods on classes regardless of path are also inspected (preserves existing decorator regime).

Constructors are NOT inspected — they run once at mount, not per frame; the forbidden-API set's intent doesn't apply.

### D-T309a-2. Drop the "missing-frame-parameter" check entirely

The current rule emits a violation when a path-matched (or decorator-tagged) function does not declare a `frame` parameter. This is dropped from `inspectUniformUpdater`.

Rationale:

1. The actual determinism hazard is **calling a forbidden API** (non-deterministic time source). The forbidden-API check directly catches that.
2. A function that doesn't take `frame` and doesn't call any forbidden API is **deterministic by definition** (within the rule's scope). Flagging it as a violation is defensive over-reach.
3. The check creates false positives in legitimate factory / dispose / mount code that lives in `clips/shader/**` for organisational reasons — the entire pressure that pushed T-383 toward the static-class workaround.
4. The check duplicated information already encoded in TypeScript types: the `UniformsForFrame<P>` type signature in `@stageflip/runtimes-shader` requires `frame`. A misshapen updater would fail typecheck before `check-determinism` runs.

**T-309 ACs that drop**: AC #5 (missing-frame-only signal), AC #6 first violation kind. AC #6 second violation kind (Date.now usage in same function) remains — it now surfaces as a single violation, not "twice."

### D-T309a-3. Output format change — drop the second violation line

When a forbidden API is called inside a function with no `frame` parameter, the rule emits ONE violation (for the API), not TWO (one for missing-frame, one for the API). The output line set is unchanged otherwise.

### D-T309a-4. Backward compatibility — existing T-383 code continues to pass

The T-383 `ShaderClipFactoryBuilder` static-class form must continue to pass under the new rule. After D-T309a-1, the class's methods are now inspected; under D-T309a-2, none of them violate (no forbidden APIs). So the new rule passes the existing code unchanged.

T-309a does NOT refactor T-383's factory. That refactor belongs in a separate small PR after T-309a merges, when an Implementer can land it without ambiguity.

### D-T309a-5. Performance budget unchanged

The class-member walk adds O(class members) per top-level class. Trivial overhead in practice. Existing <2s budget from T-309 D-T309-5 is preserved.

## Acceptance criteria

### AST walk extension

1. **Class method, path-based, positive case**: a file at `packages/runtimes/interactive/src/clips/shader/foo.ts` containing
   ```ts
   class Foo {
     bar(frame: number) { return Date.now(); }
   }
   ```
   causes the sub-rule to FAIL at `bar`'s `Date.now()` call site.
2. **Class method, path-based, negative case**: same path with
   ```ts
   class Foo {
     bar(frame: number) { return frame * 0.001; }
   }
   ```
   PASSES.
3. **Class method, decorator-tagged, regardless of path**: a file at any path with
   ```ts
   class Foo {
     /** @uniformUpdater */
     bar(frame: number) { return performance.now(); }
   }
   ```
   FAILS.
4. **Constructor not inspected**: a path-matched class whose `constructor()` calls `Date.now()` PASSES (constructors run once at mount, not per frame).
5. **Static methods inspected**: same rules apply to `static` methods on a path-matched class.

### Missing-frame check dropped

6. **Missing frame, no forbidden API, path-matched**: `function helper() { return 1; }` in `clips/shader/**` PASSES (was FAIL under T-309).
7. **Missing frame, with forbidden API, path-matched**: `function helper() { return Date.now(); }` in `clips/shader/**` FAILS with ONE violation (the API call), not two.
8. **Missing frame, decorator-tagged**: `/** @uniformUpdater */ function bar() { return Date.now(); }` FAILS with ONE violation.

### Backward compatibility

9. The existing T-383 `ShaderClipFactoryBuilder` and helpers in `packages/runtimes/interactive/src/clips/shader/` PASS under the new rule. Run `pnpm check-determinism` at HEAD post-T-309a-merge: must report `1 uniform-updater detected` (the `defaultShaderUniforms` decorator-tagged form) plus the new path-matched class methods, all PASSING. No new violations introduced on existing main-branch code.
10. The default `@uniformUpdater` (`defaultShaderUniforms`) continues to PASS. (Sanity: the dropped missing-frame check should not have been firing on it; this is an explicit pin.)

### Output format

11. Output count line `check-determinism [shader sub-rule]: PASS (N uniform-updaters detected)` reflects path-matched top-level functions + path-matched class methods + decorator-tagged functions / methods. Methods on classes are counted individually.
12. Violation lines unchanged in shape (`file:line:col` + API + hint).
13. Exit 0 on PASS; 1 on FAIL.

## Escalation triggers (CLAUDE.md §6)

- Removing the missing-frame check surfaces a real-world false negative we hadn't considered (a function that uses `frame` indirectly, e.g., via `arguments[0]`, and ALSO uses `Math.random()` — but `Math.random()` is in the forbidden-API set, so this scenario is already covered. If an analogous gap appears, escalate.).
- TypeScript's class-member iteration changes between TS versions in a way that breaks the visitor. Escalate; pin the TS version.
- The performance budget (<2s) is exceeded due to many class methods being walked. Escalate; consider memoization.
