# T-240 — `@stageflip/import-pptx` (ZIP + PresentationML parser)

**Branch**: `task/T-240-import-pptx`

## Goal

Implement the PPTX → canonical-schema parser. Read a `.pptx` file (ZIP), parse PresentationML XML for slides/layouts/masters, emit a `CanonicalSlideTree` compatible with the existing schema package. **Just the structural parser**: no group transforms, no assets, no custom geometries, no rasterization.

## Type-layer architecture (Orchestrator clarification — 2026-04-25)

`CanonicalSlideTree` is a **parser-side intermediate type**, not a `Document`. Subsequent P11 tasks (T-242 geometries, T-243 assets, T-245 rasterization) progressively narrow the parser-side types into the canonical schema-typed forms. The strict `documentSchema.parse` is only invoked **after** those siblings complete the round-trip.

Concretely, `@stageflip/import-pptx` defines its own types that mirror schema variants but loosen the fields T-240 cannot resolve:

- `ParsedAssetRef = { kind: 'resolved'; ref: AssetRef } | { kind: 'unresolved'; oocxmlPath: string }` — image elements coming out of the parser carry a `ParsedAssetRef`; T-243 resolves the unresolved variant into the schema's `AssetRef` (a `string`).
- `ParsedElement = Element | UnsupportedShapeElement` where `UnsupportedShapeElement` has `type: 'unsupported-shape'` plus parser-only fields (preset name, raw geometry XML). T-242 + T-245 narrow these into schema-typed `ShapeElement` / `ImageElement` variants.
- `ParsedSlide` = the schema's `Slide` shape but with `elements: ParsedElement[]`.
- `CanonicalSlideTree = { slides; layouts; masters; lossFlags }` — wrapper type owned by the parser package.

**Do not extend `packages/schema`**. The parser-side types live entirely inside `@stageflip/import-pptx` and are exported for downstream P11 importers to consume. Acceptance criteria #4 and #6 reference these parser-side shapes, not the canonical schema.

## Acceptance criteria

Each gets a Vitest test, written first and failing before implementation.

1. `parsePptx(buffer): Promise<CanonicalSlideTree>` accepts a `Uint8Array` of a `.pptx` ZIP and returns a typed tree.
2. Parses **slides**, **slide layouts**, **slide masters** from the OPC package; preserves their relationship.
3. Resolves text runs (`<a:r><a:t>`) into `CanonicalElement` text variants with positional info from `<p:sp><p:spPr><a:xfrm>`.
4. Resolves picture placeholders (`<p:pic>`) into image variants — image **bytes** deferred to T-243; emits `assetRef: { kind: 'unresolved', oocxmlPath: <rel-path> }`.
5. Top-level group elements (`<p:grpSp>`) emitted as group nodes **without descended transform accumulation** — that's T-241a. Test pins this: leaves under groups carry their raw `xfrm`, not the accumulated parent product.
6. Custom geometries / preset shape geometries: emit a `CanonicalElement` placeholder of variant `'unsupported-shape'` with a loss flag of code `LF-PPTX-CUSTOM-GEOMETRY`. Real handling = T-242 + T-245.
7. Malformed ZIP → throws a typed `PptxParseError` with `code: 'INVALID_ZIP'`. Malformed XML in any required part → throws with `code: 'INVALID_XML'` and the part path.
8. Five fixture PPTX files in `packages/import-pptx/test/fixtures/` round-trip parse without throwing. Pinned snapshot of the canonical tree per fixture (committed). At least one fixture per: PowerPoint (Office 365), Keynote-exported, Google-Slides-exported, LibreOffice Impress, OOXML-spec minimal.
9. Coverage ≥85% on changed files (`pnpm test --coverage`).
10. **Determinism**: parser is pure (no `Date.now`, no `Math.random`, no I/O after the buffer is read). Gate already enforces; flag here as reminder.
