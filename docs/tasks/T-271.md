# T-271 — EU Firestore region for GDPR data residency

**Branch**: `task/T-271-eu-firestore-region`

## Goal

Implement org-level data-residency routing so EU-flagged orgs read/write a dedicated EU Firestore region, with cross-region access blocked at the security-rules layer. Per `skills/stageflip/concepts/auth/SKILL.md` §"Tenant data residency": "EU-residency orgs are assigned to a dedicated Firestore region (T-271). Cross-region access is blocked at the rules level; exports run in-region."

T-271 ships:

1. **Two Firestore databases** in the Firebase project: `(default)` (US, existing) and `eu-west` (new — EU region).
2. **`Org.region: 'us' | 'eu'`** field — already in schema (`@stageflip/auth-schema`); T-271 uses it to route.
3. **Region-aware storage adapter** — `@stageflip/storage-firebase` learns to pick the right Firestore instance based on `org.region`.
4. **Rules + indexes** — `eu-west` carries the same rules as `(default)`; cross-region `read` is impossible (the EU client never sees US docs because the storage adapter never asks).
5. **Documentation** — operational runbook for org-region assignment + migration (out of scope = automated migration; in scope = manual procedure).

## Dependencies

- T-262 (auth) — merged. `Org.region` is in `@stageflip/auth-schema`.
- T-035–T-037 (`@stageflip/storage-firebase`) — merged.
- T-230 (Firebase rules) — merged. T-271 extends the rules to the `eu-west` database.
- **Operational dependency**: EU Firestore database must be provisioned in Firebase console (`europe-west3` recommended for Frankfurt, low-latency to most of EU). User has confirmed this is provisioned.

## Out of scope

| Item | Why deferred |
|---|---|
| Automated org migration between regions | Operational; rare event (legal trigger). T-271 documents the manual procedure. |
| Multi-region presence (RTDB) | RTDB has its own region; presence migration is a follow-up if EU customers complain. T-271 ships docs only for presence migration. |
| Multi-region Storage (assets) | Firebase Storage already supports per-bucket region; T-271 ships the routing in the adapter, ops provisions the EU bucket. |
| Multi-region Cloud Functions deploy | Functions auto-deploy global; T-271 documents the recommended `region` annotation for callable functions touching EU data. |
| BigQuery export region | T-273 dropped from P12. |
| Multi-region Pub/Sub for queue routing | BullMQ + Upstash Redis is single-region (Upstash supports EU); T-271 documents but doesn't implement queue routing. |

## Architectural decisions

### D-T271-1. Two Firestore databases, one Firebase project

Firebase supports multiple Firestore databases per project (the "(default)" one + named additional databases). T-271 provisions:

- `(default)`: US (`nam5` multi-region, existing) — covers US/CA/global orgs by default.
- `eu-west`: EU (`europe-west3`, Frankfurt) — covers `org.region === 'eu'` orgs.

**Why one project, not two**: shared auth (Firebase Auth is global), shared Cloud Functions, single billing surface. Cross-database `read` is impossible — each database has its own rules + access path. The "blocked at the rules level" guarantee from the skill comes for free.

### D-T271-2. Routing happens in the storage adapter

The application code does NOT know which database it's talking to. `@stageflip/storage-firebase`'s `FirebaseStorageAdapter` constructor accepts a `db: Firestore` argument; T-271 introduces a thin factory:

```ts
function getFirestoreForOrg(org: Org): Firestore {
  return org.region === 'eu' ? euFirestore : defaultFirestore;
}
```

Consumers ask for an adapter via `getStorageAdapterForOrg(org)`. The factory caches per-region adapters; consumers don't re-instantiate.

### D-T271-3. Org region is immutable post-creation (in v1)

Once an org is created with `region: 'us'`, it stays US. Same for EU. Migration is **manual** (out of scope) and requires:

1. Export the org's documents from the source region.
2. Import to the target region.
3. Update `org.region` in `users/{userId}` records that referenced it.
4. Confirm with the customer.

T-271 ships the docs + a script skeleton (`scripts/migrate-org-region.ts`) but does NOT productize the migration. Mutation of `org.region` post-creation is rejected at the schema layer (Zod refinement) and at the rules layer (rule denies write to `org.region`).

### D-T271-4. Org-creation surface picks the region

The `createOrg` Cloud Function (a follow-up; not in T-271 scope per T-262) accepts a `region` parameter. T-271 documents the contract; T-262's existing functions (invite/accept/etc.) only operate within an org so they're region-neutral.

In the meantime (until createOrg ships), test orgs are created via a seeded fixture script that assigns region.

### D-T271-5. Determinism posture

`packages/storage-firebase/**` and `packages/auth-*/**` are not in the determinism scan. T-271's changes don't introduce new determinism concerns.

## Acceptance criteria

### Region router

1. `getFirestoreForOrg(org)` returns the default Firestore for `org.region === 'us'` (or `undefined`).
2. Returns the eu-west Firestore for `org.region === 'eu'`.
3. Throws on unknown region values (defensive — Zod schema enforces, but runtime guard hardens).
4. Per-region adapter cache: requesting twice for the same region returns the same instance.
5. Constructor accepts `{ defaultFirestore, euFirestore }` injection so tests can pass mocks.

### Schema immutability

6. `orgSchema.parse({ ..., region: 'us' })` succeeds.
7. `orgSchema.parse({ ..., region: 'invalid' })` fails with a clear message.
8. **Mutation guard**: a `validateRegionTransition(prevOrg, nextOrg)` helper returns `Ok | Err` — `Err` if `prevOrg.region !== nextOrg.region`. Pin via test. (Zod can't natively express "this field is immutable across updates"; the helper is the application-side enforcement.)

### Rules

9. `firebase/firestore.rules` (or duplicated `firestore-eu.rules`) — both databases enforce the same role model from T-262. Pin via firebase-rules-unit-testing OR structural rule-text regression guard if the emulator isn't bootstrapped (deviation acceptable per T-262 precedent).
10. **No cross-region read possible**: rules tests verify that an EU-region client (auth principal) cannot read a US-region document by guessing its docId. (This is true by construction — different database = different access path — but pin a test that demonstrates it.)

### Asset routing

11. `getStorageBucketForOrg(org)` returns `gs://stageflip-eu-assets` for EU orgs, `gs://stageflip-assets` for US orgs (default). Bucket names are env-configurable.

### Backward compatibility

12. The full workspace test suite passes unchanged. Existing consumers that call `new FirebaseStorageAdapter(defaultFirestore)` directly still work; the new router is additive.
13. **Existing US orgs**: any persisted `Org` record without a `region` field defaults to `region: 'us'` via Zod's `.default('us')`. Pin via test against a fixture missing the field.

### Migration script skeleton

14. `scripts/migrate-org-region.ts --dry-run --org=<orgId> --target-region=<eu|us>` prints the planned operations without executing.
15. Script exits non-zero on missing required args.

### Documentation

16. `docs/ops/data-residency.md` documents:
   - Which legal/contractual triggers flag an org for EU residency.
   - Manual migration procedure (export/import/update).
   - What the customer sees during migration (read-only window).
   - Rollback plan if migration fails mid-flight.

18. Coverage on `packages/auth-schema/src/org.ts` immutability guard ≥95% (security primitive).
