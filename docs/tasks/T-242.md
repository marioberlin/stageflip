# T-242 — PPTX preset geometries + custom paths → schema shapes

**Branch**: `task/T-242-pptx-preset-geometries`

## Goal

Resolve every `LF-PPTX-PRESET-GEOMETRY` and `LF-PPTX-CUSTOM-GEOMETRY` flag T-240 emits today by converting OOXML preset shapes and `<a:custGeom>` payloads into schema-typed `ShapeElement` instances. Most presets resolve into the existing `'custom-path'` variant with a generated SVG `d` attribute; a small set keeps mapping into the structural kinds (`rect`, `ellipse`, `polygon`, `star`, `line`) where it makes sense.

## Dependencies

- T-240 merged (provides the `<p:sp>` walker that emits placeholder `unsupported-shape` elements + the two preset / custom flags).
- T-241a merged (no functional dependency, but shape children inside groups now carry world-space transforms).
- T-243 merged (no dependency).

## Out of scope

| Sibling | Why deferred |
|---|---|
| T-245 | Rasterization fallback for the long-tail of presets T-242 doesn't cover. |
| T-249 | Fill / stroke / theme resolution. T-242 emits `'shape'` elements with default fill/stroke; T-249 fills them in. |
| Adjustable handles (`<a:avLst>`) beyond defaults | Most presets accept adjustments via `<a:gd>` formulas. T-242 honors the most common adjustable (rounded-rect corner radius); the rest use spec defaults. Follow-up if a real fixture needs the others. |
| Reverse direction (canonical → PPTX) | Not in scope; PPTX export is a future task. |

## Architectural decisions

### Schema strategy: mostly `'custom-path'`, a few structural kinds

Schema's `ShapeKind` enum (`packages/schema/src/elements/shape.ts:22-28`) has six variants today: `rect | ellipse | line | polygon | star | custom-path`. Extending it with all 50+ preset names would couple the schema to PPTX vocabulary and bloat downstream rendering switches. Instead:

- Keep the existing five structural kinds for the presets that map cleanly (already started in T-240's `PRESET_TO_SCHEMA` table — `rect`, `roundRect`, `ellipse`, `oval`, `line`, `triangle`, `diamond`, `pentagon`, `hexagon`, `octagon`, `star4`–`star32`).
- For the rest (callouts, banners, arrows, brackets, smileyFace, sun, cloud, etc.), generate an SVG `d` attribute string and emit a `'custom-path'` variant. The schema's existing field (`packages/schema/src/elements/shape.ts:35` — `path?: string`) takes the result.
- `<a:custGeom>` likewise translates the OOXML path-language into SVG `d` and emits `'custom-path'`.

This keeps `@stageflip/schema` unchanged.

### Where the geometry library lives

New module `packages/import-pptx/src/geometries/`:

- `presets/` — one file per preset (or grouped by family). Each exports a `(box: { w: number; h: number }, adjustments?: AdjustmentMap) => string` that returns an SVG `d` attribute.
- `cust-geom/parse.ts` — translates `<a:custGeom>` (with `<a:pathLst>` / `<a:moveTo>` / `<a:lnTo>` / `<a:cubicBezTo>` / `<a:arcTo>` / `<a:close>`) into SVG `d`.
- `index.ts` — `geometryFor(prst, box, adjustments?): string | undefined`.

If `T-244` (Google Slides) later needs the same preset library, extract to `@stageflip/preset-geometries`. Not done preemptively.

### Adjustable-handle handling

`<a:avLst><a:gd name="adj1" fmla="val 12700"/></a:avLst>` exposes adjustable parameters. T-242 supports:

- **`roundRect.adj`** — corner radius, used by `roundRect` family. Affects path output.
- All other adjustments default to OOXML spec values. The presets file documents each preset's defaults.

A real fixture exposing an adjustment we don't support emits `LF-PPTX-PRESET-ADJUSTMENT-IGNORED` (new code, info severity) — the shape still renders, just at the default value.

### Coverage commitment: 50 presets

The plan calls for "50+ preset geometries". T-242 ships **exactly 50** (enumerated in the spec below). The remaining ~140 OOXML presets continue to emit `LF-PPTX-PRESET-GEOMETRY` and resolve via T-245's rasterization fallback. Adding more presets is incremental and can happen in T-242b.

The 50 presets to cover (grouped by family — see `packages/import-pptx/src/geometries/presets/index.ts` for the canonical list once implemented):

- **Already mapped to structural kinds (T-240, no path generation needed)** — 14: `rect`, `roundRect`, `ellipse`, `oval`, `line`, `triangle`, `diamond`, `pentagon`, `hexagon`, `octagon`, `star4`, `star5`, `star6`, `star8`.
- **New custom-path generators for T-242** — 36:
  - **Arrows (8)**: `rightArrow`, `leftArrow`, `upArrow`, `downArrow`, `leftRightArrow`, `upDownArrow`, `bentArrow`, `curvedRightArrow`.
  - **Callouts (6)**: `wedgeRectCallout`, `wedgeRoundRectCallout`, `wedgeEllipseCallout`, `cloudCallout`, `borderCallout1`, `borderCallout2`.
  - **Banners / Stars (6)**: `ribbon`, `ribbon2`, `verticalScroll`, `horizontalScroll`, `star10`, `star12`.
  - **Basic shapes (6)**: `parallelogram`, `trapezoid`, `chevron`, `chord`, `pie`, `donut`.
  - **Brackets / Braces (4)**: `leftBracket`, `rightBracket`, `leftBrace`, `rightBrace`.
  - **Misc (6)**: `cloud`, `sun`, `moon`, `heart`, `lightningBolt`, `noSmoking`.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

1. `geometryFor(prst, box): string | undefined` returns an SVG path string for every preset name in the 36-preset new-coverage set; returns `undefined` for unmapped presets.
2. The 14 structural-kind presets continue to emit `'shape'` elements with the right `shape` field (no regression on T-240's table).
3. The 36 newly-covered presets emit `'shape'` elements with `shape: 'custom-path'` and a non-empty `path`. Tests pin the path string for at least one canonical box size per preset.
4. `<a:custGeom>` payloads emit `'shape'` elements with `shape: 'custom-path'` and a `path` derived from the OOXML path-language. Tests pin path output for: a single `<a:moveTo>` + `<a:lnTo>` triangle, a `<a:cubicBezTo>` curve, a closed `<a:close>` polygon, and a multi-path `<a:pathLst>` with two paths.
5. **Zero** `LF-PPTX-PRESET-GEOMETRY` flags remain for any of the 50 covered presets.
6. **Zero** `LF-PPTX-CUSTOM-GEOMETRY` flags remain for `<a:custGeom>` payloads the parser successfully translates.
7. Presets outside the 50-preset set continue to emit `LF-PPTX-PRESET-GEOMETRY` + `unsupported-shape` (no regression for T-245's rasterization-fallback path).
8. **`roundRect`** with `adj1` honors the corner radius. Test: `<a:prstGeom prst="roundRect"><a:avLst><a:gd name="adj1" fmla="val 25000"/></a:avLst></a:prstGeom>` on a 100×100 box → path string with corner radius proportional to the 25000-EMU adjustment.
9. Coverage ≥85% on the new `geometries/` module. Path generation is pure (no `Date`, no `Math.random`); deterministic.
10. The full PPTX import pipeline (`parsePptx → accumulateGroupTransforms → resolveAssets`) ends with `LF-PPTX-PRESET-GEOMETRY` and `LF-PPTX-CUSTOM-GEOMETRY` cleared on a synthetic fixture using only the 50-preset set.

## Public-spec / library references

- ECMA-376 §20.1.9 — preset geometry definitions. The path-formula language (`<a:gd>` `fmla` attribute) is fully specified.
- https://learn.microsoft.com/en-us/openspecs/office_standards/ms-pptx/ — search "preset shape definitions".
- https://officeopenxml.com/drwSp-prstGeom.php — readable summary of the preset families. Cite, do not copy.
- SVG path data spec (W3C): https://www.w3.org/TR/SVG2/paths.html — output format.

The OOXML preset-geometry definitions are a public spec annex (Annex M / `presetShapeDefinitions.xml` in the ECMA-376 reference distribution). T-242's path generators **may** be derived from those definitions but **must not** be copy-pasted from any GPL/AGPL/Remotion-licensed implementation. Re-derive from the spec.

## Escalation triggers (CLAUDE.md §6)

- A real PPTX fixture exposes a preset path geometry the OOXML spec doesn't cleanly define (rare; the spec is comprehensive but sometimes ambiguous on adjustable formulas). Escalate before fudging the math.
- A real fixture exposes a `<a:custGeom>` `<a:arcTo>` parameter combination that doesn't translate cleanly to SVG `A` (arc) syntax — different parametrizations. Escalate; document the case.
- Adjustable `<a:gd>` formula evaluator turns out to be necessary (i.e., presets reference `adj` values via `pin()`/`min()`/`max()` formulas rather than constants). T-242 assumes constant adjustments only; if a fixture surfaces a real formula, escalate — full formula evaluator is its own task.

## Notes for the Orchestrator

1. **Implementation strategy**: this is a long-tail enumeration task. The Implementer should ship the geometry library scaffolding + `cust-geom/parse.ts` + `geometryFor` registry + 5–10 presets in one PR for reviewability, then add preset families in follow-up PRs. The acceptance criteria are sized for the full 50-preset set; if it lands in two PRs (e.g., `T-242a` infra + first wave, `T-242b` rest), the Reviewer should accept the split with the registry test pinning the eventual coverage list.
2. **Preset path generation reference quality**: the OOXML spec's path formulas are exact but verbose. The Implementer is expected to translate, not blindly copy. Each preset's generator should be testable against a small set of known box sizes with known SVG outputs.
3. **Coverage commitment is 50, not 50+**: the plan says "50+" but committing to 50 keeps scope honest. The remaining ~140 OOXML presets land via T-245's rasterization or a future T-242b.
