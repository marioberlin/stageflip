# T-263 — API rate limiting per user / org / key

**Branch**: `task/T-263-rate-limiting`

## Goal

Stand up the `@stageflip/rate-limit` package — distributed token-bucket rate limiting enforceable at HTTP middleware (API gateway) and per-action engine boundaries. Limits are tiered per **user**, **org**, and **api-key** with hierarchical enforcement: a request must pass all three buckets that apply. Backed by Upstash Redis (already in architecture for BullMQ queues, `docs/architecture.md:148, :339`).

T-263 ships the package + middleware + Vitest fixtures. Dispatching consumers (engine handlers, public API routes) is a follow-up — T-263 stands up the engine and pins shape.

## Dependencies

- T-262 (auth + org tenancy) — **must merge first**. Without `req.user.id`, `req.org.id`, `req.apiKey.id` the limiter has nothing to key on. If T-262 has not landed when T-263 dispatches, escalate per CLAUDE.md §6.
- Upstash Redis credentials in env — already provisioned for BullMQ queues; T-263 reuses the connection pattern.
- `@upstash/ratelimit` + `@upstash/redis` (both MIT/Apache-2.0; on whitelist).

## Architectural decisions

### D-T263-1. Algorithm: token bucket

- **Token bucket** chosen over sliding window. Reasons:
  - Token bucket smooths bursts (allows brief spikes within capacity); sliding window does not.
  - Industry-standard for API rate limiting (Stripe, GitHub, etc. converge on this).
  - `@upstash/ratelimit` provides a battle-tested implementation; no reason to roll our own.
- **Bucket params**: per-tier (see D-T263-3); each bucket has `tokens` (burst capacity) and `refillRate` (tokens per second).

### D-T263-2. Storage: Upstash Redis (existing)

- Reuse the Upstash Redis instance provisioned for BullMQ queues. No new infrastructure.
- **Key namespacing**: `ratelimit:{tier}:{id}` — e.g., `ratelimit:user:u_abc123`. Avoids collision with BullMQ's `bull:*` namespace.
- TTL: each bucket key expires after 2× its refill period when idle (Upstash auto-expires).

### D-T263-3. Three tiers, hierarchical enforcement

| Tier | Default capacity | Default refill | Notes |
|---|---|---|---|
| `user` | 60 tokens | 1 token / sec | Per-authenticated-user. Spans all sessions. |
| `org` | 600 tokens | 10 tokens / sec | Per-org aggregate. Default = 10× user limit. |
| `apiKey` | 300 tokens | 5 tokens / sec | Per-API-key (programmatic clients). Independent of user. |

A request consumes one token from each applicable bucket. **All applicable buckets must have at least one token** for the request to pass:

- Authenticated user request: `user` + `org`.
- API-key request: `apiKey` + `org` (the key is bound to an org via T-262).
- Anonymous request: not allowed at this layer (T-262 enforces auth).

If any bucket is exhausted, return 429 with `Retry-After: <seconds-until-token>` (rounded up).

**Defaults are spec-pinned**; per-tier overrides via env vars (`STAGEFLIP_RATE_LIMIT_USER_CAPACITY`, etc.) for ops tuning without redeploy.

### D-T263-4. Enforcement layer: middleware + helper

Two enforcement surfaces:

1. **HTTP middleware** (`createRateLimitMiddleware`): consumed by API route handlers. Reads `req.user.id`, `req.org.id`, `req.apiKey?.id` (per T-262 conventions); calls the limiter; on rejection emits 429 + `Retry-After` + JSON body `{ code: 'RATE_LIMITED', tier: 'user'|'org'|'apiKey', retryAfterSeconds: number }`.
2. **Engine boundary helper** (`checkAndConsume`): consumed by long-running engine handlers (e.g., AI call orchestration). Throws `RateLimitedError` with the same `tier` + `retryAfterSeconds` fields; caller decides whether to surface to user or back off internally.

Both share the underlying limiter instance.

### D-T263-5. Test isolation

- Use `@upstash/redis`' built-in mock (or a thin in-memory fake; Implementer's choice — document in PR) for vitest. Do NOT hit a real Upstash Redis from CI.
- Real-Redis integration test deferred to a smoke harness; T-263 does not block on it.

### D-T263-6. Determinism posture

`packages/rate-limit/**` is NOT clip / runtime code. `pnpm check-determinism` rules don't apply. Token bucket reads `Date.now()` for refill computation; `setTimeout` in retry/backoff paths is fine.

## Acceptance criteria

Each AC has a Vitest test, written first and failing.

### `config.ts`

1. Default bucket params per D-T263-3 are exported as constants. Pin via test.
2. Env-var overrides (`STAGEFLIP_RATE_LIMIT_<TIER>_CAPACITY`, `STAGEFLIP_RATE_LIMIT_<TIER>_REFILL`) override defaults when set; ignored when unset/empty.
3. Invalid env values (non-numeric, negative) throw at module load with a clear error message.

### `limiter.ts`

4. `new RateLimiter({ redis })` constructs with a `@upstash/redis`-compatible client (or mock).
5. `await limiter.consume({ user: 'u_a', org: 'o_a' })` returns `{ allowed: true }` when all buckets have ≥1 token; consumes one token from each.
6. When the user bucket is empty, returns `{ allowed: false, tier: 'user', retryAfterMs: <number> }`. The `retryAfterMs` is the time until the user bucket has at least one token.
7. When multiple buckets are empty, returns the bucket with the **longest** retry — caller should wait that long.
8. **Anonymous / no-user-no-apiKey** → throws `Error('rate-limit: at least one of user or apiKey required')` to fail fast (auth must run before).
9. **Per-key isolation**: consuming on `{ user: 'u_a' }` does not affect `{ user: 'u_b' }`'s bucket.
10. **Org bucket shared**: two users in the same org both consume from the org bucket. Pin via test.

### `middleware.ts`

11. `createRateLimitMiddleware({ limiter })` returns a Connect-style `(req, res, next) => void` middleware compatible with the project's HTTP layer (Implementer cross-checks the existing API stack — Express? Hono? Cloud Functions? — and matches signature).
12. Middleware reads `req.user.id`, `req.org.id`, `req.apiKey?.id`. Missing all three → throws (per AC #8).
13. On allowed: calls `next()`. On rejected: writes 429 with `Retry-After: <seconds-rounded-up>` header and JSON body matching D-T263-4.
14. **Idempotent on multiple invocations**: middleware can be applied to a router that's already wrapped by another instance (e.g., for layered limits); each instance consumes from its own buckets.

### `errors.ts`

15. `RateLimitedError extends Error` with `tier: 'user'|'org'|'apiKey'` and `retryAfterMs: number` fields.
16. JSON serialization (via `toJSON`) produces `{ code: 'RATE_LIMITED', tier, retryAfterSeconds: Math.ceil(retryAfterMs/1000) }` matching the wire shape.

### Backward compatibility

17. The full `@stageflip/storage`, `@stageflip/collab`, `@stageflip/presence`, `@stageflip/schema` test suites pass unchanged.
18. **No existing API route is modified.** T-263 ships the middleware; integration is a follow-up.

## Escalation triggers (CLAUDE.md §6)

- T-262 has not merged: STOP. The limiter cannot key on `req.user.id` if T-262 hasn't shipped the request shape.
- The project's HTTP stack is something the Implementer didn't expect (e.g., a custom dispatcher rather than Express/Hono). Escalate; the middleware signature must match what consumers will use.
- Upstash credentials missing in test env. Use the in-memory mock per D-T263-5 — do NOT hit real Upstash from CI.
- Default bucket params (D-T263-3) feel wrong for the actual API surface (e.g., 60 tokens/min is way under or over the realistic ceiling). Escalate; the defaults are spec-pinned.
- `@upstash/ratelimit` doesn't expose the bucket-internals needed for AC #6 / #7 (return time-until-token). The package exports `pending` / `remaining` / `reset` — verify these cover the spec. If not, Implementer may need to wrap with manual bucket math; escalate before going custom.
