# T-389 — `AiChatClip` `liveMount` (scoped LLM)

**Branch**: `task/T-389-ai-chat-clip-live-mount`

## Goal

Ship the second γ-live family: `AiChatClip` `liveMount`. A schema-validated `InteractiveClip` with `family: 'ai-chat'`, registered in `interactiveClipRegistry`, mountable via the T-306 `ClipFactory` contract. Wraps `@stageflip/llm-abstraction`'s provider-neutral `LLMProvider` to expose a scoped chat with a per-slide system prompt baked into the schema; streams tokens to subscribers; emits typed lifecycle telemetry.

T-389 is the third application of the **second γ pattern** (event-driven, no §3 reuse). After T-389 + T-390, the pattern has run on three different clip families (voice, ai-chat, ai-chat-static) — abstraction earns extraction *if* the duplication is genuinely "three similar lines of meaningful logic" and not just convention. Pattern-evaluation revisited per CLAUDE.md guidance.

T-389 ships `liveMount` only. **T-390 ships `staticFallback`** (captured transcript rendered as text) as a separate S-sized task.

## Architectural decisions

### D-T389-1. `AiChatClip` lives entirely in `runtimes-interactive/src/clips/ai-chat/`

No new package; no §3 runtime to reuse. Same posture as T-387 (D-T387-1). The clip is event-driven (LLM streaming, user-message events), not frame-driven. Confirms the second γ pattern across three families.

### D-T389-2. Schema narrowing — `aiChatClipPropsSchema`

```ts
// packages/schema/src/clips/interactive/ai-chat-props.ts (NEW)
export const aiChatClipPropsSchema = z.object({
  /** Per-slide system prompt baked into the schema. The clip's identity. */
  systemPrompt: z.string().min(1),
  /** LLM provider name (`openai` / `anthropic` / `gemini` / etc. — must be in PROVIDER_NAMES). */
  provider: z.string().min(1),
  /** Model identifier within the provider. */
  model: z.string().min(1),
  /** Token-cap per turn; used for backpressure. */
  maxTokens: z.number().int().positive().default(512),
  /** Temperature in [0, 1.5]. */
  temperature: z.number().min(0).max(1.5).default(0.7),
  /** Whether the chat exposes a `turn` event stream to the host (true) or runs as a one-shot Q&A (false). */
  multiTurn: z.boolean().default(true),
  /** Frame at which `staticFallback` poster should be sampled (T-390 consumes). */
  posterFrame: z.number().int().nonnegative().default(0),
}).strict();
```

Browser-safe (pure Zod). Mirrors the discriminator pattern from T-383/T-384/T-387.

### D-T389-3. Permissions — `['network']`; pre-prompt opt-in (off by default)

The clip declares `permissions: ['network']`. Per the existing shim, `network` is a no-op grant in v1 (the runtime trusts declared egress; ADR-003 §D6 follow-up adds tenant allowlists later). Pre-prompt is OFF by default for `network`-only clips because the no-op grant short-circuits without any user-visible browser dialog; pre-prompt would be redundant noise. Documented in the skill.

### D-T389-4. `liveMount` API surface

The factory mounts a small React tree:

```
<AiChatClipMount>
  <output data-role="message-stream" />        // streamed assistant tokens
  <textarea data-role="user-input" />          // user message input (optional UI; host may override)
  <button data-action="send" />                // submit user message
</AiChatClipMount>
```

All visual structure is minimally styled and exposes data attributes for app-level overrides. Per CLAUDE.md §10, no English UI strings ship in the package — the host supplies button labels, placeholder text via React props.

The factory exposes lifecycle callbacks via `MountHandle`:

```ts
interface AiChatClipMountHandle extends MountHandle {
  /** Send a user message. Returns when the assistant turn fully resolves. */
  send(userMessage: string): Promise<void>;
  /** Subscribe to streamed token events. Returns unsubscribe. */
  onTurn(handler: (event: TurnEvent) => void): () => void;
  /** Reset the conversation (drops accumulated history; system prompt persists). */
  reset(): void;
}

type TurnEvent =
  | { kind: 'user'; text: string; timestampMs: number }
  | { kind: 'assistant-token'; text: string; turnId: string; timestampMs: number }
  | { kind: 'assistant-final'; text: string; turnId: string; timestampMs: number }
  | { kind: 'error'; message: string; turnId: string };
```

### D-T389-5. `LLMChatProvider` interface — wraps `@stageflip/llm-abstraction`

```ts
// packages/runtimes/interactive/src/clips/ai-chat/llm-chat-provider.ts (NEW)
export interface LLMChatProvider {
  /** Stream a turn for the given system prompt + accumulated messages. */
  streamTurn(args: {
    systemPrompt: string;
    history: ReadonlyArray<{ role: 'user' | 'assistant'; content: string }>;
    userMessage: string;
    model: string;
    maxTokens: number;
    temperature: number;
    onToken: (token: string, turnId: string) => void;
    signal: AbortSignal;
  }): Promise<{ finalText: string; turnId: string }>;
}
```

Two implementations ship with T-389:

1. `RealLLMChatProvider({ provider })` — wraps `LLMProvider` from `@stageflip/llm-abstraction`. Constructor takes the resolved provider name; uses `createProvider(spec)` if a spec is supplied, or accepts a pre-built `LLMProvider` instance for tests.
2. `InMemoryLLMChatProvider({ scripted })` — for tests; emits a scripted token sequence.

A custom-provider seam (e.g., a tenant's self-hosted endpoint) is a future asset-gen task — Phase 14 ADR-006 covers the pattern.

### D-T389-6. No convergence test, no `frameSource` dependency

Confirms second γ pattern. The factory does NOT throw `MissingFrameSourceError`; it ignores `MountContext.frameSource`. Pin via test (mounting with `frameSource: undefined` succeeds).

### D-T389-7. Resource cleanup on dispose

`MountHandle.dispose()` MUST:

1. `signal.abort()` on any in-flight `streamTurn` call (the provider's signal-aware path).
2. Drop accumulated message history (no leaked references for GC).
3. Unsubscribe all `onTurn` handlers.
4. Be idempotent.

Less heavy than T-387's resource graph (no MediaStream, no AudioContext, no MediaRecorder), but the abort discipline matters — a leaked LLM call costs tokens.

### D-T389-8. Telemetry — lifecycle + turn events

- `ai-chat-clip.mount.start` (attributes: `family: 'ai-chat'`, provider, model)
- `ai-chat-clip.mount.success`
- `ai-chat-clip.mount.failure` (reasons: `'invalid-props'`, `'provider-unavailable'`, `'permission-denied'`)
- `ai-chat-clip.turn.started` (attribute: turnId, userMessageLength — NOT body per privacy)
- `ai-chat-clip.turn.finished` (attribute: turnId, durationMs, tokenCount, finishReason)
- `ai-chat-clip.turn.error` (attribute: turnId, errorKind from `classifyError`)
- `ai-chat-clip.dispose`

**Privacy posture**: neither user-message bodies nor assistant-completion bodies appear in telemetry attributes. Pin via test (AC #15).

### D-T389-9. `componentRef.module` value + factory layout

- Subpath export `./clips/ai-chat` added to `runtimes-interactive/package.json`.
- Factory at `clips/ai-chat/index.ts`; side-effect registers `'ai-chat'` family.
- Top-level functions (per T-309a's tightening).

### D-T389-10. Pattern-evaluation — third application

T-383 + T-384 = wrap-runtime pattern; T-387 + T-389 = standalone-event-driven pattern. T-389 is the **third** standalone-pattern application (after T-387 and the T-388 static-fallback half). Re-evaluate per CLAUDE.md "three similar lines beat a premature abstraction":

- **Convention overlap (T-387 ↔ T-389)**: schema discriminator file (5 lines), subpath export (4 lines), side-effect register (1 line), telemetry event naming convention. **Conventions, not duplication.**
- **Logic overlap**: factory skeleton (state machine, dispose, abort wiring) is structurally similar but uses different primitives (MediaRecorder vs. LLM stream). Inlining is shorter and clearer than abstracting.
- **Provider-seam pattern**: `TranscriptionProvider` (T-387) and `LLMChatProvider` (T-389) share a shape (interface + start/stream + tests via in-memory impl). This IS three similar lines of meaningful logic IF a third provider seam emerges (T-391 LiveDataClip's data-fetch seam likely). Defer extraction until T-391; document the eligibility in the skill.

**Decision**: no abstraction extraction at T-389. Skill update notes the eligibility for a future `ProviderSeam<T>` extraction at T-391.

## Acceptance criteria

### Schema

1. `aiChatClipPropsSchema` parses a valid payload (systemPrompt, provider, model, maxTokens, temperature, multiTurn, posterFrame).
2. Empty `systemPrompt` → schema error.
3. `temperature` out of `[0, 1.5]` → schema error.
4. `maxTokens` ≤ 0 → schema error.
5. `check-preset-integrity` rejects `family: 'ai-chat'` with bad props; accepts valid fixture.

### Factory + registry

6. `interactiveClipRegistry.resolve('ai-chat')` returns the factory after side-effect import.
7. Re-importing throws `InteractiveClipFamilyAlreadyRegisteredError`.
8. Mount with `MountContext.frameSource: undefined` succeeds — no `MissingFrameSourceError`.
9. Mount renders the React tree (output + textarea + button data attributes).

### Turn lifecycle

10. `send(userMessage)` calls the provider's `streamTurn` with the resolved system prompt + accumulated history + new message.
11. With multiple `onTurn` subscribers, all receive `assistant-token` events in order; final event is `assistant-final` with the full text.
12. Provider error → `onTurn` receives `error` event; `send` rejects with the same error wrapped.
13. `multiTurn: false` rejects a second `send` call with a typed `MultiTurnDisabledError`; pinned via test.
14. `reset()` drops history; subsequent `send` calls send no prior messages to the provider.

### Resource cleanup

15. `MountHandle.dispose()` calls `signal.abort()` on the in-flight call (verified via spy on the AbortController).
16. `MountHandle.dispose()` drops history (verified by re-reading internal state via a test seam).
17. `MountHandle.dispose()` unsubscribes all `onTurn` handlers.
18. `MountHandle.dispose()` is idempotent.
19. `signal.abort()` from the runtime triggers the same teardown.

### Telemetry

20. Each documented event (D-T389-8) is emitted with the documented attribute shape, observed via spy. **Neither user-message body nor assistant-completion body appears in any attribute**. Pin via grep on captured event payloads in the test.

### Permission posture

21. Mount with `permissions: ['network']` and a permissive permission shim succeeds; the shim's `network` no-op grant fires without any user-visible dialog.
