# T-251 — Schema: deck-level layouts + masters + `inheritsFrom`

**Branch**: `task/T-251-schema-layouts-masters`

## Goal

Lift the `@stageflip/schema` `Document` shape to model PPTX / Google Slides' three-tier inheritance chain (Master → Layout → Slide) so that round-trip imports preserve placeholder-driven theme inheritance instead of flattening every slide to fully-materialized geometry. Three additions:

1. `Document.layouts: SlideLayout[]` and `Document.masters: SlideMaster[]` — deck-level template stores.
2. `Slide.layoutId: string | undefined` — slide → layout pointer.
3. `ElementBase.inheritsFrom: { templateId: string; placeholderIdx: number } | undefined` — per-element placeholder reference.

At render time, the RIR compiler walks the inheritance chain: missing or `undefined` element fields are filled from the matching `placeholderIdx` on the slide's layout (or its parent master) before any further passes run. Slide-level field values always override template values.

**Must land before T-244 emits flat geometry**, otherwise the Google Slides import ships lossy and we have to migrate the document store.

## Architectural decisions

### Naming: separate `layouts` and `masters`, not unified `slideTemplates`

The plan v1.23 amendment offered "(or unified `slideTemplates`)". Decision: **separate** them. Reasoning:

- PPTX (`<p:sldLayout>` / `<p:sldMaster>`) and Google Slides (`pageType: LAYOUT` / `pageType: MASTER`) both model the two tiers separately. A unified `slideTemplates` type would force both importers to either drop the distinction (lossy round-trip) or carry a discriminator we'd later wish was a separate type.
- Inheritance direction is asymmetric: a layout points at a master (`SlideLayout.masterId`); slides point at layouts. A unified type would have to carry an optional `parentTemplateId` field that means different things at different levels — confusing.
- Editors care about the distinction: "edit master" and "edit layout" are different UI affordances. The schema should not collapse them.

### Schema additions

#### New file: `packages/schema/src/templates.ts`

```ts
import { z } from 'zod';
import { elementSchema } from './elements/index.js';
import { idSchema } from './primitives.js';
import { themeSchema } from './theme.js';

/**
 * SlideMaster — top-level template. One per "design system" the deck uses;
 * a deck typically has one master with several layouts under it. Masters
 * own placeholder elements that layouts (and ultimately slides) extend
 * via `inheritsFrom`.
 *
 * `theme` is an optional override on `Document.theme` — masters can rebrand
 * a subset of slides. Future work adds the theme-resolution pass; T-251
 * carries the field but does not implement override application.
 */
export const slideMasterSchema = z
  .object({
    id: idSchema,
    name: z.string().min(1).max(200),
    theme: themeSchema.partial().optional(),
    placeholders: z.array(elementSchema).default([]),
  })
  .strict();
export type SlideMaster = z.infer<typeof slideMasterSchema>;

/**
 * SlideLayout — second-tier template that extends one master. Layouts own
 * placeholder elements that slides extend via `inheritsFrom`. A layout's
 * placeholders inherit transitively from the master at render time.
 */
export const slideLayoutSchema = z
  .object({
    id: idSchema,
    name: z.string().min(1).max(200),
    masterId: idSchema,
    placeholders: z.array(elementSchema).default([]),
  })
  .strict();
export type SlideLayout = z.infer<typeof slideLayoutSchema>;
```

#### Modified: `packages/schema/src/elements/base.ts`

```ts
export const elementInheritsFromSchema = z
  .object({
    /** id of a SlideLayout or SlideMaster in `Document.layouts` / `Document.masters`. */
    templateId: idSchema,
    /**
     * Placeholder index. PPTX `<p:ph idx="N"/>`, Slides `placeholder.index`.
     * Multiple elements on the same template can share `placeholderIdx` if
     * the template defines a positional family (e.g., "content" placeholder

  })

export const elementBaseSchema = z
  .object({
    id: idSchema,
    name: z.string().min(1).max(200).optional(),
    transform: transformSchema,                          // stays required for now (see "Inheritance application" below)
    visible: z.boolean().default(true),
    locked: z.boolean().default(false),
    animations: z.array(animationSchema).default([]),
    inheritsFrom: elementInheritsFromSchema.optional(),  // NEW
  })
  .strict();
```

### Inheritance application: render-time, not parse-time

The schema validates that `transform`, `visible`, etc. are present on every element — even those that inherit from a placeholder. **Materialization happens at RIR compile time**, not at parse time. A new `applyInheritance` pass runs **before** `lower-tree` (which assigns z-index) and **before** `theme-resolve`:

```
parse → applyInheritance → resolveThemeRefs → resolveVariables →
expandComponents → resolveBindings → aggregateFonts → lowerTree →
finalize
```

The pass walks each slide:
1. If `slide.layoutId` is undefined, no-op for that slide.
2. Else look up `layouts[layoutId]`. If not found, drop a `LF-RIR-LAYOUT-NOT-FOUND` diagnostic (new `RIRDiagnosticCode`) and continue.
3. For each `element` in the slide whose `inheritsFrom.templateId` matches the layout id (or the layout's `masterId`):
   a. Find the matching placeholder by `placeholderIdx` on the layout, OR (transitive) on the master.
   b. **Slide overrides win**: any field set on the slide-side element keeps its value; missing or `undefined` fields are filled from the placeholder.
   c. The applied element retains the slide-side `id` (not the placeholder's). `inheritsFrom` is preserved so the editor can show "this element inherits from X".

`transform` is treated specially: if the slide-side element's `transform` is **structurally equal** to the placeholder's, it's tagged in RIR meta as "default-from-template" so the editor can show "moved" / "resized" badges later. Equality compares `position`, `size`, `rotation`, `scale`, `opacity`, `anchor` numerically (deep equal, no float epsilon for now — fixtures use round numbers).

### Pre-flight audits (call out in PR body)

T-251 confirms these via spec authoring; verifies during implementation:

1. **`GroupElement` preservation in `@stageflip/import-pptx`**: confirmed — `packages/schema/src/elements/index.ts:25-30` defines `GroupElement` with `children: Element[]` and the union at line 32-43 includes it. The PPTX importer's `walkSpTree` already emits `ParsedGroupElement` (per `packages/import-pptx/src/parts/sp-tree.ts`), and T-241a's accumulator preserves the nested structure (handover §1). **No flattening to leaves**; T-244's group-handling can rely on `GroupElement` being canonical.

2. **`TableElement` merged-cell representation**: confirmed — `packages/schema/src/elements/table.ts:14-19` defines `TableCell` with `colspan: number` and `rowspan: number` fields (defaults 1). PPTX `<a:gridSpan>` / `<a:rowSpan>` / `<a:vMerge>` and Slides `MergeTableCellsRequest` both map cleanly: `colspan = max(1, gridSpan)`, `rowspan = max(1, rowSpan)`; `vMerge="continue"` cells are absent from the cells array (covered by the parent's `rowspan`). T-244's table-handling can rely on the existing schema.

Both audits ship as documentation-only commits in this PR; no code changes from the audits.

### RIR compiler dispatch

`packages/rir/src/compile/index.ts` orchestrates the pass list. T-251 inserts `applyInheritance` at the front of the list. The new pass:

- Takes `(Document, DiagnosticSink)` and returns a `Document` with placeholders applied to slide elements.
- Is pure (no side effects, no I/O).
- Returns the input unchanged when `Document.layouts` is empty (the common-case fast path for documents that don't use templates).

### Renderer-core element dispatcher

`packages/renderer-core/src/clips/**` consume RIR elements. Because `applyInheritance` runs at compile time, the renderer sees fully-materialized elements — **no dispatcher changes required**. The renderer remains template-agnostic; this is a deliberate keep-renderers-simple choice.

### Editor-shell consumer

`packages/editor-shell` reads `documentAtom`. The new optional fields (`layouts`, `masters`, `layoutId`, `inheritsFrom`) are typed correctly via `z.infer`; consumers that don't care about templates ignore them. **No behavioral change** for the existing editor surface; future template-editing UI lands in a follow-up app PR.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### Schema layer

1. `documentSchema.parse(...)` accepts a document with `masters: []` and `layouts: []` (the back-compat default case) — no failure on existing documents.
2. `documentSchema.parse(...)` accepts a document with one `SlideMaster` containing two placeholder `TextElement`s, one `SlideLayout` extending that master, and one slide pointing at the layout.
3. `documentSchema.parse(...)` rejects a `SlideLayout` whose `masterId` does not match any `SlideMaster.id` in the same document. (Implementation: a `.refine` on `documentSchema` that walks `layouts` and asserts every `masterId` resolves.)
4. `documentSchema.parse(...)` accepts an `ElementBase` with no `inheritsFrom`; existing fixtures pass unchanged. **Pin no fixture mutation** in `roundtrip.test.ts`.
5. `documentSchema.parse(...)` accepts an `ElementBase` with `inheritsFrom: { templateId: '...', placeholderIdx: 0 }` and round-trips through `JSON.stringify` / `JSON.parse` byte-identical.
6. `SlideLayout.placeholders` and `SlideMaster.placeholders` accept any `Element` (text, image, shape, etc.); placeholders are full-shape `Element`s, not a separate placeholder type. **Pin via test** that an `ImageElement` is a valid placeholder.
7. `Slide.layoutId` is optional; existing slides without `layoutId` continue to parse.

### RIR `applyInheritance` pass

8. With `Document.layouts === []`, the pass is a no-op: input document is returned unchanged (reference equality acceptable as a fast path; structural equality required).
9. A slide with `layoutId` pointing to a layout that doesn't exist drops a `LF-RIR-LAYOUT-NOT-FOUND` diagnostic and the slide elements pass through unchanged.
10. An element with `inheritsFrom: { templateId, placeholderIdx }` whose `templateId` resolves but `placeholderIdx` doesn't match any placeholder drops a `LF-RIR-PLACEHOLDER-NOT-FOUND` diagnostic and the element passes through unchanged.
11. **Slide override semantics**: a slide-side element with `transform.position = (100, 200)` whose placeholder has `transform.position = (50, 50)` keeps `(100, 200)` post-pass. (Slide wins; placeholder fills nothing because the field was set.)
12. **Placeholder fill semantics**: a slide-side element where `name` is unset (omitted) inherits `name` from the placeholder. (Placeholder fills missing field.)
13. **Transitive inheritance via master**: a slide whose layout has no matching `placeholderIdx` walks up to the layout's `masterId` master and looks for the placeholder there. Pin via fixture: master defines placeholder #5, layout doesn't, slide element with `inheritsFrom.templateId = layoutId` and `placeholderIdx = 5` inherits from master.
14. **Slide-side `id` preservation**: the applied element retains the slide-side `id`; the placeholder's `id` is discarded.
15. **Pass purity**: the pass is deterministic — same input → same output across 1000 invocations. Pin via Vitest test.
16. The pass output passes `documentSchema.parse(...)` (i.e., applying inheritance does not produce an invalid document).

### Pass-order integration

17. The pass list in `compileRIR` runs `applyInheritance` **before** `resolveThemeRefs` (so theme tokens on placeholder values resolve correctly post-inheritance) and **before** `lower-tree` (so z-index assignment sees the materialized element list). Pin the order via a test that snapshots the pass-name list.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- The "structural equality" check on `transform` for the "default-from-template" tag (per §"Inheritance application") turns out to need float-epsilon comparison because real-world fixtures carry rounding noise. Spec assumes round numbers; if a fixture violates this, escalate before fudging.
- Adding `inheritsFrom` to `elementBaseSchema` cascades into a TypeScript inference regression in any element subtype's `.merge(...)` chain. The Zod merge pattern is brittle on optional fields with `.strict()`; if a subtype fails to compile, escalate before downgrading `.strict()`.
- The `.refine` on `documentSchema` (AC #3, asserting `SlideLayout.masterId` resolves) fires false positives on existing fixtures because they have an empty `masters: []` and a non-empty `layouts: []`. Spec assumes both arrays are populated together or both empty; escalate if a real document violates this.
- The pass-order integration test (AC #17) fails because some existing pass mutates fields the new pass should fill. Investigate before reordering — the placement at the front of the list is intentional.
- A real-world PPTX fixture (post-T-244) surfaces a placeholder shape kind that the existing `Element` union can't represent (e.g., a "footer" placeholder with no `transform`). Escalate; element-shape changes are out of scope.
- Coverage on the new pass (AC #23) is hard to reach ≥90% because some branches are only triggered by malformed input. Add fixtures that synthesize the malformed case rather than lowering the bar.
