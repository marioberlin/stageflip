# T-251 — Schema: deck-level layouts + masters + `inheritsFrom`

**Branch**: `task/T-251-schema-layouts-masters`

## Goal

Lift the `@stageflip/schema` `Document` shape to model PPTX / Google Slides' three-tier inheritance chain (Master → Layout → Slide) so that round-trip imports preserve placeholder-driven theme inheritance instead of flattening every slide to fully-materialized geometry. Three additions:

1. `Document.layouts: SlideLayout[]` and `Document.masters: SlideMaster[]` — deck-level template stores.
2. `Slide.layoutId: string | undefined` — slide → layout pointer.
3. `ElementBase.inheritsFrom: { templateId: string; placeholderIdx: number } | undefined` — per-element placeholder reference.

At render time, the RIR compiler walks the inheritance chain: missing or `undefined` element fields are filled from the matching `placeholderIdx` on the slide's layout (or its parent master) before any further passes run. Slide-level field values always override template values.

**Must land before T-244 emits flat geometry**, otherwise the Google Slides import ships lossy and we have to migrate the document store.

## Architectural decisions

### Naming: separate `layouts` and `masters`, not unified `slideTemplates`

The plan v1.23 amendment offered "(or unified `slideTemplates`)". Decision: **separate** them. Reasoning:

- PPTX (`<p:sldLayout>` / `<p:sldMaster>`) and Google Slides (`pageType: LAYOUT` / `pageType: MASTER`) both model the two tiers separately. A unified `slideTemplates` type would force both importers to either drop the distinction (lossy round-trip) or carry a discriminator we'd later wish was a separate type.
- Inheritance direction is asymmetric: a layout points at a master (`SlideLayout.masterId`); slides point at layouts. A unified type would have to carry an optional `parentTemplateId` field that means different things at different levels — confusing.
- Editors care about the distinction: "edit master" and "edit layout" are different UI affordances. The schema should not collapse them.

### Schema additions

#### New file: `packages/schema/src/templates.ts`

```ts
import { z } from 'zod';
import { elementSchema } from './elements/index.js';
import { idSchema } from './primitives.js';
import { themeSchema } from './theme.js';

/**
 * SlideMaster — top-level template. One per "design system" the deck uses;
 * a deck typically has one master with several layouts under it. Masters
 * own placeholder elements that layouts (and ultimately slides) extend
 * via `inheritsFrom`.
 *
 * `theme` is an optional override on `Document.theme` — masters can rebrand
 * a subset of slides. Future work adds the theme-resolution pass; T-251
 * carries the field but does not implement override application.
 */
export const slideMasterSchema = z
  .object({
    id: idSchema,
    name: z.string().min(1).max(200),
    theme: themeSchema.partial().optional(),
    placeholders: z.array(elementSchema).default([]),
  })
  .strict();
export type SlideMaster = z.infer<typeof slideMasterSchema>;

/**
 * SlideLayout — second-tier template that extends one master. Layouts own
 * placeholder elements that slides extend via `inheritsFrom`. A layout's
 * placeholders inherit transitively from the master at render time.
 */
export const slideLayoutSchema = z
  .object({
    id: idSchema,
    name: z.string().min(1).max(200),
    masterId: idSchema,
    placeholders: z.array(elementSchema).default([]),
  })
  .strict();
export type SlideLayout = z.infer<typeof slideLayoutSchema>;
```

### Inheritance application: pure helper in `@stageflip/schema`, called from BOTH RIR and editor

The schema validates that `transform`, `visible`, etc. are present on every element — even those that inherit from a placeholder. **Materialization happens via a single pure function `applyInheritance(doc: Document): Document` exported from `@stageflip/schema`** (new file `packages/schema/src/inheritance.ts`). Two consumers call it:

1. **RIR compiler**: a thin pass `applyInheritance` at the front of the pass list (before `theme-resolve` / `variable-resolve` / `component-expand` / `binding-resolve` / `font-aggregate` / `lower-tree` / `finalize`) is a one-line wrapper around the schema-level function plus diagnostic emission.
2. **Editor canvas**: `apps/stageflip-slide/src/components/canvas/slide-canvas.tsx` (which reads `slide.elements` directly today, **without** calling `compileRIR`) consumes a new `materializedDocumentAtom` derived atom in `@stageflip/editor-shell` that runs `applyInheritance(documentAtom)` on read. The canvas continues to read element fields directly; it just sees materialized values.

**Why a schema-level helper, not a RIR-only pass**: editor-shell + apps/stageflip-slide never invoke `compileRIR`; if inheritance lived only in RIR, the editor canvas would render slides whose `inheritsFrom` elements showed only their slide-side fields (placeholder fills missing). One source of truth in `@stageflip/schema` ensures both paths agree.

```
parse → applyInheritance (schema helper) → editor canvas reads
                       └→ RIR pass applyInheritance → resolveThemeRefs →
                          resolveVariables → expandComponents →
                          resolveBindings → aggregateFonts → lowerTree →
                          finalize
```

The helper walks each slide:
1. If `slide.layoutId` is undefined, no-op for that slide.
2. Else look up `layouts[layoutId]`. If not found, the helper passes the slide through unchanged. **Diagnostic emission is the RIR pass's job** (the schema helper has no `DiagnosticSink`); the editor surface simply renders the unmaterialized state. The RIR pass calls `applyInheritance` then walks the same template-resolution to emit `LF-RIR-LAYOUT-NOT-FOUND` / `LF-RIR-PLACEHOLDER-NOT-FOUND` for the missing references.
3. For each `element` in the slide whose `inheritsFrom.templateId` matches the layout id (or the layout's `masterId`):
   a. Find the matching placeholder by `placeholderIdx` on the layout, OR (transitive) on the master.
   b. **Slide overrides win at the top-level field granularity** (see "Override granularity" below).
   c. The applied element retains the slide-side `id` (not the placeholder's). `inheritsFrom` is preserved so the editor can show "this element inherits from X".

#### Override granularity

The override is **shallow on `ElementBase`'s top-level keys** (`name`, `transform`, `visible`, `locked`, `animations`, plus the type-specific content fields like `text`, `src`, `shape`, `path`). Nested fields inside `transform` (`position`, `size`, `rotation`, `scale`, `opacity`, `anchor`) are **NOT** field-granular: if the slide-side `transform` differs from the placeholder's at the object level (any nested field differs), the entire slide-side `transform` wins. This keeps the override rule simple and predictable; partial-transform overrides are out of scope for v1.

**`transform` "default-from-template" tag**: if the slide-side `transform` is **structurally equal** to the placeholder's, the editor renders a "default" badge so users can see which elements still match the template. Equality is numeric deep-equal (no float epsilon — fixtures use round numbers). Implemented in the editor canvas, not the schema helper.

**`animations: []` semantics**: Zod's `.default([])` materializes an empty array on every parse, so `animations: []` is **always considered "set"** at the slide level and is **never overridden** by a placeholder's animations. If a fixture surfaces a real need to inherit animations from a placeholder, lift the field's default to `undefined` in a follow-up — out of scope here.

### RIR compiler dispatch

`packages/rir/src/compile/index.ts` orchestrates the pass list (passes live in `packages/rir/src/compile/passes.ts` — **a single flat file, not a `passes/` directory**; T-251 adds the new pass into that file alongside the existing exports).

T-251 inserts `applyInheritance` at the front of the pass list:

- Takes `(Document, DiagnosticSink)` and returns a `Document` with placeholders applied to slide elements.
- Calls the schema-level `applyInheritance(doc)` for the actual materialization, then walks the slides a second time to emit `LF-RIR-LAYOUT-NOT-FOUND` / `LF-RIR-PLACEHOLDER-NOT-FOUND` diagnostics for unresolved references.
- Is pure (no side effects, no I/O).
- Returns the input unchanged when `Document.layouts` is empty (the common-case fast path for documents that don't use templates).

**`compilerDiagnosticSchema` extension** (`packages/rir/src/types.ts:354-372`): the `pass` enum is closed and currently lists `'theme-resolve' | 'variable-resolve' | 'component-expand' | 'binding-resolve' | 'timing-flatten' | 'stacking-context' | 'font-aggregate' | 'validate'`. T-251 **adds `'apply-inheritance'`** to this enum. The `code` field stays `z.string()` (open) — there is no `RIRDiagnosticCode` literal-union type to extend. The new diagnostic codes (`LF-RIR-LAYOUT-NOT-FOUND`, `LF-RIR-PLACEHOLDER-NOT-FOUND`) are documented inline in `passes.ts` next to the emission sites; no type-level extension required.

### Renderer-core element dispatcher

`packages/renderer-core/src/clips/**` consume RIR elements. Because `applyInheritance` runs at compile time as part of `compileRIR`, the renderer sees fully-materialized elements — **no dispatcher changes required**. The renderer remains template-agnostic; this is a deliberate keep-renderers-simple choice.

### Editor-shell consumer (NOT a no-op)

`packages/editor-shell` reads `documentAtom`. The new optional fields (`layouts`, `masters`, `layoutId`, `inheritsFrom`) are typed correctly via `z.infer`. **However**: the editor canvas (`apps/stageflip-slide/src/components/canvas/slide-canvas.tsx:125`) reads `slide.elements` directly without invoking `compileRIR`. A naive ship would render slides with `inheritsFrom` elements as their slide-side fields only, missing the placeholder fills.

**Fix in this PR**: add a derived atom `materializedDocumentAtom` to `packages/editor-shell/src/atoms/document.ts` (or a sibling file) that wraps `documentAtom` through the schema-level `applyInheritance(doc)` helper. The canvas migrates from reading `documentAtom` to reading `materializedDocumentAtom`. Existing slide reads that don't care about inheritance continue to work because `applyInheritance` is a no-op on documents with empty `layouts`. The `documentAtom` itself is unchanged — writes still go through it; only reads from rendering surfaces consume the materialized variant.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### Schema layer

1. `documentSchema.parse(...)` accepts a document with `masters: []` and `layouts: []` (the back-compat default case) — no failure on existing documents.
2. `documentSchema.parse(...)` accepts a document with one `SlideMaster` containing two placeholder `TextElement`s, one `SlideLayout` extending that master, and one slide pointing at the layout.
3. **No `.refine` on `documentSchema` for `masterId` resolution** — keep the parse path tolerant. Mismatched `masterId` references emit a compile-time `LF-RIR-LAYOUT-NOT-FOUND` diagnostic from the `applyInheritance` pass instead. Reasoning: persisted documents with stale references must continue to load, otherwise a small data-quality drift becomes a hard load failure.
4. `documentSchema.parse(...)` accepts an `ElementBase` with no `inheritsFrom`; existing fixtures pass unchanged. **Pin no fixture mutation** in `roundtrip.test.ts`.
5. `documentSchema.parse(...)` accepts an `ElementBase` with `inheritsFrom: { templateId: '...', placeholderIdx: 0 }` and round-trips through `JSON.stringify` / `JSON.parse` byte-identical.
6. `SlideLayout.placeholders` and `SlideMaster.placeholders` accept any `Element` (text, image, shape, etc.); placeholders are full-shape `Element`s, not a separate placeholder type. **Pin via test** that an `ImageElement` is a valid placeholder.
7. `Slide.layoutId` is optional; existing slides without `layoutId` continue to parse.

### Schema-level `applyInheritance(doc): Document` helper

8. With `Document.layouts === []`, `applyInheritance(doc)` returns the input unchanged (structural equality required; reference equality acceptable as a fast path).
9. **Slide override semantics — top-level field granularity**: a slide-side element with any `transform` set (always true under Zod's `.default()`) keeps its full `transform`. The placeholder's `transform` is **never** copied wholesale or merged at the nested-field level.
10. **Placeholder fill semantics**: a slide-side element with `name` unset (omitted from JSON, so resulting in `undefined` after parse) inherits `name` from the placeholder. Fields with explicit values (including empty strings, `false`, `0`, etc.) are NOT overridden.
11. **Transitive inheritance via master**: a slide whose layout has no matching `placeholderIdx` walks up to the layout's `masterId` master and looks for the placeholder there. Pin via fixture: master defines placeholder `idx=5`, layout doesn't, slide element with `inheritsFrom.templateId = layoutId` and `placeholderIdx = 5` inherits from master.
12. **Slide-side `id` preservation**: the applied element retains the slide-side `id`; the placeholder's `id` is discarded.
13. **`animations: []` semantics**: Zod's `.default([])` means the slide always has `animations: []` even if author omitted. Treat as "set" — `applyInheritance` never copies the placeholder's `animations`. (If a fixture surfaces a real need to inherit, lift the default to `undefined` in a follow-up; out of scope here.)
14. **Helper purity**: `applyInheritance(doc)` is deterministic — same input → same output across 1000 invocations. Pin via Vitest test.
15. The helper's output passes `documentSchema.parse(...)` (i.e., applying inheritance does not produce an invalid document).

### RIR `applyInheritance` pass

16. The RIR pass is a thin wrapper around the schema helper: it calls `applyInheritance(doc)` then walks the slides a second time to emit diagnostics for unresolved references.
17. A slide with `layoutId` pointing to a layout that doesn't exist emits one `LF-RIR-LAYOUT-NOT-FOUND` diagnostic with `pass: 'apply-inheritance'`. The slide elements pass through unchanged (the schema helper made them no-ops; the pass just records the issue).
18. An element with `inheritsFrom: { templateId, placeholderIdx }` whose `templateId` resolves but `placeholderIdx` doesn't match any placeholder (on the layout OR transitively on the master) emits one `LF-RIR-PLACEHOLDER-NOT-FOUND` diagnostic with `pass: 'apply-inheritance'`.
19. **Pass-order placement**: the pass list in `compileRIR` runs `applyInheritance` **before** `theme-resolve` (so theme tokens on placeholder values resolve correctly post-inheritance) and **before** `lower-tree` (so z-index assignment sees the materialized element list). Currently also runs **before** `expandComponents` (the existing pass is a stub; if T-249 adds component bodies that themselves use `inheritsFrom`, the order may need to flip — flag in PR body so future Implementer is aware). Pin the order via a test that snapshots the pass-name list.
20. **`compilerDiagnosticSchema.pass` enum extension**: `packages/rir/src/types.ts:360-369` adds `'apply-inheritance'` to the closed enum. (No `RIRDiagnosticCode` literal-union exists; `code` stays `z.string()`.)
