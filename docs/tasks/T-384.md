# T-384 — `ThreeSceneClip` frontier-tier primitive

**Branch**: `task/T-384-three-scene-clip`

## Goal

Ship the second frontier-tier clip family: `ThreeSceneClip`. A schema-validated `InteractiveClip` with `family: 'three-scene'`, registered in `interactiveClipRegistry`, mountable via the `ClipFactory` contract from T-306. Reuses `@stageflip/runtimes-three`'s `ThreeClipHost` so `liveMount` and `staticFallback` poster generation share a single rendering core (ADR-005 §D2 convergence by construction). Adds two primitives required by ADR-005 §D2's literal text: a **seeded PRNG** authors use in place of `Math.random()`, and a **mount-scoped `requestAnimationFrame` shim** that retargets rAF callbacks to the FrameSource clock.

T-384 replicates T-383's reuse-the-existing-runtime pattern. The pattern, first set in PR #268 + #270 (T-383 + T-309a), is now structural: every γ-core family wraps its existing §3 runtime rather than greenfielding a parallel implementation.

## Architectural decisions

### D-T384-1. Reuse `@stageflip/runtimes-three` as the rendering primitive

Same rationale as T-383 D-T383-1, applied to Three.js. The existing `runtimes-three` package is already frame-driven (no `renderer.setAnimationLoop`, no rAF inside the host), three-agnostic in its public types (authors bring their own THREE instance), and licensed-clean (three.js is MIT, in the whitelist). Two implementations would push convergence to code review.

### D-T384-2. `runtimes-interactive` adds `@stageflip/runtimes-three` as a workspace dependency

```json
"@stageflip/runtimes-three": "workspace:*"
```

The factory imports `ThreeClipHost` directly. If `ThreeClipHost` is not currently exported from `runtimes-three/src/index.ts`, T-384 adds a one-line public re-export (no behavioural change; patch bump on `@stageflip/runtimes-three`).

### D-T384-3. Schema narrowing — typed `liveMount.props` for `family: 'three-scene'`

Mirror T-383's discriminator pattern. New file:

```ts
// packages/schema/src/clips/interactive/three-scene-props.ts (NEW)
export const threeSceneClipPropsSchema = z.object({
  /**
   * Module ref to the author's `setup` callback. Same `<package>#<Symbol>`
   * shape as `componentRefSchema` from T-305 — at mount time the runtime
   * dynamic-imports and resolves the named symbol, expecting a
   * `ThreeClipSetup<P>` from `@stageflip/runtimes-three`.
   */
  setupRef: componentRefSchema,
  /** Canvas dimensions in CSS pixels. */
  width: z.number().int().positive(),
  height: z.number().int().positive(),
  /** Author-supplied props forwarded to setup + render. Schema-opaque at this layer. */
  setupProps: z.record(z.unknown()).default({}),
  /** Frame at which `staticFallback` poster should be sampled. Default 0. */
  posterFrame: z.number().int().nonnegative().default(0),
  /** PRNG seed for the ThreeSceneClip's seeded random — D-T384-5. */
  prngSeed: z.number().int().nonnegative().default(0),
}).strict();
```

Browser-safe (pure Zod). Lives at `packages/schema/src/clips/interactive/three-scene-props.ts` next to T-383's `shader-props.ts`. `check-preset-integrity` extends to parse `liveMount.props` against this schema when `family === 'three-scene'`.

The `setupRef` indirection is a notable departure from T-383 (which inlined GLSL as a string). Three.js scenes are imperative JavaScript; we cannot serialise `setup`. The runtime resolves the ref via dynamic import. This is the first preset-side use of `componentRefSchema` for a non-React-component import.

### D-T384-4. `staticFallback` posture — author-supplied poster as `ImageElement`

Same approach as T-383 D-T383-4. The `staticFallback: Element[]` is a single `ImageElement` with `src` pointing to a poster pre-rendered offline at `posterFrame` using the same `setup` and PRNG seed. T-384 tests convergence in-test by mounting both paths and comparing recorded GL state.

### D-T384-5. Seeded PRNG for author code

Per ADR-005 §D2, the wrapper supplies authors a seeded PRNG. T-384 ships a small deterministic PRNG (xorshift32 or sfc32; pick one and document) seeded from `props.prngSeed`. The PRNG is passed into the author's `setup({container, width, height, props, prng})`.

```ts
// packages/runtimes/interactive/src/clips/three-scene/prng.ts (NEW)
export interface SeededPRNG {
  /** Returns a deterministic float in [0, 1). */
  random(): number;
  /** Reset to the original seed. Useful for replay scenarios. */
  reset(): void;
}

export function createSeededPRNG(seed: number): SeededPRNG { ... }
```

Authors writing path-matched setup callbacks at `clips/three-scene/**` MUST use `prng.random()` rather than `Math.random()` — `Math.random()` is in T-309's forbidden-API set and the path-based check (T-309a-tightened) catches it. This is the **opt-in determinism path**: authors choose deterministic randomness; the PRNG seed travels with the preset and produces the same output every render.

### D-T384-6. Mount-scoped `requestAnimationFrame` shim

Per ADR-005 §D2 literal text: "The wrapper overrides `requestAnimationFrame` with a frame-driven scheduler."

Authors using third-party libraries (animation tweens, post-processing chains) inside their `setup` callback may unknowingly invoke `window.requestAnimationFrame`. The shim retargets these calls to the FrameSource clock so every animation step is frame-deterministic.

```ts
// packages/runtimes/interactive/src/clips/three-scene/raf-shim.ts (NEW)
export function installRAFShim(frameSource: FrameSource): { uninstall: () => void } {
  const originalRAF = window.requestAnimationFrame;
  const originalCAF = window.cancelAnimationFrame;
  let nextHandle = 1;
  const pending = new Map<number, (frame: number) => void>();

  window.requestAnimationFrame = (cb): number => {
    const handle = nextHandle++;
    pending.set(handle, cb);
    // Fire on the next FrameSource tick (one shot per RAF semantics).
    const unsub = frameSource.subscribe((frame) => {
      const fn = pending.get(handle);
      if (fn !== undefined) {
        pending.delete(handle);
        fn(frame); // Standard rAF passes a timestamp (DOMHighResTimeStamp); we pass `frame` instead.
        unsub();
      }
    });
    return handle;
  };

  window.cancelAnimationFrame = (handle): void => {
    pending.delete(handle);
  };

  return {
    uninstall: () => {
      window.requestAnimationFrame = originalRAF;
      window.cancelAnimationFrame = originalCAF;
    },
  };
}
```

Caveats explicitly documented in the file header:

1. **Global mutation, scoped lifetime.** The shim mutates `window.rAF`. Multiple concurrent ThreeSceneClip mounts share the shim — last installer wins; uninstall is reverse-order safe via the original-handle save. **Reviewer focus**: confirm the shim handles concurrent mounts.
2. **Timestamp argument is the frame number, not a `DOMHighResTimeStamp`.** Standard rAF signature is `cb(time: DOMHighResTimeStamp)`. The shim breaks that contract; libraries that read the argument as wall-clock will misbehave. Acceptable here because the target is frame-deterministic by construction; libraries that require true wall-clock time are incompatible with the interactive tier and authors must avoid them.
3. **The shim file lives outside `clips/three-scene/**`** (at `packages/runtimes/interactive/src/clips/three-scene/raf-shim.ts` is path-matched — but the shim itself does NOT call any forbidden API; it ONLY captures originals and installs replacements at module top-level). Verify with T-309a-tightened sub-rule output.

Alternative considered + rejected: per-mount `<iframe>` isolation. Cleaner shim semantics (no global mutation) but shipping iframes for shader-equivalent simple cases is over-engineering for v1. Re-evaluate if T-403 surfaces shim-related issues.

## Acceptance criteria

### Schema

1. `threeSceneClipPropsSchema` parses a valid payload (setupRef, dims, setupProps, posterFrame, prngSeed).
2. Invalid `setupRef.module` regex → schema error.
3. Width/height ≤ 0 → schema error.
4. `prngSeed` defaults to 0 if omitted; negative → schema error.
5. `check-preset-integrity` rejects `family: 'three-scene'` with bad props; accepts valid fixture.

### Factory

6. `interactiveClipRegistry.resolve('three-scene')` returns the factory after side-effect import.
7. Re-importing throws `InteractiveClipFamilyAlreadyRegisteredError`.
8. Factory dynamically imports `setupRef.module`, calls the resolved setup with `{container, width, height, props, prng}`, and mounts `ThreeClipHost`.
9. Setup throw → mount fails with `mount.failure` reason `'setup-throw'`; falls back via mount-harness.
10. Unresolvable `setupRef.module` → `mount.failure` reason `'setupRef-resolve'`.
11. `MountHandle.updateProps` propagates without re-mounting (the underlying scene's `render(props)` is called with the new props, not a remount).
12. `MountHandle.dispose()` calls the author's `dispose?()` callback, uninstalls the rAF shim, unsubscribes from the frame source, idempotent.
13. `signal.abort()` triggers same teardown.
14. Missing `frameSource` throws `MissingFrameSourceError` referencing ADR-005 §D2.

### Determinism + sub-rule integration (T-309a-tightened)

15. The factory ships as top-level functions (no static class). `pnpm check-determinism` PASSES; the sub-rule reports the new path-matched functions individually.
16. `prng.ts` exports `createSeededPRNG`; the path-based sub-rule does not flag it (no forbidden API; missing-frame is no longer a violation).
17. The synthetic-source AST checker, invoked on a hypothetical `clips/three-scene/foo.ts` containing `Math.random()` inside a setup callback, returns a violation (regression pin for T-309a coverage on the new path).

### Seeded PRNG

18. `createSeededPRNG(42).random()` returns the same sequence on every invocation across runs / nodes / OSes (deterministic).
19. `prng.reset()` returns to the seed-zero state; subsequent `random()` calls reproduce the original sequence.
20. Two PRNGs with the same seed produce identical sequences; different seeds produce different sequences (within statistical tolerance over N=1000 draws).

### rAF shim

21. After `installRAFShim(frameSource)`, calling `window.requestAnimationFrame(cb)` enqueues `cb`; advancing the frame source triggers `cb(frame)`.
22. `cancelAnimationFrame(handle)` removes the pending callback; advancing does not invoke it.
23. `uninstall()` restores `window.requestAnimationFrame` to the original.
24. Two concurrent `installRAFShim` calls share the shim correctly; uninstalling in reverse order restores the original (LIFO discipline).
25. Argument passed to `cb` is the frame number, not a `DOMHighResTimeStamp`. Documented as a deviation (D-T384-6 caveat 2).

### Convergence (ADR-005 §D2)

26. Running the factory at `frame=30` with PRNG seed=42 against a fixed setup produces a GL call stream identical to `ThreeClipHost` standalone at the same frame + seed. Epsilon = 0 on the call stream. Pixel-level convergence tracked in T-383a.

### Permission posture

27. Mount of `family: 'three-scene'` does not trigger any permission prompt.

### Telemetry

28. Mount emits `start` → `success` → `dispose`. Pin attribute shapes.
29. Setup throw → `failure` with reason `'setup-throw'`.
30. Bad `setupRef.module` → `failure` with reason `'setupRef-resolve'`.

## Escalation triggers (CLAUDE.md §6)

- `ThreeClipHost` requires API changes to be reusable from the interactive tier. Escalate; we may extract a lower-level primitive instead of wrapping.
- Concurrent rAF-shim installs cannot reliably uninstall in reverse without a stack discipline that breaks legitimate use cases. Escalate; we may switch to per-mount iframe isolation (alternative considered + rejected for v1; revisit if needed).
- happy-dom doesn't expose `window.requestAnimationFrame` faithfully enough to test the shim — we'd need jsdom or a real browser. Escalate; the test fixture may need a different runtime.
- `componentRefSchema` regex from T-305 rejects the `@scope/pkg/sub#Symbol` shape used by `setupRef`. Escalate; ADR-003 amendment may be needed.
- Three.js's WebGLRenderer makes calls from inside its constructor that the GL stub doesn't handle. Escalate; the convergence test may need a richer stub.
