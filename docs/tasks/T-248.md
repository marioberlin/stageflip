# T-248 — Loss-flag reporter UI

**Branch**: `task/T-248-loss-flag-reporter`

## Goal

Surface every `LossFlag` an importer emits in the editor UI so the user can:

1. **See at a glance** how many flags an import produced, broken down by severity (info / warn / error).
2. **Browse the full list** in a modal, grouped by severity and category, with the location each flag was raised against.
3. **Navigate** from a flag to the affected slide / element so they can fix or accept the issue inline.
4. **Dismiss** flags individually or in bulk for the current session.

Flags are session-ephemeral (do not persist across reloads); v1 keeps the dismissed-state in memory only. Cross-importer (PPTX / Google Slides / Hyperframes-HTML) is supported by design — the reporter consumes `LossFlag` records directly from `@stageflip/loss-flags` and renders any `source` string.

## Architectural decisions

### State: one atom in `@stageflip/editor-shell`

```ts
// packages/editor-shell/src/atoms/import-loss-flags.ts
import type { LossFlag } from '@stageflip/loss-flags';
import { atom } from 'jotai';

export const importLossFlagsAtom = atom<readonly LossFlag[]>([]);
export const dismissedLossFlagIdsAtom = atom<ReadonlySet<string>>(new Set());

/** Derived: flags minus dismissed ids; sorted severity-desc then source then code. */
export const visibleLossFlagsAtom = atom((get) => {
  const flags = get(importLossFlagsAtom);
  const dismissed = get(dismissedLossFlagIdsAtom);
  return flags
    .filter((f) => !dismissed.has(f.id))
    .slice()
    .sort(compareFlags);
});
```

Why a separate file (not `ui.ts`): mirrors the editor-shell convention of "one concern per atom file" set by `document.ts`, `selection.ts`, `undo.ts`. Future additions (counters, filters) co-locate cleanly.

Why session-ephemeral (`new Set()` reset on reload): aligns with `activeSlideIdAtom`'s session-state precedent. The reporter is a transient diagnostic surface, not document content.

### Reporter component layout

| Component | File | Role |
|---|---|---|
| `<LossFlagBadge>` | `apps/stageflip-slide/src/components/status-bar/loss-flag-badge.tsx` | Status-bar slot. Reads `visibleLossFlagsAtom`; renders an empty button when no flags, otherwise a button with the count + worst-severity color (info=blue, warn=amber, error=red). Click opens the modal. Hidden if no flags exist. |
| `<LossFlagReporter>` | `apps/stageflip-slide/src/components/dialogs/loss-flag-reporter/loss-flag-reporter.tsx` | Modal with the full list. Groups by severity → category. Each row: severity icon, source badge, code, message, location (clickable: navigates to slide/element), dismiss button. Footer: "Dismiss all" button. |
| `<LossFlagRow>` | same dir, `loss-flag-row.tsx` | One flag's row. Pure: receives the `LossFlag` + handlers via props. |

The badge and modal both live in `apps/stageflip-slide/`, not in `editor-shell/`, because the visual chrome is slide-app-specific. Future sibling apps (`stageflip-video`, `stageflip-display`) get their own badge/modal that shares the atom.

Add `<LossFlagBadge />` to `apps/stageflip-slide/src/components/status-bar/status-bar.tsx` after `<span data-testid="status-element-count" …>`. Update the TSDoc to remove "import-diagnostics badge" from the "deferred" list and add it to the "shipped" tense.

### Navigation from flag → location

When the user clicks a flag's location, the reporter:

1. If `flag.location.slideId` is present and matches a slide in the current document, set `activeSlideIdAtom` to that id.
2. If `flag.location.elementId` is present and that element exists on the active slide, set `selectedElementIdAtom` to it.
3. Close the modal.

Use existing setter atoms; do not introduce navigation primitives. If neither id matches (e.g., a layout-level flag), the click is a no-op and the row's location label gets `aria-disabled="true"` + a tooltip explaining why.

## Acceptance criteria

Each gets a Vitest / React Testing Library test, written first and failing.

### Atom layer (in `editor-shell`)

1. `importLossFlagsAtom` exported with default value `[]`.
2. `dismissedLossFlagIdsAtom` exported with default value `new Set()`.
3. `visibleLossFlagsAtom` derived: returns `importLossFlagsAtom \ dismissedLossFlagIdsAtom`, sorted severity-desc (`error` > `warn` > `info`) then by `source` ascending then by `code` ascending.
4. Setting `importLossFlagsAtom` to a new array does not clear `dismissedLossFlagIdsAtom` (re-imports respect prior dismissals — but new flags with new ids appear). If this turns out wrong in real use, change in follow-up.
5. Atoms are pure (no side effects on read); standard for jotai.

### Status-bar badge

6. With `importLossFlagsAtom = []`, the badge renders nothing visible (no DOM node, or `display: none` — Implementer's call).
7. With 3 info flags, the badge shows `3` with the info-severity color.
8. With 1 info + 2 warn flags, the badge shows `3` with the warn-severity color (worst-severity wins).
9. With 1 info + 2 warn + 1 error, the badge shows `4` with the error-severity color.
10. Clicking the badge opens the reporter modal (asserted via `<LossFlagReporter>` test-id appearing in the DOM).
11. The badge has an accessible name (`aria-label` or visible text per RTL `getByRole('button', { name: /…/ })`).

### Reporter modal

12. Empty state: when `visibleLossFlagsAtom = []`, the modal body shows the `lossFlags.modal.empty` string.
13. Non-empty: each visible flag renders one `<LossFlagRow>` with the flag's code, message, source badge, severity icon, and location.
14. Rows are grouped by severity (error → warn → info) with section headers; within each severity group, ordered by category (alpha) then source then code.
15. Per-row dismiss adds the flag's id to `dismissedLossFlagIdsAtom` (RTL test fires the click and asserts the row disappears).
16. "Dismiss all" replaces `dismissedLossFlagIdsAtom` with `new Set(importLossFlagsAtom.map(f => f.id))` (test asserts modal body shows the empty state after click).
17. Clicking a row's location label sets `activeSlideIdAtom` to `flag.location.slideId` (when present + resolvable) and `selectedElementIdAtom` to `flag.location.elementId` (when present + resolvable). Closes the modal.
18. When `flag.location.slideId` does not match any slide in `documentAtom`, the location label is `aria-disabled="true"` (test asserts).
19. The modal traps focus while open; `Escape` closes it. Reuse `<ModalShell>` (the existing precedent in `dialogs/modal-shell`).

### i18n

20. Every user-facing string uses `t('lossFlags.<key>')`. RTL test renders with the catalog and asserts the title text matches `t('lossFlags.modal.title')`. **Bare string literals in JSX are a hard fail** per CLAUDE.md §10.
21. The catalog adds every `lossFlags.*` key referenced by the components. ESLint (Biome) catches `t('not-in-catalog')` calls if the catalog enforces typed keys; if it doesn't today, that's a separate concern.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- The atom shape needs to change after Implementer review (e.g., dismissed-state turns out to need to live next to `importLossFlagsAtom` as a field rather than as a sibling atom). Spec picks separate atoms; if Implementer wants a unified shape, escalate before doing so.
- `<ModalShell>` doesn't trap focus / handle `Escape` (AC #19) and adding it requires more than a one-line change. Escalate; modal a11y is its own task surface.
- The i18n catalog lacks a typed-key narrowing mechanism and the Implementer can't enforce AC #21 mechanically. Acceptable to land without enforcement, but flag in the PR body so a future task adds it.
- Adding the `@stageflip/loss-flags` workspace dep to `apps/stageflip-slide` introduces an unexpected build / bundling regression. Escalate before working around with a re-export.
- A real flag set surfaces a renderable that breaks the layout (e.g., a 5kB `originalSnippet`). Spec assumes flags are short; if a real fixture surfaces big snippets, escalate — UI-truncation logic lands as a follow-up.
