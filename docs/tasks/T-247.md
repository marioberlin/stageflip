# T-247 — `@stageflip/import-hyperframes-html` — bidirectional Hyperframes HTML ↔ canonical (video mode)

**Branch**: `task/T-247-import-hyperframes-html`

## Goal

Convert the empty `@stageflip/import-hyperframes-html` package stub into a bidirectional importer/exporter that maps Hyperframes-style HTML decks to/from canonical `Document` (video mode).

The plan-row v1.24 at `docs/implementation-plan.md:535` says: "parse `data-*` → canonical; reverse direction too." T-247 ships both directions in one PR — the inverse pairing pattern is symmetric and the same DOM-walker code is reused.

**Hyperframes HTML format** (per `reference/hyperframes/`, Apache-2.0):

- A top-level `<div id="master-root" data-composition-id="master" data-width="..." data-height="..." data-duration="..." />` declares deck dimensions + duration (seconds).
- Inside, **composition layers** are `<div data-composition-id="..." data-composition-src="..." data-start="..." data-duration="..." data-track-index="..." />` — each is one timeline track.
- Each composition's `data-composition-src` references a sibling HTML file containing a `<template id="...">` with positioned elements.
- Elements inside compositions are `<div>`s with `class` references and `style="left:..; top:..; width:..; height:.."` inline declarations.
- Inline `<script>` blocks contain GSAP timelines (animations) and inline transcript JSON arrays (captions).

T-247 produces a `Document` with `content.mode: 'video'`, mapping:

- `master-root.data-width / data-height` → `aspectRatio` (or `{kind:'custom', w, h}`).
- `master-root.data-duration` (seconds) → `durationMs` (× 1000).
- composition layers → `Track[]` keyed by `data-composition-id`; `track.kind` derived from a heuristic (per §3 below).
- elements within each composition → `Element[]` per track, with positions derived from inline `style="left:.. top:.."` (CSS class styling is **lossy** in v1 — see Out of scope).
- inline transcript JSON → `CaptionTrack` when present.

Reverse direction (`exportHyperframes(doc)`) emits the same shape: a master-root HTML page plus one composition file per track. Animations are NOT round-trippable in v1 (they were dropped on import); reverse-direction emits no GSAP timeline.

## Out of scope

| Item | Why deferred |
|---|---|
| **CSS-class style resolution** | v1 reads inline `style="..."` declarations only. Class-styled elements (e.g., `<div class="stat-text blue-text">`) cannot recover `font-size` / `color` / `font-weight` from class definitions without a full CSS layout engine. v1 emits `LF-HYPERFRAMES-HTML-CLASS-STYLE-LOST` for any element with a non-empty `class` attribute that doesn't have a matching inline style for the same property. A future T-247-css rider could add real CSS-rule resolution. |
| **GSAP timeline parsing** | Hyperframes encodes animations via inline `<script>` blocks calling `gsap.timeline()`. Parsing JS to recover keyframes is out of scope. v1 emits `LF-HYPERFRAMES-HTML-ANIMATIONS-DROPPED` per composition with a non-empty animation script. |
| **Live transcript synthesis** | Inline transcript JSON arrays (`const TRANSCRIPT = [...]`) are recovered as a `CaptionTrack` if the format matches a known shape (start/end/text trio per word). Unrecognized shapes emit `LF-HYPERFRAMES-HTML-CAPTIONS-UNRECOGNIZED` and skip. |
| **External image bytes resolution** | T-247 emits `ParsedAssetRef.unresolved` for every `<img src>` and `data-asset-src` reference. `resolveAssets` from `@stageflip/import-pptx` (re-exported by this package or imported directly) handles the upload pass; this PR ships the structural emission only. Mirrors T-243a's split. |
| **Multi-page HTML decks (slide mode)** | Hyperframes is a video-timeline format. Mapping a multi-composition HTML deck to slide mode would require a separate spec deciding how compositions become slides. Out of scope. |
| **Reverse-direction animation emission** | The reverse direction emits HTML with elements positioned by inline styles, but no GSAP timelines (animations were dropped on import). Round-trip preserves layout, NOT animation. |
| **Reverse-direction captions emission** | v1 emits a `<script>` block with a JSON `TRANSCRIPT` array matching the recognized inbound shape; this is best-effort, not byte-accurate to the original Hyperframes producer's output. |
| **Vendored Hyperframes engine integration** | `packages/renderer-cdp/vendor/` already vendors Hyperframes' frame-capture engine for renderer use (T-117 etc.). T-247 is fresh-write — does NOT import or share code with the vendored engine. The producer-side compositional HTML format is an OOS-public convention; T-247's parser is built from the format, not from the vendored runtime. |

## Architectural decisions

### 1. Public surface

```ts
// packages/import-hyperframes-html/src/index.ts

export interface ParseHyperframesOptions {
  /**
   * Function to resolve composition-src paths to bytes. Hyperframes decks
   * are multi-file; the master HTML references compositions via
   * `data-composition-src="compositions/foo.html"`. The caller wires this
   * to whatever fetcher matches their input shape (filesystem read,
   * ZIP entry lookup, HTTP fetch).
   */
  fetchCompositionSrc: (relPath: string) => Promise<string>;
}

export interface ParseHyperframesResult {
  /** Canonical Document with content.mode === 'video'. */
  document: Document;
  /** Loss flags emitted during the parse. Empty on a perfectly clean Hyperframes input. */
  lossFlags: LossFlag[];
}

export function parseHyperframes(
  masterHtml: string,
  opts: ParseHyperframesOptions,
): Promise<ParseHyperframesResult>;

export interface ExportHyperframesOptions {
  /**
   * Function to resolve `AssetRef` -> bytes (for inline `<img src>` data-URIs
   * or for the writer's preferred output: a multi-file emission).
   */
  assets?: AssetReader;
  /**
   * Output mode:
   *   - 'multi-file' (default): returns master HTML + per-composition files; consumer writes them out.
   *   - 'inlined': inlines composition HTML into the master via `<template>` blocks.
   */
  outputMode?: 'multi-file' | 'inlined';
}

export interface ExportHyperframesResult {
  /** Master HTML bytes (UTF-8 string). */
  masterHtml: string;
  /** Map of composition-src relative path -> composition HTML. Empty in 'inlined' mode. */
  compositions: Record<string, string>;
  /** Loss flags emitted during export. */
  lossFlags: LossFlag[];
}

export function exportHyperframes(
  doc: Document,
  opts?: ExportHyperframesOptions,
): Promise<ExportHyperframesResult>;
```

`AssetReader` is the same shape as T-253-base's `AssetReader` (read-only `.get(id) => { bytes, contentType }`); re-import from `@stageflip/export-pptx` once that package's read-helper is exported, OR define a sibling locally and refactor in a future T-247-shared-reader follow-on. **Decision**: define locally in `packages/import-hyperframes-html/src/assets/types.ts` for v1; sibling refactor is mechanical and not worth coupling the two packages now.

### 2. HTML parsing — `parse5` (already in workspace)

`parse5` is the spec-compliant HTML5 parser already pulled in via `pnpm-lock.yaml` (`parse5@7.3.0`). The tree adapter produces a structured DOM the walker traverses by tag + attributes. No `cheerio` (not in workspace), no `jsdom` (heavy + browser-API surface this package doesn't need).

The walker is a small custom layer:

```
packages/import-hyperframes-html/src/dom/
  walk.ts                   # parse5 tree -> AST nodes the importer cares about
  walk.test.ts
  attrs.ts                  # data-* attr extraction helpers
  attrs.test.ts
  inline-style.ts           # parses `style="left:..; top:.."` to a Record<prop, value>
  inline-style.test.ts
```

`inline-style.ts` is a tiny parser handling exactly the Hyperframes-emitted property set: `left`, `top`, `width`, `height`, `transform`, `opacity`, `background-color`, `color`, `font-size`, `font-weight`, `border-radius`. Anything else is preserved verbatim into a `style` field on the canonical element so the reverse direction can write it back. CSS-class resolution is OOS (per "Out of scope" §1).

### 3. Track-kind heuristics

Hyperframes compositions don't carry a track-kind enum directly. The schema has `'visual' | 'audio' | 'caption' | 'overlay'`. T-247 derives via:

- `data-track-index === 0` AND `data-composition-id` matches `^main` → `'visual'` (the bottom-most composition is the A-roll).
- `data-composition-id` matches `^caption` or `^subtitle` → `'caption'`.
- Inline `<audio>` or `<video>` element with no visual children → `'audio'`.
- Otherwise → `'overlay'`.

The heuristic is documented in the reference SKILL.md so future Hyperframes producers can target it deterministically.

### 4. Element emission — positions from inline styles

Inline-style → canonical mapping (per element):

- `style="left: 540px"` + `style="top: 1360px"` → `transform.x = 540`, `transform.y = 1360` (px in the master-root coordinate system).
- `style="width: 200px"` + `style="height: 100px"` → `transform.width = 200`, `transform.height = 100`.
- Missing width/height → infer from the element's intrinsic size where possible (`<img>` natural size); otherwise emit `LF-HYPERFRAMES-HTML-DIMENSION-INFERRED` with the inferred values.
- `style="transform: translate(-50%, -50%)"` combined with absolute `left: <X>; top: <Y>; width: <W>; height: <H>` resolves to canonical `transform.x = X - W/2`, `transform.y = Y - H/2` (the canonical schema's `transformSchema` at `packages/schema/src/primitives.ts:43` has only `{x, y, width, height, rotation, opacity}` — no `anchor` field — so the importer **must** convert center-anchored CSS to top-left-anchored canonical math, not preserve an anchor token).
- `style="transform: scale(<n>)"` where n ≠ 1 → emit `LF-HYPERFRAMES-HTML-ANIMATIONS-DROPPED` (a non-identity scale on a static element is virtually always a GSAP initial state). The scale is dropped; canonical `transform` retains its `width` / `height` / defaults. There is no `transform.scale` field in the schema; the dimensions are absolute.
- `style="transform: rotate(<θ>deg)"` → `transform.rotation = θ` (the only rotation surface the schema models).
- `style="opacity: 0"` (animation initial state) → preserve as `transform.opacity = 0` only if the master-root's animation-script set surfaces a static opacity; otherwise normalize to `opacity = 1` and emit the animations-dropped flag (the `0` was a GSAP starting state).

### 5. Element types emitted

Hyperframes elements map to canonical `Element` types via tag + content shape:

- `<img>` → `ImageElement` with `src: ParsedAssetRef.unresolved` carrying the URL.
- `<video>` → `VideoElement` with `src: ParsedAssetRef.unresolved`.
- `<audio>` → `AudioElement`.
- `<div>` containing only text → `TextElement` with the inner text + parsed run styles from inline-style declarations.
- `<div>` with non-text children → `GroupElement` recursing into children.
- `<svg>` (rare in Hyperframes): emit a `ShapeElement` with `shape: 'custom'` and the SVG path string.
- Unrecognized tags: emit `LF-HYPERFRAMES-HTML-UNSUPPORTED-ELEMENT` with `originalSnippet: <tag-name>` and skip.

### 6. Captions extraction — both surfaces co-emit

Hyperframes encodes captions on **two surfaces simultaneously**:

- An inline `<script>` `const TRANSCRIPT = [{text, start, end}, ...]` array (the timed text data).
- A composition layer with `data-composition-id="captions"` (the visual styled overlay).

T-247 emits **both** when both are present, on different schema surfaces:

| Source | Canonical destination |
|---|---|
| Inline `TRANSCRIPT` JSON (recognized shape) | `videoContent.captions: CaptionTrack` (top-level — the timed text data) |
| `data-composition-id="captions"` composition layer | `Track` of kind `'caption'` (within `videoContent.tracks` — the visual overlay) |

**Cases**:

- Both present → both populate (`captions` track AND `videoContent.captions` data). The two are complementary, not redundant: the data drives downstream rendering, the visual track preserves the producer's styled appearance.
- Only inline `TRANSCRIPT` present (no caption composition) → only `videoContent.captions` populates. No `caption`-kind track in `tracks`.
- Only caption composition present (no inline `TRANSCRIPT`) → only the `caption`-kind `Track` populates. `videoContent.captions` is omitted; the consumer downstream renders without the timed text data (text content is read off the visual elements via OCR or omission).
- Neither → omit both.

The transcript JSON pattern (per `reference/hyperframes/packages 2/producer/tests/style-10-prod/src/compositions/graphics.html` lines 144+):

```js
const TRANSCRIPT = [
  { text: "We", start: 0.119, end: 0.259 },
  ...
];
```

T-247 extracts the inline `<script>` blocks, runs a regex to find `const TRANSCRIPT = [` (or `let` / `var` / `window.__transcript = [`), and uses a shape-matched JSON parse on the array. **Recognized shape (exactly)**: each array entry must be an object with the field set `{text: string, start: number, end: number}` — extra fields cause the entry to be skipped (NOT the entire array; only the offending entry); missing required fields → `LF-HYPERFRAMES-HTML-CAPTIONS-UNRECOGNIZED` for the entire array and `videoContent.captions` is omitted. Each accepted entry maps to a `CaptionSegment { startMs: start * 1000, endMs: end * 1000, text }`.

The regex + JSON-parse is deliberately conservative — it matches the exact shape Hyperframes producers emit; arbitrary inline JS isn't evaluated.

### 7. Reverse direction (`exportHyperframes`)

Emits HTML matching the imported shape:

- `<div id="master-root" data-composition-id="master" data-width="<W>" data-height="<H>" data-duration="<D>"/>` — duration in seconds (× 1/1000 from `durationMs`).
- One `<div>` per `Track`, with the composition layers' `data-*` attrs.
- Per-composition: a separate HTML file (in `'multi-file'` mode) or an inlined `<template>` (in `'inlined'` mode).
- Elements emitted as `<img>` / `<video>` / `<audio>` / `<div>` with inline `style="left:..; top:..; width:..; height:.."`.
- No `<script>` blocks for animations (out of scope).
- Captions: a single `<script>` block in the master HTML with `const TRANSCRIPT = [...]` matching the inbound shape (best-effort).

Output is **deterministic**: attribute order is fixed, whitespace is canonical (4-space indent, LF line endings), no per-call timestamps.

### 8. Loss flags taxonomy

Six new codes, all with `source: 'hyperframes-html'`:

| Code | Severity | Category | Emitted when |
|---|---|---|---|
| `LF-HYPERFRAMES-HTML-CLASS-STYLE-LOST` | `warn` | `theme` | An element has a non-empty `class` attribute without matching inline-style declarations for the same properties; CSS-class resolution is OOS in v1. |
| `LF-HYPERFRAMES-HTML-ANIMATIONS-DROPPED` | `info` | `animation` | A composition's inline `<script>` contained a GSAP timeline; the import drops it. One flag per composition. |
| `LF-HYPERFRAMES-HTML-CAPTIONS-UNRECOGNIZED` | `warn` | `other` | An inline `<script>` looked transcript-shaped but didn't match the recognized JSON pattern. The captions track is omitted. |
| `LF-HYPERFRAMES-HTML-UNSUPPORTED-ELEMENT` | `warn` | `other` | A tag the importer doesn't recognize (e.g., `<canvas>`, `<iframe>`). The element is skipped. |
| `LF-HYPERFRAMES-HTML-DIMENSION-INFERRED` | `info` | `shape` | An element lacked explicit `width` / `height`; the importer inferred from intrinsic size or surrounding context. |
| `LF-HYPERFRAMES-HTML-ASSET-MISSING` | `error` | `media` | An `<img src>` or `<video src>` reference's bytes can't be retrieved (consumer-resolved via `resolveAssets`; this flag is emitted by the resolver, not by `parseHyperframes`). Mirrors `LF-PPTX-MISSING-ASSET-BYTES` from T-243a. |

The wrapper `emitLossFlag` mirrors `@stageflip/import-pptx/src/loss-flags.ts` with `source: 'hyperframes-html'` and a `CODE_DEFAULTS` table.

### 9. Round-trip contract

T-247 ships a `roundtrip.test.ts` suite parallel to T-253-base's: every Hyperframes fixture parses through `parseHyperframes → exportHyperframes → parseHyperframes` and the second-pass `Document` is structurally equal to the first-pass under a documented equality predicate.

**Predicate exclusions + ordering rules**:

- Animations are dropped on the first pass; the second pass also has no animations. Compare both as having empty animations.
- Class-styling losses → second-pass has no class-styled elements either; the loss flag is a one-way information loss expected to recur.
- Asset bytes pass-through is a separate concern (handled by `resolveAssets`); the round-trip predicate excludes `ParsedAssetRef`.
- **Track ordering**: positional (z-order matters; `tracks[]` is index-positional).
- **Element ordering within a track**: positional (z-order within the track).
- **Loss flag ordering**: compared as a **multiset** (sorted by `code` + `location.slideId` + `location.elementId`); the second-pass loss-flag set is byte-equivalent under this multiset comparison to the first-pass set. Repeated losses (e.g., 3 `LF-HYPERFRAMES-HTML-CLASS-STYLE-LOST` flags on 3 styled elements) appear 3 times in both passes.

### 10. Determinism

The importer + exporter are pure (no I/O beyond the caller-provided `fetchCompositionSrc` / `AssetReader`). Source-level discipline test (mirrors T-253-base AC #28 + T-245 AC #19) — no `Date.now` / `Math.random` / `performance.now` / `setTimeout` etc. in `packages/import-hyperframes-html/src/**`.

10. `composition.data-start` (seconds) + `composition.data-duration` (seconds) flow through to track's elements (start offsets relative to the master). Pin via fixture.

### Element parsing

11. **`<img src="...">`** → `ImageElement` with `src: ParsedAssetRef.unresolved` carrying the URL. Pin.
12. **`<video src="...">`** → `VideoElement`. Pin.
13. **`<audio src="...">`** → `AudioElement`. Pin.
14. **`<div>` containing only text nodes** → `TextElement` with the text + inline-style-derived run styles. Pin.
15. **`<div>` with non-text children** → `GroupElement` with children. Pin: a 2-deep nested div produces a 2-deep group.
16. **Unrecognized tag** (e.g., `<canvas>`) → `LF-HYPERFRAMES-HTML-UNSUPPORTED-ELEMENT` + element skipped. Pin.

### Inline-style parsing

17. `style="left: 540px; top: 1360px"` → `transform.x = 540`, `transform.y = 1360`. Pin.
18. `style="width: 200px; height: 100px"` → `transform.width = 200`, `transform.height = 100`. Pin.
19. `style="left: 540px; top: 1360px; width: 200px; height: 100px; transform: translate(-50%, -50%)"` → canonical `transform.x === 440`, `transform.y === 1310`, `transform.width === 200`, `transform.height === 100`. The CSS center-anchor is converted to canonical top-left anchor via `x = left - width/2`, `y = top - height/2`. The schema's `transformSchema` has no `anchor` field. Pin.
20. `style="transform: scale(0)"` (or any non-identity scale on a static element) → emit `LF-HYPERFRAMES-HTML-ANIMATIONS-DROPPED`. The scale is dropped; canonical `transform` retains absolute `width` / `height` from the inline `style`. No `transform.scale` field exists in the schema. Pin.
20a. `style="transform: rotate(45deg)"` → canonical `transform.rotation === 45`. Pin (rotation is the one transform sub-property the schema models).
21. **Missing `width` / `height` on a `<div>` text element** → emit `LF-HYPERFRAMES-HTML-DIMENSION-INFERRED` with values inferred from the parent composition's child layout. Pin.
22. `style="opacity: 0"` with a sibling `<script>` containing GSAP → emit `LF-HYPERFRAMES-HTML-ANIMATIONS-DROPPED`, normalize `opacity` to 1. Pin.

### Class-style loss

23. An element with `class="stat-text blue-text"` and no inline `font-size` / `color` → emit `LF-HYPERFRAMES-HTML-CLASS-STYLE-LOST` once for that element. The element is still emitted with the inline-style-derived geometry. Pin via fixture.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- A real Hyperframes fixture surfaces a `data-*` attribute the spec doesn't enumerate (e.g., `data-clip-mask` or `data-shader`). Don't silently extend the schema; escalate with the attribute's purpose.
- The track-kind heuristic (§3) misclassifies a real fixture's composition (e.g., `data-composition-id="lower-thirds"` should be `'overlay'` but the heuristic returns `'visual'` because `track-index === 0`). Escalate; the heuristic should evolve via fixture-driven examples, not silent regex extension.
- A `<script>` block contains a `TRANSCRIPT` array whose entries vary (e.g., some have `start` / `end`, others `t0` / `t1`). Either the producer is inconsistent or the spec needs to widen the recognized shape. Escalate.
- The `parse5` tree adapter produces an unexpected node shape for the master HTML (e.g., a comment node breaks the walker). Escalate before patching the walker; parse5 7.3.0's surface is stable but unusual inputs may surface bugs.
- `style="transform: matrix(...)"` (a 2D matrix transform) appears on a real fixture. T-247 v1 only handles `translate` / `scale` / `rotate` decomposed forms; matrix decomposition is non-trivial and out of scope. Escalate; the v1 emission would either drop the matrix (silent loss) or emit an unsupported-element flag, neither is great.
- A composition's `data-composition-src` references a path outside the document root (e.g., `../../shared/captions.html`). T-247 expects compositions sibling-relative; the spec should pin path resolution rules before the Implementer guesses.
