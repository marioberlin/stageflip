# T-265 — `@stageflip/runtimes/blender` — bake-tier ClipRuntime

**Branch**: `task/T-265-blender-bake-runtime`

## Goal

Stand up a third runtime tier: **bake** — for clips that cannot run in-browser (3D scenes with heavy GPU, ray-tracing, photorealistic materials) and must pre-render frames in Blender. The bake runtime ships frames to a content-addressed cache (`bakes/{inputsHash}/frame-{N}.png` per `docs/architecture.md:330`); the renderer-cdp consumes the cached frames at video composition time.

Per ADR-003 (interactive runtime tier), three tiers exist:

1. **Frame-deterministic** (`@stageflip/frame-runtime`) — math-driven, runs in JS at any frame rate.
2. **Interactive** (`@stageflip/runtimes/interactive`) — frontier clips with `liveMount` + `staticFallback` (ADR-005).
3. **Bake** (`@stageflip/runtimes/blender` — T-265) — pre-render via Blender; consumed by composition layer.

T-265 ships:

1. **`@stageflip/runtimes/blender`** package — `BlenderClipRuntime` implementing `ClipRuntime` contract.
2. **`BlenderClip` element** — Zod schema in `@stageflip/schema` for declaring a Blender-baked clip in a document.
3. **Worker Docker image** — `services/blender-worker/` with Blender CLI + Python scripting + GPU drivers.
4. **BullMQ queue** — `stageflip:bakes` queue per architecture; consumed by the worker; produces frames to Cloud Storage.
5. **Bake job submission API** — Cloud Function `submitBakeJob({ clipDescriptor })` enqueues, returns `bakeId`.
6. **Frame-fetch API** — `getBakedFrames(inputsHash)` returns frame URLs from Cloud Storage.
7. **Determinism via `inputsHash`** — same inputs → same `bakes/{hash}/` directory; cache hits skip re-bake.

## Architectural decisions

### D-T265-1. Three-runtime tier model (recap)

Per ADR-003 §D2, each clip declares:

```ts
interface ClipRuntime {
  staticFallback: (frame: number, ctx: ClipContext) => RasterFrame;
  liveMount?: (root: HTMLElement, ctx: ClipContext) => MountHandle;
  bake?: (descriptor: ClipDescriptor) => Promise<BakeJob>; // NEW for T-265
}
```

T-265 introduces the third optional method. Clips that ship `bake` rely on the bake worker for `staticFallback` content (the worker writes frames to the cache; `staticFallback` reads them back).

### D-T265-2. `BlenderClip` element schema

```ts
export const blenderClipSchema = z.object({
  ...elementBaseSchema.shape,
  type: z.literal('blender-clip'),
  scene: z.object({
    /** Scene template id — references a built-in template like 'fluid-sim' or 'product-render'. */
    template: z.string().min(1).max(100),
    /** Template-specific parameters (typed per template). */
    params: z.record(z.unknown()).default({}),
  }),
  duration: z.object({
    durationMs: z.number().int().positive(),
    fps: z.number().int().positive().default(30),
  }),
  /** Computed from scene + duration; pinned in the schema for reproducibility. */
  inputsHash: z.string().regex(/^[a-f0-9]{64}$/),
}).strict();
```

`inputsHash` is `sha256(JSON.stringify(canonicalize({scene, duration})))` — pinned at clip-creation time. Re-rendering with the same hash hits the cache; any input change produces a new hash.

### D-T265-3. Worker contract

The Blender worker is a stateless container:

- **Input**: `BakeJob = { bakeId, inputsHash, scene, duration, outputBucket, region }`. Read from BullMQ queue.
- **Process**: pulls the scene template, injects `scene.params`, invokes Blender CLI with frame range, collects PNGs.
- **Output**: writes each frame to `gs://{outputBucket}/bakes/{inputsHash}/frame-{N}.png`. On completion, writes `bakes/{inputsHash}/manifest.json` with frame count + duration.
- **Errors**: Sentry captureError + emit `bake_failed` event; queue retries per BullMQ defaults (up to 3 attempts).

The worker is **idempotent**: if it sees a `manifest.json` already at the output path, it short-circuits successfully (cache hit at the worker side too).

### D-T265-4. Frame-fetch API

`getBakedFrames(inputsHash, { region })` returns `{ status: 'ready' | 'pending' | 'failed', frames?: string[], manifestUrl?: string }`. Implemented in the package; reads from the Storage adapter directly (no Cloud Function needed for read).

Storage adapter selection respects `region` per T-271.

### D-T265-5. Bake submission

`submitBakeJob({ clipDescriptor })` Cloud Function:

1. Verifies caller is authenticated (T-262 `requireAuth`).
2. Consumes a token from the org's bake-rate bucket (T-263). NOTE: bake is expensive — consume more tokens (10 tokens/bake submission, vs 1/regular request). Configurable.
3. Computes `inputsHash` from `clipDescriptor.scene + duration`. Verifies it matches `clipDescriptor.inputsHash` (defense against forged hashes).
4. Checks Cloud Storage for an existing `bakes/{inputsHash}/manifest.json`. If present, returns `{ status: 'ready', bakeId: '<existing>' }` — cache hit.
5. Otherwise, enqueues to `stageflip:bakes` BullMQ queue, returns `{ status: 'pending', bakeId: '<new>' }`.

### D-T265-6. Worker deployment

T-265 ships:

- `services/blender-worker/Dockerfile` — Ubuntu 22.04 + Blender 4.2 LTS + Python 3.11 + minimal Node bridge.
- `services/blender-worker/src/worker.ts` — Node entry point; pulls from BullMQ; invokes Blender via subprocess.
- `services/blender-worker/src/templates/` — built-in scene templates (fluid-sim, product-render, particle-burst).
- `services/blender-worker/scripts/render.py` — Blender Python script that takes parameters via stdin and renders to a temp dir.

Deployment of this worker is T-266 (vendor adapter). T-265 ships the image + a local-dev `docker compose` for testing.

### D-T265-7. GPU / non-GPU dual-path

The worker tries GPU rendering first (if `CUDA_VISIBLE_DEVICES` is set); falls back to CPU rendering with a soft warning emitted via Sentry. CPU is ~10x slower but produces identical output (Blender's deterministic mode).

For dev (laptops without GPU), CPU is the default. For prod (T-266 deploys with GPU), GPU is the default.

### D-T265-8. Determinism in Blender

Blender's Cycles engine has a deterministic mode (`bpy.context.scene.render.use_persistent_data = True`, fixed seed for noise samples). The worker enforces this in every render. **Same `inputsHash` → byte-identical frames** is the contract. Pin via parity test (rare — bake is too slow for the standard parity harness; pin via a single fixture in CI smoke).

### D-T265-9. Determinism posture (TS side)

`packages/runtimes/blender/**` is NOT clip / runtime code in the sense of the determinism scan (it's the runtime infrastructure, not a clip implementation). Verify scan list. The Blender worker (`services/blender-worker/**`) is also out of scope by virtue of being a service, not a package.

## Acceptance criteria

### Schema

1. `blenderClipSchema.parse({ ... })` accepts a valid BlenderClip per D-T265-2.
2. Invalid `inputsHash` (wrong format) fails parse with a clear message.
3. The `Element` union (`packages/schema/src/elements/index.ts`) includes `BlenderClip`. Existing fixtures parse unchanged.
4. `inputsHash` is verifiable: given a `BlenderClip` JSON, `computeInputsHash({ scene, duration })` returns the same hash. Pin via test with a fixture clip + expected hash.

### inputs-hash

5. `computeInputsHash` is deterministic: same input → same SHA-256 hash across 1000 invocations.
6. Field order independence: `{a: 1, b: 2}` and `{b: 2, a: 1}` produce the same hash (canonicalization is sorted-key JSON).
7. Field type sensitivity: `{ ms: 1000 }` and `{ ms: '1000' }` produce DIFFERENT hashes (no implicit coercion).

### Submit

8. `submitBakeJob` Cloud Function: requires `requireAuth`; rejects unauth'd with 401.
9. Consumes 10 tokens from the org's rate-limit bucket per submission. Pin via spy on the limiter.
10. Verifies `clipDescriptor.inputsHash` matches recomputed hash. Mismatch → 400 with `code: 'inputs-hash-mismatch'`.
11. Cache hit (manifest.json exists at the predicted path): returns `{ status: 'ready', bakeId, frames: [...] }` without enqueueing.
12. Cache miss: enqueues a BullMQ job, returns `{ status: 'pending', bakeId }`.
13. Region-aware: writes to the org's region-specific bucket per T-271.

### Fetch

14. `getBakedFrames(inputsHash, { region })`: when `manifest.json` exists, returns `{ status: 'ready', frames: string[] }` with frame URLs.
15. When manifest absent: returns `{ status: 'pending' }`.
16. When `bake_failed` marker present: returns `{ status: 'failed', error: string }`.

### Queue

17. The producer enqueues to `stageflip:bakes` (the documented architecture queue name). Job payload includes `bakeId`, `inputsHash`, `scene`, `duration`, `outputBucket`, `region`.
18. Job options: `attempts: 3`, `backoff: exponential`, `removeOnComplete: { age: 86400 }` (1 day retention for completed jobs).

### Worker

19. The worker consumes from `stageflip:bakes`. Pin via integration test (testcontainers Redis + worker).
20. **Idempotency**: if `manifest.json` already exists at the output path, the worker short-circuits successfully. Pin via fixture.
21. The worker invokes Blender via `services/blender-worker/scripts/render.py`. Pin via test with a stubbed render script that just writes empty PNGs.
22. The worker writes frames to `gs://{outputBucket}/bakes/{inputsHash}/frame-{N}.png` for N in [0, ceil(durationMs/1000 * fps)). Pin via test with mock GCS.
23. Manifest written on completion includes: `frameCount`, `fps`, `durationMs`, `outputBucket`, `region`, `completedAt`. Pin schema.
24. **GPU/CPU dual-path**: worker tries GPU; on CUDA unavailable, falls back to CPU + emits warning. Pin via env-var test.

### Determinism

25. **Same inputs → identical output** at the manifest level: two bakes with the same `inputsHash` produce the same `frameCount` + same byte-length on each frame. Pin via integration test (CI smoke; budget ~5 min for one fixture render).

### Backward compatibility

26. The full workspace test suite passes unchanged.
27. Existing `Element` consumers (RIR compiler, renderer-core dispatcher) handle `BlenderClip` either by rendering its `staticFallback` (read from cache) OR by delegating to the bake runtime. Verify dispatch path exists or is documented as a follow-up.

### Coverage + gates

28. Coverage on `packages/runtimes/blender/src/**` ≥85%.
29. Coverage on `services/blender-worker/src/**` ≥80% (Blender invocation is mocked; the orchestration logic gets full coverage).
30. `pnpm typecheck`, `pnpm lint`, `pnpm test`, `pnpm check-licenses`, `pnpm check-remotion-imports`, `pnpm check-determinism`, `pnpm check-skill-drift`, `pnpm size-limit` all green.

## Escalation triggers (CLAUDE.md §6)

- The Blender 4.2 LTS Docker image exceeds reasonable size (>4 GB after stripping). Escalate; we may need a slimmer base or accept the size.
- GPU availability in Cloud Run / dev environments is more constrained than expected (e.g., Cloud Run GPU is in preview only in some regions). Escalate; T-266 will need to deal with this either way, but if T-265 can't even ship the dual-path, escalate.
- The chosen scene templates (fluid-sim / product-render / particle-burst) require Blender add-ons that aren't in the LTS release. Escalate; we may need to bundle or pick different templates.
- Blender's deterministic mode produces non-deterministic output across CPU/GPU paths despite documented same-seed semantics. Escalate; document the invariant we can actually hold.
- The `inputsHash` canonicalization rule for floating-point parameters (e.g., 1.0 vs 1) produces different hashes across reasonable JSON serializers. Escalate; pick a canonicalization library or document the rule.
- Worker cold-start in Cloud Run exceeds 60s. Escalate; render farm deployment (T-266) will need to know.

## Notes for the Orchestrator

1. **L-sized; eight commits.** Schema + hash + queue + submit/fetch + functions + worker scaffold + worker render + skills.
2. **Reviewer focus**: AC #5–#7 (inputsHash determinism — the cache key is the whole game; field order, type sensitivity, canonicalization must all hold); AC #20 (worker idempotency — re-running on the same hash must short-circuit); AC #25 (output determinism — flaky Blender renders break caching).
3. **No production deploy in this PR.** T-266 deploys the worker. T-265 ships the image + local-dev compose.
4. **`bakes/` path is architecture-locked.** Don't accidentally version it (`bakes/v1/...`) — the cache key is supposed to be globally addressable.
5. **CI doesn't run real Blender.** Image build + worker unit tests run on every PR; full integration (`docker run` with a real render) runs nightly.
6. **Worker idempotency** (AC #20) is the single most important property. Without it, a duplicate-delivery in BullMQ produces two bakes for the same hash and one wins arbitrarily.
