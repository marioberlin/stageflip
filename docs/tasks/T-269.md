# T-269 — Load testing (K6)

**Branch**: `task/T-269-k6-load-testing`

## Goal

Stand up `tests/load/` — K6 load test suite covering the three load-bearing surfaces of the platform: **collab sync** (Yjs deltas), **render submission** (BullMQ queue ingress), and **public API** (auth + rate-limit + canonical handlers). T-269 ships scenarios + thresholds + a CI smoke job (10-min cap, low-volume); full-load runs are operational, manually triggered against a staging tenant.

## Dependencies

- T-260 (collab) — merged.
- T-261 (presence) — merged. Optional load on RTDB.
- T-262 (auth) — merged. Tests authenticate with seeded test users + api-keys.
- T-263 (rate limit) — merged. Tests respect 429 + Retry-After.
- T-264 (observability) — merged. Tests can be traced end-to-end.

## Out of scope

| Item | Why deferred |
|---|---|
| Continuous load against prod | Operational; manual invocation only. |
| Adversarial / fuzz testing | Different tool (e.g., custom + restate); future. |
| Render-farm worker load tests | T-266-era; T-269 only loads the queue ingress, not worker capacity. |
| Browser-driven load (real-user scenarios) | Playwright/Puppeteer; not K6's strength. |
| Cost / billing simulation under load | T-267-era. |
| Auto-tuning of thresholds | T-269 ships sane defaults; ops tunes from telemetry (T-264). |

## Architectural decisions

### D-T269-1. K6 + JavaScript scenarios

K6 (Apache-2.0). JavaScript-based scenarios; runs without browser. Three reasons over alternatives (Locust, Gatling, custom):

- K6 has first-class HTTP/2 + WebSocket support (we need both — collab uses WS, API is HTTP).
- Output formats include InfluxDB / Prometheus / JSON — composes with T-264's OTel pipeline.
- Tests are in JS — same language as the rest of the repo; no Python/Scala context-switch.

### D-T269-2. Three scenarios

1. **`collab-sync.js`** — opens N WebSocket connections to a test doc, fires bursts of Yjs updates, asserts P95 fan-out < 200 ms.
2. **`render-submit.js`** — N parallel HTTP POSTs to `/api/render`, asserts queue accepts < 100 ms P95 and the BullMQ depth stays bounded.
3. **`api-mixed.js`** — realistic mix of public API calls (auth, document read, document write, asset list), respects 429 + Retry-After, asserts no errors at 50 req/s steady-state.

### D-T269-3. CI smoke + ops full-load

- **CI smoke** (10 min cap, ~50 VUs each scenario): runs on every push to `main`, blocks merge on threshold breach. Pinned thresholds tight enough to catch regressions but lax enough to tolerate CI runner variance.
- **Ops full-load** (manual, 1-hour, ~500 VUs): documented in `tests/load/README.md`; ops triggers via `pnpm load:full` against a dedicated staging tenant. Results persisted to `tests/load/runs/<timestamp>.json`.

### D-T269-4. Test data

- Seeded test org with N=100 documents pre-populated.
- Seeded test users (one per VU) with API keys.
- Seeding script `tests/load/seed.ts` runs once; scenarios assume the seeded state.
- Cleanup script `tests/load/cleanup.ts` for the staging tenant.

### D-T269-5. Determinism posture

`tests/load/**` is NOT clip / runtime code. K6's runtime IS non-deterministic by design (random VU spawn, network jitter). `pnpm check-determinism` does not scan `tests/**`. Verify scan list.

### D-T269-6. Test artifacts


## Acceptance criteria

### Scenarios

1. `collab-sync.js` opens 50 WebSocket connections, runs 5 minutes, asserts fan-out P95 < 200 ms. Pin via dry-run against in-memory adapter.
2. `render-submit.js` submits 30 req/s for 5 minutes, asserts queue-accept P95 < 100 ms, error rate < 0.5%. Pin via dry-run.
3. `api-mixed.js` runs a 60/30/10 mix (read/write/auth) at 50 req/s for 5 minutes, asserts no 5xx, respects 429 + Retry-After per T-263 wire shape. Pin via dry-run.

### Thresholds + reporting

4. `thresholds.js` exports per-scenario threshold config; scenarios import. Pin via test that thresholds are within 50% of measured baseline (no future drift without intent).
5. K6 output includes summary JSON written to `tests/load/runs/<timestamp>.json`. Pin via dry-run.

### CI smoke

6. `.github/workflows/load-smoke.yml` runs all three scenarios at smoke parameters (10 VUs, 60 seconds), uses `grafana/setup-k6-action@v1`, fails the job on threshold breach.
7. CI smoke total runtime < 10 minutes (3 scenarios × 60s + setup). Verify via dry-run.

### Ops full-load (documented, not automated)

8. `tests/load/README.md` documents: env vars required (`STAGEFLIP_LOAD_TARGET`, `STAGEFLIP_LOAD_AUTH_TOKEN`), `pnpm load:full` invocation, expected runtime, output location.
9. `seed.ts` and `cleanup.ts` are idempotent; running `seed` twice doesn't double-seed; running `cleanup` after `cleanup` is a no-op.

### Auth + rate-limit handling

10. K6 helper `auth.js` accepts a JWT or API key, attaches Authorization + X-Org-Id headers per T-262 conventions.
11. K6 helper `retry-after.js` parses 429 responses, sleeps the indicated seconds, retries up to 3 times. Pin via mock.

### Backward compatibility

12. The full workspace test suite passes unchanged. `tests/load/**` is NOT included in the standard `pnpm test`; only `pnpm load:smoke` invokes it.
13. CI's existing checks (typecheck/lint/test/gates) are unaffected. The new `load-smoke` workflow is independent.

### Coverage + gates

14. K6 scenarios are not unit-tested (would be circular). Helper TS files (`seed.ts`, `cleanup.ts`) get vitest coverage ≥80%.
15. `pnpm typecheck`, `pnpm lint`, `pnpm test`, `pnpm check-licenses`, `pnpm check-remotion-imports`, `pnpm check-determinism`, `pnpm check-skill-drift`, `pnpm size-limit` all green.

## Escalation triggers (CLAUDE.md §6)

- K6's WebSocket support is insufficient for Yjs binary-frame protocol. Escalate; we may need a JS native test harness instead.
- The CI smoke runtime exceeds 10 minutes despite parameter tuning. Escalate; we may need to drop a scenario or parallelize.
- Threshold values surface real perf regressions in T-260/T-263 that weren't caught by unit tests. STOP and report — that's a finding, not a load-test issue.
