# T-406 — Chart clip family (`chart` ClipDefinition over all seven `chartKind`s)

**Branch**: `task/T-406-chart-clip-family`

## Goal

Ship a unified **`chart` ClipDefinition** in `@stageflip/runtimes-frame-runtime-bridge` that consumes the existing `ChartElement` schema (`@stageflip/schema/elements/chart.ts`) and renders all seven chart kinds — `bar`, `line`, `area`, `pie`, `donut`, `scatter`, `combo` — as frame-deterministic SVG. This is the chart primitive **Cluster E presets** (T-355–T-360) bind to, and the chart primitive the **LiveData chart-rendering follow-up** wires into the LiveDataClip pair (`^liveData-v1` ADR-005 §D1 footnote).

The clip is frame-deterministic, not interactive — each rendered SVG is a pure function of `(frame, props, themeTokens)`. Same frame number → same pixels; subject to the broad §3 rule (no `Math.random`, no `Date.now`, no `performance.now`, no `fetch`, no timers, no rAF).

T-406 ships ONE clip definition. The existing per-preset chart clips (`chartBuild`, `pieChartBuild`, `lineChartDraw` in `frame-runtime-bridge/src/clips/`) remain as-is — they are standalone preset targets with their own animation choreography, distinct from the unified family this task ships. T-406's chart clip is the consumer of `ChartElement` schema; the existing per-preset clips have their own props schemas and remain available.

## Out of scope

## Architectural decisions

### D-T406-1. Single `chart` ClipDefinition in `frame-runtime-bridge`

T-406 lives in `packages/runtimes/frame-runtime-bridge/src/clips/chart/` (new directory). It does NOT spawn a new workspace package — frame-runtime-bridge already houses every other frame-deterministic-bridge clip (kinetic-text, light-leak, the existing 3 chart clips, dashboards). The unified family is consistent with the package's existing scope.

The clipKind string is `'chart'` (matches `ChartElement.type === 'chart'`). The clip definition is `defineFrameClip<ChartProps>({ kind: 'chart', component: ChartClip, ... })`.

### D-T406-2. Props schema mirrors `ChartElement` data + kind

```ts
// packages/runtimes/frame-runtime-bridge/src/clips/chart/index.tsx (NEW)
export const chartPropsSchema = z.object({
  /** Discriminator — selects the per-kind renderer. */
  chartKind: chartKindSchema,                  // imported from @stageflip/schema
  /** Inline data; `DataSourceRef` rejected at mount time per D-T406-8. */
  data: chartDataSchema,                       // imported from @stageflip/schema
  /** Whether to render the legend (default true). */
  legend: z.boolean().default(true),
  /** Whether to render axes (default true; ignored by pie / donut). */
  axes: z.boolean().default(true),
}).strict();
```

The schema is a structural subset of `ChartElement` (drops `elementBase` fields the runtime supplies; drops the `dataSourceRefSchema` branch).

### D-T406-3. Seven per-kind renderer files

```
packages/runtimes/frame-runtime-bridge/src/clips/chart/
  index.tsx                    # NEW — unified ChartClip + dispatcher + propsSchema + ClipDefinition
  bar.tsx                      # NEW — bar renderer (vertical bars with rise + fade-in stagger)
  line.tsx                     # NEW — line renderer (stroke-dasharray draw-on)
  area.tsx                     # NEW — area renderer (line + filled polygon, sweep reveal)
  pie.tsx                      # NEW — pie renderer (per-slice arc reveal)
  donut.tsx                    # NEW — donut renderer (pie with inner cutout)
  scatter.tsx                  # NEW — scatter renderer (point fade-in stagger)
  combo.tsx                    # NEW — combo renderer (bars + line overlay; first series is bars, rest are lines)
  axes.tsx                     # NEW — shared axis renderer used by bar / line / area / scatter / combo
  legend.tsx                   # NEW — shared legend renderer used by all 7 kinds
  *.test.tsx                   # NEW — one test file per renderer + one for the dispatcher
```

`index.tsx` exports the `ChartClip` React component, the props schema, the `chartClip` `ClipDefinition`, and dispatches internally:

```tsx
function ChartClip(props: ChartProps): ReactElement {
  switch (props.chartKind) {
    case 'bar':     return <BarChart data={props.data} legend={props.legend} axes={props.axes} />;
    case 'line':    return <LineChart ... />;
    case 'area':    return <AreaChart ... />;
    case 'pie':     return <PieChart data={props.data} legend={props.legend} />;
    case 'donut':   return <DonutChart data={props.data} legend={props.legend} />;
    case 'scatter': return <ScatterChart ... />;
    case 'combo':   return <ComboChart ... />;
  }
}
```

### D-T406-4. Built-in animations per chart kind

Each renderer ships a built-in entrance animation:

| Kind | Animation | Stagger |
|---|---|---|
| bar | bars rise from baseline + fade in | per-bar |
| line | stroke-dasharray draws path L→R | per-series |
| area | line draws + fill sweeps | per-series |
| pie | per-slice arc rotates from 0 to its terminal angle | per-slice |
| donut | same as pie | per-slice |
| scatter | points fade in | per-point (deterministic order: `series.flatMap((s, i) => s.values.map((v, j) => ...))`) |
| combo | bars rise (first series) + lines draw (rest) | per-bar / per-series |

All animations complete by `floor(0.6 * durationFrames)` (configurable as a constant per renderer; v1 ships the same value across all kinds). Easing is `cubicBezier(0.16, 1, 0.3, 1)` (the existing `EASE_OUT_EXPO` shared with `chart-build.tsx`).

### D-T406-5. Theme integration via `defineFrameClip`

The clip declares `themeSlots` for the colors it needs (`primary`, `secondary`, `tertiary`, ... up to 8 series colors; `axis`, `gridline`, `text`). `defineFrameClip` injects the resolved theme tokens. Renderers read from `themeTokens.primary` / etc. — no inline color literals (a `check-skill-drift`-equivalent grep test rejects hex literals in the renderer files).

### D-T406-6. Pure SVG output

All renderers emit `<svg>` JSX. No `<canvas>`, no WebGL, no DOM measurements via `getBoundingClientRect` (which is non-deterministic). Width / height come from `useVideoConfig().width / height` (the canonical canvas dimensions per the frame-runtime contract).

### D-T406-7. Determinism contract

Renderers are pure functions of `(frame, props, themeTokens)`. The broad §3 rule applies:

- No `Math.random` / `Date.now` / `performance.now` / `fetch` / timers / rAF / Workers.
- Any randomness (e.g., scatter jitter for visual breathing room when points coincide) uses a **seeded PRNG** keyed on `props.data` content hash + chart-kind. **v1 ships NO PRNG at all** — scatter renders exact data points exactly. The content-hash key is a forward-looking design note for the future opt-in jitter prop, NOT a v1 requirement; the Implementer should not wire up a hash function in v1.
- `useCurrentFrame()` and `useVideoConfig()` are the only frame-runtime primitives consumed.

Pinned via:
- `pnpm check-determinism` (broad §3 rule already covers `frame-runtime-bridge/**`).
- Per-renderer test that renders the SVG twice with the same `(frame, props, theme)` and asserts byte-for-byte equal output.

### D-T406-8. `DataSourceRef` rejection until T-167

`chartElementSchema.data` is `z.union([chartDataSchema, dataSourceRefSchema])`. T-406's `chartPropsSchema` accepts ONLY `chartDataSchema`. The mount path (or the `defineFrameClip` props validator) rejects `DataSourceRef` with a typed error message: "Chart `data` must be inline `ChartData` in v1; `DataSourceRef` resolution lands in T-167 (data-source-bindings bundle)."

When T-167 lands, a follow-up task widens `chartPropsSchema.data` to the union and adds resolver wiring. Until then, the rejection is the contract.

### D-T406-9. Unified family does NOT replace the existing 3 standalone chart clips

`chart-build`, `pie-chart-build`, `line-chart-draw` (T-131b clips) remain unchanged. They are preset-targeting clips with their own props schemas and animation choreography. T-406 ADDS the unified family; it doesn't refactor them.

Cluster E presets (T-355–T-360) bind to T-406's unified `chart` clipKind, NOT to the standalone clips. Document this binding in the Cluster E skill (T-361, future) when those presets ship.

### D-T406-10. Pattern-evaluation — frame-deterministic vs γ-live

T-406 is the first major frame-deterministic clip family ship since γ-core (T-383 / T-384). Unlike γ-live (event-driven, host-injected provider, `MountHandle` lifecycle), T-406 is:

- Frame-deterministic (broad §3 rule applies, not the interactive-tier exemption)
- Stateless per render (no factory, no `MountHandle`, no `dispose`)
- Uses `defineFrameClip` + `ClipDefinition`, not `ClipFactory` + `MountContext`

The two patterns are intentionally distinct. Future chart enhancements stay frame-deterministic (T-167 data-source resolution; future palette overrides; future data-state transitions). Interactive-tier "chart with hover/tooltips" would be a different family.

## Acceptance criteria

### Schema + dispatcher

1. `chartPropsSchema` parses a valid payload for each of the 7 chart kinds with inline `ChartData`.
2. `data: 'ds:foo'` (a `DataSourceRef`) → schema error referencing T-167.
3. `data: { labels: [], series: [] }` (empty arrays) is accepted but renders an empty chart (no error).
4. Extra top-level fields are rejected (strict).
5. The `ChartClip` component dispatches on `chartKind` to the correct per-kind renderer (verified by snapshot or text-content assertion per kind).

### Per-kind renderers

6. **Bar** — N values render N rectangles whose heights are proportional to values; each bar's height interpolates from 0 to its terminal value over the entrance window; per-bar stagger.
7. **Line** — each series renders one `<path>` whose stroke is animated L→R via the dasharray-plus-decreasing-dashoffset technique (`stroke-dasharray = pathLength`, `stroke-dashoffset` interpolates from `pathLength` to `0` over the entrance window). Same technique as the existing `line-chart-draw.tsx` precedent. Per-series stagger.
8. **Area** — each series renders both a `<path>` (line) AND a `<polygon>` (filled area); fill sweeps in over the entrance window.
9. **Pie** — N slices fan out from 0° to their terminal angles; per-slice stagger; sum of terminal angles = 360°.
10. **Donut** — same as pie with an inner radius cutout; cutout fraction is `0.55 * outerRadius` (v1 default).
11. **Scatter** — points are rendered at `(x, y)` positions derived deterministically from `(seriesIndex, valueIndex, value)`; opacity fades in per-point.
12. **Combo** — first series renders as bars (per AC #6); subsequent series render as lines (per AC #7) overlaid.

### Animation contract

13. All per-kind animations complete by `floor(0.6 * durationFrames)` from `useVideoConfig()`. Frame `floor(0.6 * durationFrames) + 1` matches frame `durationFrames - 1` byte-for-byte (animation has settled).
14. Easing is `cubicBezier(0.16, 1, 0.3, 1)` shared with `chart-build.tsx`.
15. The `STAGGER_FRAMES` constant is exported once from `chart/constants.ts` and imported by every per-kind renderer. The dispatcher's test asserts the constant is a positive integer; renderer tests use it directly without redeclaring. (Per-renderer redeclaration with cross-file equality assertions is rejected — single source of truth.) `ENTRANCE_FRACTION` (`0.6`) and `EASE_OUT_EXPO` (`cubicBezier(0.16, 1, 0.3, 1)`) live in the same `constants.ts`.

### Theme

16. Renderers read all colors from `themeTokens` (no inline hex literals; pinned via grep test).
17. The `themeSlots` declaration on `defineFrameClip` enumerates every color the renderers consume (8 series + axis + gridline + text).

19. Each renderer's test runs the SVG render twice with the same `(frame, props, theme)` and asserts byte-for-byte equal output.
20. No `Math.random` / `Date.now` / `performance.now` reference in any renderer file (grep-pinned).

## Escalation triggers (CLAUDE.md §6)

- The `themeSlots` mechanism in `defineFrameClip` doesn't support 8+ series colors cleanly. Escalate; we may need to extend `defineFrameClip`'s themeSlots type.
- A reviewer pushes back on shipping all 7 chart kinds in one PR and wants a split (e.g., T-406a core kinds + T-406b circular/scatter/combo). Escalate; the spec authors `4 commits` precisely to allow this split if needed mid-implementation. The implementer can convert commits 2 + 3 into separate PRs if scope balloons.
- A scatter chart with overlapping points renders unreadably without jitter. Escalate; we may need to add jitter as an opt-in prop in v1 rather than deferring entirely to a future task.
- Combo chart's "first series is bars, rest are lines" rule conflicts with what Cluster E presets actually need (e.g., a preset wants line + bar + line). Escalate; we may need an explicit `seriesKind?: ChartKind[]` per-series override.
- The existing `chart-build.tsx` `EASE_OUT_EXPO` constant doesn't import cleanly into the new chart/ directory. Escalate; we may need a small refactor to extract the shared easing.
