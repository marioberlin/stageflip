# T-249 — `@stageflip/design-system` — 8-step theme learning pipeline

**Branch**: `task/T-249-design-system-theme-learning`

## Goal

Convert the `@stageflip/design-system` stub (`export {}` at `packages/design-system/src/index.ts`) into a structural pipeline that **learns a richer `LearnedTheme` from a parsed `Document`**.

Inputs: a `Document` produced by import (T-243a-c PPTX, T-244 Slides, T-247 Hyperframes) where most values are still literals (hex colors on each shape, font-family on each text run, raw spacing in transforms). Outputs: the same document mutated so that most literals are replaced with `theme:foo.bar` token refs, a `LearnedTheme` carrying the extracted tokens, and a `componentLibrary` of recurring element groupings.

The plan-row at `docs/implementation-plan.md:539` is terse: **"8-step theme learning pipeline"**. The 8 steps are defined concretely in §3 below; they're sequenced from cheapest (color clustering) to most expensive (font fetching) so consumers can stop early if they only need a subset.

T-249 owns several adjacent concerns surfaced across other tasks' "Out of scope" sections:

- **T-243c §"Out of scope" → "Font subsetting / glyph-coverage analysis"**: T-249 territory.
- **T-244 §"Out of scope" → "Custom font files referenced by Slides via Google Fonts"**: T-249 owns the font-fetch path.
- **T-251 §"Out of scope" → "Editor UI for editing layouts / masters"**: deferred again here; T-249 ships the schema + learning, not the editor UI.
- **`packages/schema/src/document.ts:42-49`**: `ComponentDefinition.body` is currently `z.unknown()`; T-249 narrows to a typed shape.

## Architectural assumptions (call out for Reviewer)

The plan-row text gives no concrete decomposition of "8 steps." This spec makes architectural decisions the Reviewer should confirm before Implementer dispatch:

1. **The 8 steps** are picked to maximize generic applicability (color/type/spacing extraction → universal) and minimize per-importer special-casing. Alternative decompositions (more steps; fewer) are possible.
2. **The pipeline mutates the input `Document`** (replaces literals with theme refs). Alternative: produce a separate `LearnedTheme` + a sidecar document, leaving the original untouched. **Decision**: in-place mutation matches the "theme:foo.bar" ref shape the schema already supports; the sidecar approach would force every consumer to merge two documents.
3. **Font fetching uses Google Fonts** as the canonical source. Alternative: extensible registry of font sources. **Decision**: v1 = Google Fonts only; T-249-fonts-extended could add others.
4. **Color clustering uses k-means in Lab color space**. Alternative: hierarchical clustering, density-based, or hand-tuned thresholds. **Decision**: k-means is the industry-standard default, deterministic with a fixed seed, and produces reasonable clusters for the typical 5-20 distinct colors per imported deck.

If any of these assumptions don't match orchestrator expectations, escalate before authoring.

## Architectural decisions

### 1. Public surface

```ts
// packages/design-system/src/index.ts

export interface LearnThemeOptions {
  /** Document to learn from (post-applyInheritance, post-resolveAssets). */
  doc: Document;
  /** Resolves a font family name to bytes. Required when the pipeline runs step 6 (font fetching). */
  fontFetcher?: FontFetcher;
  /** Asset storage for uploading fetched font bytes. Required if `fontFetcher` is set. */
  storage?: AssetStorage;
  /** Stop after step N (1-8). Default 8 (full pipeline). Useful for clients that only need color extraction. */
  stopAfterStep?: 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8;
  /** k-means seed for color clustering. Default 42. Must be set explicitly for byte-deterministic output. */
  kMeansSeed?: number;
  /** Color cluster count target. Default 8. Algorithm aims for `min(distinctColors, kMeansTargetClusters)`. */
  kMeansTargetClusters?: number;
}

export interface LearnThemeResult {
  /** Learned theme — superset of canonical Theme. */
  theme: LearnedTheme;
  /** Updated document with literals replaced by theme refs (mutation if in-place; cloned and returned otherwise — see §"Mutation strategy"). */
  document: Document;
  /** Component library extracted from recurring element groupings. */
  componentLibrary: ComponentLibrary;
  /** Loss flags emitted during the pipeline. */
  lossFlags: LossFlag[];
  /** Per-step diagnostics for auditing. */
  stepDiagnostics: StepDiagnostic[];
}

export interface LearnedTheme {
  /** Inherits from `Theme` — keeps tokens + palette compatibility. */
  tokens: ThemeTokens;
  palette: ThemePalette;
  /** Typography scale: token -> font properties. */
  typography: Record<string, TypographyToken>;
  /** Spacing scale: token -> px value. */
  spacing: Record<string, number>;
  /** Font assets: family name -> AssetRef. */
  fontAssets: Record<string, AssetRef>;
  /** Pipeline metadata for round-trip safety. */
  source: { learnedAt: Date; step: 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8; documentId: string };
}

/**
 * MUTATES `opts.doc` — replaces literal hex colors / fonts / spacing with
 * `theme:foo.bar` token refs in step 8. Callers who need the original doc
 * preserved must `structuredClone(doc)` before invoking. Concurrent reads
 * of `opts.doc` while `learnTheme` runs are unsafe; serialize at the
 * caller. See spec §3 "Mutation strategy."
 */
export function learnTheme(opts: LearnThemeOptions): Promise<LearnThemeResult>;
```

`StepDiagnostic` shape (defined alongside the public surface in `types.ts`):

```ts
export type StepDiagnostic =
  | { step: 1; kind: 'color'; clusterCount: number; distinctColors: number; }
  | { step: 2; kind: 'typography'; familyCount: number; sizeVariance: number; }
  | { step: 3; kind: 'spacing'; histogram: Record<number, number>; }
  | { step: 4; kind: 'shape-language'; histogram: Record<ShapeKind, number>; coverage: number; }
  | { step: 5; kind: 'components'; recurringCount: number; perComponentInstanceCount: Record<string, number>; }
  | { step: 6; kind: 'fonts'; fetched: number; failed: number; }
  | { step: 7; kind: 'naming'; ambiguousClusters: number; }
  | { step: 8; kind: 'writeback'; literalsReplaced: number; literalsKept: number; };
```

The discriminated union lets consumers branch on `step` for type-safe diagnostic inspection. Each variant carries the metrics named in its step's "Diagnostic" subsection.

`Document` and `Theme` types are imported from `@stageflip/schema`. The new types (`LearnedTheme`, `TypographyToken`, `ComponentLibrary`, `StepDiagnostic`, `FontFetcher`) live in `@stageflip/design-system/src/types.ts`; the schema-side `Theme` is unchanged in this PR (a future schema PR could promote `LearnedTheme` to be the canonical `Theme`, but T-249 keeps the surfaces separate to avoid coupling consumers to the richer shape).

### 2. The 8 steps

Each step is a pure function `(state) => state`; the pipeline composes them. Steps 1-5 are deterministic; steps 6-8 may emit loss flags or skip work depending on inputs.

#### Step 1 — Color extraction + clustering

Walk every element: read fills (`ShapeElement.fill`, `TableCell.fill`, etc.), strokes (`ShapeElement.stroke.color`), and text-run colors. Build a histogram of distinct hex colors weighted by element count.

Run k-means in Lab color space with `kMeansSeed` (default 42) and `kMeansTargetClusters` (default 8). Produce a palette: cluster centroid → token name (primary / secondary / accent / background / foreground / surface + spillover into `accentN`).

Token paths emitted: `theme:color.primary`, `theme:color.secondary`, etc.

**Diagnostic**: cluster size + cluster's contributing element-count.

#### Step 2 — Typography extraction

Walk every text run: collect `(font-family, font-size, font-weight, italic)` tuples. Cluster by `font-family + size`; assign tokens by frequency:

- Most frequent (font-family + size) → `theme:typography.body`
- Second-most → `theme:typography.subhead`
- Largest size with consistent family → `theme:typography.display` / `theme:typography.h1`
- Etc.

Token shape: `TypographyToken { fontFamily, fontSize, fontWeight, italic, lineHeight? }`.

**Diagnostic**: font-family count + size variance.

#### Step 3 — Spacing extraction

Walk every element's `transform`: collect deltas between sibling elements (gaps, margins, padding inferred from non-overlapping bbox edges). Cluster to a 4-8 step scale: `tight / normal / wide / x-wide` or numeric `s1 / s2 / s3 / s4`.

Token paths: `theme:spacing.tight`, `theme:spacing.normal`, etc.

**Diagnostic**: spacing histogram.

#### Step 4 — Shape primitive language

**Read-only step** — does not mutate the document or emit tokens. Counts usage of each `ShapeKind` (`rect / ellipse / line / polygon / star / custom-path`) and reports the deck's "shape language" (e.g., 80% rect + 15% ellipse + 5% line). Used by editor UI / recommendation surfaces; not consumed by the resolver.

**Diagnostic**: shape histogram with element-count + slide-coverage metrics.

#### Step 5 — Component extraction

Walk `slide.elements`; identify recurring element groupings via structural-hash matching:

1. For each slide, compute a structural fingerprint of its element tree (element types + relative positions, modulo transform offsets).
2. Hash each subtree at depth 1-3.
3. Identify hashes appearing on 3+ slides.
4. For each recurring grouping, generate a `ComponentDefinition`:
   - `id`: synthesized name (e.g., `c-titleBlock`, `c-stat-card`).
   - `body`: structured shape (NOT `z.unknown()`) — `{ slots: SlotDefinition[]; layout: LayoutDescriptor }`.
   - `slots`: identifiable per-instance overrides (text content, image src) — element types that vary across instances become slots.

This step **also** narrows the schema's `ComponentDefinition.body` from `z.unknown()` to a typed shape (per `packages/schema/src/document.ts:42`'s comment "Typed by T-249").

**Diagnostic**: recurring-grouping count + per-component instance count.

#### Step 6 — Font asset fetching

For each font-family in `theme.typography` not present in the local cache:

1. Query Google Fonts API: `https://fonts.googleapis.com/css2?family=<family>:wght@<weights>&display=swap`.
2. Parse the returned CSS for actual font file URLs (woff2/ttf).
3. Fetch font bytes via `fontFetcher`.
4. Upload via `storage.put(bytes, { contentType: 'font/woff2' | 'font/ttf' })`.
5. Write the resulting `AssetRef` into `LearnedTheme.fontAssets[family]`.

Skip this step entirely when `fontFetcher` is undefined; theme is still emitted but `fontAssets` is `{}`. Emit `LF-DESIGN-SYSTEM-FONT-FETCH-FAILED` per family that fails (network error, missing on Google Fonts, license issue).

**Glyph-coverage analysis** (from T-243c deferral): T-249 v1 fetches the full font weight/italic variants needed by the document. Subsetting (only the glyphs actually used) is OOS — a follow-on T-249-font-subsetting could add it.

#### Step 7 — Token naming + assignment

By this step, palette / typography / spacing have been clustered + tokenized. Step 7 assigns **stable names** to clusters:

- Largest color cluster (most element-coverage) → `primary`; next → `secondary`; etc. Background-like colors (lightness > 0.85) → `background`; foreground-like (lightness < 0.20) → `foreground`.
- Most-frequent typography → `body`; largest non-body size → `display` or `h1`.
- Most-frequent spacing → `normal`; smaller → `tight`; larger → `wide`.

This step is deterministic given the deterministic upstream clusters. Palette names follow a fixed mapping; ambiguity (e.g., two clusters tied for "primary") emits `LF-DESIGN-SYSTEM-AMBIGUOUS-CLUSTER`.

#### Step 8 — Token resolution / writeback

Walk the document a second time; for each literal value matching a clustered token:

- Hex color matching a palette cluster's centroid (within ΔE < 5 in Lab space) → replace with `theme:color.<token>`.
- Font-family + size matching a typography token → replace with `theme:typography.<token>`.
- Spacing value matching a spacing token (within ±2 px) → replace with `theme:spacing.<token>`.

The replacement is **in-place mutation** (per §"Architectural assumptions" #2). The returned `LearnThemeResult.document` IS the input mutated.

**Idempotence**: re-running `learnTheme` on the same document with the same opts produces the same `LearnedTheme` and the same writeback — already-tokenized refs pass through unchanged.

### 3. Mutation strategy

The pipeline mutates the input `Document`. Callers who need the original preserved can clone via `structuredClone(doc)` before calling. Documented in the public-surface JSDoc.

This matches the existing schema's `theme:foo.bar` ref shape — a document with literal hex colors and a document with `theme:color.primary` refs are both valid; the resolver handles both.

### 4. Loss flags taxonomy

Three new codes, all `source: 'design-system'`. Wrapper pattern mirrors `@stageflip/import-pptx/src/loss-flags.ts`.

| Code | Severity | Category | Emitted when |
|---|---|---|---|
| `LF-DESIGN-SYSTEM-FONT-FETCH-FAILED` | `error` | `font` | Step 6 couldn't fetch a font (network error, missing on Google Fonts, license rejected). The font-family stays in `theme.typography` but `fontAssets[family]` is undefined. |
| `LF-DESIGN-SYSTEM-AMBIGUOUS-CLUSTER` | `warn` | `theme` | Step 7 found two clusters tied for the same semantic name (e.g., two equally-large color clusters fighting for `primary`). The pipeline picks one deterministically + emits the flag for human review. |
| `LF-DESIGN-SYSTEM-COMPONENT-MERGE-FAILED` | `info` | `shape` | Step 5 detected a recurring grouping but couldn't unify into a `ComponentDefinition` (e.g., the slot identification produces inconsistent shapes across instances). The grouping is dropped from the component library. |

### 5. Determinism

The pipeline is **deterministic** given identical inputs + identical opts (notably `kMeansSeed`). Steps 1-3, 5, 7 are pure functions with no I/O. Step 4 is pure. Step 6 (font fetch) is non-deterministic by nature (network + storage); production tests use a mock `fontFetcher` that returns canned bytes.

Source-level discipline test (mirrors T-253-base AC #28 / T-245 AC #19 / T-247 AC #33 / T-246 AC #29) — no `Date.now` / `Math.random` / `performance.now` / `setTimeout` / `setInterval` in `packages/design-system/src/**` (excluding test files).

The `Date` field on `LearnedTheme.source.learnedAt` is set via `opts.modifiedAt` (or a frozen epoch fallback like T-253-base's `FROZEN_EPOCH`). Required for byte-determinism in tests.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- The 8-step decomposition's cluster boundaries (Lab ΔE < 5 for color match in step 8; 80/20 split target in step 4; 3-slide threshold for step 5 component extraction; ±2 px for spacing match) produce results that don't match a real-world fixture's intuitive theme. The thresholds are pinned but not optimized; if a real deck's expected output requires different thresholds, escalate before tuning silently.
- A real `Document` produced by an importer surfaces a value type T-249 doesn't model (e.g., a gradient fill with multiple color stops). Spec assumes single-color fills; multi-stop gradients OOS — emit `LF-DESIGN-SYSTEM-COMPONENT-MERGE-FAILED` (or a new `LF-DESIGN-SYSTEM-GRADIENT-DROPPED` if the case is common enough). Escalate.
- Google Fonts API rate-limits or returns CORS errors when fetched server-side. Spec assumes server-side execution (`apps/api`); if T-249 is invoked from a browser context, the fetch path needs a proxy. Escalate.
- The `componentDefinitionSchema.body` narrowing breaks an existing fixture (e.g., a hand-authored test document with `components: { foo: { id: 'foo', body: { custom: 'thing' } } }` no longer parses). The narrowing is intended to be additive; if backward-compat fails, investigate before relaxing.
- Step 5's structural-hash matching produces false positives (two semantically distinct groupings hash to the same value). The hash function is intentionally simple (element types + relative positions); if it's too aggressive, escalate before adding heuristics.
- Step 7's color-naming heuristics ("largest cluster → primary") produce ergonomically wrong results (e.g., a color used in 51% of elements gets named primary even though it's a background-like beige). Tune the heuristic only after escalation; the v1 deterministic rule is the contract.
