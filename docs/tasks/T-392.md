# T-392 — `LiveDataClip` `staticFallback` (cached snapshot)

**Branch**: `task/T-392-live-data-clip-static-fallback`

## Goal

Ship the `staticFallback` for `family: 'live-data'`. When the network permission gate denies (rare in v1's no-op grant world but possible after T-403's tenant allowlists land), the tenant policy refuses, or an export pipeline targets a static medium (MP4/PPTX/PDF), the mount-harness routes to a deterministic snapshot: the value of `cachedSnapshot` (a captured response payload supplied at authoring time) rendered as text. Chart rendering is **deferred to T-406's landing** — a follow-up task wires chart-aware rendering into both `liveMount` (T-391) and `staticFallback` (this task) once the chart family is on `main`.

T-392 is the `staticFallback` half of ADR-005 §D1's `LiveDataClip` row. After T-392 merges, `family: 'live-data'` is structurally complete (modulo the chart-rendering follow-up).

T-392 registers `liveDataStaticFallbackGenerator` against the **T-388a static-fallback generator registry** (already on `main`), not via a family-hardcoded harness branch.

## Architectural decisions

### D-T392-1. Schema addition — `cachedSnapshot?` on `liveDataClipPropsSchema`

T-391 shipped `liveDataClipPropsSchema` with `posterFrame` but no snapshot field. T-392 adds:

```ts
// packages/schema/src/clips/interactive/live-data-props.ts (MODIFIED)
export const liveDataClipPropsSchema = z.object({
  // ... existing T-391 fields ...
  /**
   * Captured response snapshot for the static fallback. Optional;
   * absent → an empty placeholder TextElement.
   */
  cachedSnapshot: z.object({
    /** ISO-8601 timestamp at which the snapshot was captured (display only). */
    capturedAt: z.string().min(1),
    /** Status code at capture time (display only). */
    status: z.number().int(),
    /** Parsed body. The schema accepts arbitrary JSON-shaped values. */
    body: z.unknown(),
  }).strict().optional(),
}).strict();
```

Browser-safe. Existing T-391 fixtures continue to validate (the field is optional).

### D-T392-2. Generator output — text rendering of the cached snapshot

`defaultLiveDataStaticFallback` returns:

1. A `TextElement` rendering a header line: `${endpoint} · status ${status} · captured ${capturedAt}` (truncated to a documented char cap if longer; pure transformation).
2. A `TextElement` rendering the JSON-pretty-printed `body` (truncated to a documented char cap; pure transformation).

When `cachedSnapshot` is absent, returns a single placeholder TextElement marked via `id` suffix `-placeholder` whose text is empty (host overrides via app-level i18n; same posture as T-390 D-T390-2).

Layout uses the same vertical-stacking discipline as T-390 (systemPrompt summary + per-turn elements): header at the top, body below, both clamped to fit within `(width, height)` per AC #11. The chart-rendering follow-up replaces step (2) with a chart Element when T-406 lands.

### D-T392-3. Determinism

`defaultLiveDataStaticFallback` is subject to the broad §3 rule (no rAF, no random, no time, no fetch). Pure transformation over inputs. Same posture as T-388 / T-390. JSON pretty-print uses `JSON.stringify(body, null, 2)` (V8 / SpiderMonkey / JSC implementations are spec-deterministic — same input → same output).

### D-T392-4. Telemetry

`live-data-clip.static-fallback.rendered` (attributes: `reason`, `width`, `height`, `hasSnapshot` boolean, `bodyByteLength` integer when snapshot present — NOT bodies; same privacy posture as T-390 D-T390-4 / T-391 D-T391-8). Pin via test (AC #13).

### D-T392-5. Registration via T-388a registry

`packages/runtimes/interactive/src/clips/live-data/index.ts` gains a side-effect call:

```ts
import { staticFallbackGeneratorRegistry } from '../../static-fallback-registry.js';
import { liveDataStaticFallbackGenerator } from './static-fallback.js';

interactiveClipRegistry.register('live-data', liveDataClipFactory);  // existing T-391 line
staticFallbackGeneratorRegistry.register('live-data', liveDataStaticFallbackGenerator);  // NEW
```

The harness's family-agnostic dispatch (T-388a) picks up the registration without further changes.

## Acceptance criteria

### Schema

1. `liveDataClipPropsSchema` accepts a valid `cachedSnapshot` (capturedAt, status, body).
2. `cachedSnapshot: { capturedAt: '', status: 200, body: {} }` → schema error (capturedAt min 1).
3. `cachedSnapshot: { capturedAt: 'x', status: 'oops', body: {} }` → schema error (status must be integer).
4. Existing T-391 fixtures (without `cachedSnapshot`) continue to validate.

### Generator determinism

5. `defaultLiveDataStaticFallback({ width: 640, height: 360, endpoint: 'https://x', cachedSnapshot: { capturedAt: 't', status: 200, body: {a:1} } })` returns the same Element[] byte-for-byte across two calls.
6. Different `endpoint` → different markup (header text differs).
7. Absent `cachedSnapshot` → single placeholder element (no header / body elements).

### Registry integration

8. After importing `clips/live-data`, `staticFallbackGeneratorRegistry.resolve('live-data')` returns the generator.
9. The harness routes empty `staticFallback` for `family: 'live-data'` clips to the generator (verified end-to-end via mount-harness test mirroring T-388 AC #12 / T-390 AC #9).
10. Authored `staticFallback` (non-empty) wins; the generator is still called with `reason: 'authored'` for telemetry (T-388a D-T388a-3 contract).

### Bounds

11. Each TextElement's transform fits within the clip's bounds (no element extends past `width × height`). High-snapshot-volume case included (large body) — overflow drops the body element rather than emitting an out-of-bounds transform. Mirror T-390 AC #11's overflow-guard discipline.
12. Vertical layout: body element's `transform.y` is greater than header's `transform.y`.

### Telemetry

13. `live-data-clip.static-fallback.rendered` emits with documented attribute shape; `hasSnapshot` boolean and `bodyByteLength` integer, NOT bodies.

### Coverage + gates

14. Coverage on every new file ≥ 85 %.
15. `pnpm typecheck`, `pnpm lint`, `pnpm test`, `pnpm check-licenses`, `pnpm check-remotion-imports`, `pnpm check-determinism`, `pnpm check-skill-drift`, `pnpm gen:tool-skills:check`, `pnpm check-preset-integrity`, `pnpm size-limit` all green.
