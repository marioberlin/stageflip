# T-261 — Presence via Realtime Database

**Branch**: `task/T-261-presence-rtdb`

## Goal

Stand up the `@stageflip/presence` package — the second collab plane per ADR-006 §D5 (canonical state on Firestore via Yjs/T-260; ephemeral presence on RTDB). Presence carries cursors, selection, and idle status across collaborating clients with 10–50 ms perceived latency, and is **never** persisted beyond the session.

T-261 ships the package + an RTDB-backed adapter + the in-memory adapter for tests. Editor consumption is a follow-up; T-261 stands up the engine and pins shape via vitest.

## Dependencies

- **Required reading**: ADR-006 §D5 (presence boundary). The presence layer must NOT use `y-protocols/awareness` and must NOT instantiate `Y.Awareness`.
- T-260 (`@stageflip/collab`) — merged. Presence is a sibling, not a consumer; T-261 must compose with `CollabClient.status` but does not import it.

## Architectural decisions

### D-T261-1. Shape (locked by `concepts/collab/SKILL.md:50–57`, refined here)

```ts
export interface Presence {
  userId: string;
  /** Stable color across sessions — derived from userId hash, see D-T261-4. */
  color: string;
  /** Last-seen wall-clock ms. Server validates against staleness; client renews. */
  lastSeenMs: number;
  cursor?: {
    slideId: string;
    x: number;
    y: number;
    /** Cursor coords are slide-local, in canonical-pixel units (per RIR). */
  };
  selection?: {
    /** IDs of selected elements on the active slide. */
    elementIds: string[];
  };
  status?: 'active' | 'idle' | 'away';
}
```

**Why `lastSeenMs` and not server-computed**: clients write their own heartbeat; server-side TTL (D-T261-3) compares to server-time. This keeps the client-side liveness computation cheap (no extra read).

### D-T261-2. Heartbeat + idle/away thresholds

- **Heartbeat cadence**: 10 s. Client writes its presence record (or a `lastSeenMs` field update) every 10 s.
- **Status transitions** (computed client-side, stored in `status` field):
  - `active` while there's been a cursor or selection change in the last 30 s.
  - `idle` after 30 s of no input.
  - `away` after 5 minutes of no input.
- **Server-side TTL**: a presence record older than 30 s is considered stale. Stale records are filtered from `subscribe()` reads but NOT eagerly deleted server-side (RTDB rules + a periodic Cloud Function clean idle entries; out of scope for T-261, follow-up infra task). Filtering is sufficient for v1.
- **Hard disconnect**: RTDB's `onDisconnect().remove()` removes the presence record on socket close. This is RTDB's killer feature for presence; T-261 must wire it correctly (per-record, per-connection).

### D-T261-3. Cursor coalescing

Mouse-move can fire 60 fps. Naive write-through saturates RTDB and burns quota. The `PresenceClient.setCursor(...)` API debounces to **50 ms** (20 fps wire rate). The local-state observable updates instantly (UI sees no lag); the wire write is debounced. Selection changes are NOT debounced — they're user-intent operations and rare.

### D-T261-4. Color assignment

Stable per `userId` across docs and sessions:

```ts
function colorForUserId(userId: string): string {
  const palette = [
    '#3b82f6', '#10b981', '#f59e0b', '#ef4444', '#8b5cf6',
    '#ec4899', '#06b6d4', '#84cc16', '#f97316', '#6366f1',
    '#14b8a6', '#eab308',
  ]; // 12 colors, all WCAG-AA on white, distinguishable to deuteranopes
  let hash = 0;
  for (const ch of userId) hash = (hash * 31 + ch.charCodeAt(0)) | 0;
  return palette[Math.abs(hash) % palette.length]!;
}
```

Pure function. Pinned via test for stability across releases (palette order is part of the API contract — changing it shifts every user's color).

### D-T261-5. Adapter contract

Mirrors the `@stageflip/storage` shape but smaller:

```ts
export interface PresenceAdapter {
  /** Write/refresh this user's presence record. */
  set(docId: string, userId: string, presence: Presence): Promise<void>;
  /** Remove this user's record. Used on intentional client teardown. */
  remove(docId: string, userId: string): Promise<void>;
  /** Subscribe to presence changes for a doc. Yields the full peer map per change. */
  subscribe(docId: string, opts?: { signal?: AbortSignal }): AsyncIterable<Map<string, Presence>>;
  /** Register an onDisconnect handler so the record is removed on socket close. */
  registerDisconnectCleanup(docId: string, userId: string): Promise<void>;
}
```

`registerDisconnectCleanup` is RTDB-specific in concept but deliberately part of the contract so the in-memory adapter can simulate it (via `AbortController` + cleanup), and so any future Postgres-LISTEN/NOTIFY adapter can implement it via session-state tracking.

### D-T261-6. Determinism posture

`packages/presence/**` is NOT clip / runtime code. The `pnpm check-determinism` rules (no `Date.now`, `Math.random`, `setTimeout`, etc.) do not apply. Presence freely uses `Date.now()` for `lastSeenMs`, `setInterval` for heartbeat, `setTimeout` for debounce.

Verify scan list at `scripts/check-determinism.ts`. If `packages/presence/**` ends up in scope, that's a config bug — escalate (do NOT add `// determinism-safe` escapes).

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### `presence.ts` shared types + helper

1. `colorForUserId(userId)` is pure: same input → same output, deterministically. Pin against fixture: `colorForUserId('alice') === '#10b981'` (or whatever the hash maps to — Implementer fixes the expected output once and pins it; the palette order is now part of the API contract).
2. `colorForUserId` returns one of the 12 palette colors — never a random value or empty string.
3. `Presence` type round-trips through `JSON.stringify` / `JSON.parse` with no field loss.

### In-memory adapter (`in-memory.ts`)

4. `set(docId, userId, p)` then `subscribe(docId)` yields a map with `{ userId: p }` on first iteration.
5. `set` with a different `userId` adds; second `subscribe` consumer also sees both.
6. `remove(docId, userId)` deletes — next yielded map omits the user.
7. `registerDisconnectCleanup(docId, userId)` registers cleanup; calling `signal.abort()` on the subscribe-loop's signal triggers `remove(docId, userId)` for that user (in-memory simulates RTDB's `onDisconnect`).
8. Per-doc isolation: a `set('docA', 'alice', p)` does NOT show up in `subscribe('docB')`.
9. Multi-subscriber fan-out: two `subscribe(docId)` consumers both receive every change.

### RTDB adapter (`firebase-rtdb.ts`)

10. Constructor accepts a `Database` instance from `firebase-admin/database`. Pin via fixture using either `firebase-admin`'s built-in mock OR the `@firebase/rules-unit-testing` library — Implementer's call, document the choice.
11. `set` writes to `/presence/{docId}/{userId}`. Verify via direct DB read.
12. `subscribe` uses `child_added` / `child_changed` / `child_removed` events; yields the full peer map after each change. Reuse the `AbortSignal` plumbing pattern from `InMemoryStorageAdapter.subscribeUpdates`.
13. `registerDisconnectCleanup` calls `ref.onDisconnect().remove()` for the user's path. Pin via test: simulate disconnect (test framework's API), verify the path is gone.
14. **Stale filter**: `subscribe` filters records where `Date.now() - lastSeenMs > 30_000` from the yielded map. Pin via fixture using time-mocking.

### PresenceClient (`client.ts`)

15. `new PresenceClient({ adapter, docId, userId, color? })` constructs synchronously; `await client.connect()` writes the first presence record AND registers the disconnect cleanup.
16. `client.setCursor({ slideId, x, y })` updates the local presence and writes through the adapter, **debounced to 50 ms** (D-T261-3). Pin via fake timers: 10 calls within 50 ms produce 1 wire write; the local observable still emits 10 times.
17. `client.setSelection({ elementIds })` is NOT debounced — every call writes through.
18. `client.subscribe()` returns `AsyncIterable<Map<string, Presence>>` excluding the local user (peers only). Pin via test that asserts the local user's `userId` is not in any yielded map.
19. **Heartbeat**: `connect()` schedules a 10 s `setInterval` that updates `lastSeenMs` (and `status` per D-T261-2 thresholds). Pin via fake timers: 35 s elapsed → 3 heartbeats issued; status transitions `active → idle → idle` (depending on input).
20. **Idle/away transitions** (D-T261-2): with no `setCursor` / `setSelection` calls for 30 s, status flips to `idle`; for 5 min, `away`. Pin via fake timers.
21. `client.dispose()` cancels the heartbeat interval, calls `adapter.remove(...)`, aborts the subscribe loop, and is idempotent.
22. **Color**: if not provided in constructor, derived via `colorForUserId(userId)`. If provided, used as-is.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- The RTDB Admin SDK's `onDisconnect` semantics interact badly with the test framework (firebase-admin's mock vs `rules-unit-testing` emulator) such that AC #13 is infeasible to test deterministically. Escalate; do NOT bypass the test by mocking `onDisconnect` at the surface — that misses the whole point.
- The color palette (D-T261-4) has accessibility issues that the Implementer's review surfaces (e.g., a deuteranope tester says #10b981 and #84cc16 are confusable). Escalate; the palette is part of the API contract and changing it shifts every user's color.
- A real-world fixture surfaces a presence rate that the 50 ms cursor debounce can't keep up with (e.g., trackpad inertia firing at 240 Hz). Escalate; widening the debounce window has UX implications that warrant Orchestrator sign-off.
- `pnpm check-determinism` flags `packages/presence/**`. Escalate; D-T261-6 says it's out of scope. Don't add escape comments.
- The `Presence.cursor` units (slide-local canonical pixels) conflict with how the editor canvas reports cursor positions (which may be DOM pixels or some DPI-aware variant). Escalate; the editor consumer (T-261-era follow-up) will need a transform either way, but the engine should pick one canonical unit and stick to it.

## Notes for the Orchestrator

1. **M-sized; expect six commits.** The Presence type + helper + in-memory adapter + RTDB adapter + client + skill update map cleanly to six commits. RTDB adapter tests are the trickiest piece; budget extra time.
2. **Reviewer focus**: AC #13 (RTDB `onDisconnect` actually triggers on socket-close, not just on intentional `dispose`) — get this wrong and the presence database silently fills with ghosts. AC #16 (cursor debounce wire-vs-local — local must emit on every call; only the wire write is debounced). AC #18 (subscribe excludes the local user — common bug to ship a feedback loop where the user sees their own cursor).
3. **No editor-shell changes in this PR.** Editor consumption is a follow-up. T-261's contract pins shape; downstream apps wire it.
4. **`firebase-admin`** is already in workspace via `@stageflip/storage-firebase`. No new dep, no whitelist amendment. The `pnpm check-licenses` pass should be a no-op.
5. **Heartbeat cadence (10 s)** vs **stale TTL (30 s)** — the 3:1 ratio is intentional. A single missed heartbeat doesn't immediately ghost the user; two consecutive misses do. If a follow-up Implementer wants to tune cadence, they must keep the ratio.
6. **D-T261-3 (cursor debounce) is at 50 ms** = 20 fps wire rate. This was chosen because (a) RTDB cost scales with writes, (b) human eye perceives <80 ms as instantaneous for cursor movement, (c) the local observable updates at full rate so animation feels smooth. Don't tune without considering all three.
7. **D-T261-4 color palette** is part of the API contract. Once shipped, every user gets a deterministic color. Changing the palette order shifts every user's color across releases — a visible UX regression. Treat as immutable post-T-261.
