# T-252 — `@stageflip/export-google-slides` — render-diff convergence loop

**Branch**: `task/T-252-export-google-slides`

## Goal

Convert a canonical `Document` into a Google Slides presentation via the Slides API's `presentations.batchUpdate`. The exporter is the **peer of `@stageflip/export-pptx`** (T-253-base) — same input shape, different output target.

The challenge that distinguishes T-252 from T-253-base: **Google Slides server-side rendering silently drifts from any one-shot batchUpdate write**. A `CreateShape` request followed by `InsertText` + `UpdateTextStyle` can produce a slide whose rendered pixels differ from the canonical's intent by more than the user would accept (text autofit changes the font size, padding inferred differently, theme tokens resolve to different actual colors). Round-trip fidelity requires a **render-diff convergence loop**:

```
1. Plan batchUpdate requests in preference order (update > duplicate > create).
2. Apply the planned requests via batchUpdate.
3. Fetch the resulting rendered slide PNG via presentations.pages.getThumbnail (T-244's same endpoint).
4. Diff against a renderer-cdp golden produced from the canonical document.
5. For out-of-tolerance objects, plan adjustment requests (translate / size / font-size / spacing nudges).
6. Apply adjustments. Repeat from step 3 up to max-iterations (default 3).
7. Residual out-of-tolerance objects emit as image-fallback via T-245's rasterizeFromThumbnail
   with LF-GSLIDES-EXPORT-FALLBACK.
```

The loop bounds API call cost (3 iterations × N elements ≈ 100-500 calls per medium deck) while reaching pixel-perfect-tolerance for the elements Slides server-side rendering can match.

**Three tier modes** select different fidelity-vs-editability tradeoffs (per spec §"Tier modes" below); **hybrid is the default**.

## Out of scope

| Item | Why deferred |
|---|---|
| **Animations export** | Slides doesn't model the same animation surface as PPTX; mapping is non-trivial. T-252 v1 emits `LF-GSLIDES-EXPORT-ANIMATIONS-DROPPED` per slide that had any. |
| **Speaker notes export** | Slides has notes (`Page.notesProperties`); v1 deferred. Emit `LF-GSLIDES-EXPORT-NOTES-DROPPED`. |
| **Custom fonts upload** | Slides supports custom fonts via the Drive API; v1 emits `LF-GSLIDES-EXPORT-FONT-SUBSTITUTED` when a canonical font isn't in the supported family list. |
| **Theme override per master** | T-252 v1 only writes to the existing default master; per-master theme overrides are a future task (T-249's territory). |
| **Custom geometry shapes** (`shape: 'custom-path'`) | Slides has no custom-path equivalent. v1 image-fallback via T-245. |
| **Live collaboration awareness** | Concurrent edits to the target presentation can clobber T-252's writes; v1 assumes exclusive write access. A `presentations.get` etag check could detect concurrent edits but is OOS. |
| **Iterative refinement of stub fixtures** | The convergence loop's tolerances are tuned against the canned `renderer-cdp` golden + canned `presentations.pages.getThumbnail` mock responses. Real-world deck testing happens at integration time. |
| **Deck-level metadata round-trip** (`Document.meta.title`) | Slides supports `presentations.update` for metadata; v1 wires title only. Other metadata fields (createdAt, updatedAt) are managed by Slides server-side. |
| **Tables with rotated cells** | Slides API doesn't support cell rotation; emit `LF-GSLIDES-EXPORT-TABLE-ROTATION-LOST`. |

## Architectural decisions

### 1. Public surface

```ts
// packages/export-google-slides/src/index.ts

export type ExportTier = 'fully-editable' | 'hybrid' | 'pixel-perfect-visual';

export interface ExportGoogleSlidesOptions {
  /** Auth provider; same shape as T-244's. */
  auth: GoogleAuthProvider;
  /** Existing presentationId to overwrite. If undefined, creates a new presentation. */
  presentationId?: string;
  /** Renderer-CDP wrapper for producing the canonical-side golden PNGs. */
  renderer: RendererCdpProvider;
  /** Tier mode. Default 'hybrid'. */
  tier?: ExportTier;
  /** Max convergence iterations per slide. Default 3. */
  maxIterations?: number;
  /** API base URL override (testing only). */
  apiBaseUrl?: string;
  /** Per-tolerance overrides (testing only — defaults are spec'd in §3). */
  tolerances?: Partial<ConvergenceTolerances>;
}

export interface ExportGoogleSlidesResult {
  /** The presentationId the deck was written to. */
  presentationId: string;
  /** Loss flags emitted during export. */
  lossFlags: LossFlag[];
  /** Per-slide convergence outcomes for auditing. */
  outcomes: SlideExportOutcome[];
  /** Total Slides API calls made (for cost auditing). */
  apiCallsMade: number;
}

export interface SlideExportOutcome {
  slideId: string;
  /** Number of convergence iterations actually run. */
  iterations: number;
  /** Per-element residual count after convergence (0 = perfect). */
  residualCount: number;
  /** Final per-element diff metrics (text bbox px, image px, perceptual diff %). */
  finalMetrics: { textBboxPx: number; imageShapePx: number; perceptualDiffPct: number };
}

export interface RendererCdpProvider {
  /** Render one slide of the canonical document to PNG bytes at the given size. */
  renderSlide(doc: Document, slideId: string, sizePx: { width: number; height: number }): Promise<Uint8Array>;
}

  doc: Document,
  opts: ExportGoogleSlidesOptions,

```

The `RendererCdpProvider` interface decouples T-252 from the renderer-cdp package's full surface — production wires through `@stageflip/renderer-cdp`'s render-fixture-test pipeline; tests use a stub returning canned PNG bytes per slideId.

### 2. Tier modes

| Mode | Convergence loop runs? | Image-fallback for residuals? | Editability of output |
|---|---|---|---|
| `fully-editable` | NO | NO (residuals stay as canonical-best-effort native objects) | Maximum (every element is a Slides native shape/text/image; users can edit freely; visual fidelity may drift) |
| `hybrid` (default) | YES | YES (T-245 image-fallback for residuals) | High — most elements native; only out-of-tolerance residuals become images |
| `pixel-perfect-visual` | YES (but with `maxIterations: 1` to skip the loop overhead — visual is fixed by image-fallback anyway) | YES, **aggressively**: emits ALL elements as image-fallbacks; deck is structurally one rasterized image per slide | Minimum — users can't edit content; pixel-perfect visual match guaranteed |

Default `hybrid` matches the `~90% deterministic + ~10% AI-or-fallback` pattern that the importer side uses (T-244 + T-246).

### 3. Convergence tolerances

Per plan-row v1.24:

```ts
interface ConvergenceTolerances {
  /** Text element bounding-box position tolerance in pixels. Default 2. */
  textBboxPositionPx: number;
  /** Text element bounding-box size tolerance in pixels. Default 3. */
  textBboxSizePx: number;
  /** Image / shape element position+size tolerance in pixels. Default 1. */
  imageShapePx: number;
  /** Whole-slide perceptual diff threshold (0..1, where 0.01 == 1%). Default 0.02 (2%). */
  perceptualDiffThreshold: number;
}
```

Per-element diff:
- **Text**: extract bbox from the rendered Slides PNG via simple connected-components on the rendered text region; compare against the canonical golden's text bbox. Position OR size out → element is residual.
- **Image / shape**: same connected-components on solid-region; tighter tolerance.
- **Whole-slide perceptual**: pixel-difference normalized against image area. Below threshold → all-good even if individual elements drift slightly.

The diff implementation is in `packages/export-google-slides/src/diff/` — uses the same `pngjs` workspace dep + a small connected-components implementation (or reuses one from `@stageflip/rasterize` if T-245's Implementer surfaced one).

### 4. Renderer-CDP golden

The canonical-side baseline must match the dimensions Slides will return from `presentations.pages.getThumbnail`. T-244's spec §3 says `LARGE` thumbnails are 1600 × auto-height (per aspect). T-252's `RendererCdpProvider.renderSlide` is invoked at the same dimensions per slide.

**The renderer-cdp wrapper does NOT run a full Chrome instance**: production wiring uses `@stageflip/renderer-cdp`'s reference-render fixture path (the same one used by `pnpm parity` and the slide app's e2e). Tests use a stub returning canned PNG bytes per slideId. **Production deployment of the renderer**: T-231 (Cloud Run render worker, already merged) provides the infrastructure; T-252 calls it via a small HTTP wrapper.

### 5. Plan-then-apply pattern

For each slide, T-252 builds a `SlideExportPlan` that lists `presentations.batchUpdate` requests in dependency order, then applies as one transaction:

```
1. Read existing presentation state via presentations.get.
2. For each canonical element on the slide, emit one or more batchUpdate requests:
   a. If element has inheritsFrom AND a matching placeholder exists on the slide's layout/master:
      → emit UpdateShapePropertiesRequest against the inherited shape (preserves theme bindings, font inheritance).
   b. Else if a "similar" object exists on the target slide (same type, near-bbox, near-text):
      → emit a duplicate-then-modify pair (DuplicateObjectRequest + UpdateShapePropertiesRequest + InsertTextRequest + UpdateTextStyleRequest).
   c. Else:
      → emit CreateShapeRequest (or CreateImageRequest, CreateTableRequest, etc.) + InsertTextRequest + UpdateTextStyleRequest.
3. Group export: child shapes are created FIRST, then a single GroupObjectsRequest binds them.
4. Tables: emit CreateTableRequest, then per-cell InsertTextRequest, then MergeTableCellsRequest for spans.
5. Apply the plan via presentations.batchUpdate.
6. Then run the convergence loop (§"Convergence loop" below).
```

The "preference order" (a → b → c) is the optimization target the plan-row v1.24 names. **Reasoning**: option (a) preserves theme + font inheritance and is the cheapest in API-call count; (b) is intermediate (duplicate is one API call, then modify is several); (c) is the most expensive and doesn't preserve theme bindings.

### 6. Convergence loop

For each slide, after the initial plan applies:

```
let iter = 0;
while (iter < maxIterations) {
  iter++;
  const apiPng = await fetchSlideThumbnail(slideId, ...);
  const goldenPng = await renderer.renderSlide(doc, slideId, { width: apiPng.width, height: apiPng.height });
  const diff = computeDiff(apiPng, goldenPng, tolerances);
  if (diff.allElementsInTolerance) break;
  const adjustments = planAdjustments(diff, slidePlan);
  if (adjustments.length === 0) break; // convergence stalled — no further nudges to try
  await batchUpdate(adjustments);
}
// At loop exit, residual out-of-tolerance elements emit LF-GSLIDES-EXPORT-FALLBACK
// and are image-rasterized via T-245's rasterizeFromThumbnail.
const residuals = computeFinalResiduals(...);
for (const r of residuals) {
  // Crop the canonical golden at the element's bbox; rasterize via T-245;
  // upload to Slides as an Image element via CreateImageRequest;
  // remove the original element via DeleteObjectRequest;
  // emit LF-GSLIDES-EXPORT-FALLBACK.
}
```

**Adjustment planning** (`planAdjustments`): given a per-element diff (`text bbox shifted by 5px right, font-size shrunk by 2pt`), produce the corresponding `UpdateShapePropertiesRequest` / `UpdateTextStyleRequest`. The math is the inverse of the observed delta — no model-based optimization, just "move the element by the negative of the observed shift."

If a single element's adjustments don't converge after `maxIterations` (oscillates around a target), it falls into image-fallback.

### 7. Image-fallback via T-245

When an element doesn't converge within tolerance:

1. Crop the canonical golden PNG at the element's bbox via `rasterizeFromThumbnail(goldenPng, elementBbox, { paddingPx: 0 })`.
2. Upload the cropped bytes via Slides Drive API (`drive.files.create` with `mimeType: 'image/png'`).
3. Emit a `CreateImageRequest` with the uploaded image's `contentUrl`.
4. Emit a `DeleteObjectRequest` for the original native element.
5. Emit `LF-GSLIDES-EXPORT-FALLBACK` (severity `warn`, category `media`) on the residual.

**Group-element residuals — Slides cascades child deletes**: if the residual element is a `GroupElement`, emitting a single `DeleteObjectRequest` against the group's `objectId` deletes the group AND every child object in one operation (per Slides API spec). The Implementer **must NOT** emit per-child `DeleteObjectRequest`s — they would target objects that no longer exist after the parent's delete cascades, surfacing as `LF-GSLIDES-EXPORT-API-ERROR` (per AC #25). The image-fallback for a group residual replaces the entire group structure with a single rasterized image element. Pin via fixture (`hybrid-with-residual` extended OR a new `group-residual` fixture).

The `pixel-perfect-visual` tier mode skips the convergence loop entirely and rasterizes everything — every element becomes an image-fallback from the start. This produces a deck that's visually byte-identical to the canonical golden but has zero editability.

### 8. Loss flags taxonomy

Eight new codes, all `source: 'gslides'`. Wrapper auto-fills source via T-244's existing `loss-flags.ts`.

| Code | Severity | Category | Emitted when |
|---|---|---|---|
| `LF-GSLIDES-EXPORT-FALLBACK` | `warn` | `media` | An element couldn't converge within tolerance and was rasterized via T-245. One per residual element. |
| `LF-GSLIDES-EXPORT-API-ERROR` | `error` | `other` | A `presentations.batchUpdate` request returned a 400/403/500. The slide is left in a partial state; one flag per failed request. |
| `LF-GSLIDES-EXPORT-CONVERGENCE-STALLED` | `warn` | `other` | The convergence loop exited because adjustments planned to zero (no further nudges) before tolerance was reached. The element falls into image-fallback per §7. |
| `LF-GSLIDES-EXPORT-ANIMATIONS-DROPPED` | `info` | `animation` | One per slide that had any `ElementBase.animations`. |
| `LF-GSLIDES-EXPORT-NOTES-DROPPED` | `info` | `other` | One per slide that had non-empty `Slide.notes`. |
| `LF-GSLIDES-EXPORT-FONT-SUBSTITUTED` | `warn` | `font` | A canonical font isn't in Slides' supported family list; the writer picked the closest match. |
| `LF-GSLIDES-EXPORT-TABLE-ROTATION-LOST` | `warn` | `shape` | A table cell had non-zero rotation; Slides' table API doesn't support per-cell rotation. |
| `LF-GSLIDES-EXPORT-CUSTOM-GEOMETRY-DEGRADED` | `warn` | `shape` | A `ShapeElement` with `shape: 'custom-path'` falls into image-fallback (Slides has no custom-path equivalent). |

### 9. Determinism

The export is **non-deterministic by nature** — Slides server-side rendering is the source of variability. T-252's tests use:

- A stub `RendererCdpProvider` returning canned PNG bytes per slideId.
- A stub Slides API client returning canned `Thumbnail` responses + canned `batchUpdate` echo responses.
- A stub `AssetReader` (for image bytes).

Source-level discipline test (mirrors T-253-base AC #28 + T-245 AC #19 + T-247 AC #33 + T-246 AC #29) — no `Date.now` / `Math.random` / `performance.now` etc. in `packages/export-google-slides/src/**`.

### 10. Round-trip with T-244

T-252 ships a `roundtrip.test.ts` parallel to T-253-base's: fixtures export → re-import via `parseGoogleSlides` (T-244), with the second-pass `Document` structurally equal under a documented predicate.

Predicate exclusions:

- Animations dropped → second pass has empty animations.
- Notes dropped → empty notes.
- Image-fallback elements: the second pass sees `ImageElement`s, not the original element types; the predicate accepts this transformation when the `LF-GSLIDES-EXPORT-FALLBACK` flag fires, but pins that no other elements were silently rasterized.

The round-trip suite uses canned API responses to avoid live calls in CI.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### Public surface

1. `exportGoogleSlides(doc, opts)` returns `Promise<ExportGoogleSlidesResult>` with the four documented fields. Pin via TypeScript + sanity test.
2. `opts.presentationId` undefined → creates a new presentation; defined → overwrites the existing presentation (verified via mock api client recording the `create` vs. `batchUpdate` calls). Pin both branches.
3. `opts.tier` defaults to `'hybrid'`. Pin via test that omits the option and verifies hybrid behavior on a fixture.

### Tier modes

4. **`fully-editable`**: no convergence loop runs (`outcomes[*].iterations === 0`), no image-fallback (`outcomes[*].residualCount === 0`). Residual elements stay as native objects with whatever drift Slides server-side rendering introduced. Pin.
5. **`hybrid`** (default): convergence loop runs up to `maxIterations`; out-of-tolerance residuals image-fallback. Pin via fixture with one residual.
6. **`pixel-perfect-visual`**: no convergence iterations beyond 1; ALL elements become image-fallbacks regardless of would-be-converged status. Pin via fixture verifying every element on output is an `ImageElement`.

### Plan emission preference order

7. **Option (a) — `inheritsFrom`-aware update**: an element with `inheritsFrom = { templateId, placeholderIdx }` resolving to a slide layout/master placeholder emits `UpdateShapePropertiesRequest` (NOT `CreateShapeRequest`). Pin via fixture (`placeholder-aware`).
8. **Option (b) — duplicate similar**: an element with no `inheritsFrom` but a similar object on the target slide (same type, bbox center within 50 px, text 80% Levenshtein-similar) emits `DuplicateObjectRequest` + modifications. Pin via fixture (`duplicate-similar`).
9. **Option (c) — create from scratch**: an element with no `inheritsFrom` and no similar target emits `CreateShapeRequest`. Pin.
10. **Group export ordering**: a `GroupElement` with N children emits N child-create requests FIRST, then a single `GroupObjectsRequest` referencing the just-created object IDs. Pin via fixture.
11. **Table emission**: a `TableElement` with M rows × N cols emits one `CreateTableRequest` + (M × N) `InsertTextRequest`s + (per-merged-cell) `MergeTableCellsRequest`s. Pin.

### Convergence loop

12. Loop iterates exactly `maxIterations` times when all iterations have residuals (`outcomes[*].iterations === maxIterations`). Pin.
13. Loop exits early when `diff.allElementsInTolerance` returns true. Pin via fixture where the first iteration converges.
14. Loop exits with `LF-GSLIDES-EXPORT-CONVERGENCE-STALLED` when `planAdjustments` returns an empty array (no further nudges). The stalled element ALSO falls into image-fallback per §7, emitting **both** `LF-GSLIDES-EXPORT-CONVERGENCE-STALLED` AND `LF-GSLIDES-EXPORT-FALLBACK` on the same residual (per Notes for the Orchestrator §6). Pin both flags + verify the residual was rasterized via T-245 + the original element was deleted from the slide.
15. Per-element tolerance: text bbox position drift within `textBboxPositionPx: 2` is in-tolerance; > 2 px is residual. Pin both branches.
16. Per-element tolerance: image/shape drift within `imageShapePx: 1` is in-tolerance; > 1 px is residual. Pin.
17. Whole-slide perceptual diff: < `perceptualDiffThreshold: 0.02` (2%) is in-tolerance even if individual elements drift. Pin via fixture with sub-threshold per-element drift summing to < 2% pixel-area difference.

### Image-fallback

18. A residual element triggers `rasterizeFromThumbnail` (T-245) on the canonical golden, with `paddingPx: 0`. Pin via mock `@stageflip/rasterize` recording the call.
19. The cropped image bytes are uploaded via Drive API (mock api client records `drive.files.create` with `mimeType: 'image/png'`). Pin.
20. The original native element is deleted via `DeleteObjectRequest`; the new image element is created via `CreateImageRequest` with the uploaded `contentUrl`. Pin.
21. Each residual emits exactly one `LF-GSLIDES-EXPORT-FALLBACK`. Pin.

### Loss flags

23. All eight `GSlidesExportLossFlagCode` variants resolve to a `CODE_DEFAULTS` entry. Pin via test that iterates the union.

25. API errors on a `batchUpdate` request emit one `LF-GSLIDES-EXPORT-API-ERROR` per failed request, with the request id and the API's error message in `originalSnippet`. Pin.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- The Slides API documented `batchUpdate` request types diverge from what the spec assumes (e.g., `DuplicateObjectRequest` doesn't preserve theme bindings as expected). Investigate first; if the API's actual behavior breaks the preference-order optimization, escalate before downgrading to "always create."
- The renderer-cdp golden produces dimensions that don't match Slides' returned `Thumbnail` dimensions (e.g., aspect-ratio rounding). Investigate the upstream renderer; the diff math depends on exact-dimension matches.
- The convergence loop oscillates: an adjustment moves an element 2 px right, the next iteration moves it 2 px left, repeating forever (until maxIterations). Spec says this falls into image-fallback; if the oscillation indicates a real bug in `planAdjustments`, escalate before silently accepting image-fallback for every iter.
- Image-fallback bytes upload fails because Drive API quota is hit. The `LF-GSLIDES-EXPORT-API-ERROR` flag covers this; but if the per-slide quota cost is higher than expected (e.g., 10+ uploads per slide), escalate — the design assumes ~1-2 fallbacks per slide.
- The "similar object" heuristic in option (b) — same type, near-bbox, near-text — produces false positives (e.g., two unrelated buttons get duplicated because they share text). Escalate; the heuristic may need tightening.
- A real `presentations.batchUpdate` response has `replies` field shapes the spec didn't enumerate. Don't silently extend the parser; escalate.
- An element type emitted by the canonical Document has no Slides equivalent (e.g., a `VideoElement`). Spec says image-fallback; if the rasterized image-fallback can't represent the video correctly (e.g., it'd be a still frame), escalate.
