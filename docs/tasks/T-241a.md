# T-241a — PPTX nested group transform accumulator

**Branch**: `task/T-241a-pptx-group-transform-accumulator`

## Goal

Walk the parser-side group tree emitted by T-240 (`@stageflip/import-pptx`) and accumulate each group's `<a:xfrm>` transform into descendant transforms so leaf children carry world-space coordinates. Resolves every `LF-PPTX-NESTED-GROUP-TRANSFORM` flag T-240 currently emits at every group node.

The plan describes T-241a as "the #1 source of OOXML parse failures" — get the math right.

## Type-layer additions (parser-side, no schema change)

`ParsedGroupElement` gains two parser-side fields capturing the OOXML child-coordinate space:

- `groupOrigin: { x: number; y: number }` — from `<a:grpSpPr><a:xfrm><a:chOff x="…" y="…"/>`. Defaults to `{ x: 0, y: 0 }` when absent.
- `groupExtent: { width: number; height: number }` — from `<a:grpSpPr><a:xfrm><a:chExt cx="…" cy="…"/>`. Defaults to the group's own `transform.{width,height}` when absent.

Both stored in EMU-derived px (matches T-240's `emuToPx` convention) so accumulation math stays unit-consistent.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

1. `accumulateGroupTransforms(tree: CanonicalSlideTree): CanonicalSlideTree` returns a new tree with leaf-element transforms in world space. Pure function: no I/O, no Date, no Math.random.
2. After accumulation, **zero** `LF-PPTX-NESTED-GROUP-TRANSFORM` flags remain in `tree.lossFlags`. T-241a's commit also removes the flag emission from `parts/sp-tree.ts:parseGroup` so downstream callers don't see ghost flags.
3. Single-group accumulation: a group at world `(100, 100)` with `chOff=(0,0)` `chExt=group.ext` containing a child at local `(10, 10)` produces a child whose world transform reads `(110, 110)`.
4. **Child-coord scaling**: when `chExt ≠ group.ext`, child positions scale by `group.ext / chExt`. Example: group `ext=(200,200)`, `chExt=(100,100)`, child local `(50,50)` → child world `(group.x + 100, group.y + 100)`.
5. **Rotation accumulation**: group rotation `r_g` + child rotation `r_c` → child world rotation `r_g + r_c` (modulo 360, but preserved as-stored). Position is rotated around the group's center per the OOXML spec.
6. **Multi-level nesting**: group-in-group accumulates correctly across N levels. Test pins N=3.
7. **Group node retention**: the group node itself remains in the tree (callers that want to render group bounds still can); its `transform` is left as the group's own world transform, not zeroed.
8. Coverage ≥85% on the new `transforms/` module.
9. The `transforms/` pass is **idempotent**: running it twice produces the same output as running it once. Test pins this — guards against accidental double-application.
10. Determinism gate (`pnpm check-determinism`) still passes; no new forbidden-API uses.

## Affine math reference (cite the public OOXML spec — do not vendor)

- ECMA-376 §20.1.7.5 (`<a:xfrm>` for groups), §20.1.7.6 (`<a:chOff>`), §20.1.7.7 (`<a:chExt>`).
- https://learn.microsoft.com/en-us/openspecs/office_standards/ms-pptx/ — search "group transform".
- https://officeopenxml.com/drwSp-rotate.php — clean prose summary of rotate semantics; cite, don't copy.

The world transform of a child element nested in a group is composed as:

```
scale_x = group.ext.cx / chExt.cx
scale_y = group.ext.cy / chExt.cy

local_x = child.off.x − chOff.x
local_y = child.off.y − chOff.y

world_x = group.off.x + scale_x * local_x
world_y = group.off.y + scale_y * local_y

world_w = scale_x * child.ext.cx
world_h = scale_y * child.ext.cy

world_rotation = group.rotation + child.rotation
```

Multi-level nesting recurses outside-in: apply the outermost group's transformation first, then the next, etc. Implementer should derive this independently from the spec, not from this summary.
