# T-267 — Stripe billing + usage metering — SPEC ONLY

**Status**: SPEC ONLY. Per user direction (2026-04-28), implementation is deferred to the final phase. This document fixes the architecture so any future Implementer can pick it up without re-deciding.

**Branch (when implementing)**: `task/T-267-stripe-billing`

## Goal

Define the billing + metering architecture so:

1. **Plan tiers** (Free / Pro / Enterprise) gate features and resource quotas.
2. **Usage events** (renders, AI calls, asset storage GB-months) are metered per org and reconciled to Stripe at month-end.
3. **Billing surface** (subscription management, invoice history) is org-admin-accessible via Stripe Customer Portal embed or callable functions.
4. **Usage rate-limits** (T-263) integrate with plan tiers — Free tier gets the smallest buckets, Enterprise gets configurable.
5. **Pause/cancel** flows handle dunning + grace period without losing data.

T-267's deferral is intentional: until we have usage data from real customers, the plan-tier numbers (price, quotas) are speculative. The architecture lands now; the dollar-figure parameters land at implementation.

## Dependencies (when implementing)

- T-262 (auth + tenancy) — merged. Org has `id`, `ownerId`, `plan` field; T-267 extends the plan field's semantics.

## Architectural decisions

### D-T267-1. Three plan tiers

| Tier | Audience | Concurrency | AI calls/mo | Storage GB | Render minutes/mo | Price* |
|---|---|---|---|---|---|---|
| **Free** | Solo / hobbyist | 1 user | 100 | 1 GB | 10 | $0 |
| **Pro** | Small team | 5 users | 5,000 | 50 GB | 500 | $TBD/mo |
| **Enterprise** | Customizable | Unlimited | Unlimited | Custom | Custom | Sales contract |

*Prices are placeholders. The user owns the price decision at implementation time.

Plan limits are surfaced in `OrgPlan` schema (extends `Org.plan` from T-262):

```ts
export const orgPlanSchema = z.object({
  tier: z.enum(['free', 'pro', 'enterprise']),
  /** Stripe subscription id; null for Free tier. */
  stripeSubscriptionId: z.string().nullable(),
  /** Stripe customer id; created on first paid plan. */
  stripeCustomerId: z.string().nullable(),
  /** Effective dates of the current period. */
  currentPeriodStart: z.string().datetime(),
  currentPeriodEnd: z.string().datetime(),
  /** Resource limits — copied from tier defaults but Enterprise can override. */
  limits: z.object({
    users: z.number().int().nonnegative(),
    aiCallsPerMonth: z.number().int().nonnegative(),
    storageGB: z.number().nonnegative(),
    renderMinutesPerMonth: z.number().int().nonnegative(),
    rateLimitUserCapacity: z.number().int().positive(),
    rateLimitOrgCapacity: z.number().int().positive(),
  }),
  /** Status — drives feature gating + read-only-mode. */
  status: z.enum(['active', 'past_due', 'canceled', 'paused']),
});
```

### D-T267-2. Usage event model

Two layers of metering:

1. **Hot path**: every API call that consumes resources logs a usage event via the observability logger (T-264). Format:
   ```
   { event: 'usage', kind: 'ai-call' | 'render' | 'storage-write', orgId, amount, costUnit, ts }
   ```
   - `amount` in the event's natural unit (1 AI call, 0.3 render minutes, 0.5 GB).
   - `costUnit` is human-readable for ops (`'1-call'`, `'minutes'`, `'GB-months'`).
   - Goes to pino + auto-correlated to OTel trace.

2. **Reconciliation**: a daily Cloud Function reads the previous day's usage events from the structured log archive (Cloud Logging → BigQuery sink — note: T-273 is dropped, so this needs a different sink; spec says use Firestore instead):
   - Aggregates per-org, per-event-kind.
   - Compares against `org.limits` for the period.
   - Reports overage to Stripe via Stripe Metering API.
   - Writes the daily aggregate to `orgs/{orgId}/usage/{YYYY-MM-DD}` for visibility.

**Note on T-273 drop**: the original spec assumed BigQuery export. With BigQuery dropped, we use Firestore for the daily aggregates. Hot-path logs still go to pino; reconciliation reads from a new `usage-events` Firestore collection (each high-cost call writes both a log line AND a Firestore doc; cheaper events log only).

### D-T267-3. Stripe integration topology

```
Client (auth-client)
  ├─ org-admin clicks "Upgrade"
  ↓
Cloud Function: createCheckoutSession(orgId, tier)
  ├─ creates Stripe checkout URL
  ↓
Client redirects to Stripe Checkout
  ↓
On success: Stripe webhook → Cloud Function: handleStripeWebhook
  ├─ updates orgs/{orgId}/plan to match the new subscription
  ├─ updates org.limits per the new tier
  ↓
Subsequent API calls see the new plan via T-262's JWT claims (refreshed)
```

Webhook idempotency via Stripe's `event.id` + a `processed-events` Firestore collection.

### D-T267-4. Customer portal embed

For self-serve subscription management (cancel, upgrade, update card), Stripe Customer Portal is the primary surface. T-267 ships a Cloud Function that mints a one-time portal URL (`createPortalSession(orgId)`) and the client redirects to it.

### D-T267-5. Past-due / read-only mode

When `org.status === 'past_due'` (Stripe webhook flips it after dunning failures):

- Read operations succeed.
- Write operations return 402 Payment Required with a structured body pointing at the portal.
- Renders are NOT processed (queue submissions return 402).
- Auth is unaffected (users can still log in).

After 14 days past_due → `canceled` → org enters read-only mode. After 30 days canceled → data archived (separate operational task; out of T-267 scope).

### D-T267-6. Rate-limit integration

T-263's rate limiter accepts capacity overrides. T-267 wires:

```ts
const limiter = new RateLimiter({
  redis,
  config: orgPlanLimitsToRateLimitConfig(org.plan.limits),
});
```

Per-org Enterprise customers can negotiate higher caps; the limits are stored in `org.plan.limits` and the limiter reads them.

### D-T267-7. Determinism + observability

`packages/billing/**` is NOT in determinism scan. Stripe API calls are network-driven.

Every Stripe call is wrapped in an OTel span; failures call `captureError`. Webhook processing is idempotent (D-T267-3).

## Acceptance criteria (when implementing)

### Plan schema

1. `orgPlanSchema.parse({ tier: 'free', limits: { ... } })` accepts; pin defaults match D-T267-1.
2. Tier defaults are exported as constants; Enterprise tier can override `limits` at any value.
3. Plan transitions (Free → Pro → Enterprise) are validated; downgrade to Free is allowed; transitions across tiers preserve `currentPeriodStart`/`End`.

### Stripe integration

4. `createCheckoutSession(orgId, tier)` requires `requireRole('admin')` (T-262); creates a Stripe Checkout URL with the correct line items per tier; returns `{ url, sessionId }`.
5. `createPortalSession(orgId)` requires `admin`; returns `{ url }` to Stripe Customer Portal.
6. `handleStripeWebhook` HTTP endpoint: verifies the webhook signature; handles `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`; updates the org's plan accordingly; returns 200.
7. Webhook idempotency: replaying the same `event.id` is a no-op.

### Usage events

8. `writeUsageEvent({ orgId, kind, amount })` writes a Firestore doc + a structured log line. Pin via test.
9. The reconciler aggregates per-org per-day; produces `orgs/{orgId}/usage/{YYYY-MM-DD}` docs; reports overages to Stripe Metering API for paid plans.
10. **Free tier**: overage triggers a soft-limit response (rate-limited 429 from T-263 with `code: 'PLAN_LIMIT_EXCEEDED'`); paid tiers bill the overage.

### Read-only mode

11. `org.status === 'past_due'` → write operations return 402 with `{ code: 'PAYMENT_REQUIRED', portalUrl }`.
12. `org.status === 'canceled'` → all writes return 402; reads succeed; renders rejected.
13. State transitions emit Sentry breadcrumbs via T-264.

### Rate-limit integration

14. `orgPlanLimitsToRateLimitConfig(limits)` converts `OrgPlan.limits` to the `RateLimiter` config shape.
15. Enterprise tier with custom limits applies them at limiter init.

### Portal hooks

16. `useOrgPlan()` React hook (in `@stageflip/auth-client` or new `@stageflip/billing-client`) returns the current plan + status reactively.
17. `usePlanLimits()` returns derived limits per the current plan.

### Backward compatibility

18. The full workspace test suite passes unchanged.
19. Existing orgs (created before T-267) default to `tier: 'free'` via Zod default.

## Escalation triggers (when implementing)

- Stripe Webhook signature verification fails with the chosen webhook secret. Probably a config issue; verify before implementing custom signature logic (Stripe SDK provides `constructEvent` — use it).
- Stripe Metering API has different semantics than expected (e.g., aggregation windows). Escalate; spec assumes monthly aggregation, but Metering API may report differently.
- Past-due → read-only mode interferes with offline-first editor flows (writes queue locally). Escalate; we may need to differentiate "payment required" from "transient error" at the editor layer.
- The 402 response shape conflicts with rate-limit's 429 wire shape (both denying writes, different reasons). Verify the auth-middleware error layer can return either cleanly.
