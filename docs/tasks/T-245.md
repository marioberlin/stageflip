# T-245 — Slide rasterization primitive (`@stageflip/rasterize`)

**Branch**: `task/T-245-rasterize-primitive`

## Goal

Ship `rasterizeFromThumbnail(pageImage, bboxPx, opts) => RasterizedAsset` — a pure-TS PNG-crop primitive that takes a rendered slide page (a PNG `Uint8Array`) plus a pixel-space bounding box, crops the box (with optional padding), and returns the cropped bytes plus a content-hashed asset id ready for `AssetStorage.put`.

The plan-row v1.24 at `docs/implementation-plan.md:533` says the primitive "lives in `@stageflip/import-google-slides` or a new thin shared package." **Decision**: ship a new shared package `@stageflip/rasterize`. Reasoning:

- Both **import** (T-244 image-fallback for unresolvable elements) and **export** (T-252 image-fallback for residuals after the render-diff convergence loop) consume it.
- Burying it inside `@stageflip/import-google-slides` would force `@stageflip/export-google-slides` to depend on the importer — a cycle in spirit if not in package graph.
- The surface is intentionally tiny (one function + types). A 1-function package is justified when the alternative is a backwards directional dependency.

The primitive has **no awareness of import or export semantics** — no Slides API knowledge, no Drawing ML knowledge, no loss flags. It just crops PNG bytes deterministically. Consumers attach the loss-flag emission and the `AssetStorage` upload at their layer.

## Dependencies

- T-243a + T-243-storage-adapter merged. The `AssetStorage` interface from `@stageflip/import-pptx` provides the `.put(bytes, { contentType })` shape T-245's consumers will call. **T-245 itself does NOT call `AssetStorage`** — it returns bytes + a content hash; the consumer uploads.
- T-244 spec merged (#200). T-244 declares T-245 as the materialization path for `LF-GSLIDES-IMAGE-FALLBACK`. T-245 ships the primitive; T-244's Implementer wires it later as a separate post-walk pass.
- T-247-loss-flags merged. Open-`source` `LossFlag` shape is what consumers use; T-245 is loss-flag-free.

**Does NOT depend on**: T-244 implementation (T-245 is a primitive, not a parser), T-246, T-251, T-252, T-253.

**Blocks**: T-244 implementation's image-fallback path (declared in T-244 spec §"Out of scope" → "The slide rasterization primitive" row); T-252 (`@stageflip/export-google-slides` — image-fallback after render-diff residual).

## Out of scope

| Item | Why deferred |
|---|---|
| `AssetStorage.put` invocation | T-245 returns `{ bytes, contentType, contentHashId }`; consumers (`@stageflip/import-google-slides`, `@stageflip/export-google-slides`) wire the upload at their layer. Mirrors T-243a's separation between `parsePptx` (structural) and `resolveAssets` (I/O). |
| Loss-flag emission | The primitive is shape-agnostic; flags belong to consumers. |
| Re-encoding to formats other than PNG | v1 emits PNG only. WebP / AVIF deferred — not needed for image-fallback paths. |
| Resampling / scaling | The primitive crops pixel-exact slices. If the consumer needs a different resolution, it does that at a separate layer. T-244 always uses the rendered PNG at its actual returned dimensions per T-244 spec §3 — no upstream scale. |
| Color-profile handling | PNG bytes flow through unchanged at the pixel level; ICC profile chunks pass through (`pngjs` preserves them). |
| Animated / multi-frame source | Source is a single-frame PNG. APNG / GIF would need a different surface. |
| Native-binding implementations (`sharp`, `@napi-rs/canvas`) | Pure-JS via `pngjs` keeps the package install footprint small and the license whitelist clean. `sharp` is LGPL-3.0; using it would require dynamic-linking ADR + per-package whitelist (per `THIRD_PARTY.md` §1). The primitive's compute is light (one PNG decode + crop + encode per call); pure-JS is fine. |

## Architectural decisions

### 1. Public surface

```ts
// packages/rasterize/src/index.ts

export interface RasterizeOptions {
  /**
   * Padding in pixels added to each side of the bbox before cropping.
   * Helps the consumer recover anti-aliased edges and small overflows that
   * the source bbox didn't quite include. Default 16.
   */
  paddingPx?: number;
  /**
   * PNG compression level for the re-encode (0–9). Pinned default 6
   * matches `pngjs`'s reasonable compromise between size + speed.
   * Lower → larger output but faster encode. Required for byte-determinism;
   * different levels produce different bytes.
   */
  compressionLevel?: number;
  /**
   * PNG filter type for the re-encode. Pinned default `pngjs.constants.PNG_ALL_FILTERS`
   * (per-row adaptive). Required for byte-determinism.
   */
  filterType?: number;
}

export interface RasterizedAsset {
  /** PNG bytes of the cropped region. */
  bytes: Uint8Array;
  /** Content type — always `'image/png'` in v1. */
  contentType: 'image/png';
  /**
   * Content hash of the bytes (sha256, hex, full 64 chars). Consumers can
   * pass this to `AssetStorage.put` as a hint, but the storage's returned
   * id is the source of truth (mirrors T-243a's pattern).
   */
  contentHashId: string;
  /** Actual cropped dimensions (post-padding, post-clamp). */
  width: number;
  height: number;
}

export interface BboxPx {
  /** Top-left x in source-pixel coordinates (NOT EMU). */
  x: number;
  y: number;
  width: number;
  height: number;
}

export function rasterizeFromThumbnail(
  pageImage: Uint8Array,
  bboxPx: BboxPx,
  opts?: RasterizeOptions,
): Promise<RasterizedAsset>;

export type { BboxPx, RasterizeOptions, RasterizedAsset };
```

### 2. Algorithm

```
1. Decode pageImage → ImageData (width, height, RGBA pixel buffer) via pngjs.
2. Compute the padded bbox:
     padded.x = max(0, bboxPx.x - paddingPx)
     padded.y = max(0, bboxPx.y - paddingPx)
     padded.width  = min(image.width  - padded.x, bboxPx.width  + 2 * paddingPx)
     padded.height = min(image.height - padded.y, bboxPx.height + 2 * paddingPx)
3. Allocate cropped pixel buffer (padded.width × padded.height × 4 bytes).
4. Copy rows from the source's RGBA buffer.
5. Encode cropped pixel buffer → PNG bytes via pngjs (pinned compressionLevel + filterType).
6. Compute sha256 of the encoded bytes.
7. Return { bytes, contentType: 'image/png', contentHashId, width: padded.width, height: padded.height }.
```

The **clamp behavior** (step 2) is the only non-trivial part. A bbox extending off the source image is clamped to the visible region — this matches the consumer expectation that elements partially off-canvas (which can happen via animation interpolation or imported decks) still produce a finite cropped asset.

### 3. Determinism

The primitive must be byte-deterministic: same `pageImage` + `bboxPx` + `opts` → byte-identical `RasterizedAsset.bytes` and `contentHashId`.

Sources of non-determinism that the spec pins:

- **`compressionLevel` + `filterType`**: pinned defaults via Zod-validated options. A consumer override is allowed but the validator rejects unsupported values.
- **PNG metadata**: `pngjs` preserves the source PNG's color type, bit depth, and gamma when re-encoding the cropped output via `PNG.write()`. The crop output's metadata matches the source's; no per-call clock or random bytes.
- **No timestamps in output**: PNG writes don't include creation timestamps unless explicitly opted in. `pngjs.PNG`'s default doesn't.
- **Hash function**: sha256 via `node:crypto`'s `createHash` — already used by `@stageflip/loss-flags` for stable id generation; deterministic.

This package is **not** in CLAUDE.md §3's determinism-restricted scope, but the source-level discipline applies (no `Date.now` / `Math.random` / `performance.now` in package code). Add a grep test mirroring T-253-base AC #28.

### 4. Validation

Inputs are validated before any work:

- `pageImage` must be non-empty `Uint8Array` with valid PNG signature (first 8 bytes match `89 50 4E 47 0D 0A 1A 0A`). Throw `RasterizeError({ code: 'INVALID_PNG' })` otherwise.
- `bboxPx`: `width > 0`, `height > 0`, `x >= 0`, `y >= 0`. The bbox MAY extend past the source's bounds (clamped per §2 step 2), but a fully-out-of-bounds bbox throws `RasterizeError({ code: 'BBOX_OUT_OF_BOUNDS' })` — emitting a 0-pixel asset isn't useful.
- `paddingPx`: must be `>= 0`. NaN / negative throws.

### 5. Error taxonomy

```ts
export type RasterizeErrorCode =
  | 'INVALID_PNG'           // Source bytes don't decode
  | 'BBOX_OUT_OF_BOUNDS'    // Bbox doesn't intersect the image at all
  | 'BBOX_INVALID'          // Negative width/height/x/y or NaN
  | 'OPTIONS_INVALID'       // compressionLevel out of 0–9, etc.
  | 'ENCODE_FAILED';        // pngjs encode threw

export class RasterizeError extends Error {
  code: RasterizeErrorCode;
  cause?: unknown;
  constructor(code: RasterizeErrorCode, message: string, cause?: unknown);
}
```

Errors do NOT emit loss flags. The primitive is shape-agnostic; consumers translate errors into their domain's flags.

### 6. Tests-first contract

Per CLAUDE.md §3, every AC gets a Vitest test failing first. The fixtures are tiny (~few-hundred-byte PNGs hand-built via `pngjs.PNG.sync.write` to avoid binary fixtures in git).

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### Public surface

1. `rasterizeFromThumbnail(pageImage, bboxPx)` returns a `Promise<RasterizedAsset>` with the four documented fields. Pin via TypeScript compilation + a sanity test.
2. The default `paddingPx` is 16. Pin: a 100×100 source with bbox `{x:40, y:40, width:20, height:20}` produces a cropped output of `{width: 20+32, height: 20+32}` = 52×52.
3. The default `compressionLevel` is 6. Pin via test that overrides explicitly and gets different bytes.
4. The default `filterType` is `pngjs.constants.PNG_ALL_FILTERS` (4). Pin via test.

### Crop algorithm

5. **Pixel-exact crop**: a source PNG with a known checkerboard pattern, cropped at a non-padded bbox, produces an output whose pixel buffer matches the source's region byte-identical. Pin via fixture: 16×16 source with alternating-color 4×4 tiles, bbox `{x:4, y:4, width:8, height:8}` (`paddingPx: 0`) → 8×8 crop with the expected tile pattern.
6. **Padding**: with `paddingPx: 4`, the same bbox produces a 16×16 crop with 4 extra pixels on each side, matching the source's surrounding region.
7. **Clamp on top-left**: a bbox starting at `{x:-10, y:-10, width:20, height:20}` (or with positive coords + padding that extends past origin) clamps to `{x:0, y:0, width:10, height:10}` (the visible region only). Pin via fixture.
8. **Clamp on bottom-right**: a bbox extending past the source's right/bottom edge clamps similarly. Pin.
9. **Bbox exactly at edges**: a bbox `{x:0, y:0, width:imageW, height:imageH}` (with `paddingPx: 0`) produces an output byte-identical to the source. Pin.

### Validation

10. **Invalid PNG signature**: bytes that don't start with `89 50 4E 47 0D 0A 1A 0A` → throws `RasterizeError({ code: 'INVALID_PNG' })`. Pin.
11. **Empty `pageImage`**: zero-byte input → throws same code. Pin.
12. **Bbox fully out of bounds**: a bbox that doesn't intersect the source (e.g., `{x:1000, y:1000, width:1, height:1}` on a 100×100 source) → throws `RasterizeError({ code: 'BBOX_OUT_OF_BOUNDS' })`. Pin.
13. **Negative bbox dimensions**: `width: -1` or `height: -1` → throws `RasterizeError({ code: 'BBOX_INVALID' })`. Pin.
14. **NaN in bbox**: any NaN field throws same code. Pin.
15. **Negative `paddingPx`**: throws `RasterizeError({ code: 'OPTIONS_INVALID' })`. Pin.
16. **`compressionLevel` out of 0–9**: throws same code. Pin.

### Determinism

17. **Byte-identical across two calls**: same `pageImage` + `bboxPx` + default opts → two calls produce byte-equal `bytes`, byte-equal `contentHashId`. Pin via `Buffer.from(b1).equals(Buffer.from(b2))`.
18. **Different `compressionLevel` → different bytes**: a sanity test that the option actually flows through. Pin.
19. **Source-level discipline**: the package's `src/**` does NOT call `Date.now` / `Date()` / `new Date()` (without arg) / `Math.random` / `performance.now` / `setTimeout` / `setInterval`. Pin via grep test mirroring T-253-base AC #28.

### Content hash

20. `contentHashId` is exactly 64 hex chars (sha256). Pin.
21. `contentHashId(b1) !== contentHashId(b2)` when the bytes differ (e.g., different bbox on the same source). Pin.
22. The hash matches `node:crypto`'s `createHash('sha256').update(bytes).digest('hex')`. Pin via direct comparison.

### Errors

23. `RasterizeError` is an `Error` subclass with `.code: RasterizeErrorCode` and optional `.cause`. Pin via `instanceof` + property access.
24. The error message includes the error code and a human-readable description. Pin via fuzzy match.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- `pngjs.PNG.sync.write` produces non-byte-identical output across two consecutive calls with the same input + same pinned options. The fix is option-pinning; if that doesn't work, investigate `pngjs` version pinning or escalate before adding a custom DEFLATE.
- The PNG signature check (AC #10) fails on a real Slides thumbnail because Google rotates the export endpoint and starts returning JPEG bytes. Escalate; T-244's spec says PNG only, and the consumer should validate before calling T-245.
- A PNG with a palette (color type 3) decodes through `pngjs` but the cropped output's color type differs from the source's. The spec says metadata passes through unchanged; if `pngjs.write` insists on RGBA output for cropped data, document the deviation in the PR body and pin the actual behavior in tests rather than silently emitting truecolor.
- A PNG with `iCCP` (color profile) chunks loses the chunk on re-encode through `pngjs`. The spec assumes pass-through; if `pngjs` strips it, escalate before fudging — color-profile-aware consumers may rely on it.
- The `compressionLevel: 6` default produces output larger than expected for image-fallback paths (e.g., a 600×400 crop at 200 KB). The consumer can override; don't change the default to optimize for one consumer.
