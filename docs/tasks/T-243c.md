# T-243c — PPTX embedded font extraction → AssetStorage

**Branch**: `task/T-243c-pptx-font-extraction`

## Goal

Extend `@stageflip/import-pptx` to (1) parse `<p:embeddedFont>` entries from `ppt/presentation.xml` during the structural pass and (2) resolve their bytes through the existing `AssetStorage` interface. After this PR, a PPTX with embedded fonts round-trips through `parsePptx → resolveAssets` into a `CanonicalSlideTree` whose `embeddedFonts` collection carries resolved `AssetRef`s — no `LF-PPTX-UNRESOLVED-FONT` flags emit on success.

Mirrors the architecture T-243a (#172) shipped for images and T-243b (#193) specs for videos: structural pass emits a parser-side record with `ParsedAssetRef.unresolved`; the post-walk `resolveAssets` pass uploads the bytes and rewrites the ref.

**Schema-side integration is out of scope** — this PR ships parser-side discovery + storage upload only. Wiring resolved font assets into `Document.embeddedFonts` (or any equivalent canonical surface) lands in a follow-up after T-251 (#194) merges, since T-251 owns the foundational schema-shape work and a parallel addition would conflict.

## Architecture

### Parser-side (extends `parts/presentation.ts` + new `embedded-fonts.ts`)

`<p:embeddedFontLst>` is a deck-level element inside `/ppt/presentation.xml` (per ECMA-376 §19.2.1.5). Each `<p:embeddedFont>` has the shape:

```xml
<p:embeddedFontLst>
  <p:embeddedFont>
    <p:font typeface="Custom Family" panose="020B0604020202020204"/>
    <p:regular r:id="rId10"/>
    <p:bold r:id="rId11"/>
    <p:italic r:id="rId12"/>
    <p:boldItalic r:id="rId13"/>
  </p:embeddedFont>
  <!-- … more <p:embeddedFont> entries … -->
</p:embeddedFontLst>
```

- `<p:font typeface="…">` carries the family name (the same string text elements reference via `fontFamily`).
- The four optional face children (`<p:regular>`, `<p:bold>`, `<p:italic>`, `<p:boldItalic>`) each carry a relId that resolves to a TTF/OTF/EOT part inside the ZIP. Not every face is required; a font may ship only `<p:regular>`.
- Optional attributes `<p:embeddedFont embeddingType="restricted">` exist; T-243c ignores them (out of scope).

T-243c's parser change:

1. **New file**: `packages/import-pptx/src/parts/embedded-fonts.ts`. Exports `readEmbeddedFonts(entries: ZipEntries): ParsedEmbeddedFont[]`. Reads `ppt/presentation.xml`, walks `<p:embeddedFontLst>/<p:embeddedFont>`, resolves relIds via `presentation.xml.rels`, returns one `ParsedEmbeddedFont` per entry.
2. **Modified**: `packages/import-pptx/src/parsePptx.ts` calls `readEmbeddedFonts` once during deck parsing and attaches the result to `CanonicalSlideTree.embeddedFonts`.
3. **New `LossFlagCode`**: `LF-PPTX-UNRESOLVED-FONT`. One flag per font family with at least one unresolved face.

`readEmbeddedFonts` is pure (no network, no `Date`).

### Type-layer additions

In `packages/import-pptx/src/types.ts`:

```ts
/**
 * One PPTX-embedded font family. Faces are optional — a font may ship
 * only `regular`, or `regular` + `bold`. Each face's `bytes` reference is
 * unresolved at parser exit; T-243c's `resolveAssets` extension uploads
 * the bytes and rewrites to `{ kind: 'resolved', ref: AssetRef }`.
 */
export interface ParsedEmbeddedFont {
  /** PPTX `<p:font typeface="…">` value — matches text elements' `fontFamily`. */
  family: string;
  /** Optional PANOSE classification (kept opaque; renderers may ignore). */
  panose?: string;
  /** Per-face byte references. Face is absent when the PPTX didn't ship it. */
  faces: {
    regular?: ParsedAssetRef;
    bold?: ParsedAssetRef;
    italic?: ParsedAssetRef;
    boldItalic?: ParsedAssetRef;
  };
}
```

`CanonicalSlideTree` (currently at `types.ts:174-186`) extends with one new optional field:

```ts
export interface CanonicalSlideTree {
  slides: ParsedSlide[];
  layouts: Record<string, ParsedSlide>;
  masters: Record<string, ParsedSlide>;
  lossFlags: LossFlag[];
  embeddedFonts?: ParsedEmbeddedFont[];        // NEW
  transformsAccumulated?: boolean;
  // …
}
```

Optional with implicit `undefined` default (treated as "no embedded fonts" by consumers); existing fixtures parse unchanged.

`LossFlagCode` (the PPTX-local union in `types.ts:117-125`) extends with one new member:

```ts
export type LossFlagCode =
  | …existing…
  | 'LF-PPTX-UNRESOLVED-FONT';                 // NEW
```

`CODE_DEFAULTS` in `loss-flags.ts` adds `{ severity: 'info', category: 'font' }`.

### Asset-resolution-side (extends `assets/resolve.ts`)

The existing `resolveAssets` walker dispatches by `element.type` for in-slide assets. **Embedded fonts are deck-level, not per-element.** T-243c adds a separate post-element walk at the end of `resolveAssets`:

```ts
// After the per-element walk completes:
if (tree.embeddedFonts !== undefined) {
  for (const font of tree.embeddedFonts) {
    for (const faceKey of ['regular', 'bold', 'italic', 'boldItalic'] as const) {
      const ref = font.faces[faceKey];
      if (ref?.kind !== 'unresolved') continue;
      // Read bytes, upload, rewrite ref. Same pattern as the image branch.
    }
  }
}
```

After all faces resolve, drop the family's `LF-PPTX-UNRESOLVED-FONT` flag from `tree.lossFlags`. If any face remains unresolved (missing bytes), emit `LF-PPTX-MISSING-ASSET-BYTES` (existing flag) for that face only and keep the family flag.

`tree.assetsResolved` (the existing idempotence marker) covers font assets too — re-running `resolveAssets` is a no-op.

### Content-type extension

`packages/import-pptx/src/assets/content-type.ts` extends with the standard font MIMEs:

```ts
'.ttf':   'font/ttf',
'.otf':   'font/otf',
'.eot':   'application/vnd.ms-fontobject',
'.woff':  'font/woff',
'.woff2': 'font/woff2',
```

Source: IANA media-type registry (`https://www.iana.org/assignments/media-types/media-types.xhtml#font`). PowerPoint typically embeds TTF; the others cover the long tail.

## Acceptance criteria

Each gets a Vitest test, written first and failing.

### Parser layer

1. `<p:embeddedFontLst>` with one `<p:embeddedFont>` carrying `<p:font typeface="Custom"/>` + `<p:regular r:id="rId10"/>` parses to one `ParsedEmbeddedFont` with `family === 'Custom'`, `faces.regular.kind === 'unresolved'`, `faces.regular.oocxmlPath` matching the relId's resolved target.
2. **All four faces** populated: `<p:regular>`, `<p:bold>`, `<p:italic>`, `<p:boldItalic>` all parse to `ParsedAssetRef.unresolved`. Pin via fixture.
3. **Partial faces**: a `<p:embeddedFont>` with only `<p:regular>` parses to `faces: { regular: …, bold: undefined, italic: undefined, boldItalic: undefined }`. Pin via fixture.
4. **PANOSE preserved**: `<p:font typeface="X" panose="ABC123"/>` produces `panose: 'ABC123'`. Absent `panose` produces `undefined`.
5. **No `<p:embeddedFontLst>`**: returns `embeddedFonts: undefined` on `CanonicalSlideTree` (no empty-array convention; absence is undefined).
6. **Multiple fonts**: two `<p:embeddedFont>` entries produce a 2-entry `ParsedEmbeddedFont[]`, in document order.
7. `LF-PPTX-UNRESOLVED-FONT` emits with `severity: 'info'`, `category: 'font'`, `originalSnippet` set to the family name (one flag per family).
8. **Unresolved relId**: a `<p:regular r:id="rIdMissing"/>` whose relId doesn't resolve in `presentation.xml.rels` drops the face (sets to `undefined`) and emits an `LF-PPTX-UNRESOLVED-FONT` for the family if no faces resolved at all; otherwise emits per-face nothing extra (the flag already covers the family-level concern).

### Asset-resolution layer

9. `resolveAssets` reads bytes for each face, calls `storage.put(bytes, { contentType: 'font/ttf' })` (or appropriate MIME), and rewrites each face's `ParsedAssetRef` to `{ kind: 'resolved', ref: 'asset:<id>' }`.
10. `LF-PPTX-UNRESOLVED-FONT` drops from `tree.lossFlags` after all faces of that family resolve. If any face's bytes are missing from the ZIP, the family's flag stays AND a per-face `LF-PPTX-MISSING-ASSET-BYTES` emits.
11. `inferContentType` pins for **all 5 added extensions**: `.ttf` → `'font/ttf'`, `.otf` → `'font/otf'`, `.eot` → `'application/vnd.ms-fontobject'`, `.woff` → `'font/woff'`, `.woff2` → `'font/woff2'`. Pin via test.
12. **Idempotence**: running `resolveAssets` twice is a no-op (reuses the existing `assetsResolved: true` marker).
13. **Empty-fonts fast path**: a tree with `embeddedFonts: undefined` skips the font-walk entirely (no flag drops, no `storage.put` calls). Pin via test that `storage.put` mock is called zero times.

### End-to-end

14. The full pipeline (`parsePptx → resolveAssets`) on a synthetic fixture with one image, one video, and one embedded font (regular face only) emits **zero** unresolved-asset/font flags after the resolve pass (assuming bytes are present in the fixture ZIP).
15. Existing 138+ tests in `@stageflip/import-pptx` continue to pass with no expected-output changes (no regression).

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- A real PPTX fixture's `<p:embeddedFont>` carries an `<p:embeddedFontDataId>` shape T-243c's parser doesn't recognize (the four named children — regular/bold/italic/boldItalic — are the documented set, but a fixture might use the generic `<p:embeddedFontDataId>` complex type directly).
- A fixture's font relId resolves to a part **outside** the `/ppt/fonts/` convention (e.g., a font in `/ppt/media/` or `/word/fonts/`). The relationship resolver doesn't care about path conventions, so this should "just work", but call it out if a fixture exposes a path that breaks the test.
- A fixture's `panose` attribute is malformed (e.g., not 10 bytes hex). Spec says we keep `panose` opaque; if Zod validation breaks somewhere unexpectedly, escalate.
- A `<p:embeddedFont embeddingType="restricted">` fixture surfaces and the team needs a license-compliance pre-flight before bytes upload. T-243c uploads any embedded font bytes; downstream license review is out of scope. Escalate before implementing a runtime license-check.
- A gate fails after 2 rounds of debugging.
