# T-260 — ChangeSets + CRDT (Yjs) sync layer

**Branch**: `task/T-260-yjs-changesets-sync`

## Goal

Stand up the `@stageflip/collab` package — the first task that exercises the storage contract's delta methods (`applyUpdate` / `subscribeUpdates`) introduced in T-025/T-026 and dormant since Phase 1.

Per **ADR-006** (must read before implementing), T-260 ships:

1. A **shaped Y.Doc binding** of `Document` (D1).
2. A **storage-contract-backed Yjs Provider** (D2; no y-websocket).
3. A **command layer** that emits both Yjs transactions and `ChangeSet`s (D3).
4. A **`compact(docId)`** helper that crystallizes a fresh `DocumentSnapshot` from current Y.Doc state (D4).
5. **Error handling** per D8 (reconnect-with-backoff, status observable).

T-260 does **not** modify `apps/stageflip-*` or `packages/editor-shell` — collab consumption by the editor is a follow-up. This task is the package + tests + skill update.

## Architectural decisions

All deferred to **ADR-006**. This spec restates only the most load-bearing summary:

- **Y.Doc shape (D1)**: top-level `Y.Map` mirrors `Document`. `slides`, `elements`, `animations` are `Y.Array<Y.Map>`. Long-form text (`TextElement.text`, `slide.notes`) is `Y.Text`. Theme tokens, master/layout structural ids, and `meta.id` are single-writer plain JSON inside the parent map.
- **Transport (D2)**: storage adapter IS the transport. Provider wraps `applyUpdate` / `subscribeUpdates` / `getSnapshot`. No new server.
- **ChangeSet layer (D3)**: every command emits both a Y.Doc transaction and a `ChangeSet` (RFC 6902 ops with `actor` + `parentVersion` + `createdAt`). Y.Text per-keystroke debounces to one collapsed `replace` ChangeSet per transaction-close.
- **Snapshot cadence (D4)**: server-driven, T-260 ships the `compact()` helper; the worker that calls it is operational.
- **No Yjs awareness (D5)**: presence is RTDB (T-261).
- **No determinism gate (D7)**: collab is not clip/runtime code; `Date.now()`, `crypto.randomUUID()`, `Math.random()` (via Yjs) are fine.

## Acceptance criteria

Each AC has a Vitest test, written first and failing.

### Binding (`binding.ts`)

1. `documentToYDoc(doc: Document): Y.Doc` produces a Y.Doc whose `toJSON()` round-trips through `documentSchema.parse(...)` byte-identical to the input. Pin via fixture covering text / image / shape / group / table elements + a deck with `masters` + `layouts` + `inheritsFrom`.
2. `yDocToDocument(ydoc: Y.Doc): Document` is the inverse of `documentToYDoc`; structural equality required.
3. **TextElement.text** lives in a `Y.Text` (not a plain string). Pin: two clients edit the same text run concurrently — the merged result contains both insertions in source-order, no overwrites.
4. **Slide.notes** lives in a `Y.Text`. Same merge semantics as #3.
5. **Slide.elements** is a `Y.Array<Y.Map>`. Pin: two clients add an element to the same slide concurrently — both elements present after merge; order is deterministic given Yjs client-id ordering.
6. **`Document.theme`, `Document.variables`, `Document.components`, `Document.masters`, `Document.layouts`** are stored as plain JSON values inside the parent `Y.Map`. Pin: a concurrent edit to `theme.colors.primary` from two clients results in last-writer-wins (single-writer convention; CRDT machinery does not apply).
7. **`meta.id` is treated as immutable** post-construction — the binding throws if `documentToYDoc(doc1)` is called twice with different `doc1.meta.id` against the same Y.Doc. Pin via test.
8. The binding handles **all element types currently in `@stageflip/schema/src/elements/`** (text, image, shape, group, table — and any added since spec authoring; Implementer cross-checks the union).

### Provider (`provider.ts`)

9. `new YjsStorageProvider({ ydoc, storage, docId })` subscribes to `storage.subscribeUpdates(docId)` on construction and applies incoming updates via `Y.applyUpdate(ydoc, bytes)`. Pin via fixture.
10. The provider observes local Y.Doc updates (via `ydoc.on('update', ...)`) and forwards them via `storage.applyUpdate(docId, bytes)`. Pin: a local `ydoc.transact(...)` results in exactly one `applyUpdate` call (transaction-coalesced).
11. **Local-update debounce**: rapid local updates within 50 ms collapse to a single `applyUpdate` call. Pin via fake timers.
12. **Origin filtering**: updates received from `subscribeUpdates` are applied with origin `provider` (a stable symbol). The local-update observer must NOT re-emit updates whose origin is `provider` (would create an echo loop). Pin via test that mounts two providers against the same in-memory adapter and verifies no infinite echo.
13. **Bootstrap**: on construction, if `storage.getSnapshot(docId)` returns a non-null snapshot whose `content` is a `Uint8Array`, apply it to the Y.Doc before the subscribe loop reads its first message.
14. **Bootstrap from JSON `content`**: if `getSnapshot` returns a snapshot whose `content` is a parsed `Document` (legacy / migration case), the provider calls `documentToYDoc(content)` and merges into the existing Y.Doc.
15. **Reconnect with backoff** (D8.1): if the `subscribeUpdates` async iterator throws, the provider rebuilds the subscription. Backoff is exponential: 200 ms, 400 ms, 800 ms, ..., capped at 30 s, with ±25% jitter. Pin via fake timers + fault injection.
16. **Status observable**: `provider.status` is one of `'syncing' | 'synced' | 'error'`. Transitions: `syncing` until first remote-update applied or 500 ms of quiet (whichever first); `error` while reconnect is in progress; `synced` once subscribe loop is healthy. Pin transitions.
17. **Dispose**: `provider.dispose()` aborts the subscribe loop (via `AbortController` passed to `subscribeUpdates`), removes the local-update observer, and is idempotent.

### CollabClient (`client.ts`)

18. `new CollabClient({ docId, storage, actor })` is the consumer-facing API. Construction is synchronous; `await client.hydrate()` returns once the provider transitions to `synced`.
19. `client.document` returns the current `Document` view (computed via `yDocToDocument`); the value is stable-equal across reads when no mutations have occurred.
20. `client.command(name, args)` runs a registered command from `commands/`. Throws if `name` is unknown. Pin via test.
21. `client.dispose()` calls `provider.dispose()` and clears the in-memory Y.Doc.

### Commands (`commands/`)

22. **`addSlide`** appends a slide to `content.slides`. Emits one Y.Doc transaction AND one `ChangeSet` with one `add` op at path `/content/slides/-`. Pin via test that asserts both fan-outs.
23. **`removeSlide(slideId)`** removes the slide. ChangeSet ops: one `remove` at `/content/slides/<idx>`. Y.Doc transaction: `slides.delete(idx, 1)`.
24. **`reorderSlides(fromIdx, toIdx)`** uses `Y.Array.move`-equivalent semantics (delete + insert in same transaction). ChangeSet ops: one `move` op.
25. **`addElement(slideId, element)`** + **`removeElement(slideId, elementId)`** + **`updateElementTransform(slideId, elementId, transform)`** — same shape: one Y.Doc transaction, one ChangeSet. Pin shape of each.
26. **`setTextRun(slideId, elementId, text)`** updates a Y.Text by **diffing the new text against the current Y.Text content** and emitting minimal `Y.Text.insert` / `Y.Text.delete` calls (NOT `replace whole text`). Pin: a single-character insertion in the middle of a 1000-char run produces one `insert` and zero `delete`s.
27. **ChangeSet debounce on Y.Text edits**: rapid `setTextRun` calls (within a 250 ms window) collapse into a single `ChangeSet` with one `replace` op carrying the final text. The **Yjs updates are NOT debounced** at this layer (per AC #11, that lives in the provider; here the debounce is on ChangeSet emission only). Pin via fake timers.
28. **All commands carry the constructor-supplied `actor`** in their emitted `ChangeSet`. Pin via test.
29. **`parentVersion` semantics**: `ChangeSet.parentVersion` is the most recent snapshot version observed by the client at command-emit time, OR `0` if no snapshot has loaded. Pin via test that exercises pre-snapshot, post-snapshot, post-rebase paths.

### Snapshot helper (`snapshot.ts`)

30. `compact(docId, ydoc, storage): Promise<DocumentSnapshot>` calls `Y.encodeStateAsUpdate(ydoc)`, builds a `DocumentSnapshot` with monotonically-increasing `version`, calls `storage.putSnapshot(docId, snapshot)`, and returns it.
31. **Concurrent compaction safety**: two concurrent `compact()` calls on the same docId must not produce snapshots with the same `version`. Pin via fixture using a deterministic `Date.now` mock; the second compaction either overwrites cleanly or throws — Implementer's call, document the choice.
32. **Compact + replay round-trip**: after `compact()`, a fresh provider hydrating from the new snapshot reaches the same `Document` view as the source provider. Pin via fixture.

## Escalation triggers (CLAUDE.md §6)

Stop and report instead of guessing if:

- ADR-006 is ambiguous on a binding decision (D1) for an element type that the spec didn't explicitly enumerate. Escalate; the ADR is the source of truth, not the spec.
- The `Y.Text` merge semantics for `TextElement.text` produce a result that violates the schema's `text` shape (e.g., schema requires structured runs with formatting; Y.Text is plain string). Escalate; the schema may need a discriminated `text` variant before binding can proceed.
- The reconnect-with-backoff path (AC #15) interacts with `InMemoryStorageAdapter`'s lack of persistence in a way that makes deterministic testing infeasible. Escalate; do NOT modify the adapter (out of scope per AC #35).
- The `compact()` race (AC #31) requires extending the storage contract with a CAS-on-version method. Escalate; the storage contract is frozen for T-260.
- `pnpm check-determinism` flags `packages/collab/**` because the scan list is broader than expected. Escalate; ADR-006 D7 says this is out of scope. The fix is updating the scan list with reviewer concurrence, not adding `determinism-safe` comments.
- A real-world fixture (e.g., a 10 MB persisted document with 50k Yjs updates) makes hydration take >5 s. Escalate; the snapshot-compaction worker is operationally out of scope but if the design itself is the bottleneck, surface it before shipping.

## Notes for the Orchestrator

1. **L-sized; expect six commits.** Binding + provider are the heavy substantive code. CollabClient + commands + snapshot are mechanical wiring on top. Skill update is small.
2. **Reviewer focus**: AC #1–#8 (binding round-trip, especially Y.Text concurrent-merge semantics, AC #3 / #4); AC #10 / #11 / #12 (provider transaction-coalescing + origin filtering — the echo-loop trap is the most common shipping bug); AC #26 (text-diff insert/delete vs. naive replace — naive replace would lose all concurrent edits and is silently catastrophic).
3. **No editor-shell changes in this PR.** T-261 (presence) and follow-ups consume `@stageflip/collab`. The acceptance criteria deliberately do not require editor wiring; the package must stand up on its own with vitest.
4. **`yjs` MIT** is on the whitelist (`THIRD_PARTY.md` §1: MIT permitted). No ADR-level whitelist amendment required. The Implementer adds an inventory entry in `THIRD_PARTY.md` §1's auto-populated list (or `pnpm check-licenses` may emit a "new MIT dep, no further action" log line — confirm by running locally).
5. **Determinism scope** (`scripts/check-determinism.ts`): the script today scans `packages/frame-runtime/**`, `packages/runtimes/**/src/clips/**`, and `packages/renderer-core/src/clips/**`. `packages/collab/**` is correctly out of scope. Implementer should NOT broaden the scope; if the linter complains, it's a check-determinism config bug, escalate.
6. **In-memory adapter quirk**: `InMemoryStorageAdapter.applyUpdate` is fan-out-only (no persistence past the live subscribers' buffer). This makes the reconnect-with-backoff test (AC #15) infeasible to test against a real reconnect flow with this adapter — fault-injection via a wrapping adapter is the right pattern. Implementer should NOT modify the in-memory adapter (AC #35) — wrap it.
7. **Origin filtering (AC #12)** is a known Yjs footgun. The provider must use a stable origin symbol and the local-update observer must early-return when `transaction.origin === providerOrigin`. Get this wrong and you ship an infinite-echo bug that only surfaces with two clients connected to the same adapter.
8. **Future T-261 consumer**: the `CollabClient.status` observable shape (D8 / AC #16) needs to compose with presence status. Implementer should expose `status` as a typed observable (`{ subscribe(cb): () => void }` or jotai atom — match the editor-shell convention) so T-261 can read it without rework.
