# T-394 — `WebEmbedClip` `staticFallback` (poster-frame screenshot)

**Branch**: `task/T-394-web-embed-clip-static-fallback`

## Goal

Ship the `staticFallback` for `family: 'web-embed'`. When the network permission gate denies, the tenant policy refuses, or an export pipeline targets a static medium (MP4/PPTX/PDF), the mount-harness routes to a deterministic poster: an `ImageElement` rendering the `posterImage` data URL supplied at authoring time. When `posterImage` is absent, a single placeholder TextElement (empty text) is emitted.

T-394 is the `staticFallback` half of ADR-005 §D1's `WebEmbedClip` row. After T-394 merges, `family: 'web-embed'` is structurally complete.

T-394 registers `webEmbedStaticFallbackGenerator` against the **T-388a static-fallback generator registry** (already on `main`).

## Out of scope

| Item | Why deferred |
|---|---|
| Live screenshot capture at authoring time | Authoring concern. T-394 accepts the host's pre-captured screenshot via `posterImage`. Capture pipeline (browser-side `html2canvas`-style, or server-side Puppeteer / Playwright) is a future asset-gen task. |
| Video poster (MP4-frame poster instead of static image) | Different transport contract; the schema accepts a single image URL. |
| Localised placeholder copy | App-level i18n catalogue (CLAUDE.md §10). |
| Multi-resolution poster (responsive image set) | Single image URL in v1. |
| External hosted poster URL with fetch-at-export | Determinism — the data URL is the authoring-time bake; an external fetch at export time would intersect determinism. v1 enforces this at the schema layer (the `posterImage.src` refine accepts ONLY `data:` URLs). Widening to `http(s)` is gated on a determinism-skill ruling for the frame-runtime image-fetch path. |

## Architectural decisions

### D-T394-1. Schema addition — `posterImage?` on `webEmbedClipPropsSchema`

T-393 shipped `webEmbedClipPropsSchema` with `posterFrame` but no image field. T-394 adds:

```ts
// packages/schema/src/clips/interactive/web-embed-props.ts (MODIFIED)
export const webEmbedClipPropsSchema = z.object({
  // ... existing T-393 fields ...
  /**
   * Captured poster screenshot for the static fallback. Optional;
   * absent → an empty placeholder TextElement (host overrides via
   * app-level i18n).
   *
   * v1 accepts ONLY `data:` URLs (inline-baked at authoring time).
   * `http(s):` URLs are out of scope — they would intersect the
   * determinism floor (the frame-runtime would have to fetch the
   * image at export time, which `check-determinism` forbids inside
   * `clips/**`). The "External hosted poster URL with fetch-at-export"
   * deferral in §"Out of scope" tracks this; widening the schema is
   * a separate task gated on a determinism-skill ruling for the
   * external-image fetch path.
   */
  posterImage: z.object({
    /** Data URL pointing at the poster image. v1: `data:` only. */
    src: z.string().refine(
      (s) => s.startsWith('data:'),
      'posterImage.src must be a data: URL in v1; http(s) URLs are deferred per the out-of-scope table',
    ),
    /** MIME type hint (display only; not enforced against `src`). */
    contentType: z.enum(['image/png', 'image/jpeg', 'image/webp']).optional(),
  }).strict().optional(),
}).strict();
```

Browser-safe. Existing T-393 fixtures continue to validate (the field is optional).

### D-T394-2. Generator output — single `ImageElement` rendering the poster

`defaultWebEmbedStaticFallback` returns:

1. When `posterImage` is supplied: a single `ImageElement` filling the clip's bounds with `src = posterImage.src`. The element's `transform` matches the canvas dimensions (`x: 0, y: 0, width, height`).
2. When `posterImage` is absent: a single placeholder `TextElement` with empty text (host overrides via app-level i18n; same posture as T-390 D-T390-2 / T-392).

Layout is trivial — no vertical-stacking discipline like T-390 / T-392 because there is only one element either way.

### D-T394-3. Determinism

`defaultWebEmbedStaticFallback` is subject to the broad §3 rule (no rAF, no random, no time, no fetch). Pure transformation over inputs. Same posture as T-388 / T-390 / T-392.

The `posterImage.src` is passed through verbatim — the generator does NOT fetch the URL. v1 accepts ONLY `data:` URLs (the schema refine in D-T394-1 enforces this); the renderer decodes the data URL inline. `http(s):` URLs are deferred per §"Out of scope" — accepting them would push the fetch question to the frame-runtime export path, which is `check-determinism`-restricted.

### D-T394-4. Telemetry

`web-embed-clip.static-fallback.rendered` (attributes: `reason`, `width`, `height`, `hasPoster` boolean, `posterSrcLength` integer when poster present — NOT the URL itself; same privacy posture as T-390 D-T390-4 / T-392 D-T392-4 / T-393 D-T393-8). Pin via test (AC #11).

The `posterSrcLength` integer is included so telemetry consumers can roughly distinguish a tiny placeholder data URL from a substantial screenshot. The full `src` string is NOT included — even though the URL is configuration, a 50KB data URL would balloon every telemetry event. The integer length suffices.

### D-T394-5. Registration via T-388a registry

`packages/runtimes/interactive/src/clips/web-embed/index.ts` gains a side-effect call:

```ts
import { staticFallbackGeneratorRegistry } from '../../static-fallback-registry.js';
import { webEmbedStaticFallbackGenerator } from './static-fallback.js';

interactiveClipRegistry.register('web-embed', webEmbedClipFactory);  // existing T-393 line
staticFallbackGeneratorRegistry.register('web-embed', webEmbedStaticFallbackGenerator);  // NEW
```

The harness's family-agnostic dispatch (T-388a) picks up the registration without further changes.

## Acceptance criteria

### Schema

1. `webEmbedClipPropsSchema` accepts a valid `posterImage` (data URL).
2. `posterImage: { src: 'https://cdn.example.com/poster.png' }` → schema error (v1 rejects http(s) per the out-of-scope deferral; the refine fires).
3. `posterImage: { src: 'relative/path.png' }` → schema error (refine rejects).
4. `posterImage: { src: 'data:image/png;base64,xxx', extra: true }` → schema error (strict shape).
5. Existing T-393 fixtures (without `posterImage`) continue to validate.

### Generator determinism

6. `defaultWebEmbedStaticFallback({ width: 640, height: 360, posterImage: { src: 'data:image/png;base64,abc' } })` returns the same Element[] byte-for-byte across two calls.
7. Different `posterImage.src` → different markup.
8. Absent `posterImage` → single placeholder TextElement (no ImageElement).

### Registry integration

9. After importing `clips/web-embed`, `staticFallbackGeneratorRegistry.resolve('web-embed')` returns the generator.
10. Authored `staticFallback` (non-empty) wins; the generator is still called with `reason: 'authored'` for telemetry (T-388a D-T388a-3 contract).

### Telemetry

11. `web-embed-clip.static-fallback.rendered` emits with documented attribute shape; `hasPoster` boolean and `posterSrcLength` integer, NOT the URL itself.

### Bounds

12. The ImageElement's transform fits within the clip's bounds (no element extends past `width × height`). Trivial because the element fills the canvas exactly.

## Escalation triggers (CLAUDE.md §6)

- A reviewer requests `posterImage.src` accept a relative URL (e.g., `/assets/posters/x.png`). Escalate; relative URLs intersect deployment-target ambiguity (relative to the export's host vs. the app shell vs. the iframe origin).
- The schema's `data:` URL refine rejects a legitimate data URL shape (e.g., a non-base64 percent-encoded SVG). Escalate; we may need to widen the refine to include all `data:` shapes.
- A consumer requests `http(s):` URL acceptance for `posterImage.src` (hosted-asset workflow). Escalate per the out-of-scope row — this needs a determinism-skill ruling on the frame-runtime image-fetch path before the schema can widen.
- An ImageElement with empty `src` fails the schema's required-non-empty refine (if the schema enforces one). Escalate.
