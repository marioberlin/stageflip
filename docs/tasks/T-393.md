# T-393 ŌĆö `WebEmbedClip` `liveMount` (sandboxed iframe)

**Branch**: `task/T-393-web-embed-clip-live-mount`

## Goal

Ship the fourth ╬│-live family: `WebEmbedClip` `liveMount`. A schema-validated `InteractiveClip` with `family: 'web-embed'`, registered in `interactiveClipRegistry`, mountable via the T-306 `ClipFactory` contract. Mounts a sandboxed `<iframe>` whose `src` is a clip-supplied URL; exposes a typed `MountHandle` with `reload()`, `postMessage()`, and origin-filtered `onMessage()`; emits typed lifecycle telemetry.

T-393 is the **fourth application of the second ╬│ pattern** (event-driven, no ┬¦3 reuse). Reaffirms the T-391 D-T391-10 ruling ŌĆö no `ProviderSeam<T>` extraction even at the fourth family ŌĆö because `WebEmbedClip` does not even have a provider seam (the iframe is the transport; the browser's `<iframe>` element is the runtime). Documented in D-T393-10.

T-393 ships `liveMount` only. **T-394 ships `staticFallback`** (poster-frame screenshot rendered as `ImageElement`) as a separate S-sized task.

## Architectural decisions

### D-T393-1. `WebEmbedClip` lives entirely in `runtimes-interactive/src/clips/web-embed/`

No new package; no ┬¦3 runtime to reuse. Same posture as T-387 / T-389 / T-391. The clip is event-driven (postMessage events), not frame-driven. Confirms the second ╬│ pattern across five families.

### D-T393-2. Schema narrowing ŌĆö `webEmbedClipPropsSchema`

```ts
// packages/schema/src/clips/interactive/web-embed-props.ts (NEW)
export const webEmbedClipPropsSchema = z.object({
  /** Absolute URL of the embedded page. The clip does not resolve relative paths. */
  url: z.string().url(),
  /**
   * `sandbox` attribute tokens. Default `[]` ŌĆö fully sandboxed (no scripts,
   * no same-origin, no forms, no popups). Authors opt in to specific
   * capabilities by adding tokens (`'allow-scripts'`, `'allow-same-origin'`,
   * `'allow-forms'`, etc.). The schema does not enforce a token allowlist ŌĆö
   * the security review (T-403) decides which tokens are permitted at
   * tenant level. Values are passed verbatim into the iframe's `sandbox`
   * attribute as a space-separated string.
   */
  sandbox: z.array(z.string()).default([]),
  /**
   * Origins permitted to dispatch via `onMessage`. Empty / undefined ŌåÆ
   * `onMessage` never fires. The clip's own `postMessage()` outbound is
   * NOT origin-filtered (host sends; iframe receives via the browser's
   * standard postMessage rules), but the inbound side IS ŌĆö every
   * `event.origin` MUST appear in this allowlist or the event is
   * dropped. Same posture as the T-391 `LiveData` headers field ŌĆö
   * documentation, not security theatre; the real defence is the
   * iframe's `sandbox` attribute + the host's CSP.
   */
  allowedOrigins: z.array(z.string().url()).optional(),
  /** Optional iframe-width override. Defaults to the clip transform's width. */
  width: z.number().int().positive().optional(),
  /** Optional iframe-height override. Defaults to the clip transform's height. */
  height: z.number().int().positive().optional(),
  /** Frame at which `staticFallback` poster is sampled (T-394 consumes). */
  posterFrame: z.number().int().nonnegative().default(0),
}).strict();
```

Browser-safe (pure Zod). Mirrors the discriminator pattern from T-391.

### D-T393-3. Permissions ŌĆö `['network']`; pre-prompt off

The clip declares `permissions: ['network']`. Per the existing shim, `network` is a no-op grant in v1; tenant allowlists land later (T-403). Pre-prompt is OFF by default for `network`-only clips per the T-389 / T-391 precedent. Documented in the skill.

### D-T393-4. `liveMount` API surface

The factory mounts a single `<iframe>` element with the resolved sandbox attribute. The factory does NOT expose a host-overridable visual surface ŌĆö the iframe IS the surface. Hosts that want to overlay UI compose at the application layer (the iframe is sized to the clip transform).

The factory exposes lifecycle callbacks via `MountHandle`:

```ts
interface WebEmbedClipMountHandle extends MountHandle {
  /** Reload the iframe (host-driven; equivalent to setting `iframe.src = iframe.src`). */
  reload(): void;
  /**
   * Send a postMessage to the iframe's contentWindow. The factory handles
   * the targetOrigin ŌĆö it forwards the URL's origin verbatim, NOT '*' (the
   * host knows which page they embedded; reflexive '*' would defeat the
   * scoping).
   */
  postMessage(message: unknown): void;
  /**
   * Subscribe to inbound postMessage events from the iframe, filtered by
   * `allowedOrigins`. Returns unsubscribe.
   */
  onMessage(handler: (event: WebEmbedMessageEvent) => void): () => void;
}

interface WebEmbedMessageEvent {
  /** Origin of the message (always one of `allowedOrigins`). */
  origin: string;
  /** Message data. The clip does NOT validate; subscribers may. */
  data: unknown;
}
```

`postMessage` is a no-op (with telemetry) when called pre-mount or post-dispose. `onMessage` handlers fire only after `mount.success` and stop firing after `dispose`.

### D-T393-5. NO provider seam ŌĆö the iframe is the transport

Unlike Voice (T-387) / AiChat (T-389) / LiveData (T-391), T-393 has no `WebEmbedProvider` interface. The browser's `<iframe>` element IS the runtime; the factory's job is to create + dispose the element. There is no host-injected client to abstract.

This is the architectural reason D-T393-10 reaffirms the T-391 ruling: the four ╬│-live families now span four shapes (streaming-with-callback, request/response, no-seam-just-DOM), and a `ProviderSeam<T>` abstraction over them would be either too narrow (cancellation only) or fight all four concretes.

### D-T393-6. No convergence test, no `frameSource` dependency

Confirms second ╬│ pattern. The factory does NOT throw `MissingFrameSourceError`; it ignores `MountContext.frameSource`. Pin via test (mounting with `frameSource: undefined` succeeds).

### D-T393-7. Resource cleanup on dispose

`MountHandle.dispose()` MUST:

1. Remove the `window` postMessage event listener (the clip's filter is a closure on `state` ŌĆö leaking it would keep `state` alive after dispose).
2. Set `iframe.src = 'about:blank'` BEFORE detaching ŌĆö this halts the embedded page's scripts, network requests, and timers. Detaching alone does not stop a running embedded page in some browsers (Firefox in particular).
3. Detach the iframe from the DOM.
4. Drop the subscriber set.
5. Be idempotent.

Less heavy than T-389's LLM-stream graph or T-391's per-fetch AbortController, but the iframe-stops-via-about:blank step is load-bearing ŌĆö without it, an embedded page keeps running in the background after the host considers the clip disposed.

### D-T393-8. Telemetry ŌĆö lifecycle + message events

- `web-embed-clip.mount.start` (attributes: `family: 'web-embed'`, `url`, `sandbox` (joined string), `hasAllowedOrigins` (boolean))
- `web-embed-clip.mount.success`
- `web-embed-clip.mount.failure` (reasons: `'invalid-props'`, `'permission-denied'`)
- `web-embed-clip.message.outbound` (attributes: `byteLength` integer, `targetOrigin`)
- `web-embed-clip.message.received` (attributes: `origin`, `byteLength` integer; **NEVER the message body**)
- `web-embed-clip.message.dropped` (attributes: `origin`, `reason: 'origin-not-allowed' | 'pre-mount' | 'post-dispose'`)
- `web-embed-clip.reload`
- `web-embed-clip.dispose`

**Privacy posture**: postMessage payload bodies NEVER appear in telemetry attributes. `byteLength` (computed via `JSON.stringify(message).length` for outbound; via `JSON.stringify(event.data).length` for inbound) is an integer length only. Same posture as T-391 D-T391-8 / T-389 D-T389-8. Pin via test (AC #16).

The `url` IS included on `mount.start` ŌĆö it is configuration (which page is being embedded), not user content. The same way `endpoint` is on `live-data-clip.mount.start`.

- Subpath export `./clips/web-embed` added to `runtimes-interactive/package.json`.

### D-T393-10. Pattern-evaluation ŌĆö fourth application reaffirms the T-391 ruling

T-391 D-T391-10 ruled out `ProviderSeam<T>` extraction at the third application. T-393 is the **fourth** standalone-pattern application and reaffirms the ruling. The four seams now span four shapes:

- `TranscriptionProvider.start({ stream, language, partial, onTranscript })` ŌĆö streaming + discriminated-union callback.
- `LLMChatProvider.streamTurn({ ..., onToken, signal })` ŌĆö streaming + per-token callback.
- `LiveDataProvider.fetchOnce({ ..., signal })` ŌĆö request/response.
- **No provider seam** for `WebEmbedClip` ŌĆö the browser's `<iframe>` is the transport.

A generic `ProviderSeam<T>` over four shapes including "no seam at all" is incoherent. Skill update reaffirms: "The four frontier-clip implementations span shapes from streaming-with-callback to raw-DOM; there is no extractable abstraction. Future families follow the convention without inheriting a generic shape."

## Acceptance criteria

### Schema

1. `webEmbedClipPropsSchema` parses a valid payload (url, sandbox, allowedOrigins, width, height, posterFrame).
2. Non-URL `url` ŌåÆ schema error.
3. Non-array `sandbox` ŌåÆ schema error.
4. Non-URL entry inside `allowedOrigins` ŌåÆ schema error.
5. `check-preset-integrity` invariant 13 rejects `family: 'web-embed'` with bad props; accepts valid fixture.

### Factory + registry

6. `interactiveClipRegistry.resolve('web-embed')` returns the factory after side-effect import.
7. Re-importing throws `InteractiveClipFamilyAlreadyRegisteredError`.
8. Mount with `MountContext.frameSource: undefined` succeeds ŌĆö no `MissingFrameSourceError`.

### Iframe lifecycle

9. Mount creates exactly one `<iframe>` element under the supplied root, with `src` matching `props.url` and `sandbox` matching the joined token list.
10. The iframe's width / height attributes match `props.width` / `props.height` when supplied; otherwise they fall back to the clip transform's dimensions.
11. `reload()` re-assigns `iframe.src` (testable by spying on the assignment) and emits `web-embed-clip.reload` telemetry.
12. `postMessage(msg)` calls `iframe.contentWindow.postMessage(msg, targetOrigin)` where `targetOrigin` is the URL's origin.
13. `onMessage` handlers fire ONLY for events whose `event.source === iframe.contentWindow` AND `event.origin Ōłł props.allowedOrigins`. Events failing either check are dropped + logged via `web-embed-clip.message.dropped`.
14. `onMessage` returns an unsubscribe function that cleanly removes the handler.

### Resource cleanup

15. `MountHandle.dispose()` removes the `window.message` event listener (verified by spy on `window.removeEventListener`), sets `iframe.src = 'about:blank'`, detaches the iframe, and is idempotent.

### Telemetry

16. Each documented event (D-T393-8) is emitted with the documented attribute shape, observed via spy. **postMessage payload bodies NEVER appear in any attribute**; `byteLength` is an integer length. Pin via grep on captured event payloads.

### Permission posture

17. Mount with `permissions: ['network']` succeeds. The shim's `network` grant callback fires synchronously; `MountContext.permissionPrePrompt` is unset (the no-op grant skips the pre-prompt React render cycle). Both assertable via spy on `permissionShim.mount` and a `MountContext` shape check.

## Escalation triggers (CLAUDE.md ┬¦6)

- A reviewer pushes back on the "no provider seam" stance and wants T-393 to introduce a `WebEmbedProvider` interface for testability. Escalate; the iframe IS the testable surface (jsdom / happy-dom support `<iframe>` and `postMessage` end-to-end). Adding a seam where none is needed creates more abstraction churn than it pays for.
- The `iframe.src = 'about:blank'` halt step turns out to be unreliable in some browser (e.g., Safari preserving running scripts despite the navigation). Escalate; we may need an additional `iframe.contentWindow.location.replace('about:blank')` step.
- The `event.source === iframe.contentWindow` filter has cross-realm pitfalls (the iframe's `contentWindow` may be revoked after navigation). Escalate; we may need a stable identifier (e.g., a known sentinel field in the message body, set by host-controlled bootstrap script).
- Tenant policy wants to enforce origin allowlists at the schema layer (rejecting `family: 'web-embed'` clips whose `url` origin isn't in a tenant allowlist). Escalate; this is a T-403 concern, not a T-393 concern.
