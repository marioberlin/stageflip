# T-266 — Render farm deployment — adapter pattern (vendor-agnostic)

**Branch**: `task/T-266-render-farm-adapter`

## Goal

Stand up the **render-farm adapter abstraction** — a `RenderFarmAdapter` contract that abstracts over CoreWeave / Paperspace / self-hosted-K8s vendor choices. Pick a vendor lazily; the adapter contract decouples the choice from the consumer (the bake worker from T-265).

User decision (2026-04-28): **spec the adapter / channel concept** so we can swap vendors without rewriting. Vendor choice deferred to whenever first prod load demands it.

T-266 ships:

1. **`@stageflip/render-farm`** package — `RenderFarmAdapter` interface + `InMemoryRenderFarmAdapter` (for local dev + tests) + a stub `KubernetesRenderFarmAdapter` (interface only, raises `NotImplementedError`).
2. **Adapter selector** — `getRenderFarmAdapter(env: Env)` returns the configured adapter based on env vars.
3. **Worker → adapter wiring** — the bake worker (T-265) calls into the adapter for "scale up" / "scale down" / "submit job" instead of vendor-specific APIs. (T-265 ships the worker; T-266 adds the adapter abstraction layer.)
4. **Vendor choice doc** — `docs/ops/render-farm-vendors.md` evaluating CoreWeave / Paperspace / self-hosted-K8s, with cost/throughput/ops tradeoffs.

T-266 does NOT pick a vendor or deploy to one. It ships the adapter and one fake implementation (in-memory).

## Dependencies

- T-265 (Blender bake-tier ClipRuntime) — must merge first. T-265's worker is what gets deployed via T-266's adapter.
- T-264 (observability) — merged. Adapter implementations emit OTel spans for scaling / submission.
- ADR-003 (interactive runtime tier) — ratified. T-265's bake tier is the consumer.

## Out of scope

| Item | Why deferred |
|---|---|
| Actual vendor deployment (CoreWeave / Paperspace / etc.) | This is the WHOLE POINT of the adapter pattern. Vendor choice + provisioning is a separate operational task once T-265 ships and we have real bake load to observe. |
| Auto-scaling logic (worker count vs queue depth) | Adapter exposes scale-up/down APIs; the autoscaler is a future task, possibly built on the adapter. |
| Multi-vendor concurrent deployment (failover) | Single-vendor for v1; multi-vendor is far future. |
| Cost optimization (spot instances, preemption handling) | Vendor-specific implementation; lands per-vendor adapter as it ships. |
| Per-job priority / preemption | Future feature. |
| Egress optimization (pre-warming workers near GCS region) | Operational; out of scope. |

## Architectural decisions

### D-T266-1. Adapter contract

```ts
export interface RenderFarmAdapter {
  /** Submit a job to the farm. Adapter is responsible for queuing / scaling. */
  submitJob(job: RenderFarmJob): Promise<{ jobId: string }>;
  /** Cancel a queued or running job. Best-effort; running jobs may complete. */
  cancelJob(jobId: string): Promise<void>;
  /** Query job state. */
  getJobStatus(jobId: string): Promise<RenderFarmJobStatus>;
  /** Stream worker logs (best-effort; some vendors don't expose this). */
  streamLogs?(jobId: string, opts?: { signal?: AbortSignal }): AsyncIterable<string>;
  /** Adapter capabilities — which features are real vs stubs. */
  readonly capabilities: RenderFarmCapabilities;
}

export interface RenderFarmJob {
  /** BullMQ job id from T-265's queue. The farm runs the worker against this. */
  bakeId: string;
  /** Container image (worker image from T-265). */
  image: string;
  /** Resource requirements. */
  resources: { cpu: number; memoryGB: number; gpu?: 'cuda' | 'rocm' | 'cpu-only' };
  /** Env vars to pass to the worker (REDIS_URL, ORG_ID, INPUTS_HASH, OUTPUT_BUCKET). */
  env: Record<string, string>;
  /** Optional metadata for telemetry (org_id, region). */
  tags?: Record<string, string>;
}

export interface RenderFarmJobStatus {
  jobId: string;
  state: 'queued' | 'running' | 'succeeded' | 'failed' | 'canceled';
  startedAt?: string;  // ISO
  finishedAt?: string;
  error?: string;
  logsAvailable: boolean;
}

export interface RenderFarmCapabilities {
  /** Supports streaming logs. */
  streamingLogs: boolean;
  /** Supports GPU types beyond cpu-only. */
  gpuTypes: Array<'cuda' | 'rocm' | 'cpu-only'>;
  /** Supports sub-minute scale-up. */
  fastScaleUp: boolean;
  /** Maximum concurrent jobs the adapter can run. */
  maxConcurrentJobs: number;
}
```

### D-T266-2. Two implementations in this task

1. **`InMemoryRenderFarmAdapter`** — runs the worker in the same Node process via `child_process.spawn`. Used for local dev + tests. Capabilities: `cpu-only` GPU (because dev machines vary), streaming logs (just the spawned process's stdout), `maxConcurrentJobs: 4`.

2. **`KubernetesRenderFarmAdapter` stub** — class skeleton with the methods, all of which throw `NotImplementedError('K8s adapter requires a vendor configuration; see docs/ops/render-farm-vendors.md')`. The shape exists so consumers compile against it; the implementation lands when a vendor is picked.

### D-T266-3. Selector

```ts
export function getRenderFarmAdapter(env: NodeJS.ProcessEnv): RenderFarmAdapter {
  const kind = env.STAGEFLIP_RENDER_FARM_ADAPTER ?? 'in-memory';
  switch (kind) {
    case 'in-memory': return new InMemoryRenderFarmAdapter(...);
    case 'k8s':       return new KubernetesRenderFarmAdapter(...);   // throws on call
    default: throw new Error(`unknown render-farm adapter: ${kind}`);
  }
}
```

Production env sets `STAGEFLIP_RENDER_FARM_ADAPTER=k8s` (or a future vendor key) and the K8s adapter (when implemented) does the real work.

### D-T266-4. Vendor evaluation rubric

`docs/ops/render-farm-vendors.md` evaluates each vendor along:

| Vendor | Cost / GPU-hr | GPU types | Fast scale | Ops surface | Lock-in risk |
|---|---|---|---|---|---|
| **CoreWeave** | $0.40–2.40 | A40/H100 | <30s | Medium (their CLI + K8s) | Low (K8s native) |
| **Paperspace** (DigitalOcean) | $0.30–3.00 | A100/H100 | ~60s | Low (managed) | Medium (Gradient platform) |
| **Self-hosted K8s** (GKE / EKS) | varies (provisioned) | any (you bring) | depends | High (full ops) | None (your cluster) |

The recommendation in v1 is **self-hosted K8s on GKE** because:

- Already on GCP (Firebase + Storage + future Cloud SQL) — single-cloud surface.
- GKE Autopilot reduces ops burden.
- Spot instances for non-critical bakes.
- Full control over GPU type + region.

But the adapter pattern means we can swap to CoreWeave (cheapest GPU/hr) for cost, or to Paperspace (managed simplicity) without rewriting.

### D-T266-5. Determinism + observability

`packages/render-farm/**` is NOT in determinism scan. Adapter impl uses `Date.now()`, network I/O, subprocess spawning — all expected.

Every adapter method emits an OTel span with attributes: `jobId`, `state`, `vendor`, `duration_ms`. Failures call `captureError`.

## Acceptance criteria

### Contract

1. `RenderFarmAdapter` interface from D-T266-1 is exported. Pin shape via type-level test.
2. `RenderFarmJob`, `RenderFarmJobStatus`, `RenderFarmCapabilities` types are exported.

### `InMemoryRenderFarmAdapter`

3. `submitJob(job)` spawns the worker as a child process, returns a `jobId`. Pin via test using a stub worker script (echoes the env back).
4. `getJobStatus(jobId)` returns the current state; transitions through `queued → running → succeeded` for a successful run.
5. `cancelJob(jobId)` sends SIGTERM to the child process; status transitions to `canceled`.
6. `streamLogs(jobId, { signal })` yields stdout lines from the spawned process. Pin via test.
7. `capabilities`: `{ streamingLogs: true, gpuTypes: ['cpu-only'], fastScaleUp: true, maxConcurrentJobs: 4 }`.
8. **Concurrency limit**: submitting a 5th job when 4 are running queues the 5th; first to finish triggers the queued one. Pin via test.

### `KubernetesRenderFarmAdapter` stub

9. Constructor accepts an opaque config object.
10. Every method throws `NotImplementedError` with a clear message pointing to `docs/ops/render-farm-vendors.md`.
11. `capabilities` returns a stub with `maxConcurrentJobs: 0` (signals "not deployed").

### Selector

12. `getRenderFarmAdapter(env)`: env not set OR `'in-memory'` → `InMemoryRenderFarmAdapter`. `'k8s'` → `KubernetesRenderFarmAdapter`. Unknown → throws.
13. Selector is pure: same env → same adapter type (instance reuse acceptable as a fast path).

### Worker integration

14. The blender worker's "I started"/"I finished" markers (D-T266-5) are written to stdout in a parseable format the in-memory adapter can read for state transitions. Pin via integration test (worker + adapter together).

### Telemetry

15. Every adapter method emits an OTel span with `vendor`, `jobId`, `state`, `duration_ms` attributes. Pin via mock OTel tracer.
16. Failed jobs call `captureError` with the job context.

### Backward compatibility

17. The full workspace test suite passes unchanged.
18. T-265's worker tests continue to pass — the worker is unchanged at the contract level (just now emits state markers; existing tests don't break).

### Documentation

19. `docs/ops/render-farm-vendors.md` documents:
   - The adapter contract (recap).
   - The three vendor candidates + cost/throughput/ops table.
   - Recommendation rationale (GKE for v1).
   - Migration path between vendors.

## Escalation triggers (CLAUDE.md §6)

- The blender worker's worker.ts requires invasive changes to emit state markers cleanly (more than the "light modification" planned). Escalate; the contract may need to evolve.
- The K8s stub needs a non-trivial config shape that's hard to design without knowing the actual K8s deployment. Escalate; we may want to keep the stub config truly opaque (`Record<string, unknown>`) until a real vendor is picked.
- The in-memory adapter's child_process approach is incompatible with how the blender-worker is packaged (e.g., the worker expects to be invoked as a Cloud Run task, not a subprocess). Escalate; the in-memory adapter may need a different strategy (e.g., directly calling the worker's main function).
